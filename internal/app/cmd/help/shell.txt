Open an interactive shell into a running sandbox.

Automatically detects the sandbox substrate (EC2 or ECS Fargate) and
dispatches the correct AWS CLI command — SSM Session Manager for EC2,
ECS Exec for Fargate containers. No need to know which substrate is
running; km shell handles the abstraction.

Usage:
  km shell <sandbox-id>

Examples:
  # Open a shell into a sandbox
  km shell sb-a1b2c3d4

Prerequisites:
  - AWS CLI v2 installed with Session Manager plugin
  - AWS credentials configured (profile: klanker-terraform)
  - Sandbox must be in running state

How It Works:

  EC2 Sandboxes:
    Discovers the EC2 instance ID from sandbox metadata, then runs
    'aws ssm start-session' to open an interactive terminal via SSM
    Session Manager. Requires the SSM agent running on the instance
    (installed by default in sandbox AMIs).

  ECS Fargate Sandboxes:
    Discovers the ECS cluster and task ARN from sandbox metadata, then
    runs 'aws ecs execute-command' to open an interactive /bin/bash
    session in the running container. Requires ECS Exec enabled on the
    task (configured by default in sandbox task definitions).

Error Handling:
  - Sandbox not found: exits with error and guidance
  - Sandbox stopped/suspended: exits with message to start or top up budget
  - Unknown substrate: exits with error listing supported substrates
  - AWS CLI not installed: exits with installation guidance
