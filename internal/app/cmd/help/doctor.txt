Check platform health and bootstrap verification.

Runs all platform health checks in parallel and prints a structured report
showing pass, warn, fail, and skip results. Exits with code 1 if any check
reports an ERROR, enabling CI gating patterns like:

  km doctor && km create profile.yaml

Usage:
  km doctor [flags]

Examples:
  # Run all health checks (default human-readable output)
  km doctor

  # Output results as JSON for programmatic consumption
  km doctor --json

  # Show only failures and warnings (suppress OK results)
  km doctor --quiet

  # Use in CI pipelines — exit 0 on all-pass, exit 1 on any error
  km doctor && echo "Platform healthy" || echo "Platform has issues"

  # Clean up safe stale resources (KMS keys, IAM roles, schedules, Slack queues)
  km doctor --dry-run=false

  # Also delete detached orphan EBS volumes (separate opt-in; in-use volumes
  # are never auto-deleted — terminate the owning EC2 instance first)
  km doctor --dry-run=false --delete-ebs

  # Also delete stale Slack inbound SQS queues + their SSM parameters (queues
  # whose owning sandbox row has been removed from DynamoDB)
  km doctor --dry-run=false --delete-sqs

  # Also sweep orphan S3 prefixes — artifacts/{sandbox-id}/ and
  # transcripts/{sandbox-id}/ for sandboxes no longer in DynamoDB
  km doctor --dry-run=false --delete-s3

  # Also delete per-sandbox Lambdas (budget-enforcer, github-token-refresher)
  # whose owning sandbox is gone from DynamoDB. Platform Lambdas are never
  # touched.
  km doctor --dry-run=false --delete-lambdas

  # Combine all opt-ins for a full cleanup pass
  km doctor --dry-run=false --delete-ebs --delete-sqs --delete-s3 --delete-lambdas

  # Same thing, shortcut form
  km doctor --dry-run=false --with-deletes

  # Preview every potential delete without acting (combine the meta-flag with
  # the default --dry-run=true)
  km doctor --with-deletes

  # Run regional checks against every configured region
  km doctor --all-regions

Flags:
  --json          Output results as a JSON array of CheckResult objects
  --quiet         Suppress OK results; show only warnings and errors
  --dry-run       Default true. With --dry-run=false, cleanup-capable checks
                  delete safe stale resources (KMS keys, IAM roles, schedules).
                  EBS volumes, SQS queues, and S3 prefixes each require their
                  own --delete-ebs / --delete-sqs / --delete-s3 opt-in.
  --delete-ebs    With --dry-run=false, also delete detached orphan EBS volumes
                  (state=available). In-use volumes are never auto-deleted —
                  terminate the owning EC2 instance first via 'km destroy
                  <sandbox-id> --remote --yes' or the AWS console.
  --delete-sqs    With --dry-run=false, also delete stale Slack inbound SQS
                  queues and their per-sandbox SSM parameters. Queue creation
                  has a 60-second AWS-side cooldown, so the explicit opt-in
                  prevents racing with sandbox provisioning.
  --delete-s3     With --dry-run=false, also sweep orphan S3 prefixes:
                    - artifacts/{sandbox-id}/  (rendered profile, userdata,
                      exported logs — accumulates because km destroy never
                      cleans this prefix)
                    - transcripts/{sandbox-id}/  (Phase 68 conversation
                      transcripts)
                  Only prefixes whose sandbox is gone from DynamoDB are
                  swept. Live sandboxes are never touched.
  --delete-lambdas
                  With --dry-run=false, also delete per-sandbox Lambda
                  functions whose owning sandbox row is gone from DynamoDB:
                    - {prefix}-budget-enforcer-{sandbox-id}
                    - {prefix}-github-token-refresher-{sandbox-id}
                  Platform Lambdas (create-handler, ttl-handler, slack-bridge,
                  ecs-spot-handler, email-create-handler) are NEVER touched —
                  the check uses a per-sandbox component allowlist, not a
                  denylist, so a future platform Lambda can't accidentally
                  be classified as orphan.
  --with-deletes  Shortcut for --delete-ebs --delete-sqs --delete-s3
                  --delete-lambdas. Pair with --dry-run=false for a full
                  cleanup pass; with the default --dry-run=true it just
                  makes every report-only check consistently surface what
                  it would delete. Per-resource flags can still be added
                  individually — --with-deletes is a strict superset.
  --all-regions   Run regional checks (VPC, AMIs) against PrimaryRegion plus
                  every region listed in KM_REPLICA_REGION (CSV).

Check Categories:

  Config
    Verifies km-config.yaml is loaded and all required fields are present:
    domain, account IDs (management, terraform, application), SSO start URL,
    and primary region.

  Credentials
    Calls STS GetCallerIdentity for the configured AWS profile to verify
    authentication is active. Skipped when no profile is configured.

  State Bucket
    Verifies the S3 state bucket (KM_STATE_BUCKET) exists and is accessible.

  DynamoDB Tables
    - Budget table (km-budgets): ERROR if missing — required for cost tracking
    - Identity table (km-identities): WARN if missing — optional feature

  KMS Key
    Checks that the km-platform KMS key alias exists for state bucket encryption.

  SCP (Sandbox Containment)
    Verifies the km-sandbox-containment SCP is attached to the application
    account in AWS Organizations. Skipped if management account ID is not set.

  GitHub App Config
    Reads /km/config/github/app-client-id and /km/config/github/installation-id
    from SSM Parameter Store. Returns WARN (not ERROR) if missing — GitHub App
    integration is optional.

  VPC (per region)
    Checks for km-managed VPCs (tagged km:managed=true) in the primary region
    and any replica region (KM_REPLICA_REGION env var).

  Active Sandboxes
    Lists running sandboxes and reports count by status. Informational only.

Exit Codes:
  0   All checks passed (OK or WARN or SKIPPED)
  1   One or more checks reported ERROR

Output Symbols:
  ✓   OK      — check passed
  ⚠   WARN    — check found a potential issue (non-blocking)
  ✗   ERROR   — check failed; platform may not function correctly
  -   SKIPPED — check skipped (missing config or credentials)

Remediation:
  Each error and warning includes an actionable remediation step. Run
  'km doctor' to see what needs to be fixed before running 'km create'.
