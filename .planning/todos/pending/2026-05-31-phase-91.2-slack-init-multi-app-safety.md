## Problem

When a Slack workspace has multiple installed apps (common scenario — e.g. a
GitHub integration app + the klankermaker app), `km slack init --force` and
`km slack rotate-signing-secret` silently accept whichever value the operator
pastes without verifying it belongs to the SAME app as the bot token already in
SSM. The signing secret has no in-band validation path — it only manifests as
"every Slack event returns 401 signature mismatch" at runtime, which looks
identical to "bot dispatches nothing" to the user.

Discovered live during Phase 91 UAT (2026-05-31): operator with two Slack apps
in their workspace (`KlankerMakerNotification` and `GH-KPH`) pasted the GH-KPH
app's signing secret during `km slack init --force`. SSM was overwritten,
bridge Lambda cached the wrong value, and every inbound event failed signature
verification for ~30 minutes before the symptom was traced. See conversation
trace at `git show 42e5b9c~5..42e5b9c` for the diagnostic flow.

## Proposed solution

Two independent improvements:

### 1. Pre-overwrite confirmation in `km slack init` / `rotate-signing-secret`

Before persisting a newly-prompted bot token or signing secret, fetch the
current SSM value's metadata and display a confirmation prompt:

```
Current bot in SSM: klankermakernotificat (U0B1Q96BX24, team T0B0D7QLQ2H)
⚠ Existing signing secret found (version 5, last modified 2026-05-25).
The signing secret you paste MUST be from the SAME Slack app as the bot above.

In Slack admin: Apps → KlankerMakerNotification → Basic Information → Signing Secret

Overwrite? [y/N]
```

For `km slack init --force` and `km slack rotate-signing-secret`, this displays
the existing values so the operator can sanity-check the source app before
clobbering. The bot context (user_id, team_id) makes app identity unambiguous.

Default to `[N]` — explicit confirmation required. `--yes` flag skips the
prompt for scripted use.

### 2. `km doctor` check for bot-token / signing-secret coherence

The signing secret cannot be validated in isolation (no API call returns it).
But the bot token CAN be validated via `auth.test`, and the response includes
`bot_id` + `team_id`. Cache these at SSM alongside `bot_user_id` (Phase 91
already caches user_id at `{prefix}slack/bot-user-id` — extend to write
`{prefix}slack/bot-id` and `{prefix}slack/team-id`).

Doctor check: call `auth.test` against the bot token; verify the live response's
`bot_id`/`team_id` match the cached values. Mismatch → WARN with a hint that
the bot token was rotated to a different app's token and the signing secret
likely needs rotating too.

This doesn't directly detect a signing-secret/bot-token MISMATCH (they're
independent values from two different Slack admin pages), but it catches the
upstream pattern: operators who paste the wrong app's signing secret often
also paste the wrong app's bot token next time. A consistent bot-token/team-id
trail is a strong proxy.

### 3. Stretch: synthetic signature probe in doctor

Optional. Sign a known payload locally with the SSM signing secret, POST to
the bridge Lambda's URL with that signature, and check the response. If
signature verification passes, the secret is correct. If it returns 401, the
secret is wrong.

This requires the doctor to know the bridge URL (already in SSM at
`{prefix}slack/bridge-url`) and to construct a valid Slack-shaped envelope.
Adds runtime cost to every `km doctor` run, so should be gated behind a
`--probe-signing-secret` flag rather than always-on.

## Out of scope

- **Self-recovery from signature mismatch on the bridge side.** The bridge
  could in theory detect a sustained 401-only pattern and reload SSM, but
  that's a separate observability problem and conflates "operator misconfigured"
  with "operator rotated mid-flight."
- **Replacing the interactive prompt entirely.** Some operators rely on the
  bare prompt; keep it as the default.
- **Migration of existing single-app installs.** They keep working as-is —
  the new code paths only run when the SSM values are already present.

## Origin

Raised during Phase 91 UAT debugging on 2026-05-31. Cost: ~30 minutes of
"why does the bot ignore me" → tracing through bot identity, channel binding,
mention-scan logic, bot_user_id lookups, before the bridge CloudWatch logs
revealed the actual symptom (`WARN events: signature check failed`). The
operator confirmed they had two apps installed and had pasted the wrong one's
signing secret.

The diagnostic chain that uncovered this is itself documented in the conversation
log (see `git show 4d49544..42e5b9c` and the surrounding session transcript) —
the pattern of "operator pastes wrong app's secret" is now identified, but the
preventive UX is not yet in place.

Defer until a future operator hits the same trap OR until we add a third Slack
knob that goes through `km slack init`.
