<objective>
Upgrade Phase 95's fire-and-forget relay into a claim-aware scatter-gather and add
the cross-install running-channels lister. This plan owns the CONTRACT layer:

1. Change the `PeerRelayer` interface so `Broadcast` returns `[]PeerClaimResult`
   (the breaking change) and update ALL implementers + test doubles.
2. Change `postToPeer` to read + parse the peer's JSON response, with the LOCKED
   rollout-safety rule: any legacy `"ok"` / non-JSON / HTTP-error response maps to
   `Claimed:true` (conservative — never produce a false orphan).
3. Add the peer-side relayed-request response: a relayed request that misses
   locally returns `{claimed:false, channels:[...]}` (this install's running
   sandbox channels); one that owns the channel returns `{claimed:true}`.
4. Add `DDBRunningChannelLister` (a `dynamodb:Scan` of `km-sandboxes` for
   `state=running` AND `slack_channel_id` present) behind a narrow `DDBScanAPI`
   interface.

Plan 03 consumes these types to wire the front-door orphan reply. This plan does
NOT change the front-door miss path's reply behavior — only the relayed-request
(peer-side) response and the gather plumbing.

Purpose: Establish the scatter-gather contract + lister so Plan 03 has
`[]PeerClaimResult` + `RunningChannelLister` to build the reply.
Output: New interfaces/types, the upgraded relayer, the lister adapter, the
peer-side JSON response.
</objective>
