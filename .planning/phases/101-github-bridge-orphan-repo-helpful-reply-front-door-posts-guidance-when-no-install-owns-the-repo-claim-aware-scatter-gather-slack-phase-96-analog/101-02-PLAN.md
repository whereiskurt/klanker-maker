<objective>
Upgrade the GitHub bridge relayer from Phase-100 fire-and-forget to the Phase-96 Slack claim-aware
scatter-gather shape — MINUS the channel list. Three changes in `pkg/github/bridge`:

1. `interfaces.go`: add `PeerClaimResult{ PeerURL string; Claimed bool }` (drop `Channels` — there is
   no repo list in the GitHub orphan comment) and CHANGE `PeerRelayer.Broadcast`'s return type from
   `error` to `([]PeerClaimResult, error)`. This is a RETURN-TYPE CHANGE to the existing method (exactly
   how Slack evolved its relayer from Phase 95→96), not a new method. Update the interface doc — REMOVE
   the Phase-100 "do not add a claim-result slice return here" note (interfaces.go:7-9), which is now superseded.

2. `relayer.go`: add `peerRelayResponse{ Claimed bool `json:"claimed"` }`; rewrite `Broadcast` to collect
   `[]PeerClaimResult` from a buffered channel after `wg.Wait()`; add `postToPeer` that reads the body
   and parses `{claimed:bool}`. THE SAFETY INVARIANT (GH-ORPHAN-ROLLOUT): any transport error, non-2xx
   status, OR unparseable/legacy (`"ok"`) body ⇒ `Claimed:true`. Only an explicit `{"claimed":false}`
   counts as unclaimed. Keep `wg.Wait()`, `relayBroadcastTimeout`, header forwarding, and `X-KM-Relayed:1`
   UNCHANGED. Empty `PeerURLs` ⇒ `(nil, nil)`.

3. `relayer_test.go`: extend with claim-tally + rollout-safety tests.

CRITICAL — DROP all channel/repo-list machinery (`Channels`, `SandboxChannelInfo`, `RunningChannelLister`).
Those exist in Slack ONLY because humans join channels; there is nothing to "join" on a GitHub repo.

This plan is INDEPENDENT of Plan 01 (config) — both are Wave 1. Plan 03 (Wave 2) consumes this new
signature in `webhook_handler.go` (front-door tally) and `cmd/km-github-bridge/main.go`.
Output: PeerClaimResult + claim-aware Broadcast + extended relayer tests.
</objective>
