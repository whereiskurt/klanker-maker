# Phase 101 — UAT: GitHub bridge orphan-repo helpful reply (default router)

**Requirement:** GH-ORPHAN-E2E (manual-only — requires two live installs sharing one
GitHub App webhook + real GitHub delivery + a real PR/issue on an unowned repo; cannot
be exercised by unit mocks).
**Status:** draft — operator runs this during `/gsd:verify-work` or post-deploy.
**Mode:** live, two installs, one GitHub App.

This runbook verifies the end-to-end behavior of the Phase 101 orphan-repo helpful
reply: a PR comment on a repo **no install owns**, delivered by GitHub to the **`kph`**
front-door bridge, produces **exactly ONE** guidance comment from the front door; a
comment on an **owned** repo produces no guidance comment (owner claimed it); and a
second @-mention within the cooldown window produces no second comment.

Synthetic install names (`kph`, `sec`) and `on.aws` Function URLs throughout.

---

## Pre-conditions

- Two km installs in the same account, distinct `resource_prefix`: **`kph`** (front
  door, `github.default_router: true`) and **`sec`** (peer).
- Both installs running Phase-101 Lambda code (`make build-lambdas` + `km init
  --dry-run=false` applied on each).
- A single **GitHub App** (one webhook URL pointing at `kph`'s bridge Function URL;
  one shared webhook secret stored in SSM on both installs).
- A test repo **owned by neither install** (e.g. `acme/widgets`) with the GitHub App
  installed and a PR or issue open for testing.
- A test repo **owned by `sec`** (e.g. `sec-org/demo`) — listed under
  `sec`'s `github.repos:` — for the negative (no-orphan-comment) check.
- An allowlisted GitHub login to author test comments.

---

## Setup

### 1. One GitHub App, creds in BOTH installs

Create one GitHub App (`km github manifest` renders the manifest). Subscribe it to
the `issue_comment` webhook event. Paste the **same** App creds + webhook secret into
**both** installs:

```bash
# In the kph install context:
km github init        # caches App client-id / private key / webhook-secret under /kph/config/github/

# In the sec install context:
km github init        # SAME App creds + webhook-secret under /sec/config/github/
```

Both installs now hold an identical copy of the App webhook secret — each can
independently HMAC-verify `X-Hub-Signature-256` on a relayed body.

### 2. Install the App on both test repos

In the GitHub App settings, install the App on **both** test repos:
- `acme/widgets` (the unowned repo — both `kph` and `sec` omit it from `github.repos:`)
- `sec-org/demo` (the owned repo — listed under `sec`'s `github.repos:`)

### 3. Unique repo ownership

Each install lists only **its own** repos in `github.repos:`

```yaml
# kph km-config.yaml
github:
  default_router: true   # Phase 101: enable orphan guidance comment
  peer_bridges:
    - https://sec000.lambda-url.us-east-1.on.aws/
  repos:
    - match: "kph-org/*"
      profile: github-review

# sec km-config.yaml
github:
  peer_bridges:
    - https://kph000.lambda-url.us-east-1.on.aws/
  repos:
    - match: "sec-org/*"
      profile: github-review
```

Neither install lists `acme/widgets` → it is the unowned test repo.
`sec-org/demo` is owned by **`sec` only** → the owned test repo for Test C.

### 4. Point the App webhook at the front door

In the GitHub App settings, set the **Webhook URL** to the **front-door** install's
bridge URL (`https://kph000.lambda-url.us-east-1.on.aws/`). GitHub delivers every
`issue_comment` event to `kph`.

### 5. Deploy BOTH installs (NOT `--sidecars`)

`github.default_router` → `KM_GITHUB_DEFAULT_ROUTER` and `github.peer_bridges` →
`KM_GITHUB_PEER_BRIDGES` are env-block changes, so a full terragrunt apply is required:

```bash
# On kph (front door):
make build-lambdas       # CLEAN rebuild — make build alone does NOT rebuild the bridge zip
km init --dry-run=false  # full apply — updates the Lambda environment.variables block

# On sec (peer):
make build-lambdas
km init --dry-run=false  # sec needs Phase-101 code to emit {"claimed":bool} JSON
```

> **NOT `km init --sidecars`:** it rebuilds the zip + cold-starts but does NOT update
> the Lambda env block. (Memory: `project_km_init_lambdas_doesnt_deploy`,
> `feedback_km_init_full_apply`.)

### 6. Verify the front-door env var

```bash
# Confirm KM_GITHUB_DEFAULT_ROUTER reached the kph Lambda:
aws lambda get-function-configuration \
  --function-name kph-github-bridge \
  --query 'Environment.Variables.KM_GITHUB_DEFAULT_ROUTER'
# Expected: "true"
```

### 7. Open test issues/PRs

Create or identify:
- A PR on `acme/widgets` (unowned repo) — note the PR number (e.g. `#7`).
- A PR on `sec-org/demo` (owned repo) — note the PR number.

## Test A — Unowned repo: one guidance comment posted

**Scenario:** @-mention the bot on the **unowned** repo.

1. On a PR in `acme/widgets` (e.g. PR `#7`), post a comment from the allowlisted
   login that @-mentions the bot:

   ```
   @km-bot review this PR
   ```

### Expected (PASS criteria)

- [ ] **Delivery → front door.** GitHub delivers the `issue_comment` webhook to `kph`
      (the App's configured webhook URL).
- [ ] **Front door relays + tallies.** `kph` runs `Resolve("acme/widgets")` → no
      match → broadcasts to `sec`. `sec` returns `{"claimed": false}` (no match).
      `kph` tallies: zero claims → orphan.
- [ ] **Exactly ONE guidance comment.** `kph` posts one comment on PR `#7` in
      `acme/widgets` naming `github.repos:` wiring and `km init`. The comment text
      references `docs/github-bridge.md`.
- [ ] **No duplicate comment.** Only one comment from the bot (not two).
- [ ] **No 👀 reaction.** Neither install posts a 👀 (no dispatch — the repo is
      unowned, so no sandbox turn runs).
- [ ] **No sandbox dispatch.** `km agent list` on any sandbox shows no new run for
      this event.

### How to observe

- **GitHub UI:** the PR `#7` comment thread shows exactly one guidance comment from
  the bot.
- **`kph` (front door) logs:** `claimResults` tally line (0 claimed), orphan-comment
  post line, NO dispatch, NO 👀.
- **`sec` (peer) logs:** HMAC verify OK, `Resolve` miss, returns `{"claimed": false}`.

---

## Test B — Cooldown: no repeat comment within 3600s

**Scenario:** @-mention the bot AGAIN on the **same PR** within 3600 seconds of Test A.

1. On the **same PR** (`acme/widgets` PR `#7`), immediately after Test A, post another
   @-mention:

   ```
   @km-bot please review again
   ```

### Expected (PASS criteria)

- [ ] **No second guidance comment.** The cooldown key
      `gh-router-cooldown:acme/widgets#7` is still active (TTL 3600s). The front door
      detects the active cooldown and skips posting a second comment.
- [ ] **No 👀, no dispatch** (same as Test A — unowned repo, no sandbox turn).
- [ ] **`kph` logs:** orphan tally shows zero claims again, but a cooldown-active log
      line indicates the comment was suppressed.

> To test after the cooldown expires: wait 3600 seconds (or manually delete the DDB
> nonce key `gh-router-cooldown:acme/widgets#7` from the `kph-bridge-nonces` table),
> then re-run. Expect a NEW guidance comment to appear.

---

## Test C — Owned repo: no guidance comment

**Scenario:** @-mention the bot on an **owned** repo.

1. On a PR in `sec-org/demo` (owned by `sec`, delivered to `kph` front door), post:

   ```
   @km-bot review this PR
   ```

### Expected (PASS criteria)

- [ ] **Delivery → front door.** `kph` receives the webhook.
- [ ] **Front door relays.** `kph` runs `Resolve("sec-org/demo")` → no match (not in
      `kph`'s repos) → broadcasts to `sec`.
- [ ] **`sec` claims.** `sec` matches `sec-org/demo` → dispatches the turn → returns
      `{"claimed": true}`.
- [ ] **No guidance comment.** `kph` tallies one claim → `anyClaimed = true` → does
      NOT post any guidance comment.
- [ ] **Exactly ONE 👀.** `sec` (the owner) posts the 👀 on the PR comment. `kph`
      posts none.
- [ ] **One sandbox turn.** `km agent list` on a `sec` sandbox shows one new run.
      No run on any `kph` sandbox.

### How to observe

- **GitHub UI:** the PR comment shows one 👀 and NO guidance comment from the bot.
- **`kph` logs:** relay line (broadcast to `sec`), claim tally (`claimed:true`), no
  guidance-comment post, no 👀.
- **`sec` logs:** HMAC verify OK, dispatch line, 👀 posted, `{"claimed":true}` response.

## Test D — Rollout-safe / mixed fleet (optional)

**Scenario:** `sec` is still running Phase-100 code (returns plain `200 "ok"`, no JSON).

> This is a pre-upgrade rollback test. Only applicable if you can temporarily downgrade
> `sec`'s Lambda to Phase-100 code.

1. Deploy Phase-100 code to `sec`'s Lambda (no `{"claimed":bool}` response).
2. @-mention the bot on `acme/widgets` PR `#7` (unowned repo, cooldown cleared).

### Expected (PASS criteria)

- [ ] **No false guidance comment.** `kph` receives `sec`'s plain 200 (legacy body),
      tallies it as `claimed:true` (rollout-safe) → `anyClaimed = true` → **no orphan
      comment posted**.
- [ ] **No false dispatch.** No sandbox turn, no 👀.
- [ ] **`kph` logs:** relay to `sec`, legacy/non-JSON 200 tallied as claimed, no
      guidance-comment post.

> This test confirms the **rollout-safe invariant**: upgrading one install at a time is
> safe. A false orphan comment should NEVER appear while peers are on mixed versions.
