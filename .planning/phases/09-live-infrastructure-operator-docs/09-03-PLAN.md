Create a comprehensive operator guide at the repo root. The guide must cover these sections in order:

**1. Prerequisites**
- AWS Organizations with 3 accounts: management, terraform, application (explain each role)
- AWS SSO (IAM Identity Center) configured with at least one permission set for operator access
- Domain name registered and Route53 hosted zone created (note: zone can be in management or application account)
- Required CLI tools: aws-cli v2, terragrunt >= 0.67, terraform >= 1.5, Go >= 1.25, Docker (for ECR push), sops
- Required env vars summary table: KM_DOMAIN, KM_ARTIFACTS_BUCKET, KM_ACCOUNTS_MANAGEMENT, KM_ACCOUNTS_TERRAFORM, KM_ACCOUNTS_APPLICATION, KM_ROUTE53_ZONE_ID, KM_OPERATOR_EMAIL, KM_REGION (default us-east-1)

**2. Initial Configuration**
- Run `km configure` — document what it prompts for (domain, account IDs, SSO start URL, region) and what file it writes (~/.km/config.yaml or ./km-config.yaml)
- Verify AWS access: `aws sts get-caller-identity` in each account context
- Set environment variables (document shell export commands for each KM_* var)

**3. Build Artifacts**
- `make build-lambdas` — produces build/ttl-handler.zip and build/budget-enforcer.zip (arm64 for Graviton Lambda)
- `make sidecars` — cross-compiles sidecar binaries and uploads to S3
- `make ecr-push` — builds Docker images for ECS sidecars and pushes to ECR

**4. Deploy Shared Infrastructure**
Document the REQUIRED ordering:
  a. Network: `cd infra/live/use1/network && terragrunt apply`
  b. DynamoDB budget table: `cd infra/live/use1/dynamodb-budget && terragrunt apply`
  c. SES domain: `cd infra/live/use1/ses && terragrunt apply` (requires KM_ROUTE53_ZONE_ID)
  d. TTL handler Lambda: `cd infra/live/use1/ttl-handler && terragrunt apply` (requires build/ttl-handler.zip to exist)
- Note: dynamodb-budget, ses, and ttl-handler can run in any order but ALL must complete before first `km create`
- Note: SES requires domain verification — check SES console for verification status after apply
- Note: SES inbound email is only supported in us-east-1, us-west-2, eu-west-1

**5. First Sandbox**
- Create a test profile or use a built-in: `km validate profiles/open-dev.yaml`
- Run `km create profiles/open-dev.yaml`
- Verify with `km list` and `km status <sandbox-id>`
- Clean up with `km destroy <sandbox-id>`

**6. Verification Checklist**
Numbered checklist an operator can walk through:
  1. DynamoDB table exists: `aws dynamodb describe-table --table-name km-budgets`
  2. SES domain verified: `aws ses get-identity-verification-attributes --identities sandboxes.${KM_DOMAIN}`
  3. TTL Lambda exists: `aws lambda get-function --function-name km-ttl-handler`
  4. Sidecar binaries in S3: `aws s3 ls s3://${KM_ARTIFACTS_BUCKET}/sidecars/`
  5. ECR images present: `aws ecr list-images --repository-name km-dns-proxy`

**7. Troubleshooting**
Common issues:
- "exec format error" in Lambda logs: rebuild with `make build-lambdas` (arm64 required)
- SES "not verified": wait up to 72 hours for DKIM propagation; check Route53 records
- Terragrunt "state bucket not found": ensure S3 state bucket and DynamoDB lock table exist (created during initial AWS account setup)
- `KM_ROUTE53_ZONE_ID` missing: look up with `aws route53 list-hosted-zones`

Do NOT include emojis. Use plain markdown headers and numbered steps. Keep it practical — this is an ops runbook, not marketing material.
