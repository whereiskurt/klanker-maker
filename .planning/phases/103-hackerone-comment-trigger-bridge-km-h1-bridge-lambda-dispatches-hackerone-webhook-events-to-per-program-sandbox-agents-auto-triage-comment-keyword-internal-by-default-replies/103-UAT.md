## Part 1 — Deploy sequence (operator runs locally)

Run from the repo root with `AWS_PROFILE=klanker-application` (default install).
The ordering below is load-bearing — two memory footguns are encoded in it.

```bash
# 1. make build BEFORE km init — FOOTGUN #6 (project_make_build_precedes_km_init).
#    Plan 08 added dynamodb-h1-threads + lambda-h1-bridge to regionalModules(); a
#    STALE km silently SKIPS the new modules and bakes mock dependency ARNs
#    (account 000000000000) into dependents → runtime AccessDenied. Always rebuild
#    the km binary first.
make build

# 2. Clean rebuild of the Lambda zips + sidecars (incl. km-h1-bridge.zip + km-h1).
make build-lambdas

# 3. Full apply — NOT --sidecars — FOOTGUN #7 (feedback_km_init_full_apply).
#    The new lambda-h1-bridge module + its live unit + the KM_H1_* env block + IAM
#    require a real terragrunt apply. `--sidecars` only rebuilds zips + cold-starts;
#    it does NOT create the new module or update the Lambda env block.
km init --dry-run=false

# 4. Refresh the management Lambdas for the new notification.h1.inbound schema field
#    (Plan 09). This is the schema-field refresh; safe to run after the full apply.
km init --sidecars

# 5. Mint the webhook secret + store Basic-Auth creds in SSM; print the Function URL.
#    Provide the API username + token (above) when prompted. The token is NEVER
#    written to the repo — it lands only in SSM under /{prefix}/config/h1/*.
km h1 init   # follow prompts; copy the printed Function URL + webhook secret

# 6. Sanity-check the stored config (secret redacted).
km h1 status

# 7. Provision the H1 triage sandbox(es) for the target program.
km create profiles/h1-triage.yaml
```

> **Do NOT reference any production HackerOne program in any command.** Use only
> the Sandbox programs `prodsec_klanker_maker_test_h1b` /
> `prodsec_klanker_maker_test_h1r`.

### Paste the Function URL into the HackerOne Webhooks UI

HackerOne has no App-install model — webhooks are configured **per program** in the UI:

**Engagements → Program → Settings → Automation → Webhooks**

1. Add a webhook.
2. **Payload URL** = the Function URL printed by `km h1 init`.
3. **Secret** = the webhook secret printed by `km h1 init`.
4. Select the events you want (at minimum `report_created` + `report_comment_created`),
   or "send everything".
5. Save, then click **Test request**.
6. Open the **Recent deliveries** tab — confirm a **200** for the test delivery
   (the bridge HMAC-verified a valid `X-H1-Signature`).

## Part 2 — Six-step reply-visibility UAT (HackerOne UI)

Configure an `h1.programs:` entry in `km-config.yaml` for the target Sandbox
program with **2 targets** (multi-target fanout), `events: [report_created]`, and
an `allow:` list. Re-run `km init --dry-run=false` if you changed the env block,
then submit a test report in the Sandbox program.

| # | Step | Action | PASS criteria |
|---|------|--------|---------------|
| 1 | **Webhook accepted** | Click **Test request** in the program Webhooks UI | **200** in Recent-Deliveries with a valid `X-H1-Signature` (no 401) |
| 2 | **Auto-triage → INTERNAL** | Trigger `report_created` (submit a test report) | An **INTERNAL** triage comment appears on the primary target's analysis — **NOT** researcher-visible |
| 3 | **@handle /command → reply** | As an **allowlisted** user, comment `@handle /reply_to_researcher <msg>` on the report | A reply is posted (the command was honored) |
| 4 | **Allowlisted external = primary-only** | (same as #3) | The reply is **RESEARCHER-VISIBLE** and posted **exactly ONCE** (primary target only), despite the 2 fanout targets |
| 5 | **Non-allowlisted → downgraded** | As a **NON-allowlisted** user, comment `@handle /reply_to_researcher <msg>` | The reply is **DOWNGRADED to INTERNAL** (or blocked) — **NOT** researcher-visible. ⚠️ A researcher-visible reply here is a **P0 safety bug** |
| 6 | **Loop-guard sanity** | Observe the report over a few minutes / repeated events | The report does **NOT** accrue repeated identical internal comments (Pitfall 4) |

### Multi-target fanout expectation (steps 2 & 4)

With **N targets**, a single trigger produces **N internal analyses** (one per target,
each in its own `(report_id, target)` thread row) but **exactly ONE external reply**
(primary target only). N internal + 1 external is the pass shape — N external replies
is a fanout-leak bug.

## Part 3 — Watch notes during UAT

- **Rate-limit / 429 (Pitfall 5):** under burst (e.g. many rapid comments), the
  HackerOne customer API may return **429**. Confirm `km-h1` backs off and retries
  rather than wedging the poller. A sustained 429 with no progress is a regression.
- **Loop guard (Pitfall 4):** the inbound FIFO uses a **1800s** visibility timeout
  (deliberately longer than the 30s Slack/GitHub queues) so a long triage turn is
  not re-delivered mid-flight. If you see the SAME internal triage comment posted
  repeatedly on one report, the turn is being re-delivered — check the DLQ depth
  (`km doctor`) and the visibility-timeout setting.
- **DLQ:** the `{prefix}-h1-inbound-*.fifo` queues carry the shared per-install DLQ
  + `RedrivePolicy(maxReceiveCount=3)`. A poison message lands in the DLQ after 3
  receives instead of wedging the queue (`project_inbound_poller_fifo_poison_wedge`).
