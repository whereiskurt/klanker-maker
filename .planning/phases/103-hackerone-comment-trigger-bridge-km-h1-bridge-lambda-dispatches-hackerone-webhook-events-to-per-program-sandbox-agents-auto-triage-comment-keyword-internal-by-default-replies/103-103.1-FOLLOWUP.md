# Phase 103.1 — `km doctor` HackerOne bridge checks

## Origin

Flagged as a deferred fast-follow in Phase 103 (`docs/h1-bridge.md` § Troubleshooting:
"`km doctor` does not yet have HackerOne checks — a candidate fast-follow") and the
verifier's out-of-scope note. Operator requested it as a follow-up after the docs landed.

The Slack (`doctor_slack.go`) and GitHub (`doctor.go::checkGitHub*`) bridges each have a
`km doctor` group that validates their SSM-backed config + inbound queues. The H1 bridge
shipped without one — so a misconfigured deploy (missing webhook secret, missing Basic-Auth
creds, un-deployed bridge URL) is invisible until a live webhook silently fails.

## Goal

Add an **H1 doctor group** that mirrors the GitHub group: validate the SSM-backed H1 config
and the per-sandbox inbound queues, **gated dormant** — every check SKIPs silently when
`h1.programs` is empty (byte-identical doctor output to a non-H1 install).

## Scope

New file `internal/app/cmd/doctor_h1.go` + `doctor_h1_test.go`, plus minimal wiring into
`runDoctor` and the `DoctorConfigProvider` interface. All checks are gated on
`len(GetH1Programs()) > 0`.

### Checks (mirror `checkGitHub*` in `doctor.go`)

| Check | Verifies | Level on failure |
|---|---|---|
| `checkH1WebhookSecret` | `/{prefix}/config/h1/webhook-secret` present in SSM | WARN — "run `km h1 init`" |
| `checkH1APICreds` | `/{prefix}/config/h1/api-username` **and** `/{prefix}/config/h1/api-token` present | WARN — "run `km h1 init --api-username … --api-token …`" |
| `checkH1BridgeURL` | `/{prefix}/config/h1/bridge-url` present + valid HTTPS | WARN — "run `km init --dry-run=false` to deploy the H1 bridge Lambda" |
| `checkH1ProgramsValidity` (pure config, no AWS) | each program has a non-empty `handle` + ≥1 `target`; `default_command` (if set) names a defined `commands` key; a comment-keyword-only program (no `events`) has a `bot_handle` (install or per-program) | WARN per offending program |

Gating: when `len(cfg.H1.Programs) == 0`, the group SKIPs silently (mirror the GitHub
"Empty slice means the GitHub bridge is not configured" pattern at `doctor.go:265`).

> **Note — inbound DLQ already covered.** `doctor_inbound_dlq.go` already sweeps every
> `{prefix}-*-inbound-*.fifo` DLQ generically, so `h1-inbound` queues are covered without a
> new check. Confirm this during execution; only add an H1-specific queue check if the
> generic sweep excludes them.

## Tasks

1. **Interface + adapter** — add `GetH1Programs() []appcfg.H1ProgramEntry` to
   `DoctorConfigProvider` and `appConfigAdapter` (mirror `GetGithubRepos`). Add
   `GetSsmPrefix()`-based reads (already present).
2. **`doctor_h1.go`** — implement the four checks above, using the existing `SSMReadAPI`
   seam and `CheckResult` shape (copy the structure of `checkGitHubConfig` /
   `checkGitHubWebhookSecret` / `checkGitHubBridgeURL`).
3. **Wire into `runDoctor`** — call the H1 group where the GitHub group is dispatched, gated
   on `len(provider.GetH1Programs()) > 0`. Group header: "HackerOne bridge".
4. **`doctor_h1_test.go`** — mirror `doctor_slack_inbound_test.go`: mock SSM; assert PASS when
   all params set, WARN on each missing param, SKIP when `h1.programs` empty, and the
   `checkH1ProgramsValidity` table cases (missing target, bad default_command, no
   handle/bot_handle).

## Verification

- `go test ./internal/app/cmd -run 'H1Doctor|DoctorH1|checkH1' -count=1` green.
- `km doctor` on a configured install shows the "HackerOne bridge" group; on a non-H1
  install shows nothing new (dormant).
- `make build` succeeds.

- DLQ depth check if the generic `doctor_inbound_dlq.go` already covers `h1-inbound`.
- `km doctor --backfill-tags` interaction.
