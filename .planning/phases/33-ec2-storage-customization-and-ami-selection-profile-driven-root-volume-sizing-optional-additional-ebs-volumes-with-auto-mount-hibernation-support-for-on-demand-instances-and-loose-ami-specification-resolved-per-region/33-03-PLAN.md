  <action>
    1. Add semantic validation in the EC2 service HCL generation function:
       - If `p.Spec.Runtime.AdditionalVolume != nil && p.Spec.Runtime.Substrate == "ecs"`: return error "additionalVolume is not supported for ECS substrate"

    2. Extend `ec2HCLParams` with:
       ```go

       AdditionalVolumeEncrypted  bool
       ```

    3. Populate new params from profile:
       - If `p.Spec.Runtime.AdditionalVolume != nil`: set SizeGB from `.Size`, Encrypted from `.Encrypted`
       - Otherwise: both remain zero-values (0, false)

    4. Add to `ec2ServiceHCLTemplate` locals (alongside root_volume_size_gb etc.):
       ```
       additional_volume_size_gb     = {{ .AdditionalVolumeSizeGB }}
       additional_volume_encrypted   = {{ .AdditionalVolumeEncrypted }}
       ```

    5. Extend `userDataParams` with:
       ```go
       AdditionalVolumeMountPoint string  // empty = no additional volume
       ```

    6. Populate in `generateUserData`:
       - If `p.Spec.Runtime.AdditionalVolume != nil`: set from `.MountPoint`

    7. Add the auto-mount section to `userDataTemplate`. Place it after the filesystem bind-mount section (section 2.5) as section 2.6 or after the most recent numbered section. Use the template conditional pattern:
       ```
       {{- if .AdditionalVolumeMountPoint }}
       # ============================================================
       # X.X. Additional EBS volume: format and mount
       # ============================================================
       echo "[km-bootstrap] Waiting for additional EBS volume to attach..."
       DEVICE=""
       for i in $(seq 1 30); do
         for dev in /dev/xvdf /dev/nvme1n1; do
           if [ -b "$dev" ]; then
             DEVICE="$dev"
             break 2
           fi
         done
         sleep 2
       done

       if [ -n "$DEVICE" ]; then
         if ! blkid "$DEVICE" &>/dev/null; then
           echo "[km-bootstrap] Formatting $DEVICE as ext4..."
           mkfs.ext4 -F "$DEVICE"
         fi
         mkdir -p "{{ .AdditionalVolumeMountPoint }}"
         DEVICE_UUID=$(blkid -s UUID -o value "$DEVICE")
         if ! grep -q "$DEVICE_UUID" /etc/fstab 2>/dev/null; then
           echo "UUID=${DEVICE_UUID} {{ .AdditionalVolumeMountPoint }} ext4 defaults,nofail 0 2" >> /etc/fstab
         fi
         mount -a
         chown sandbox:sandbox "{{ .AdditionalVolumeMountPoint }}" 2>/dev/null || true
         echo "[km-bootstrap] Additional volume mounted at {{ .AdditionalVolumeMountPoint }}"
       else
         echo "[km-bootstrap] WARNING: additional EBS volume device not found after 60s"
       fi
       {{- end }}
       ```

       Key details:
       - Probes BOTH /dev/xvdf (Xen) and /dev/nvme1n1 (Nitro NVMe) for device detection
       - Idempotent: checks blkid before formatting (safe for reattached volumes)
       - Uses UUID in fstab (not device path) for persistence
       - `nofail` option prevents boot hang if volume detaches
       - `chown sandbox:sandbox` follows existing sandbox user pattern
  </action>

    1. In `variables.tf`, add two new variables:
       ```hcl
       variable "additional_volume_size_gb" {
         type        = number
         description = "Additional EBS data volume size in GB. 0 = no additional volume."
         default     = 0
       }

       variable "additional_volume_encrypted" {
         type        = bool
         description = "Whether the additional EBS volume should be encrypted."
         default     = false
       }
       ```

    2. In `main.tf`, add the EBS volume resource (count-gated):
       ```hcl
       resource "aws_ebs_volume" "additional" {
         count             = var.additional_volume_size_gb > 0 ? 1 : 0
         availability_zone = local.effective_azs[0]
         size              = var.additional_volume_size_gb
         encrypted         = var.additional_volume_encrypted
         type              = "gp3"
         tags = {
           "km:sandbox-id" = var.sandbox_id
           Name             = "km-sandbox-${var.sandbox_id}-data"
         }
       }
       ```

    3. Add the volume attachment resource. This must handle both spot and on-demand instances:
       ```hcl
       resource "aws_volume_attachment" "additional" {
         count       = var.additional_volume_size_gb > 0 ? 1 : 0
         device_name = "/dev/xvdf"
         volume_id   = aws_ebs_volume.additional[0].id
         instance_id = length(local.ec2_ondemand_map) > 0 ? aws_instance.ec2_ondemand[keys(local.ec2_ondemand_map)[0]].id : aws_spot_instance_request.ec2spot[keys(local.ec2spot_map)[0]].spot_instance_id
         force_detach = true
       }
       ```

       IMPORTANT: For spot instances, use `.spot_instance_id` (the actual EC2 instance ID), NOT the spot request ID. The `wait_for_fulfillment = true` flag already present ensures spot_instance_id is populated. Use a conditional to select between on-demand and spot based on which map is populated.

    4. Place these resources after the existing instance resources (after `aws_instance.ec2_ondemand` block) for logical grouping.
