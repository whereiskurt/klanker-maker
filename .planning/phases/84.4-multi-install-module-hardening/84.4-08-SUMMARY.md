## Issues Encountered

### Gap 1 (TIER-1, BLOCKING): ssm-session-doc shared-resource destruction at teardown

`infra/modules/ssm-session-doc/v1.0.0/main.tf` hardcodes `KM-Sandbox-Session`. Both installs imported the same AWS resource. When rg was torn down, its module deleted the document from AWS, breaking `km shell` on the km install. Required manual re-apply of km's ssm-session-doc module.

**Impact:** Any second install using the `terragrunt import` workaround will destroy the first install's SSM session document when torn down. This is worse than "second install can't apply" — it's the workaround introducing cross-install destruction risk.

**84.4.1 fix:** Migrate to `infra/modules/ssm-session-doc/v2.0.0/` with `${prefix}-Sandbox-Session` naming.

### Gap 2 (CO-PRIMARY, BLOCKING): SCP AND-composition

km install's `km-sandbox-containment` SCP (attached to application account) trusts only `km-*` role ARNs. The rg install's `rg-create-handler` is denied by km's SCP for `ec2:CreateSecurityGroup` and `iam:CreateRole`. No rg sandbox can be created without manual SCP modification.

**Impact:** The core Phase 84.4 thesis — two installs can run independently in the same account — is broken at the sandbox lifecycle layer.

**84.4.1 fix:** Design and implement a cross-install SCP composition model (union of trusted roles, per-install OUs, or suffix-pattern allow).

### Gap 3 (PRIMARY): Plan 06 auto-import does not fire for shared-domain second install

`detectSharedSESState` returned `registerID = true` despite the SES domain identity already existing in AWS. The auto-import path was skipped. Route53 rejected all three DKIM CNAME creates with `InvalidChangeBatch`.

**Impact:** Plan 06's headline deliverable — frictionless second-install bootstrap on shared SES domain — does not work in practice. Manual `terragrunt import` required.

**84.4.1 fix:** Make `detectSharedSESState` probe AWS `GetEmailIdentity`; OR: separate the DKIM-record-import gate from the `registerID` gate.

## RESEARCH.md Predictions Violated

From the 84.4 RESEARCH.md (not re-read, inferred from plan design):

1. **Auto-import prediction violated:** Plan 06 was designed to make `km bootstrap --shared-ses` on a second install fully automatic for shared-domain scenarios. The unit tests passed; the integration scenario failed.

2. **SCP independent-attach prediction violated:** Plan 02 modeled each install's SCP as independent. The implicit assumption was that per-install SCPs don't interact. The AND-composition of SCPs was not analyzed and breaks the sandbox lifecycle.

3. **v2.0.0 module completeness prediction violated:** Plans 02-05 migrated scp, efs, s3-replication — but `ssm-session-doc` (and other v1.0.0 modules) were not audited for shared-name patterns. The audit test scans v2.0.0 directories only; pre-existing v1.0.0 modules with hardcoded `KM-`/`km-` names remained invisible.
