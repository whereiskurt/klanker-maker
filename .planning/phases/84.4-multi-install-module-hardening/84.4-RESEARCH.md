### Pattern 3: SCP 5KB Limit Guard — `terraform_data` `precondition`

Terraform 1.4+ supports `lifecycle.precondition` on `terraform_data` resources. This is the right HCL pattern for plan-time assertions without AWS API calls:

```hcl
# Recommended: terraform_data precondition (Terraform >= 1.4, current req is >= 1.6)
resource "terraform_data" "scp_size_guard" {
  lifecycle {
    precondition {
      condition     = length(data.aws_iam_policy_document.sandbox_containment.json) <= 5000
      error_message = "SCP policy exceeds 5,000 bytes (5,120-byte AWS limit minus 120-byte buffer). Current size: ${length(data.aws_iam_policy_document.sandbox_containment.json)} bytes. Remove a trusted ARN or consolidate statements."
    }
  }
}
```

Alternatively, a `variable` validation block works at plan-time too but only validates variable inputs, not computed values. The `terraform_data` + `precondition` pattern is correct here because `length(...)` references the rendered `data` source output.

### Pattern 4: HCL Static Analysis Test Mechanics

**What `hcl/v2/hclsyntax` provides:**

```go
// Source: https://pkg.go.dev/github.com/hashicorp/hcl/v2/hclsyntax
// Confidence: HIGH (official package docs)

// Parse a .tf file
parser := hclsyntax.NewParser()
file, diags := hclsyntax.ParseConfig(src, filename, hcl.InitialPos)

// Walk the body for resource blocks
body := file.Body.(*hclsyntax.Body)
for _, block := range body.Blocks {
    if block.Type != "resource" { continue }
    // block.Labels[0] = resource_type, block.Labels[1] = resource_name
    attrs := block.Body.Attributes
    if nameAttr, ok := attrs["name"]; ok {
        switch expr := nameAttr.Expr.(type) {
        case *hclsyntax.LiteralValueExpr:
            // Literal string — check for "km-" hardcode
            if strings.Contains(expr.Val.AsString(), "km-") {
                t.Errorf(...)
            }
        case *hclsyntax.TemplateExpr:
            // Interpolation like "${var.resource_prefix}-foo"
            // Verify it references var.resource_prefix
        case *hclsyntax.ScopeTraversalExpr:
            // Pure variable reference like var.resource_prefix
        }
    }
}
```

**Key type-assertion facts (confirmed from hclsyntax docs):**
- `*hclsyntax.LiteralValueExpr` — a bare quoted string with no interpolation. `.Val` is a `cty.Value`. Use `.AsString()`.
- `*hclsyntax.TemplateExpr` — `"${var.resource_prefix}-sandbox-containment"`. Contains `Parts []Expression`.
- `*hclsyntax.ScopeTraversalExpr` — a pure traversal like `var.resource_prefix`. `.Traversal` is `hcl.Traversal`.
- A template that references `var.resource_prefix` has at least one `Part` that is a `*hclsyntax.ScopeTraversalExpr` whose `.Traversal[0].Name == "var"` and `.Traversal[1].Name == "resource_prefix"`.

**Directory walk pattern:**
```go
func walkModulesV2(repoRoot string) []string {
    var modules []string
    pattern := filepath.Join(repoRoot, "infra", "modules", "*", "v*.0.0")
    matches, _ := filepath.Glob(pattern)
    for _, m := range matches {
        if strings.HasSuffix(m, "v1.0.0") {
            continue  // skip historical reference
        }
        modules = append(modules, m)
    }
    return modules
}
```

**Important:** `filepath.Glob` does NOT support `**` — use two-level glob `*/v*.0.0` or `filepath.WalkDir` with skip logic.

**IAM policy document Resource ARNs in heredoc JSON:**
The SCP module uses `data.aws_iam_policy_document` (HCL resource), NOT heredoc JSON. Its `Resource = ["*"]` fields don't contain role ARNs — role ARNs appear only in `condition.values`. The audit test for IAM role ARNs inside `condition` values requires traversing `hclsyntax.ObjectConsExpr` for `condition` block attributes, then string-checking the `values` list.

For `s3-replication/v1.0.0/main.tf`, the IAM policy is a `jsonencode({...})` call — this is an `hclsyntax.FunctionCallExpr` with the JSON object as argument. The `Resource` field values (e.g., `var.source_bucket_arn`) are variables, not literals. No audit failure expected there. The role names (`aws_iam_role.replication.name`, `aws_iam_role_policy.replication.name`) are the load-bearing audit targets.

### Pattern 5: DKIM Auto-Import in `runBootstrapSharedSES`

**Existing pattern to follow:** `runBootstrapSharedSES` already:
1. Calls `ExportTerragruntEnvVars(loadedCfg)` — confirmed at line 342
2. Builds `ses.Client` from `awspkg.LoadAWSConfigInRegion` — confirmed at line 368-375
3. Calls `ApplyTerragruntFunc(ctx, sesDir)` — the apply happens AFTER auto-detect

**Where DKIM auto-import plugs in:** Between the apply (line 414) and the return (line 416). After `ses-shared-rule-set` is applied, the DKIM tokens exist in SES state but the Route53 records may already exist (pre-Phase 84) outside tfstate. The import dance brings them into tfstate so subsequent plan runs show zero diff.

**Alternative:** Before apply, as a pre-apply import sequence. This is the correct approach: if DKIM records exist in AWS but not in tfstate, importing before apply prevents "resource already exists" errors. The import must happen before apply, not after.

**Correct flow:**
```
1. detectSharedSESState → registerID = false (domain already exists)
2. ses.GetIdentityDkimAttributes(domain) → []DkimToken{token0, token1, token2}
3. For each token: runner.Import(sesDir, "aws_route53_record.dkim[N]", "<zone_id>_<token>._domainkey.<domain>_CNAME")
4. route53.ListResourceRecordSets(hostedZoneId) → check for MX and _amazonses TXT
5. If MX exists: runner.Import(sesDir, "aws_route53_record.mx[0]", "<zone_id>_<domain>_MX")
6. If TXT exists: runner.Import(sesDir, "aws_route53_record.ses_verification[0]", "<zone_id>__amazonses.<domain>_TXT")
7. ApplyTerragruntFunc(ctx, sesDir)
```

**Route53 import ID format** (confirmed from `ses-shared-rule-set/v1.0.0/main.tf` lines 70-82):
- General format: `<hosted_zone_id>_<record_name>_<type>`
- MX: `Z123456_sandboxes.example.com_MX`
- TXT: `Z123456__amazonses.sandboxes.example.com_TXT` (note: double underscore because record name starts with `_amazonses`)
- DKIM CNAME: `Z123456_<token>._domainkey.sandboxes.example.com_CNAME`

**`Runner.Import()` method (must be added to `pkg/terragrunt/runner.go`):**
```go
// Import runs `terragrunt import <address> <id>` inside dir.
// Used by runBootstrapSharedSES to bring pre-existing AWS resources
// (DKIM CNAMEs, MX, TXT) into foundation tfstate before apply.
func (r *Runner) Import(ctx context.Context, dir, address, id string) error {
    cmd := r.buildCommand(ctx, dir, "import", address, id)
    return r.runCommand(ctx, cmd)
}
```

**SES API choice: Go SDK (already imported), not shell-out.** The `ses.Client` is already constructed in `runBootstrapSharedSES`. `GetIdentityDkimAttributes` takes `Identities []string`, returns `DkimAttributes map[string]types.IdentityDkimAttributes`. `IdentityDkimAttributes.DkimTokens []string` contains the three DKIM tokens. No need to shell out to `aws ses get-identity-dkim-attributes`.

**Route53 check: existing `route53.Client` in `unbootstrap.go` is a reference.** The auto-import logic needs a Route53 client — can be constructed the same way as in `bootstrap.go`'s `runBootstrap` path.

### Anti-Patterns to Avoid

- **Don't add `moved {}` blocks for the default install.** CONTEXT.md is explicit: with `resource_prefix = "km"` (the backward-compat default), interpolation produces byte-identical names. No Terraform state migration is needed for the `km` install.
- **Don't declare `required_providers` in `v2.0.0/main.tf`.** Confirmed pattern: `root.hcl`'s `generate "provider"` block is the single source of truth. `s3-replication` terragrunt.hcl explicitly notes "no root include to avoid duplicate generate provider blocks" — its provider is inline. Verify per-module whether they use `include "root"` or a standalone `generate "provider"`.
- **Don't import DKIM records using `import {}` blocks in HCL** (the H9 precedent from Plan 84.1). DKIM token names are not knowable at plan time without having the identity in state already. Use imperative `terragrunt import` calls from Go code instead.
- **Don't use `os.Exec("aws", "ses", ...)` shell-out for DKIM tokens.** The `ses.Client` is already constructed in `runBootstrapSharedSES`. Use `sesClient.GetIdentityDkimAttributes(...)` directly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Parse HCL attribute types | string matching / regex on raw `.tf` text | `hcl/v2/hclsyntax` type assertions | Raw text can't distinguish `"km-foo"` (literal) from `"${km-foo}"` (interpolation with literal-looking content) |
| SCP 5KB plan-time guard | Go test that renders policy and checks byte count at test time only | `terraform_data` + `lifecycle.precondition` | Precondition fires at every `terraform plan`; Go test is a secondary gate |
| Inventory diff tooling | Custom JSON comparison library | `aws` CLI + `jq` + `sort` + `diff` | AWS CLI returns consistent JSON per service; jq tag-filter patterns are well-understood; no new Go dependency needed |

**Key insight:** The inventory diff is operator-run UAT (not a CI test). A checked-in `scripts/inventory-diff.sh` is cleaner than inlining 50 lines of shell in PLAN.md. But whether it becomes a script or inline shell is Claude's discretion per CONTEXT.md.

## Common Pitfalls

### Pitfall 1: EFS `creation_token` Is the Identity — SG `name` Is NOT

AWS EFS: `creation_token` uniquely identifies a filesystem. If two installs share the same `creation_token`, the second `apply` fails with "EFS filesystem with that creation token already exists." This is the exact failure mode the `whereiskurt` probe hit (`km-shared-use1` collision).

AWS Security Groups: `name` must be unique within a VPC. Two installs in different VPCs can have the same SG name without collision. Two installs sharing a VPC (which currently don't occur — each install gets its own VPC from `km bootstrap`) also need unique SG names. The `${var.resource_prefix}-efs-${var.region_label}` naming makes both safe.

**Warning sign:** Any plan output showing "resource already exists" or "creation token already in use" during `km init` on the second install.

### Pitfall 2: SCP Policy Byte Count — AWS Measures the JSON, Not the HCL

AWS enforces a 5,120-byte limit on the JSON-serialized SCP policy content. The `data.aws_iam_policy_document` Terraform resource produces compact JSON (no extra whitespace). The `length()` function in the precondition measures UTF-8 bytes of that compact JSON. The `kph` prefix (3 chars) is shorter than `km` (2 chars), so a `kph` policy is slightly larger. The `whereiskurt` prefix (11 chars) is the worst case. Both must pass the 5,000-byte threshold (with 120-byte buffer). The companion Go test for the 4KB (not 5KB) threshold is an additional safety margin.

**Warning sign:** `precondition` failure during `km init --plan` after adding trusted ARNs.

### Pitfall 3: `hcl/v2` vs `hcl v1` — Different APIs

`github.com/hashicorp/hcl v1.0.0` (currently in `go.mod` as indirect) is the legacy HCL1 format parser. Terraform uses HCL2 (`github.com/hashicorp/hcl/v2`). The `hclsyntax` package is in `hcl/v2` ONLY. If the test imports `hcl v1` by mistake, it will silently fail to parse Terraform files (which are HCL2 format).

**Prevention:** `import "github.com/hashicorp/hcl/v2/hclsyntax"` — the `/v2/` is load-bearing. Run `go mod tidy` after adding the dependency.

### Pitfall 4: `filepath.Glob` Cannot Match Two-Level Wildcards

`filepath.Glob("infra/modules/*/*/main.tf")` works (two `*` at different depths). But `"infra/modules/**/*.tf"` does NOT work — `**` is not supported by Go's `filepath.Glob`. Use two-level glob or `filepath.WalkDir` with depth-tracking.

**Prevention:** Test the walk function against the actual `infra/modules/` tree before writing assertions.

### Pitfall 5: `terragrunt import` Requires Init'd Backend

`terragrunt import` (unlike `terraform import`) requires the backend to be initialized. The runner's `buildCommand` already sets `TG_BACKEND_BOOTSTRAP=true` and `TG_NON_INTERACTIVE=true`, which handles auto-init. However, import needs the module's `.terragrunt-cache` to be initialized with the provider. If called against a fresh clone, `terragrunt import` may fail with "Backend not initialized." Add a `Reconfigure` (init -reconfigure) call before the import sequence as a safety measure, following the pattern from `RunUninitWithDeps`.

### Pitfall 6: Route53 Import ID Double-Underscore for `_amazonses` Records

Confirmed from `ses-shared-rule-set/v1.0.0/main.tf` line 76:
```hcl
id = "${var.hosted_zone_id}__amazonses.${var.email_domain}_TXT"
```
The double underscore (`__`) is because the record name itself starts with `_amazonses` and the import ID format concatenates `<zone_id>_<name>_<type>`. One `_` is the separator, one is part of the record name. This is correct and confirmed against the actual module.

### Pitfall 7: `BuildSCPPolicy` in `bootstrap.go` Still Uses `km-` Hardcodes

After `scp/v2.0.0/` is created with `${var.resource_prefix}` templating, the Go function `BuildSCPPolicy` in `bootstrap.go` (lines 1071-1092) still hardcodes `km-ecs-spot-handler`, `km-budget-enforcer-*`, etc. This function is called by `runShowPrereqs` for displaying the SCP document to the operator — it doesn't apply the SCP, but it will display incorrect ARNs for non-default-prefix installs. The planner should include a task to update `BuildSCPPolicy` to accept `resourcePrefix string` as a parameter.

### Pitfall 8: `km uninit` Does Not Touch SCP or Foundation

`km uninit` calls `regionalModules()` in reverse — confirmed to include `efs`, `s3-replication`, `ses`, but NOT `scp` (SCP is in `infra/live/management/scp/`, a separate path from `infra/live/use1/...`). The `whereiskurt` SCP must be manually detached and deleted. `km unbootstrap` deletes SSM params, S3 buckets, KMS key/alias, and optionally Route53 zone — it does NOT detach/delete the SCP. This is correct behavior (SCP is Management account-global, not regional) but the planner must explicitly encode the manual SCP steps.

```go
// TestModuleNamesUseResourcePrefix asserts that every v2.0.0+ module in
// infra/modules/ uses ${var.resource_prefix} in resource-name positions
// and does not contain hardcoded "km-" literals.
func TestModuleNamesUseResourcePrefix(t *testing.T) {
    repoRoot := findRepoRootForTest(t)
    modules := walkModulesV2(t, repoRoot)
    if len(modules) == 0 {
        t.Fatal("no v2.0.0+ modules found — check walk path")
    }
    for _, modDir := range modules {
        entries, _ := os.ReadDir(modDir)
        for _, e := range entries {
            if !strings.HasSuffix(e.Name(), ".tf") { continue }
            checkTFFile(t, filepath.Join(modDir, e.Name()))
        }
    }
}

func walkModulesV2(t *testing.T, repoRoot string) []string {
    t.Helper()
    basePath := filepath.Join(repoRoot, "infra", "modules")
    entries, err := os.ReadDir(basePath)
    if err != nil { t.Fatalf("ReadDir: %v", err) }
    var result []string
    for _, mod := range entries {
        if !mod.IsDir() { continue }
        verDir := filepath.Join(basePath, mod.Name())
        vEntries, _ := os.ReadDir(verDir)
        for _, ver := range vEntries {
            if !ver.IsDir() { continue }
            if ver.Name() == "v1.0.0" { continue }  // skip historical reference
            if strings.HasPrefix(ver.Name(), "v") {
                result = append(result, filepath.Join(verDir, ver.Name()))
            }
        }
    }
    return result
}
```

## Open Questions

1. **`bootstrap.go` `BuildSCPPolicy` update scope**
   - What we know: `BuildSCPPolicy` (line 441) mirrors SCP module trusted_arns but hardcodes `km-`. It's called for display-only purposes (`runShowPrereqs`), not for actual SCP apply.
   - What's unclear: Does `runShowPrereqs` need to display accurate ARNs for non-`km` prefixes in Phase 84.4's scope, or is display-only correctness deferred?
   - Recommendation: Update `BuildSCPPolicy` to accept `resourcePrefix string` parameter — small change, prevents confusing IAM prereq output for `rg` prefix UAT.

2. **`s3-replication` IAM role name reference audit**
   - What we know: CONTEXT.md says "audit `infra/` for any references to `km-s3-replication-*` role names." The `s3-replication/v1.0.0/main.tf` policy uses `aws_iam_role.replication.arn` (not the name string) in the replication config — so the policy itself is clean.
   - What's unclear: Whether any other module, compiler, or live wiring references the `km-s3-replication-` role name as a literal string.
   - Recommendation: `grep -rn "km-s3-replication" infra/ internal/ pkg/ cmd/` as a planner-encoded pre-task. Expect zero hits beyond `s3-replication/v1.0.0/main.tf`.

3. **`s3-replication` provider alias handling in v2.0.0**
   - What we know: `s3-replication/v1.0.0/main.tf` declares `provider "aws" { alias = "replica" ... }` at the top of the module. The live wiring (`infra/live/use1/s3-replication/terragrunt.hcl`) explicitly does NOT use `include "root"` to avoid duplicate `generate "provider"` blocks.
   - What's unclear: Does `v2.0.0` need any provider handling changes? Almost certainly not — just copy the module structure.
   - Recommendation: Copy v1.0.0 provider block verbatim into v2.0.0. No change needed.

4. **DKIM import idempotency when DKIM records are already in tfstate**
   - What we know: `terragrunt import` on an already-imported resource fails with "resource already exists in state." The auto-import code needs to check tfstate first.
   - What's unclear: Best way to check — read tfstate JSON via S3, or use `terragrunt state list` output.
   - Recommendation: Use the `FoundationStateReader.StateOwns(addr)` pattern already established in Phase 84.1 (`detectSharedSESState` uses it for the rule set and domain identity). Extend it to check `aws_route53_record.dkim[0]`, `[1]`, `[2]`, `aws_route53_record.mx[0]`, `aws_route53_record.ses_verification[0]`. Only import records NOT already in state.
