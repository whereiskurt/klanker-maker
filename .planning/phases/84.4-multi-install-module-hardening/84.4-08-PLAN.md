    ## Phase 84.4 — Multi-install module hardening

    As of Phase 84.4, `infra/modules/scp/`, `infra/modules/efs/`, and
    `infra/modules/s3-replication/` have v2.0.0 directories that template all
    resource names from `${var.resource_prefix}`. A second `km` install in the same
    AWS account/Organization can apply alongside the first without resource-name
    collisions, and the second install's SCP correctly guards its own
    `${prefix}-*` lambdas (not the first install's `km-*` lambdas).

    ### AWS 5-SCP-per-target limit

    Each install owns `${var.resource_prefix}-sandbox-containment` SCP attached to
    its own application account. AWS allows up to 5 SCPs per OU/account, so up to
    5 km installs can share a single application account before hitting this limit.

    ### Overlap semantics

    When two installs share an application account, both `${prefix}-sandbox-containment`
    SCPs attach independently. SCPs combine with AND-logic for Deny statements, but
    each install's Deny condition is gated by `ArnLike` on `${prefix}-*` role
    patterns — so the km install's SCP cannot accidentally deny the rg install's
    lambdas (and vice versa). Cross-install denies don't fire.

    ### Second-install bootstrap sequence

    To stand up a second install (e.g. `resource_prefix: rg`) in an account that
    already has a `km` install:

    ```
    # 1. Clone the repo to a sibling working directory
    git clone <repo-url> klanker-maker-rg
    cd klanker-maker-rg
    make build

    # 2. Configure
    ./bin/km configure
    # Set resource_prefix: rg; choose email_subdomain (shared or separate)

    # 3. Plain bootstrap (SCP, KMS, artifacts, state)
    ./bin/km bootstrap --dry-run=false

    # 4. Shared SES (auto-imports DKIM/MX/TXT if domain identity exists)
    ./bin/km bootstrap --shared-ses --dry-run=false

    # 5. Plan + apply
    ./bin/km init --plan
    ./bin/km init --dry-run=false
    ```

    `km bootstrap --shared-ses` auto-detects when the domain identity already exists
    in AWS (from a sibling install). It fetches the DKIM tokens via
    `ses.GetIdentityDkimAttributes`, then imports `aws_route53_record.dkim[0..2]`
    (and `mx[0]`, `ses_verification[0]` if those records exist) into the second
    install's foundation tfstate before apply. No manual `terragrunt import`
    invocations are required.

    ### 5,120-byte SCP limit guard

    `infra/modules/scp/v2.0.0/main.tf` includes a `terraform_data.scp_size_guard`
    precondition that trips at plan-time if the rendered SCP policy JSON exceeds
    5,000 bytes (120-byte buffer below the AWS 5,120-byte hard limit). If a future
    addition to `trusted_arns_*` would bust the limit, the plan fails with a clear
    error before AWS rejects the apply.

    ### Companion teardown

    `km uninit --dry-run=false` destroys regional resources (efs, s3-replication,
    ses, network, etc.). `km unbootstrap --dry-run=false` destroys the install's
    foundation (state bucket, KMS alias, artifacts bucket). The install's SCP is
    NOT touched by either — manually detach via:

    ```
    AWS_PROFILE=<management> aws organizations detach-policy \
      --policy-id <id> --target-id <application-account>
    AWS_PROFILE=<management> aws organizations delete-policy --policy-id <id>
    ```
