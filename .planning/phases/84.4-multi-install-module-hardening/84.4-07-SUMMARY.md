## Issues Encountered

1. **SCP dual-tracking (architectural learning):** Both the canonical km install and the whereiskurt probe install tracked the same `km-sandbox-containment` SCP (p-cvd490xt) in terraform state. This was not a teardown blocker once identified, but is a significant multi-install onboarding constraint. Step 0 (SCP state rm) was added to the teardown procedure as a CRITICAL prerequisite. Future OPERATOR-GUIDE.md multi-install section must warn: if the second install's SCP module was initialized by importing an existing policy, always run `terragrunt state rm` on the three SCP state resources before `km uninit`.

2. **EFS non-issue:** The whereiskurt EFS apply had failed before filesystem creation (Plan 05 finding — creation_token collision with canonical km EFS). Only the security group `km-efs-use1` existed, not an actual EFS filesystem. The SG was destroyed by `km uninit` as part of the VPC teardown cascade. No manual EFS cleanup was required.

3. **KMS 7-day soft-delete:** The whereiskurt KMS key (d40b51c6-e319-4d2c-9d51-511ab08cfa4c) entered PendingDeletion state. Scheduled hard-deletion: 2026-05-25. This is expected behavior for `km unbootstrap` — key material inaccessible immediately, AWS deletes after 7 days. Not a teardown failure.
