# Phase 84.4: Multi-install module hardening — `infra/modules/` source fixes — Context

**Gathered:** 2026-05-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Make every `infra/modules/**/*.tf` safe for parallel `km` installs in the same AWS account/Organization. Today only the SES path is multi-install-safe (Phase 84 shared rule set + per-install rules); SCP, EFS, S3-replication, and ~15 other modules hard-code the literal `km-` in resource names — second install collides at apply, and even after import the SCP's *policy content* (trusted_arns_*) still denies the second install's `${prefix}-*` lambdas at runtime, so second-install sandboxes can't actually be created.

**Delivers:**
- SCP module bumped to `v2.0.0`: `${prefix}-sandbox-containment` per-install policy, `trusted_arns_*` templated from `var.resource_prefix`
- EFS module bumped to `v2.0.0`: per-install `${prefix}-shared-${region}` filesystem + SG + mount targets
- S3-replication module bumped to `v2.0.0`: prefix-namespaced IAM role + policy names
- KM-literal audit via Go HCL static-analysis test (`pkg/terragrunt/modulehygiene_test.go`) wired into `make test`
- `runBootstrapSharedSES` auto-detect + auto-import for foundation DKIM/MX/TXT records
- Whereiskurt probe install cleanly torn down + a fresh-prefix UAT (`rg` or equivalent) covering bootstrap → apply → sandbox create → destroy → uninit → unbootstrap with byte-identical AWS-CLI inventory diff against the existing `km` install

**Does NOT deliver:**
- ECS / ec2spot resource-name prefix-namespacing (those literals don't currently collide because they include `${var.sandbox_id}` — deferred unless a follow-up phase needs them)
- Wrapper-level UX fixes (Phase 84.3's domain)
- Cross-install data sharing surfaces (foreign-install EFS mount, cross-install IAM trust, etc.)
- Migration tooling to rename an existing install's prefix (`km` → something else)

</domain>

<decisions>
## Implementation Decisions

### SCP architecture (LOCKED)

**Per-install SCP.** Each install owns `${var.resource_prefix}-sandbox-containment` attached to its own `application_account_id`.

- `infra/modules/scp/v1.0.0/` stays in tree as historical reference; new work goes in `v2.0.0/`.
- `aws_organizations_policy.sandbox_containment.name = "${var.resource_prefix}-sandbox-containment"`
- `trusted_arns_instance/iam/ssm` locals derive `${var.resource_prefix}-ecs-spot-handler`, `${var.resource_prefix}-budget-enforcer-*`, `${var.resource_prefix}-ec2spot-ssm-*`, `${var.resource_prefix}-github-token-refresher-*`, `${var.resource_prefix}-ttl-handler` — module-internal templating, NOT an operator-overridable variable. (Module logic stays in sync with the lambdas it guards; both update when prefix changes.)
- Two installs sharing the same `application_account_id`: each install's SCP attaches independently to that account (AWS allows up to 5 SCPs per OU/account; AND-logic for Deny across overlapping SCPs is safe here because each install's Deny condition is gated by ArnLike on `${prefix}-*` role patterns, so cross-install denies don't fire). Document the AWS 5-SCP/target limit + the overlap semantics in OPERATOR-GUIDE.md Multi-install section.
- **5,120-byte SCP limit guard:** Add a `precondition` block (or equivalent `terraform_data` validation) at plan time that asserts `length(data.aws_iam_policy_document.sandbox_containment.json) <= 5000` (120-byte buffer for AWS overhead). Fails the plan if a future trusted_arns addition would bust the limit. Companion Go unit test in the module's test harness renders the policy with synthetic `kph` (3-char) and `whereiskurt` (11-char) prefixes and asserts both render under 4KB.

### EFS architecture (LOCKED)

**Per-install EFS.** Each install owns `${var.resource_prefix}-shared-${region}` filesystem + `${var.resource_prefix}-efs-${region}` security group + mount targets.

- `infra/modules/efs/v1.0.0/` stays in tree; new work goes in `v2.0.0/`.
- `aws_efs_file_system.shared.creation_token = "${var.resource_prefix}-shared-${var.region_label}"`
- `aws_security_group.efs.name = "${var.resource_prefix}-efs-${var.region_label}"`
- All `Name` tags follow the same `${var.resource_prefix}-*` pattern; tags include `"km:resource-prefix" = var.resource_prefix` (Phase 82 convention).
- **No `moved {}` blocks needed for default-install rename.** With `var.resource_prefix = "km"` (the backward-compat default per Phase 82), interpolation produces `km-shared-${region}` and `km-efs-${region}` — byte-identical to today. AWS resource identity preserved (creation_token unchanged). Verified by `km init --plan` against the existing `km` install showing zero-diff for the efs module — added as a planner task ("Wave N: verify default-install zero-diff").
- Mount targets land in the install's own `vpc_id` + `subnet_ids` (wired through live wiring's `inputs = {}`). Two installs in different VPCs each get their own filesystem with mount targets in their own VPC. Two installs sharing a VPC each get their own filesystem with mount targets in that VPC (AWS allows multiple filesystems per VPC).
- Sandbox-side mount: sandboxes only mount THEIR OWN install's filesystem (compiler plumbs `efs_dns` / `efs_id` per-sandbox from the same install's outputs). No cross-install mount surface.

### S3-replication architecture (LOCKED)

**Universally prefix-namespaced.** `infra/modules/s3-replication/v1.0.0/` → `v2.0.0/`.

- `aws_iam_role.replication.name = "${var.resource_prefix}-s3-replication-${var.source_bucket_name}"`
- `aws_iam_role_policy.replication.name = "${var.resource_prefix}-s3-replication-policy"`
- Audit `infra/` for any references to `km-s3-replication-*` role names (in IAM policies, sandbox userdata, compiler-emitted templates, etc.). If found, those references must use the new prefix-namespaced names too — add to PLAN.md task list at planning time.

### Audit / regression enforcement (LOCKED)

**Go HCL static-analysis test.** New file `pkg/terragrunt/modulehygiene_test.go`.

```go
func TestModuleNamesUseResourcePrefix(t *testing.T) {
  modules := walkModulesV2("infra/modules")  // skip v1.0.0 dirs (historical)
  for _, mod := range modules {
    files := parseHCLFiles(mod)  // github.com/hashicorp/hcl/v2/hclsyntax
    for _, attr := range collectResourceNameAttrs(files) {
      if isLiteral(attr.Expr) && containsKmDash(attr.Value) {
        t.Errorf("%s: %s has hardcoded km- literal: %q",
          mod, attr.Position, attr.Value)
      }
      if isInterpolation(attr.Expr) {
        assert.True(t, refersTo(attr.Expr, "var.resource_prefix"))
      }
    }
  }
}
```

- Walks each `infra/modules/*/v2.0.0/*.tf` (and any other `v2.0.0+` directories), parses with `hclsyntax`, collects "resource-name-position" attributes, and asserts each is either an interpolation containing `var.resource_prefix` or on the allow-list. No AWS calls, no `terragrunt init` — runs in <1s. Wired into `make test`.
- **"Resource-name positions" (subject to audit):** `name`, `creation_token`, `alias`, `policy_name`, and IAM policy document `Resource` ARNs that reference roles by name.
- **NOT audited (allow-list, no fail):** comments, `variable.default` values (those legitimately set `"km"` as the backward-compat default), `Component` tag values, `"km:manager"` / `"km:component"` / `"km:purpose"` tag values, description strings, sentinel strings. Phase 82 set the precedent that `km:manager = "km-cluster"` is a label, not a resource name.
- `infra/modules/*/v1.0.0/` directories are EXCLUDED from the audit walk — those keep their literals for historical reference. The audit only enforces v2.0.0+ modules.
- The audit applies to ses/v2.0.0/ too (the existing Phase 84 v2.0.0 module is already prefix-clean — the audit codifies that going forward).

### Module versioning strategy (LOCKED)

**Bump to v2.0.0, keep v1.0.0 in tree.** Matches Phase 84 SES precedent.

- `infra/modules/scp/v2.0.0/`, `infra/modules/efs/v2.0.0/`, `infra/modules/s3-replication/v2.0.0/`
- Live wiring (`infra/live/<region>/scp/terragrunt.hcl` etc.) flips `source` from `v1.0.0` to `v2.0.0`.
- Each `v2.0.0` directory ships with a `test/` subdirectory containing a minimal Go test or HCL test harness that instantiates the module with synthetic prefixes and asserts via static analysis (mirrors `cluster-irsa/v1.0.0/test_harness.tf.skip` pattern).

### Backward-compat default (LOCKED)

`variable "resource_prefix" { default = "km" }` in each `v2.0.0/variables.tf`. With the default, interpolation produces names byte-identical to today's `km-*` literals — existing single-install operators see zero diff at `terragrunt plan`.

### Variable plumbing (LOCKED)

`var.resource_prefix` is wired through from the live wiring's `inputs = {}` block. Same pattern as Phase 80 cluster-irsa and Phase 84 ses-shared-rule-set.

### Foundation DKIM/MX/TXT auto-import (LOCKED — carried over from Phase 84.3 #4)

**`runBootstrapSharedSES` auto-detect + auto-import.**

When `runBootstrapSharedSES` detects the domain identity exists in AWS (`aws ses list-identities --identity-type Domain`) but no `aws_route53_record.dkim` entries are present in the foundation tfstate:

1. Shell out: `aws ses get-identity-dkim-attributes --identities sandboxes.${domain}` to obtain the 3 DKIM tokens.
2. 3× `terragrunt import 'aws_route53_record.dkim[0]' ...` (and `[1]`, `[2]`) — note the `count`-indexed addressing matches `ses-shared-rule-set/v1.0.0/main.tf` (resources use `count = var.register_domain_identity ? 3 : 0`).
3. Conditional `aws route53 list-resource-record-sets --hosted-zone-id ${zone_id}` lookup for the MX record + verification TXT.
4. If found, `terragrunt import aws_route53_record.mx` + `terragrunt import aws_route53_record.ses_verification`.

Replicates the manual runbook in `OPERATOR-GUIDE.md` § Phase 84.1 (already zsh-fixed in commit `69704cf`). On a second install in a shared-domain account, operator runs `km bootstrap --shared-ses --dry-run=false` and the imports happen automatically — no manual `terragrunt import` calls needed.

### Whereiskurt probe teardown (LOCKED)

**Standard sequence + AWS-CLI inventory diff.**

1. **BEFORE snapshot:** `aws` CLI inventory of resources in account 052251888500 tagged or named for either prefix. Cover: EFS filesystems, EFS mount targets, EFS security groups, SCPs attached to the application account, IAM roles/policies matching `whereiskurt-*` or `km-*`, Lambda functions, DynamoDB tables, S3 buckets, KMS key aliases, Route53 records, SES rules. Save as `before.json`.
2. `km uninit --dry-run=false` from the `klanker-maker-whereiskurt/` working directory — regional teardown.
3. Manual SCP detach + delete (whereiskurt SCP): `aws organizations detach-policy --policy-id <id> --target-id <app-account>` + `aws organizations delete-policy --policy-id <id>`. (Manual because the whereiskurt install's apply was incomplete — SCP is imported into state but the live module's `lifecycle.prevent_destroy = false` may or may not be present.)
4. Manual EFS delete: whereiskurt EFS was orphaned by the failed apply (collided with km's `creation_token`). `aws efs delete-mount-target` then `aws efs delete-file-system`.
5. `km unbootstrap --dry-run=false` — foundation teardown. Deletes whereiskurt state bucket, KMS key alias, etc.
6. **AFTER snapshot:** Re-run the same `aws` CLI inventory. Save as `after.json`.
7. **Diff:** All `whereiskurt-*` named/tagged resources gone. All `km-*` named/tagged resources byte-identical to BEFORE (modulo timestamps). Any unexpected delta → halt and investigate before fresh-prefix UAT runs.

This protocol IS the planner's first task ("Wave 0: whereiskurt teardown"). The whereiskurt install is a probe — do NOT attempt to "finish" it.

### Fresh-prefix UAT scope (LOCKED — full lifecycle)

**End-to-end:** bootstrap → all 17 regional modules green → sandbox create → sandbox destroy → uninit → unbootstrap → AWS-CLI inventory diff against the `km` install (`km` install byte-identical before and after).

Concrete steps the planner must encode in PLAN.md:
1. Pick a fresh prefix (e.g. `rg`), clone `klankrmkr` into `klanker-maker-rg/`.
2. `km configure` — operator-driven; produces `km-config.yaml` with `resource_prefix: rg` and the Phase 84.3-derived artifacts/state-bucket names.
3. `km bootstrap --dry-run=false` (plain — SCP/KMS/artifacts/state).
4. `km bootstrap --shared-ses --dry-run=false` (foundation SES — auto-imports DKIM/MX/TXT if domain identity already exists).
5. `km init --plan` — destroy-class gate must trip zero protected resources. Verify all 17 regional modules pass.
6. `km init --dry-run=false` — all 17 regional modules green.
7. `km create profiles/learn.yaml` — sandbox launches, reaches healthy state, `initCommands` run successfully, observable in `km otel`.
8. `km destroy <sandbox-id> --remote --yes` — sandbox cleanly torn down.
9. `km uninit --dry-run=false` — regional teardown clean.
10. `km unbootstrap --dry-run=false` — foundation teardown clean.
11. **AWS-CLI inventory diff against the `km` install:** every `km-*`/`km`-tagged resource byte-identical to a snapshot taken before the UAT started.

This is the load-bearing test for closure criterion (g) in the roadmap goal. It proves both "second install works at runtime" AND "second install can be cleanly removed without affecting the first install."
