<objective>
Add a `--no-sign` flag to km-send so the same script can deliver plain email to external (Gmail/non-sandbox) recipients without trying to fetch a signing key or stamp X-KM-* headers.

Purpose: Today km-send unconditionally fetches the Ed25519 key from SSM (line 873) and signs (line 884). Sending to `whereiskurt@gmail.com` works only by accident (the signing succeeds but the headers are noise to Gmail). Per Phase 57 goal, external email should be a first-class path: skip SSM, skip openssl, omit X-KM-* headers — but KEEP the KM-AUTH safe-phrase appender (it's an authentication mechanism independent of Ed25519 and is needed for the inbound reply path validated by km-mail-poller in Plan 03).

Output: 4 surgical edits inside the `pkg/compiler/userdata.go` km-send heredoc (lines ~688-919) that turn 3 of the Wave 0 RED tests GREEN: `TestUserData_KmSend_NoSignFlag`, `TestUserData_KmSend_NoSignOmitsHeaders`, `TestUserData_KmSend_NoSignKeepsAuthPhrase`. All pre-existing `TestKmSend*` tests in `userdata_test.go` (SSMFetch, OpensslSign, SESv2Send, PKCS8Prefix, PresentWhenEmailSet) MUST remain GREEN — the signing block is GUARDED, not removed.

Honors locked decision: "km-send `--no-sign` skips Ed25519 SSM key fetch and X-KM-* headers, enabling plain email to Gmail/external addresses; KM-AUTH safe-phrase appending stays active." (roadmap Phase 57 goal, line 1248)
</objective>
