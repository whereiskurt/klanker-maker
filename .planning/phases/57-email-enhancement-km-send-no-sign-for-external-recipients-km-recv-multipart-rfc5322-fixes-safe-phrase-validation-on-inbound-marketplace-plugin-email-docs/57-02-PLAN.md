<objective>
Fix three closely-coupled km-recv bugs that prevent external (Gmail) email from being readable inside a sandbox:

1. RFC 5322 folded headers (Subject across multiple lines, X-KM-Signature wrapped past the 998-char SES boundary) parse as truncated/empty.
2. multipart/alternative body extraction — Gmail's `multipart/mixed -> multipart/alternative -> {text/plain, text/html}` layout silently produces an empty body because `extract_body()` only scans one boundary level deep.
3. No visible distinction between sandbox-signed mail and external mail — operators and agents have no signal that an email is unsigned/external.

Purpose: Phase 57 promotes external email to a first-class path. Plan 01 delivers OUTBOUND --no-sign; Plan 02 delivers INBOUND parsing for the corresponding replies + Gmail-originated messages. Together with Plan 03's safe-phrase gate, they close the external-email loop.

Output: Three surgical edits inside the `pkg/compiler/userdata.go` km-recv heredoc (lines ~941-1353) that turn the 4 Wave 0 RED tests GREEN: `TestUserData_KmRecv_FoldedHeaders`, `TestUserData_KmRecv_MultipartAlternative`, `TestUserData_KmRecv_NestedMultipart`, `TestUserData_KmRecv_ExternalDisplay`, plus `TestUserData_KmRecv_ExternalJSONField`. Pre-existing km-recv tests (TestKmRecvContainsDynamoDBLookup, TestKmRecvContainsOpensslVerify, TestKmRecvContainsMailDir, TestKmRecvContainsSPKIDERPrefix, TestKmRecvPresentWhenEmailSet, TestKmRecvAbsentWhenNoEmail) MUST remain GREEN.

Honors locked decision: "km-recv gets RFC 5322 folded header parsing, multipart/alternative body extraction (Gmail HTML emails), and `--from-external` display hint" (roadmap Phase 57 goal). Per RESEARCH.md, the "--from-external" CLI flag is implemented as automatic detection — no new flag — based on absence of X-KM-Sender-ID header. This deviation is explicitly allowed in the planning context.

        # unfold_headers: RFC 5322 header unfolding via awk one-liner.
        # Continuation lines (starting with space or tab) are joined to the
        # preceding header line with a single space replacing the leading whitespace.
        # MUST be applied ONLY to the header section — never to the body, because

        unfold_headers() {
