<objective>
Finish the resource_prefix migration for the per-sandbox CloudWatch log groups so they are
CREATED with the same dynamic prefix that teardown DELETES with. This stops the log-group leak
at the source on non-default-prefix installs (the root cause behind the ~271 orphaned groups on
the kph install) while remaining a byte-identical no-op on the default km install. Includes the
lockstep create-handler IAM fix and a scoped SCP-coupling investigation.

Purpose: Make new sandboxes self-clean at teardown instead of relying solely on the doctor
safety net; close the project_teardown_prefix_asymmetry gap for log groups.
Output: Three TF modules + two compiler files migrated, golden byte-identity test for the default
install, and a documented decision on the SCP role-pattern coupling.

ROLLOUT NOTE (surface in the SUMMARY, not a task): the TF + Lambda changes require
`make build-lambdas` (clean — see project_km_init_skips_existing_lambda_zips) + `km init --dry-run=false`
to deploy. Existing sandboxes are NOT retroactively renamed — they keep their legacy `km-` groups,
which the Plan 94-02 doctor check reclaims at teardown via its both-names match.
</objective>
