<objective>
Add the flag surface and resolution/guard logic for scoped `km init`:
- Two package-level curated allowlist slices: `scopedCheapAllowlist` (tier-1:
  lambda-github-bridge, lambda-slack-bridge, lambda-h1-bridge, email-handler) and
  `scopedGatedAllowlist` (tier-2: ses).
- A `resolveScopedModule(...)` helper that maps the `--only` value and the four
  sugar aliases (`--github`/`--slack`/`--h1`/`--email`) to a single canonical module
  name, validates it against the combined allowlist, and errors (listing the allowed
  set) on unknown/out-of-allowlist input.
- Five new Cobra flags on `km init`: `--only <string>`, `--github`, `--slack`,
  `--h1`, `--email` (bool).
- A mutual-exclusion guard in the dispatch block: scoped flags are incompatible with
  `--plan`, `--sidecars`, `--lambdas` (and with each other — at most one entry point).

This plan does NOT implement the apply path (`runInitScoped`/`RunInitScopedWithRunner`)
— that is Plan 03. To keep the dispatch compiling and testable here, wire the scoped
branch to call a package-level function variable `runInitScopedFunc` that Plan 03
will define; in THIS plan, declare it as a stub `var runInitScopedFunc = func(...) error { return fmt.Errorf("not implemented") }`
so the guard/resolution tests can run without the apply path. Plan 03 replaces the
stub assignment with the real implementation.

Purpose: Deliver the operator-facing flag contract + validation, independently testable.
Output: Updated init.go (flags, allowlist, resolver, guard) + real test assertions.
