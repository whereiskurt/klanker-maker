**`km doctor` now surfaces a WARN with a copy-paste `aws dynamodb update-item` recovery command when an S3 state object's MD5 disagrees with the DynamoDB lock-table Digest — the post-Ctrl-C breakage diagnosed in Phase 84 UAT observation #8.**

## Decisions Made

All five decisions captured in the frontmatter `key-decisions` field. Highlights:

- **Detection-only, not auto-repair.** Digest repair mutates terragrunt's source-of-truth metadata; making it a doctor side-effect would let a `km doctor` run silently rewrite the same field that subsequent terragrunt operations rely on as a tamper-detection beacon. The Remediation field gives operators the exact command to run deliberately.
- **WARN, not ERROR.** Doctor's ERROR exit code is intended for "this install is broken right now" — but a digest mismatch doesn't corrupt state, it just blocks terragrunt's safety check. The S3 state IS correct. WARN matches the existing `checkSESRules` orphan-rules pattern from Phase 84.
- **`dynamodb.NewScanPaginator` (H7).** Specifically called out in plan-checker rev 1; the aws-sdk-go-v2 documentation is explicit that `Scan` does not auto-paginate and a single call caps at 1MB.

## Deviations from Plan

The plan executed mostly as written; one Rule-3-class fix and two minor naming/API choices.

### Auto-fixed Issues

**1. [Rule 3 - Blocking] `dynamodbtypes` import alias instead of plan's `ddbtypes`**
- **Found during:** Task 1 GREEN
- **Issue:** Plan's prose suggested `ddbtypes` as the import alias; the rest of the codebase (`budget.go`, `budget_test.go`, `at_test.go`) consistently uses `dynamodbtypes`.
- **Fix:** Used `dynamodbtypes` to match the established convention.
- **Files modified:** `internal/app/cmd/doctor.go`, `internal/app/cmd/doctor_state_digest_test.go`.
- **Verification:** `go test ./internal/app/cmd/ -run "TestCheckStateLockDigest"` all pass.
- **Committed in:** `54baaf3` (Task 1 GREEN).

**2. [Rule 3 - Blocking] Two separate DoctorDeps fields instead of reusing S3Client/DynamoClient**
- **Found during:** Task 2 GREEN
- **Issue:** Plan suggested `s3Client := s3.NewFromConfig(awsCfg)` reusing existing-scope variables, but the existing `deps.S3Client` is typed `S3HeadBucketAPI` (only HeadBucket/HeadObject) and `deps.DynamoClient` is `DynamoDescribeAPI` (only DescribeTable). Neither supports the new check's operations.
- **Fix:** Added two new narrowly-typed fields `StateLockS3Client S3StateReader` and `StateLockDDBClient LockDigestReader` to `DoctorDeps`; wired both in `initRealDepsWithExisting`.
- **Why this is better than widening:** Keeps the existing narrow interfaces minimal — broadening `S3HeadBucketAPI` to include `GetObject` would pollute all unrelated checks. Two new narrow fields cost nothing at runtime.
- **Files modified:** `internal/app/cmd/doctor.go`.
- **Verification:** `TestBuildChecks_IncludesStateLockDigest` and `TestBuildChecks_StateLockDigest_NilClientsSkippedCleanly` both PASS.
- **Committed in:** `ef96321` (Task 2 GREEN).

**3. [Rule 1 - Bug] Use `errors.As` with `*s3types.NoSuchKey` (concrete type) instead of `*s3.NoSuchKey`**
- **Found during:** Task 1 GREEN (compile-time)
- **Issue:** AWS SDK v2 puts the typed exception in the `types` subpackage (`s3types`); plan's snippet implied root `s3` package.
- **Fix:** Used `var noSuchKey *s3types.NoSuchKey; errors.As(gErr, &noSuchKey)` matching the existing `var rnf *dynamodbtypes.ResourceNotFoundException` pattern in `budget.go`.
- **Files modified:** `internal/app/cmd/doctor.go`.
- **Verification:** `TestCheckStateLockDigest_S3StateMissing` returns CheckWarn with "orphan" message.
- **Committed in:** `54baaf3` (Task 1 GREEN).
