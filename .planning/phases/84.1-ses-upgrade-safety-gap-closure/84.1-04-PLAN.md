Close the THREE highest-impact gaps from Phase 84 UAT in one tightly-coupled Wave 2 plan:

**GAP-2** (`km bootstrap` not idempotent): Foundation `aws_ses_receipt_rule_set.shared` has `lifecycle.prevent_destroy = true` AND `count = var.register_shared_rule_set ? 1 : 0`. Second bootstrap invocation: auto-detect sees rule set exists → `KM_REGISTER_SHARED_RULESET=false` → count flips 1→0 → terraform plans destroy → `prevent_destroy` blocks → bootstrap errors. Per `84-10-UAT.md` line 147: "register-flag semantics are incompatible with prevent_destroy. Likely fix: register flags should mean 'manage this' not 'create-only-first-time'; once foundation has the resource in state, the flag should stay true."

**GAP-3** (auto-detect operates on AWS reality, not state ownership): On in-place upgrade from Phase 82.x, `sandboxes.example.com` domain identity existed in AWS (owned by regional v1.0.0 state). Auto-detect saw it and set `register_domain_identity=false` → foundation skipped creating → regional v1.0.0 destroy then nuked it. Net data loss: domain identity, DKIM, MX, verification TXT, active rule set pointer. See `84-10-UAT.md` line 149.

**GAP-6** (foundation does not import existing v1.0.0 resources): The dominant failure mode for any operator upgrading from Phase 82.x. The fix is some combination of (a) foundation `terraform import` blocks for the shared resources OR (b) regional v2.0.0 `removed {}` blocks that suppress the destroy. We do BOTH — they are complementary: import blocks bring the resource into foundation state; removed blocks ensure regional state release does not destroy the AWS resource. See `84-10-UAT.md` line 155.

Per CLAUDE.md "Phase 84" section: "Shared rule set: `sandbox-email-shared` — account-shared, owned by `infra/modules/ses-shared-rule-set/v1.0.0/`, has `lifecycle.prevent_destroy = true`." The `prevent_destroy` invariant MUST be preserved by this plan (per quality_gate).

Purpose: Make the Phase 82.x → 84 in-place upgrade safe — zero AWS resource loss, zero manual recovery. After this plan, the Phase 84 UAT recovery procedure (manual terragrunt apply with forced env vars + manual `aws dynamodb update-item`) is eliminated.

Output: Foundation module + regional v2.0.0 module that survive both fresh apply AND in-place upgrade from Phase 82.x with no destroy of shared resources.

Terraform import block syntax (Terraform 1.5+):
```hcl
import {
  to = aws_ses_domain_identity.sandbox[0]   # target address — note [0] because of count
  id = var.email_domain                      # SES domain identity uses the domain as the AWS resource ID
}
```
Multiple import blocks at module top-level are allowed; they apply on the next `terragrunt apply` and are then ignored. Operators can leave them in place — they have no effect after the resource is in state.

Terraform removed block syntax (Terraform 1.7+):
```hcl
removed {
  from = aws_ses_domain_identity.sandbox
  lifecycle { destroy = false }
}
```
Tells terraform "this resource is no longer in this module — forget the state entry, do NOT destroy the AWS object". Exactly what we need in regional v2.0.0 so the v1.0.0 → v2.0.0 transition doesn't destroy shared resources.
