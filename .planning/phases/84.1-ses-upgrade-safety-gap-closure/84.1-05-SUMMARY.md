## Decisions Made

- **Introduced `passed-with-caveats` UAT status** — operator chose the minimal UAT scope (Steps 1+2 only) explicitly to limit operational burden. Pure `passed` would have misrepresented the scope; pure `diagnosed` would have implied new gaps when in fact existing gaps were merely re-deferred. The new status honestly captures "work is complete, no regressions found, but the most operationally-heavy verifications were intentionally deferred". Documented in the UAT log + in this SUMMARY's frontmatter so future readers know what it means.
- **GAP-6 verification is structural, not empirical** — `removed{}` block syntax + addressing validates via `terraform validate`; `terragrunt plan` against post-recovery regional state (no v1.0.0 resources present) shows zero destroys of shared SES resources. The in-anger scenario (v1.0.0 resources actually in regional state for the removed blocks to suppress) was NOT reconstructed because no Phase 82.x snapshot was available and the operator chose not to construct one. The load-bearing distinction is documented in the UAT log so future readers know what "verified" means in this context.
- **All 11 requirement IDs marked Complete despite partial empirical scope** — the SHIPPED CODE closes the gaps; the deferred items are verification-of-the-fix, not the fix itself. The empirical gap is captured in Phase 84.2's roadmap evolution entry rather than holding Phase 84.1 open indefinitely (which would have blocked Phase 84.2 from starting and created a circular dependency since Phase 84.2's `km init --plan` flag is the very tool that would make the empirical verification trivial).
- **Helper script committed as chore commit, not docs commit** — `uat-gap6-check.sh` is operator-runnable bash, not docs. Standalone (no Makefile wiring) because Phase 84.2 will replace it with the `km init --plan` flag's destroy-class gate. Committed by continuation agent on operator's behalf since the script existed unstaged in the working tree per operator's UAT-response instructions.
- **Operator anomaly during Step 2 prep folded into Phase 84.2** — `km bootstrap --shared-ses --dry-run=true` reported `creating` for shared rule set + domain identity on an account that already has them. This may indicate GAP-3's `FoundationStateReader` wiring isn't activating in the bootstrap dry-run code path. Investigation deferred to Phase 84.2 rather than escalated as a Phase 84.1 regression because (a) the structural state-precedence wiring landed correctly per Plan 84.1-04's Self-Check, and (b) the dry-run anomaly is a separate investigation surface from the in-apply behavior the wiring was designed to fix.
