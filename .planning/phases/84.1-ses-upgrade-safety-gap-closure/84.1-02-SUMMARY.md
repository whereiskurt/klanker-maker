**Bounded terragrunt runner with cmd.Cancel/WaitDelay + quiet-mode heartbeat, per-module Apply timeout in km init's RunInitWithRunner (via ModuleTimeoutFunc), and bootstrap-side BootstrapApplyTimeout — closes GAP-4 silent-hang + GAP-5 unbounded-Apply + plan-checker rev 1 H6 bootstrap mirror**

## Accomplishments

- **GAP-4 closed** (silent multi-minute hang from 84-10-UAT.md lines 53-72): the runner now emits a heartbeat in quiet mode (`  ... still running (Xs elapsed, pid=N)`) at `Runner.ProgressInterval` cadence (15s default), so a wedged terragrunt surfaces as visible progress instead of an indefinite silent stall. The PID in the heartbeat lets operators `ps` / `kill -9` the orphan if recovery is needed.
- **GAP-5 closed**: every regional module Apply in `RunInitWithRunner` is now wrapped in `context.WithTimeout(ctx, ModuleTimeoutFunc(mod.name))`. Defaults are conservative upper bounds (10min for network + ses-shared-rule-set; 5min for SES + lambda modules; 3min for everything else) — well above any normal apply duration. The ses-module Reconfigure call also gets a 2-min bound via `reconfigureTimeout`.
- **Plan-checker rev 1 H6 closed** (bootstrap mirror): `km bootstrap --shared-ses` now bounds `defaultApplyTerragrunt` (Reconfigure + Apply) by `BootstrapApplyTimeout` (10min default). Without this, `km bootstrap` had the same indefinite-hang surface as `km init` did before this plan.
- **Actionable timeout errors**: every timeout path produces an error string that (a) names which module wedged, (b) points operators at OPERATOR-GUIDE.md § Phase 84.1 state-digest recovery (runbook ships in Plan 84.1-05), and (c) wraps `context.DeadlineExceeded` so callers can `errors.Is`-detect timeouts vs real terragrunt failures.
- **Ctrl-C feels responsive**: `cmd.Cancel = func() error { return cmd.Process.Signal(os.Interrupt) }` plus `cmd.WaitDelay = 5 * time.Second` gives terragrunt 5s to release state locks gracefully before SIGKILL. Per plan-checker rev 1 L15, `os.Interrupt` (not `syscall.SIGTERM`) keeps the runner Windows-portable.

## Decisions Made

- **Heartbeat at the runner layer, not the call-site layer**: putting the heartbeat goroutine inside `runBounded` means every km command that wraps terragrunt (`km init`, `km bootstrap`, `km destroy`, `km cluster add`, `km slack init`) inherits the heartbeat automatically. Alternative (per-caller wiring) would have duplicated the goroutine code in 6+ places.
- **`ModuleTimeoutFunc` is a package-level var, not a const-fronted function**: gives tests a clean sub-second override without changing the public surface or introducing a flag. Same pattern as the existing `ApplyTerragruntFunc` (in `bootstrap.go`) and `InitSESPreflight` (in `init.go`) test seams.
- **Single `BootstrapApplyTimeout` bounds the WHOLE Reconfigure+Apply pair in `defaultApplyTerragrunt`**: one budget for one module-apply, matching `RunInitWithRunner`'s per-module-Apply treatment in Task 2. Alternative (separate bound per call) would have allowed 20 minutes of wall-clock time (10 reconfigure + 10 apply), which is not what an operator would expect from a "10-minute bootstrap apply timeout".
- **`cmd.Cancel` uses `os.Interrupt`, NOT `syscall.SIGTERM`** (plan-checker rev 1 L15): `syscall.SIGTERM` is undefined on Windows builds, which would break cross-compile. `os.Interrupt` maps to SIGINT on Unix and CTRL_BREAK_EVENT on Windows — both achieve the "give the child a chance to clean up before SIGKILL" semantics we want.
- **Stub package-level vars in RED so tests compile**: the alternative ("RED = tests don't compile because the symbol doesn't exist") is technically a valid RED but produces noisier git history. Adding the minimal field/var declaration in the RED commit (with a TODO docstring) keeps the diff focused on the implementation that lands in GREEN.
