### Human Verification Required

Four behaviors require live AWS infrastructure. These should be performed by the operator after running `make build` and `km init --sidecars`:

**1. BDM collision avoidance on EC2 launch**

**Test:** Bake an AMI from a sandbox that had `additionalVolume` set. Then `km create` a new sandbox pointing at that AMI ID with the same `additionalVolume` profile.
**Expected:** Terraform plan shows `device_name = "/dev/sdg"` (not "/dev/sdf") on `aws_volume_attachment.additional`; `km create` succeeds without EC2 `InvalidBlockDeviceMapping` error.
**Why human:** Requires a real baked AMI with /dev/sdf occupied in its BDMs and a live EC2 launch — the compiler-side logic is verified; the EC2 API acceptance requires real infrastructure.

**2. Login shell not deadlocked during FIFO setup race**

**Test:** `km create profiles/dc34.ami.yaml --remote`; immediately run `km shell <sandbox-id>` as soon as the instance reaches running state (before km-audit-log has started).
**Expected:** PROMPT_COMMAND completes in under 500ms; the shell is responsive.
**Why human:** The 100ms timeout bound is verified by code inspection; confirming the race condition is eliminated requires real SSM session timing.

**3. km-audit-log recovers after /run/km wipe**

**Test:** SSM into a running sandbox. Run `sudo rm -rf /run/km && sudo systemctl restart km-audit-log`. Watch `journalctl -u km-audit-log -f`.
**Expected:** Retry messages appear (up to ~30s), then "reading from audit pipe" log line; audit events flow after re-login.
**Why human:** Requires systemd + real FIFO on a running EC2 instance.

**4. Sidecars carry new sandbox identity after bake-relaunch**

**Test:** Bake AMI from sandbox A. Create sandbox B from that AMI ID. On B, run `systemctl show km-audit-log km-dns-proxy km-http-proxy km-tracing -p Environment`.
**Expected:** All four units show `SANDBOX_ID=<B>`, not sandbox A's ID.
**Why human:** Requires two real sandboxes and a bake-relaunch cycle.
