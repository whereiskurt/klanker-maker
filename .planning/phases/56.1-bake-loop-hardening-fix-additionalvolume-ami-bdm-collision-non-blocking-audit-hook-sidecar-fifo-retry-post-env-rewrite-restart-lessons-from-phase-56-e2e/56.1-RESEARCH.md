# Phase 56.1: Bake-Loop Hardening — Research

**Researched:** 2026-04-26
**Domain:** EC2/Terraform BDM collision, Linux FIFO boot ordering, systemd unit sequencing
**Confidence:** HIGH (all findings sourced from direct code inspection)

## Summary

Phase 56.1 fixes four concrete runtime bugs discovered when running an AMI-backed sandbox end-to-end after Phase 56 completed. Every bug has a clear root cause traceable to a specific file and line range in the current codebase. No new AWS APIs or Go libraries are required — all fixes are targeted edits to existing files.

**Bug 1 (BDM collision):** `infra/modules/ec2spot/v1.0.0/main.tf` hardcodes `/dev/sdf` as the `aws_volume_attachment.additional` device name (line 582). When the sandboxed AMI already occupies `/dev/sdf` in its own block device mappings (as a baked AMI of a sandbox that had an additionalVolume does), EC2 rejects the launch. The fix requires fetching the baked AMI's BDMs via `DescribeImages` at compiler time, detecting the collision, and passing an alternative device name through to Terraform.

**Bug 2 (audit-hook deadlock):** `pkg/compiler/userdata.go` writes the `_km_audit()` PROMPT_COMMAND hook (line 487) with a bare redirect `> /run/km/audit-pipe`. When `km-audit-log` hasn't created the FIFO yet (a common race on first login), this write blocks indefinitely because no reader has opened the pipe. The 2>/dev/null guard only suppresses stderr, not the blocking open. The fix: change the redirect to use `timeout 0.1` or add `> /dev/null 2>&1` as a non-blocking path fallback.

**Bug 3 (sidecar FIFO-retry):** `sidecars/audit-log/cmd/main.go` attempts to open the FIFO once (line 178: `os.OpenFile(pipePath, os.O_RDWR, 0)`). If it fails, it logs an error and falls back to stdin — but stdin is empty on EC2. On a baked AMI the boot sequence may rewrite `/run/km/` state after the sidecar starts. The fix adds a startup retry loop (mkdir -p, mkfifo, open with backoff) in main.go or equivalently in the systemd unit's `ExecStartPre=`.

**Bug 4 (post-env-rewrite restart):** `pkg/compiler/userdata.go` writes `/etc/profile.d/km-identity.sh` and `/etc/profile.d/km-profile-env.sh` early in cloud-init (sections 2.8-2.9, lines 147-215), but the systemd services start *after* `daemon-reload` at line 523 — which happens *after* the env files are written. So for the **first boot** of a stock AMI this is fine. On a **baked AMI resumed from snapshot**, systemd units may have been running during the bake, and on re-launch they pick up stale environments from the AMI's `/etc/environment` or from the previous unit state. The fix: add `EnvironmentFile=-/etc/profile.d/km-identity.sh` to each sidecar's systemd unit, or issue an explicit `systemctl restart` of all km-* sidecars after the env files are written.

**Primary recommendation:** Fix all four bugs in a single phase with 3 plans: Plan 01 targets the BDM collision (Go compiler + Terraform), Plan 02 targets the audit-hook deadlock + FIFO-retry (userdata.go + audit-log main.go + systemd units), Plan 03 targets the post-env-rewrite restart (systemd EnvironmentFile= or userdata restart sequence). Each plan needs `make build` + `km init --sidecars` to deploy.

## Architecture Patterns

### Recommended Fix Patterns

#### Fix 1: BDM Collision Detection in the Compiler

The compiler must detect whether the baked AMI's BDMs collide with the additionalVolume device name. The device name `/dev/sdf` is currently hardcoded in `main.tf`. There are two viable approaches:

**Approach A (compile-time inspection, PREFERRED):** Add a `GetAMIBDMDevices(ctx, client, amiID) ([]string, error)` helper to `pkg/aws/ec2_ami.go` that calls `DescribeImages` and returns the `DeviceName` strings from all `BlockDeviceMappings`. In `pkg/compiler/service_hcl.go`'s `generateEC2ServiceHCL`, when both `isRawAMIID(p.Spec.Runtime.AMI)` is true AND `additionalVolume` is set, call this helper and pick the next non-colliding device name from the ordered sequence: `/dev/sdf`, `/dev/sdg`, `/dev/sdh`, …, `/dev/sdp`. Pass the selected device name as a new HCL variable `additional_volume_device_name` to the module.

**Approach B (Terraform-side, fallback):** Add a `data.aws_ami` lookup in `main.tf` when `var.ami_id != ""` to read the BDMs, then compute the device name locally. This is harder to test and adds Terraform complexity.

Approach A is preferred because it keeps logic in Go (testable), avoids an extra data source in Terraform, and aligns with how `isRawAMIID` was already introduced.

**New variable in `variables.tf`:**
```hcl
variable "additional_volume_device_name" {
  type        = string
  description = "Device name for the additional EBS volume. Defaults to /dev/sdf."
  default     = "/dev/sdf"
}
```

**Patch to `main.tf` line 582:**
```hcl
resource "aws_volume_attachment" "additional" {
  count       = var.additional_volume_size_gb > 0 ? 1 : 0
  device_name = var.additional_volume_device_name   # was: "/dev/sdf"
  ...
}
```

**HCL template addition in `ec2ServiceHCLTemplate`:**
```
    additional_volume_device_name = "{{ .AdditionalVolumeDeviceName }}"
```

**EC2 device name rotation order (AL2023/xvd* namespace):**
- `/dev/sdf` → kernel sees `/dev/xvdf`
- `/dev/sdg` → kernel sees `/dev/xvdg`
- Through `/dev/sdp` (AWS maximum: sdf-sdp for additional volumes)
- Boot volume is always `/dev/xvda` (root) so sda/sdb reserved; NVMe instances expose `/dev/nvme[1-N]n1` anyway — userdata.go section 2.6 already handles the NVMe probe (lines 80-91)

**GetAMIBDMDevices helper signature:**
```go
// pkg/aws/ec2_ami.go — add to EC2AMIAPI interface
// DescribeImages already in interface; no new interface methods needed.

// AMIBDMDeviceNames fetches the block device mapping DeviceName strings for an AMI.
// Returns nil, nil if the AMI has no BDMs (e.g. instance-store or brand-new ID).
// Used by the compiler to detect /dev/sdf collision before emitting additionalVolume HCL.
func AMIBDMDeviceNames(ctx context.Context, client EC2AMIAPI, amiID string) ([]string, error)
```

The function calls `DescribeImages` with `ImageIds: []string{amiID}` and extracts `BlockDeviceMappings[*].DeviceName`. `DescribeImages` is already in `EC2AMIAPI`. The compiler can call this at HCL generation time — but it needs an EC2 client passed in. The current `generateEC2ServiceHCL` signature does not accept an EC2 client; the caller (`Compile` in `compiler.go`) must pass one.

**Key constraint:** The compiler only needs BDM inspection when `isRawAMIID(p.Spec.Runtime.AMI) == true` AND `p.Spec.Runtime.AdditionalVolume != nil`. This covers the exact failure case.

**Alternative (no AWS call at compile time):** The compiler emits `/dev/sdf` as today, but adds a safeguard in userdata.go section 2.6: if `/dev/sdf` is already mounted (exists and is block device) when the additional volume attach code runs, try `/dev/sdg`, etc. This is a userdata-only fix that avoids changing the compiler/Terraform, but it doesn't prevent EC2 from refusing to launch when BDMs collide at the Terraform `aws_volume_attachment` level. The EC2 launch failure happens before any userdata runs — so the userdata workaround alone is not sufficient. The Terraform layer must receive the correct device name.

**Recommended path:** Both the Terraform variable change AND the compiler-time detection. Use a simple fallback in `generateEC2ServiceHCL`: if no EC2 client is available (nil), default to `/dev/sdf` with a warning log. This keeps the function testable without AWS.

#### Fix 2: Non-Blocking Audit Hook (userdata.go)

**Current code (userdata.go line 487):**
```bash
> /run/km/audit-pipe 2>/dev/null
```

**Problem:** A bare `>` on a named FIFO (O_WRONLY) blocks until a reader opens the other end. `2>/dev/null` redirects stderr but the blocking happens at the `open()` syscall for stdout, before any shell code runs.

**Fix option A (preferred — timeout wrapper):**
```bash
printf '...\n' | timeout 0.1 tee /run/km/audit-pipe > /dev/null 2>/dev/null || true
```
`timeout 0.1` kills the write after 100ms if no reader; `|| true` prevents PROMPT_COMMAND failures.

**Fix option B (O_NONBLOCK via bash):**
```bash
{ printf '...\n' > /run/km/audit-pipe; } 2>/dev/null &
disown
```
This forks the write into a background subshell. If the pipe is unread the subshell hangs orphaned but the user's shell continues. Messier — leaves zombie processes.

**Fix option C (check pipe before writing):**
```bash
[ -p /run/km/audit-pipe ] && printf '...\n' > /run/km/audit-pipe 2>/dev/null || true
```
The `[ -p /run/km/audit-pipe ]` check passes as long as the FIFO exists, but the write still blocks if no reader. Insufficient.

**Fix option D (tee with no-hang open):**
Open the FIFO with `exec 3<>/run/km/audit-pipe` (O_RDWR prevents blocking), then `printf > /run/km/audit-pipe`. Because the FIFO is opened O_RDWR in main.go line 178, this is also how the sidecar opens it — avoiding the blocking-writer-without-reader problem at both ends.

**Recommendation: Fix option A (timeout 0.1)** — it is the most readable and self-documenting. The heartbeat writer in the same script (line 514) needs the same treatment.

#### Fix 3: Sidecar FIFO-Retry (audit-log main.go)

**Current code (main.go lines 177-185):**
```go
if pipePath := envOr("KM_AUDIT_PIPE", ""); pipePath != "" {
    f, openErr := os.OpenFile(pipePath, os.O_RDWR, 0)
    if openErr != nil {
        log.Error().Err(openErr).Str("pipe", pipePath).Msg("audit-log: failed to open audit pipe")
    } else {
        inputReader = f
    }
}
```

**Problem:** Single attempt. If `/run/km/audit-pipe` doesn't exist (reboot, baked AMI first boot, systemd ordering race), the sidecar silently falls through to `inputReader = os.Stdin`. On EC2, stdin is `/dev/null` — the sidecar processes zero events and the audit log is silent.

**Fix (retry loop with backoff):**
```go
if pipePath := envOr("KM_AUDIT_PIPE", ""); pipePath != "" {
    var pipeFile *os.File
    for attempt := 1; attempt <= 10; attempt++ {
        // Ensure /run/km exists
        if mkErr := os.MkdirAll(filepath.Dir(pipePath), 0755); mkErr != nil {
            log.Warn().Err(mkErr).Msg("audit-log: cannot create pipe dir, retrying")
        }
        // Re-create FIFO if missing
        if _, statErr := os.Stat(pipePath); os.IsNotExist(statErr) {
            _ = syscall.Mkfifo(pipePath, 0666)
            _ = os.Chown(pipePath, os.Getuid(), os.Getgid())
        }
        f, openErr := os.OpenFile(pipePath, os.O_RDWR, 0)
        if openErr == nil {
            pipeFile = f
            break
        }
        log.Warn().Int("attempt", attempt).Err(openErr).Msg("audit-log: retrying pipe open")
        time.Sleep(time.Duration(attempt) * 500 * time.Millisecond)
    }
    if pipeFile != nil {
        inputReader = pipeFile
        log.Info().Str("pipe", pipePath).Msg("audit-log: reading from audit pipe")
    } else {
        log.Error().Str("pipe", pipePath).Msg("audit-log: giving up on pipe after retries, reading stdin")
    }
}
```

**Systemd complement:** Also add `Restart=on-failure` to `km-audit-log.service` in userdata.go (currently only `Restart=always` — confirm). The systemd unit is written at lines 430-453 of userdata.go.

#### Fix 4: Post-Env-Rewrite Restart

**Root cause:** On first boot from a baked AMI, the sidecar systemd units inherit the environment from the time of the bake — not the environment after the new sandbox's cloud-init rewrites `/etc/profile.d/km-identity.sh`. The systemd units do NOT have `EnvironmentFile=` set; their `Environment=` lines are hardcoded at unit-write time from the compiler template (see userdata.go lines 385-416, 430-453).

**Key observation:** The systemd unit `Environment=` lines ARE already populated with the correct `SANDBOX_ID`, `CW_LOG_GROUP`, etc. at template render time by the Go compiler. This is correct for a fresh provisioning. The problem only arises when a **baked AMI** is used: the AMI was created with the old sandbox's identity baked into the unit files under `/etc/systemd/system/km-*.service`. When the AMI is launched as a NEW sandbox with a DIFFERENT sandbox ID, the unit files on disk still contain the OLD sandbox ID. The `cloud-init` script (userdata) runs and rewrites `km-identity.sh`, but systemd has already loaded the stale unit from disk.

**Fix (preferred — restart sidecars after env write, in userdata.go):**
After the `systemctl daemon-reload` call at line 523, add:
```bash
# Post-env-rewrite restart: sidecars may have started with stale identity from baked AMI.
# Restart to pick up new SANDBOX_ID and CW_LOG_GROUP written in sections 2.8-2.9.
systemctl restart km-dns-proxy km-http-proxy km-audit-log km-tracing 2>/dev/null || true
```

**Fix (alternative — EnvironmentFile= in unit templates):**
Change the systemd unit templates in userdata.go to read environment at runtime:
```
EnvironmentFile=-/etc/profile.d/km-identity.sh
```
The `-` prefix makes the directive optional (no error if missing). Then remove the hardcoded `Environment=SANDBOX_ID=` line. This is cleaner for the baked-AMI case but changes the unit file structure non-trivially.

**Recommendation:** The `systemctl restart` after daemon-reload is simpler and lower risk. EnvironmentFile= is the more principled long-term fix but requires removing the hardcoded Environment= lines and ensuring the profile.d files set the exact var names the sidecars expect (they do — `SANDBOX_ID` appears in both the unit and km-identity.sh).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Device name collision detection | Custom NVMe/xvd translation table | Use AWS canonical device name list (sdf-sdp) | AWS documents valid additional volume names as /dev/sd[f-p] |
| Non-blocking FIFO write | O_NONBLOCK fcntl() in bash | `timeout 0.1 tee /dev/null < pipe` pattern or timeout wrapper | Standard POSIX shell pattern; no bash extensions needed |
| Retry-with-backoff in Go | Custom retry framework | Simple for-loop with `time.Sleep` | stdlib only; no external retry package needed |
| Systemd ordering of cloud-init vs sidecars | Custom systemd generator | `EnvironmentFile=` + restart-after pattern | systemd native feature; no ordering changes needed |

## Common Pitfalls

### Pitfall 1: Operator-declared device name vs AMI-collision skip

**What goes wrong:** Fixing the BDM collision by silently skipping the additionalVolume (i.e., emitting `additional_volume_size_gb = 0`) is tempting but dangerous. The operator explicitly declared an additionalVolume — dropping it silently causes confusing "my data volume disappeared" bugs.

**How to avoid:** Always emit the additionalVolume. Detect collision and pick the next non-colliding device name instead of skipping. Log a `[km-bootstrap] WARNING: /dev/sdf collides with AMI BDM, remapping to /dev/sdg` message.

**Warning signs:** Any code path that sets `additionalVolume.Size = 0` or omits `additional_volume_size_gb` in the HCL when a raw AMI ID is present.

### Pitfall 2: `generateEC2ServiceHCL` requires an EC2 client for BDM lookup

**What goes wrong:** `generateEC2ServiceHCL` is called in tests without real AWS access. Adding an EC2 client parameter directly will break all existing tests in `ec2_storage_test.go`.

**How to avoid:** Pass the BDM device names as a pre-computed parameter rather than an EC2 client. The caller (`pkg/compiler/compiler.go`) can resolve BDMs before calling `generateEC2ServiceHCL`. The function signature stays testable: tests pass `nil` or empty slice for BDM names, which defaults to `/dev/sdf`.

**Warning signs:** Any test that requires mocking an EC2 client inside `generateEC2ServiceHCL` — this means the design leaked AWS dependencies into the pure HCL-generation layer.

### Pitfall 3: FIFO mkfifo permissions on baked AMI

**What goes wrong:** The baked AMI may have `/run/km/audit-pipe` as a stale FIFO from the previous sandbox session (if the bake happened during an active session). On re-launch, `stat` sees the path exists but the file system is read-only or the old FIFO is in an inconsistent state.

**How to avoid:** In the FIFO-retry loop, always remove and re-create the FIFO rather than reusing the existing one: `os.Remove(pipePath)` followed by `syscall.Mkfifo(pipePath, 0666)`. The userdata.go section that runs `mkdir -p /run/km && mkfifo /run/km/audit-pipe` (lines 420-422) also runs on re-launch since it's in cloud-init, so the FIFO should be fresh — but the sidecar retry defensively handles the case where it races ahead.

**Warning signs:** FIFO already exists but open returns `ENXIO` (no readers/writers on a stale one).

### Pitfall 4: systemctl restart during cloud-init races with sidecar startup

**What goes wrong:** Calling `systemctl restart km-audit-log` in the userdata script before the sidecar binary has been downloaded (section 5 downloads from S3) restarts a unit that hasn't started yet — or worse, restarts before the new FIFO is created.

**How to avoid:** Place the `systemctl restart` call AFTER section 5 (sidecar download) and AFTER the FIFO creation (lines 420-422 in userdata.go), not immediately after daemon-reload.

**Warning signs:** `systemctl status km-audit-log` shows "activating" at the time restart is called.

### Pitfall 5: `timeout` not always available in restricted profiles

**What goes wrong:** `timeout` is a coreutils command. Some minimal container images don't have it. On AL2023 and Ubuntu it is always present. The sandbox EC2 environment is a full Linux instance so this is not a concern.

## Code Examples

### Selecting next non-colliding device name

```go
// Source: project code convention (pkg/compiler patterns)
// deviceNameAfterBDMs returns the first device name from candidateDevices
// that does not appear in amiDevices.
// candidateDevices is the ordered sequence of valid AWS additional volume names.
// Returns "" if all candidates are occupied (should never happen in practice: 10 slots).
var additionalVolumeDeviceCandidates = []string{
    "/dev/sdf", "/dev/sdg", "/dev/sdh", "/dev/sdi",
    "/dev/sdj", "/dev/sdk", "/dev/sdl", "/dev/sdm",
    "/dev/sdn", "/dev/sdo", "/dev/sdp",
}

func pickAdditionalVolumeDevice(amiDevices []string) string {
    occupied := make(map[string]bool, len(amiDevices))
    for _, d := range amiDevices {
        occupied[d] = true
    }
    for _, candidate := range additionalVolumeDeviceCandidates {
        if !occupied[candidate] {
            return candidate
        }
    }
    return "/dev/sdf" // unreachable in practice
}
```

### AMI BDM device name fetch (new helper in pkg/aws/ec2_ami.go)

```go
// Source: extends existing pkg/aws/ec2_ami.go pattern
// AMIBDMDeviceNames returns the device names from all BlockDeviceMappings for an AMI.
// Returns nil,nil when the AMI has no BDMs or ami_id is empty.
// Returns nil,err when DescribeImages fails.
func AMIBDMDeviceNames(ctx context.Context, client EC2AMIAPI, amiID string) ([]string, error) {
    if amiID == "" {
        return nil, nil
    }
    out, err := client.DescribeImages(ctx, &ec2.DescribeImagesInput{
        ImageIds: []string{amiID},
    })
    if err != nil {
        return nil, fmt.Errorf("describe images %s for BDM: %w", amiID, err)
    }
    if len(out.Images) == 0 {
        return nil, nil
    }
    var devices []string
    for _, bdm := range out.Images[0].BlockDeviceMappings {
        if bdm.DeviceName != nil {
            devices = append(devices, *bdm.DeviceName)
        }
    }
    return devices, nil
}
```

### Non-blocking FIFO write in bash (userdata.go hook)

```bash
# Source: project userdata.go _km_audit pattern — fix for Bug 2
_km_audit() {
  local cmd
  cmd=$(HISTTIMEFORMAT= history 1 | sed 's/^ *[0-9]* *//')
  [ -z "$cmd" ] && return 0
  cmd=$(printf '%s' "$cmd" | sed 's/\\/\\\\/g; s/"/\\"/g; s/	/ /g' | head -c 500)
  printf '{"timestamp":"%s","sandbox_id":"%s","event_type":"command","source":"shell","detail":{"command":"%s","user":"%s"}}\n' \
    "$(date -u +%Y-%m-%dT%H:%M:%SZ)" "{{ .SandboxID }}" "$cmd" "$(whoami)" \
    | timeout 0.1 tee /run/km/audit-pipe > /dev/null 2>/dev/null || true
}
```

## Open Questions

1. **Does the compiler have access to AWS credentials at `km create` time?**
   - What we know: `km create` calls `terragrunt apply`. AWS credentials are present.
   - What's unclear: Does `pkg/compiler/compiler.go` have the region + credentials context needed to call `ec2.DescribeImages` before calling `generateEC2ServiceHCL`?
   - Recommendation: Look at `compiler.go` Compile function signature. If it takes a `context.Context` and `config.Config`, the EC2 client can be constructed there using existing `pkg/aws` patterns. If not, the compiler's `Compile` function needs a new optional EC2 factory parameter.

2. **Can the baked AMI have `/dev/xvdf` (NVMe alias) rather than `/dev/sdf` in its BDMs?**
   - What we know: AWS stores the requested device name in BDMs, not the kernel-visible NVMe name. So the BDM will show `/dev/sdf` even on NVMe instances.
   - What's unclear: Whether older AMIs or user-created AMIs might list `/dev/xvdf` directly.
   - Recommendation: Normalize both `/dev/sdf` and `/dev/xvdf` as equivalent in `pickAdditionalVolumeDevice` by treating xvd* as sd* mappings in the occupied set.

3. **Should the FIFO-retry loop be in main.go or as an ExecStartPre= in the systemd unit?**
   - What we know: An `ExecStartPre=` script would run as km-sidecar user and could `mkfifo` before the Go binary starts.
   - What's unclear: The km-sidecar user was created in the same userdata; on AMI re-launch it persists. The FIFO is in tmpfs (`/run/`) which is reset on each boot. `ExecStartPre=` is a viable alternative.
   - Recommendation: Implement retry in main.go (testable in unit tests, no new bash scripts). Add a comment referencing the alternative in the systemd unit template.
