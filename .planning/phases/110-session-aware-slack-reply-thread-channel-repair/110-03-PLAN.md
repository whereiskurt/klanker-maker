<objective>
Add the sandbox-side `km-slack reply` subcommand. It resolves the destination
thread via a first-hit-wins chain and then reuses the existing signed post path.
The sandbox NEVER reads DDB — session-id resolution goes through the Plan 02
bridge `lookup-thread` action.

Resolution chain (first hit wins):
1. `--thread <ts>` (requires `--channel`)
2. `$KM_SLACK_THREAD_TS` (set by the inbound poller)
3. session id — `--session <id>` or auto-detected newest session file
   (Claude: `~/.claude/projects/**/<id>.jsonl` by mtime; Codex: `~/.codex` best-effort,
   branched on `$KM_AGENT`) → bridge `lookup-thread` → (channel_id, thread_ts)
4. fallback: top-level post to `$KM_SLACK_CHANNEL_ID`

`--body /file` and `--render plain|mrkdwn|blocks` are honored (same as `post`).

Purpose: Let a resumed agent reply into its own Slack thread.
Output: reply subcommand wired into the dispatch table.
</objective>
