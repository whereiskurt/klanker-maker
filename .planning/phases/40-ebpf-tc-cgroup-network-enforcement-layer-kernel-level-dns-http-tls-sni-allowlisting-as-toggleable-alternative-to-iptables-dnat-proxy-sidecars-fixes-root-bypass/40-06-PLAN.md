<objective>
Create the ring buffer audit consumer, the `km ebpf-attach` CLI command, wire eBPF cleanup into `km destroy`, and create root bypass verification test scenarios.

Purpose: The audit consumer converts BPF ring buffer deny events into structured JSON logs for the existing audit pipeline. The `km ebpf-attach` command is the entry point invoked by EC2 user-data to orchestrate the full eBPF setup: create enforcer, start DNS resolver, begin audit logging, and run until signaled to stop. The destroy command integration ensures BPF artifacts are cleaned up when a sandbox is destroyed. Root bypass verification ensures the security promise holds: even with root inside the sandbox, BPF enforcement cannot be circumvented.

Output: `pkg/ebpf/audit/` consumer, `km ebpf-attach` command, destroy.go eBPF cleanup, integration test scaffolding.
</objective>

**pkg/ebpf/audit/audit.go** (`//go:build linux`):

```go
type Consumer struct {
    reader    *ringbuf.Reader
    sandboxID string
    logger    zerolog.Logger
}
```

`NewConsumer(eventsMap *ebpf.Map, sandboxID string, logger zerolog.Logger) (*Consumer, error)`:
1. Create `ringbuf.NewReader(eventsMap)`
2. Return Consumer

`(c *Consumer) Run(ctx context.Context) error`:
1. Loop reading from ring buffer (blocking `c.reader.Read()`)
2. On `ringbuf.ErrClosed` or ctx.Done: return nil
3. Deserialize `record.RawSample` into `bpfEvent` struct using `binary.Read` with `binary.LittleEndian`
4. Convert event to structured JSON log:
   ```json
   {
     "timestamp": "2026-03-31T12:00:00Z",
     "sandbox_id": "sb-abcd1234",
     "pid": 12345,
     "src_ip": "10.0.1.5",
     "dst_ip": "93.184.216.34",
     "dst_port": 443,
     "action": "deny",
     "layer": "connect4",
     "comm": "curl",
     "event_type": "ebpf_network_deny"
   }
   ```
5. Log via zerolog at Warn level for denies, Debug level for allows/redirects
6. IP addresses converted from uint32 network byte order to dotted notation via `net.IP`

`(c *Consumer) Close() error` — close ring buffer reader (unblocks Read())

Helper functions:
- `actionString(action uint8) string` — "deny", "allow", "redirect"
- `layerString(layer uint8) string` — "connect4", "sendmsg4", "egress_skb", "sockops"
- `uint32ToIP(ip uint32) net.IP` — convert network byte order uint32 to net.IP

**pkg/ebpf/audit/audit_test.go**:
- `TestActionString` — verify all action constants map to strings
- `TestLayerString` — verify all layer constants map to strings
- `TestUint32ToIP` — verify network byte order conversion

```go
var ebpfAttachCmd = &cobra.Command{
    Use:   "ebpf-attach",
    Short: "Attach eBPF network enforcement to sandbox cgroup",
    Long:  "Loads BPF programs, attaches to sandbox cgroup, starts DNS resolver, and runs audit consumer. Invoked by EC2 user-data bootstrap.",
    RunE:  runEbpfAttach,
    Hidden: true,  // internal command, not user-facing
}
```

Flags:
- `--sandbox-id` (required)
- `--dns-port` (default 5353)
- `--http-proxy-port` (default 3128)
- `--firewall-mode` (default "block"; values: log, allow, block)
- `--allowed-dns` (comma-separated DNS suffixes)
- `--allowed-hosts` (comma-separated hosts for L7 proxy)
- `--proxy-hosts` (comma-separated hosts needing L7 proxy)
- `--cgroup` (cgroup path, default auto-detected)

`runEbpfAttach(cmd *cobra.Command, args []string)`:
1. Parse flags
2. Create Enforcer via `ebpf.NewEnforcer(cfg)` with:
   - FirewallMode from --firewall-mode flag
   - ProxyPID = uint32(os.Getpid())
   - MITMProxyAddr = 127.0.0.1 in network byte order
3. Pre-populate initial CIDR allowlist:
   - 169.254.169.254/32 (IMDS)
   - VPC CIDR (10.0.0.0/8 or from IMDS)
4. Start DNS resolver via `resolver.NewResolver()` in goroutine
5. Start audit consumer via `audit.NewConsumer()` in goroutine
6. If --proxy-hosts specified, mark resolved IPs for L7 proxy
7. Wait for SIGTERM/SIGINT
8. On signal: close audit consumer, stop resolver, close enforcer (unpin handled by km destroy)

Register `ebpfAttachCmd` as subcommand of root command in the existing cmd registration pattern. Follow the same pattern as other internal/app/cmd/*.go files.

**internal/app/cmd/destroy.go** — Add eBPF cleanup to the sandbox destroy path:

1. Add import for the ebpf pin package:
   ```go
   ebpfpkg "github.com/whereiskurt/klankrmkr/pkg/ebpf"
   ```

   Note: Use a build-tag-aware approach. Since `pkg/ebpf/pin.go` has `//go:build linux`, the import and call must be conditional. Create a thin helper file `internal/app/cmd/destroy_ebpf_linux.go` with `//go:build linux` that calls the cleanup, and `internal/app/cmd/destroy_ebpf_other.go` with `//go:build !linux` that is a no-op. Alternatively, if the destroy command only runs on the host (not inside sandboxes), and the host is always Linux (EC2), then a direct import with `//go:build linux` on the helper is sufficient.

   Preferred approach: Add a helper function in destroy.go that is platform-safe:
   ```go
   // cleanupEBPF removes any pinned BPF programs and maps for the sandbox.
   // This is a no-op for proxy-mode sandboxes that never had eBPF attached.
   // Implemented in destroy_ebpf_linux.go / destroy_ebpf_other.go.
   ```

   **destroy_ebpf_linux.go** (`//go:build linux`):
   ```go
   package cmd

   import ebpfpkg "github.com/whereiskurt/klankrmkr/pkg/ebpf"

   func cleanupEBPF(sandboxID string, logger zerolog.Logger) {
       if !ebpfpkg.IsPinned(sandboxID) {
           logger.Debug().Str("sandbox_id", sandboxID).Msg("No pinned eBPF programs found, skipping cleanup")
           return
       }
       logger.Info().Str("sandbox_id", sandboxID).Msg("Cleaning up pinned eBPF programs and maps")
       if err := ebpfpkg.Cleanup(sandboxID); err != nil {
           logger.Warn().Err(err).Str("sandbox_id", sandboxID).Msg("eBPF cleanup failed (non-fatal, instance will be terminated)")
       } else {
           logger.Info().Str("sandbox_id", sandboxID).Msg("eBPF cleanup complete")
       }
   }
   ```

   **destroy_ebpf_other.go** (`//go:build !linux`):
   ```go
   package cmd

   import "github.com/rs/zerolog"

   func cleanupEBPF(sandboxID string, logger zerolog.Logger) {
       // eBPF is Linux-only; no cleanup needed on other platforms
   }
   ```

2. In the destroy command's RunE function, call `cleanupEBPF(sandboxID, logger)` EARLY in the teardown sequence — before EC2 instance termination, before Terragrunt destroy. The BPF cleanup needs the bpffs filesystem which is on the instance, but since `km destroy` runs from the operator's machine (not the instance), the cleanup call uses SSM or is a best-effort local check. Actually, since BPF programs are pinned to bpffs ON the EC2 instance, and `km destroy` runs from the operator's laptop, the cleanup must happen via SSM RunCommand or during instance shutdown. The correct approach:

   - Check if sandboxID has pinned BPF programs by calling `IsPinned()` which checks `/sys/fs/bpf/km/{sandboxID}/`. This only works if km destroy runs ON the instance.
   - Since km destroy runs from the operator's machine, the eBPF cleanup actually happens automatically when the EC2 instance is terminated (bpffs is an in-memory filesystem that disappears with the instance).
   - However, for the "both" mode where proxy sidecars coexist, and for `km stop`/`km resume` scenarios, we should still wire in the cleanup.

   **Revised approach:** Add `cleanupEBPF()` call in the destroy path. On non-Linux (operator laptop), it's a no-op. On Linux (if km destroy is ever run ON the instance, e.g., during automated cleanup), it does the real cleanup. The primary cleanup mechanism for remote destroy is instance termination which removes bpffs automatically.

   Add the call right after the lock check and before the main teardown begins:
   ```go
   // Best-effort eBPF cleanup — no-op on non-Linux or proxy-mode sandboxes
   cleanupEBPF(sandboxID, logger)
   ```

3. Also add `internal/app/cmd/destroy_ebpf_linux.go` and `internal/app/cmd/destroy_ebpf_other.go` to the plan's files list.

**pkg/ebpf/enforcer_integration_test.go** (`//go:build linux && integration`):

These are integration tests that require root on an EC2 instance. They use the `integration` build tag so they don't run in normal CI. They document and verify the security guarantees of EBPF-NET-12.

```go
// TestRootCannotBypassEBPF verifies that a root process inside the sandbox
// cgroup cannot bypass eBPF network enforcement.
//
// Prerequisites:
// - Must run as root on EC2 (AL2023, kernel 6.1+)
// - BPF programs must be loaded and attached to test cgroup
//
// Test scenarios:
// 1. iptables -F has no effect on eBPF enforcement
// 2. Direct connect() to blocked IP returns EPERM
// 3. Raw socket to blocked IP is dropped by cgroup_skb/egress
// 4. Cannot call bpf() syscall to detach programs (EPERM, no CAP_BPF)
// 5. DNS query for blocked domain returns NXDOMAIN
// 6. Hardcoded IP (bypassing DNS) is blocked by connect4 LPM_TRIE check
```

Test structure:
1. `TestRootIPTablesFlushIrrelevant`:
   - Create enforcer, attach to test cgroup
   - Allow 1.1.1.1, block everything else
   - Run `iptables -F -t nat` (this would break iptables DNAT enforcement)
   - Verify connect to blocked IP (93.184.216.34) from cgroup still returns EPERM
   - This proves eBPF enforcement is independent of iptables

2. `TestRootDirectConnectBlocked`:
   - Create enforcer with block mode
   - Attempt `net.Dial("tcp", "93.184.216.34:443")` from cgroup process
   - Verify error contains EPERM / connection refused

3. `TestRootCannotDetachBPF`:
   - Document that `bpf()` syscall requires CAP_BPF in the **init** (host) user namespace
   - Sandbox processes run in the host user namespace but without CAP_BPF
   - Even `CAP_NET_ADMIN` is insufficient to detach cgroup BPF programs
   - The test verifies by attempting `link.AttachCgroup()` from a non-privileged process in the cgroup and expecting EPERM

4. `TestDNSBlockedDomain`:
   - Configure allowlist with only ".github.com"
   - From cgroup process: resolve "evil.com" via DNS query
   - Verify NXDOMAIN response

5. `TestHardcodedIPBlocked`:
   - Do NOT resolve any domain
   - Attempt connect to a specific IP not in allowlist
   - Verify EPERM from connect4

Each test creates its own cgroup, attaches BPF programs, runs the test, and cleans up. Tests are independent and can run in any order.

Include detailed comments explaining WHY each test proves the security guarantee. These tests are documentation as much as verification.
