# Phase 91.3 — Bypass mention-only in bot-engaged threads

## Origin

Raised by operator during live Phase 91 Slack UAT (2026-05-31, same session as 91.1):

> "It's working when I @ the KlankerMakerNotification — and it will reply and start a
> thread, but when I reply to the reply, it shouldn't require another @ — once a klanker
> @'d and it replies, all future back and forth in that thread should not require another
> @ to that sandbox."

This is a Phase 91 design gap, not a bug. Phase 91's mention scan applies uniformly to all
messages, including thread replies. But threads are 1:1 conversations with the bot by
construction — once the bot is engaged in a thread, requiring another @-mention on every
reply is unnatural and breaks the conversational flow.

## What shipped

### 1. `pkg/slack/bridge/events_interfaces.go`
* `SlackThreadStore` interface gains `LookupSandbox(ctx, channelID, threadTS) (sandboxID, err)`.
* Distinct from the existing `Get` (which returns `claude_session_id` — set by the poller
  AFTER the agent replies). `LookupSandbox` returns `sandbox_id` — set by `Upsert` at
  dispatch time, so the bypass fires AS SOON AS the first mention-triggered dispatch
  enqueues, not after the poller responds.
* This bridges the user-perceived latency window between sending an @-mention and the
  agent replying. If the user sends a follow-up message in the thread BEFORE the agent
  responds, the bypass still applies (the thread row was already written at upsert).

### 2. `pkg/slack/bridge/aws_adapters.go`
* `DDBThreadStore.LookupSandbox` implementation: same GetItem call shape as `Get`,
  different attribute extracted (`sandbox_id` vs `claude_session_id`).
* No new GSI. No new field. No schema change to `km-slack-threads`. The `sandbox_id`
  attribute was already written by `Upsert` (aws_adapters.go:903) from day one of
  Phase 67 — Phase 91.3 just exposes a read path for it.

### 3. `pkg/slack/bridge/events_handler.go` step 4b
* The mention-only block grows a thread-bypass branch BEFORE the substring scan:
  ```go
  if msg.ThreadTS != "" {
      sb, err := h.Threads.LookupSandbox(ctx, msg.Channel, msg.ThreadTS)
      if err != nil { /* fail-open: treat as unknown thread; mention required */ }
      else if sb != "" { bypassed = true }
  }
  if !bypassed { /* existing mention scan */ }
  ```
* Fail-open on DDB error matches `isBotLoop`'s policy.
* Logs `events: mention-only: bypassed for engaged thread` at Debug level so the
  bypass is traceable in CloudWatch.

### 4. Tests (`pkg/slack/bridge/events_handler_test.go`)
* `fakeThreads` stub extended with `sandboxByThread map[string]string` (key:
  `channel|threadTS` → `sandbox_id`).
* New `TestEventsHandler_MentionOnly_ThreadBypass` (5 cases):
  - Engaged thread, no mention → dispatched (bypass works)
  - Engaged thread, with mention → dispatched (bypass + match)
  - Thread reply but no row → skipped (no bypass)
  - Top-level, no mention → skipped (bypass never applies)
  - Top-level, with mention → dispatched (normal mention scan)

### 5. Docs
* `docs/slack-notifications.md` § Phase 91 § Troubleshooting: new row for "bot ignores
  thread reply when engaged" and a callout for the inverse (`thread-bypass routes
  unrelated thread chatter to sandbox`).
* `CLAUDE.md` § Phase 91: bullet noting the Phase 91.3 thread-bypass semantics.

## Deployment

This is a Lambda code change — bridge handler logic changed. Requires:

```sh
make build                            # rebuild km binary (incl. lambdas)
./km init --dry-run=false             # terragrunt apply redeploys km-slack-bridge Lambda
```

`km init --sidecars` does NOT update Lambda code in the bridge — that's terragrunt-owned.
(Same constraint as Phase 91.1's env-block changes.)

## Edge cases worth knowing

* **Race: user sends thread reply before bot's agent reply lands.** `claude_session_id`
  isn't set yet. With `LookupSandbox` (Phase 91.3) the bypass fires anyway because
  `sandbox_id` is written synchronously at dispatch time. The agent will see both
  messages in its run.

* **Thread bound to a destroyed sandbox.** Same channel-lookup path applies after the
  bypass (events_handler.go line 201). If the sandbox was destroyed, `SandboxRoutingInfo`
  will have empty `QueueURL` and the handler returns 200 silently. Bypass let the
  mention check pass, but downstream lookup still gates dispatch. No leakage to
  unintended sandboxes.

* **Mistaken @-mention engaging a thread the user didn't intend to be bot-owned.** Once
  the row is upserted, the thread is bot-owned forever (until row TTL of 30 days). If
  the user wants to opt out of the bypass for a thread, the only path today is to start
  a new top-level message. A future Phase 91.4 could add a "release thread" command,
  but that's overkill for current scale.

## Why an in-place follow-up and not a decimal phase

Same rationale as 91.1: ~150 lines of code (counting tests), single subsystem, no new
external dependency. The semantic is a natural extension of Phase 91's mention scan, not
a new behaviour. Captured here as a `FOLLOWUP.md` for traceability.
