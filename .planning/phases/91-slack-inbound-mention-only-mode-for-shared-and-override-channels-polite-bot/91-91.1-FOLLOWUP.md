# Phase 91.1 ‚Äî km-config.yaml drives KM_SLACK_MENTION_ONLY; auto-read KM_SLACK_BOT_USER_ID

## Origin

Raised by operator immediately after Phase 91 docs landed (2026-05-30, same session).
Question: "Is there edits to km-config.yaml that make the ENV vars not needed on the init
and creates?"

Phase 91 shipped with this rollout sequence:

```sh
make build
export KM_SLACK_MENTION_ONLY=true
km slack init --force
export KM_SLACK_BOT_USER_ID=$(aws ssm get-parameter --name /${KM_RESOURCE_PREFIX}/slack/bot-user-id --query Parameter.Value --output text)
km init --dry-run=false
```

Two `export` lines that the operator has to type before every `km init`. They drift out of
sync with the desired install policy on a hand-typed shell, easy to forget, and not the
shape of how the rest of `km-config.yaml` values flow (`operator_email`, `resource_prefix`,
`artifacts_bucket` etc. all flow through viper ‚Üí `ExportTerragruntEnvVars` automatically).

## What shipped

## Observations worth carrying forward

- **`km init --sidecars` does NOT update Lambda env blocks.** It rebuilds + uploads
  binaries and forces a cold-start (via `upsertLambdaEnvVar` setting `TOOLCHAIN_VERSION`),
  but the bridge Lambda's `environment.variables` block ‚Äî including the Phase 91
  `KM_SLACK_MENTION_ONLY` and `KM_SLACK_BOT_USER_ID` ‚Äî is owned by the Terraform module
  and only updates on full terragrunt apply (`km init` without `--sidecars`).
  Phase 91's original rollout text used `km init --sidecars` which was wrong; Phase 91.1
  docs fix this.

- **Two orderings, both valid.** `km slack init --force` ‚Üí `km init` is the clean path
  (no doctor WARN, no info line). `km init` ‚Üí `km slack init --force` is the "discovered"
  path the operator hit in practice ‚Äî non-fatal, but requires a second `km init` to flow
  the cached SSM value into the Lambda env. The "Either ordering works" table in
  `docs/slack-notifications.md` ¬ß Phase 91 documents this.

- **SSM `ParameterNotFound` returns HTTP 400, not 404.** The `EnsureSlackBotUserIDFromSSM`
  helper's `[info]` line surfaces this as a smithy error string. Acceptable for an info
  line; if we ever want to suppress noise on first-install, branching on
  `errors.As(err, &smithytypes.ParameterNotFound{})` would let us silently skip instead.

## Hotfix `4d49544` ‚Äî viper merge-loop allowlist (km v0.3.770)

After the first `km init --dry-run=false` apply landed on the application account, the
Lambda's `Environment.Variables` showed `KM_SLACK_MENTION_ONLY="false"` despite
`slack: mention_only: true` in `km-config.yaml`. `KM_SLACK_BOT_USER_ID` flowed correctly
(SSM auto-read worked).

**Root cause:** `internal/app/config/config.go::Load()` runs a v2‚Üív viper merge loop with a
hardcoded key allowlist (lines 334-364). Only keys in that allowlist get copied from the
km-config.yaml viper instance to the main viper instance. The Phase 91.1 initial commit
added the `SlackConfig` struct, viper `IsSet`/`GetBool` read, and `ExportTerragruntEnvVars`
emission ‚Äî but missed adding `slack.mention_only` to the merge allowlist. As a result,
`v.IsSet("slack.mention_only")` returned false in the main viper instance and
`cfg.Slack.MentionOnly` stayed `nil` ‚Üí `ExportTerragruntEnvVars` emitted nothing ‚Üí
terragrunt's `get_env("KM_SLACK_MENTION_ONLY", "false")` fallback won ‚Üí Lambda env got
`"false"`.

**Why initial tests didn't catch it:** the Phase 91.1 unit tests
(`slack_mention_only_export_test.go`) constructed `*config.Config` directly with
`Slack.MentionOnly` populated, bypassing the yaml‚Üíviper‚Üícfg load chain entirely. Test gap
on a class of bug: any future nested-key config addition that misses the merge-loop entry
would have the same silent failure mode.

**Fix:** add `"slack.mention_only"` to the merge-loop allowlist + three end-to-end
integration tests that load from a real yaml file and assert `cfg.Slack.MentionOnly`:

* `TestLoadSlackMentionOnly_True` ‚Äî yaml has `mention_only: true` ‚Üí `*cfg.Slack.MentionOnly == true`
* `TestLoadSlackMentionOnly_False` ‚Äî yaml has `mention_only: false` ‚Üí `*cfg.Slack.MentionOnly == false`
* `TestLoadSlackMentionOnly_Absent` ‚Äî yaml omits the `slack:` block ‚Üí `cfg.Slack.MentionOnly == nil`

These three tests would have failed at PR time and caught the bug pre-deploy. Going
forward, any new nested-key config addition should follow this template: struct +
viper-read + merge-loop allowlist + end-to-end yaml-load test.
