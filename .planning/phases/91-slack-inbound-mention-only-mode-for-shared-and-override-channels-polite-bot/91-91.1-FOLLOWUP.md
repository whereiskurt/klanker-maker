# Phase 91.1 ŌĆö km-config.yaml drives KM_SLACK_MENTION_ONLY; auto-read KM_SLACK_BOT_USER_ID

## Origin

Raised by operator immediately after Phase 91 docs landed (2026-05-30, same session).
Question: "Is there edits to km-config.yaml that make the ENV vars not needed on the init
and creates?"

Phase 91 shipped with this rollout sequence:

```sh
make build
export KM_SLACK_MENTION_ONLY=true
km slack init --force
export KM_SLACK_BOT_USER_ID=$(aws ssm get-parameter --name /${KM_RESOURCE_PREFIX}/slack/bot-user-id --query Parameter.Value --output text)
km init --dry-run=false
```

Two `export` lines that the operator has to type before every `km init`. They drift out of
sync with the desired install policy on a hand-typed shell, easy to forget, and not the
shape of how the rest of `km-config.yaml` values flow (`operator_email`, `resource_prefix`,
`artifacts_bucket` etc. all flow through viper ŌåÆ `ExportTerragruntEnvVars` automatically).

## What shipped

- Called from `runInit` immediately after `ExportTerragruntEnvVars(cfg)` (uses the already-
  loaded `awsCfg` from the cred-validation step above).

## Observations worth carrying forward

- **`km init --sidecars` does NOT update Lambda env blocks.** It rebuilds + uploads
  binaries and forces a cold-start (via `upsertLambdaEnvVar` setting `TOOLCHAIN_VERSION`),
  but the bridge Lambda's `environment.variables` block ŌĆö including the Phase 91
  `KM_SLACK_MENTION_ONLY` and `KM_SLACK_BOT_USER_ID` ŌĆö is owned by the Terraform module
  and only updates on full terragrunt apply (`km init` without `--sidecars`).
  Phase 91's original rollout text used `km init --sidecars` which was wrong; Phase 91.1
  docs fix this.

- **Two orderings, both valid.** `km slack init --force` ŌåÆ `km init` is the clean path
  (no doctor WARN, no info line). `km init` ŌåÆ `km slack init --force` is the "discovered"
  path the operator hit in practice ŌĆö non-fatal, but requires a second `km init` to flow
  the cached SSM value into the Lambda env. The "Either ordering works" table in
  `docs/slack-notifications.md` ┬¦ Phase 91 documents this.

- **SSM `ParameterNotFound` returns HTTP 400, not 404.** The `EnsureSlackBotUserIDFromSSM`
  helper's `[info]` line surfaces this as a smithy error string. Acceptable for an info
  line; if we ever want to suppress noise on first-install, branching on
  `errors.As(err, &smithytypes.ParameterNotFound{})` would let us silently skip instead.

## Why an in-place follow-up and not a decimal phase

This change is purely a UX wrapper over Phase 91's already-shipped mechanisms ŌĆö no new
behaviour, no new requirements, no new test surface beyond the config-load roundtrip.
Inserting a `91.1` phase via `/gsd:insert-phase` would have produced a one-plan, three-task
phase that adds ceremony without adding clarity. The operator asked for it inline, the
implementation was 60 lines, and the documentation update naturally lives in the same
sections as Phase 91 itself. Tracked here as a `FOLLOWUP.md` so traceability survives.
