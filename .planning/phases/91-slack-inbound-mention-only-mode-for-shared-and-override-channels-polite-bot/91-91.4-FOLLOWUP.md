# Phase 91.4 — First-only reactor toggle (notifySlackInboundReactAlways)

## Origin

Raised by operator during live UAT immediately after Phase 91.3 landed:

> "I'm just thinking of one more use case where the operator doesn't want reactions
> AFTER the first time. So, only the FIRST time :eyes: land on the message, the reply
> threads in, and the back and forth goes back and forth without any more :eyes:
> this would be like a polite notifySlackInboundMentionOnly and
> 'notifySlackInboundReactAlways' and it could be true/false. I'd add false to the
> polite profile to test."

The Phase 91.3 thread-bypass made thread continuations frictionless (no re-@-mention),
but each continuation still got a 👀 reaction. For workspaces where multiple humans are
chatting alongside the bot in a thread, the 👀-spam adds visual noise without signal.
The user's signal-vs-noise expectation: 👀 on the engagement (the moment the bot "saw"
the request), silent thereafter for back-and-forth.

## What shipped

### 1. Profile field — `pkg/profile/`

* `types.go::CLISpec.NotifySlackInboundReactAlways *bool` — tri-state, sibling of
  `NotifySlackInboundMentionOnly`:
  - `nil` → default true (current chatty-reactor)
  - `&true` → explicit chatty-reactor
  - `&false` → first-only reactor
* `schemas/sandbox_profile.schema.json` — added the field as optional boolean.
* `pkg/profile/profile_cli_mention_test.go::TestCLISpec_NotifySlackInboundReactAlways_RoundTrip` — 3 cases.

### 2. Compiler — `pkg/compiler/userdata.go`

* `resolveReactAlways(*CLISpec) bool` — simpler than `resolveMentionOnly`: explicit
  override wins, otherwise true. Defensive nil-cli → true (preserves chatty default).
* Emission in the existing `notifyEnv` block: gated on
  `NotifySlackEnabled == &true` (same gate as `KM_SLACK_MENTION_ONLY`), emits
  `KM_SLACK_REACT_ALWAYS=true|false` into per-sandbox userdata.
* `userdata_mention_test.go::TestResolveReactAlways` — 4 cases (nil cli, override nil,
  override &true, override &false).

### 3. Install-level config — `internal/app/config/config.go`

* `SlackConfig` struct grows `ReactAlways *bool` (Phase 91.1 added `MentionOnly`;
  mirrors that exactly).
* viper merge-loop allowlist gets `slack.react_always` entry.
* viper IsSet load: `cfg.Slack.ReactAlways = &val` when key present.
* `config_test.go::TestLoadSlackReactAlways_{True,False,Absent}` — 3 cases.

### 4. ExportTerragruntEnvVars — `internal/app/cmd/init.go`

* New block immediately after the 91.1 `KM_SLACK_MENTION_ONLY` emission. Same shape:
  drift WARN to stderr when env disagrees with yaml; otherwise set env if unset.
* `slack_mention_only_export_test.go::TestExportTerragruntEnvVars_SlackReactAlways_{Absent,True,False,EnvWins}` — 4 cases.

### 5. Bridge handler — `pkg/slack/bridge/events_handler.go`

* `EventsHandler.ReactAlways bool` field (default true via Go zero-value would be
  false, but `WireMentionOnly` sets the explicit default of true — see below).
* Step 10 reactor block grows the gate:
  ```go
  if !h.ReactAlways && msg.ThreadTS != "" {
      h.log().Debug("events: reaction skipped (react-always=false, thread reply)", ...)
  } else {
      // existing 👀 post
  }
  ```
* Top-level messages (`msg.ThreadTS == ""`) always react regardless of ReactAlways —
  that's the engagement signal. Only thread replies in first-only mode go silent.
* `events_handler_test.go::TestEventsHandler_ReactAlways` — 4 cases covering the
  truth table:
  - react_always=true, top-level → reacts
  - react_always=true, thread reply → reacts
  - react_always=false, top-level → reacts (engagement)
  - react_always=false, thread reply → silent

### 6. Lambda cold-start wiring — `cmd/km-slack-bridge/main.go`

* `WireMentionOnly` (renamed mentally but kept for backwards compat) now also reads
  `KM_SLACK_REACT_ALWAYS`. Default-true semantic encoded as
  `h.ReactAlways = os.Getenv("KM_SLACK_REACT_ALWAYS") != "false"` — empty env, "true",
  or any garbage value all preserve chatty-reactor. Only the literal string "false"
  flips it.
* Cold-start log line extended with `react_always` field.

### 7. Lambda Terraform — `infra/modules/lambda-slack-bridge/v1.0.0/`

* `infra/live/use1/lambda-slack-bridge/terragrunt.hcl` — `slack_react_always =

## Truth table

| ReactAlways | msg.ThreadTS | Outcome |
|---|---|---|
| true (default) | "" (top-level) | 👀 posted |
| true (default) | "1.0" (reply) | 👀 posted |
| false | "" (top-level) | 👀 posted (engagement) |
| false | "1.0" (reply) | **silent dispatch** |

## Architecture note: install-level vs per-sandbox

The profile field `notifySlackInboundReactAlways` flows into per-sandbox userdata
(via the compiler's `notifyEnv` block) but does NOT alter the bridge Lambda's
runtime behaviour. The bridge is install-level — there is exactly one
`KM_SLACK_REACT_ALWAYS` env var on the bridge Lambda, driven by `km-config.yaml`
`slack.react_always`. Profile-level granularity will arrive when a future phase
extends `SandboxRoutingInfo` to carry per-sandbox bridge config (similar to how
Phase 91.3's `LookupSandbox` extended thread-store routing).

This is the same trade-off Phase 91 itself made for `NotifySlackInboundMentionOnly`
— operators who want per-profile bridge behaviour should hold off until that
phase ships. Phase 91.4 is install-level by design, matching the proven 91/91.1
architecture.

## Deployment

Lambda code change (bridge handler grew the gate) + Lambda env block change
(`KM_SLACK_REACT_ALWAYS` added). Requires:

```sh
make build                            # km v0.3.773+
./km init --dry-run=false             # full terragrunt apply — Lambda code + env update
```

`km init --sidecars` is NOT sufficient (same constraint as 91.1 / 91.3).

## Why an in-place follow-up (and not a decimal phase)

Same rationale as 91.1 and 91.3: surgical extension to Phase 91 architecture, no new
external dependency, ~250 lines of code (counting tests + Terraform), all within the
established Slack bridge subsystem. Inserting `/gsd:insert-phase 91.4` would have
generated a single-plan phase that adds ceremony without adding clarity.
