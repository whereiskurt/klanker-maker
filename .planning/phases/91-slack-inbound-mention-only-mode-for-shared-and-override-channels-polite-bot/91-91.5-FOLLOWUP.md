# Phase 91.5 — Per-sandbox `notifySlackInboundReactAlways` override

## Origin

Raised by operator immediately after Phase 91.4 landed:

> "Is there a way to override the KM_SLACK_REACT_ALWAYS per sandbox? Ideally I could
> set it in the profile/learn.v2.polite.yaml per sandbox too."

Phase 91.4 added the profile field `cli.notifySlackInboundReactAlways` for symmetry
but documented it as forward-compat — the bridge's runtime behaviour was install-level
only. The user wants the profile field to actually drive behaviour per sandbox.

Phase 91.5 closes that gap: the profile field is now first-class. Different profiles
in the same install can opt into different reactor behaviour.

## Architecture

Three-piece wiring along the existing channel→sandbox lookup path:

```
Profile (cli.notifySlackInboundReactAlways)
   │
   │ km create → provisionSlackInboundQueue (create_slack_inbound.go)
   ▼
km-sandboxes row .slack_react_always  ← written ONLY when profile sets it explicitly
   │
   │ Slack event → events handler step 6 → DDBSandboxByChannel.FetchByChannel
   ▼
SandboxRoutingInfo.ReactAlways *bool  ← tri-state: nil = use install default
   │
   │ events_handler.go step 10
   ▼
Effective: info.ReactAlways if non-nil else h.ReactAlways (install-level env var)
```

The absent-attribute semantic is load-bearing: when a profile omits the field, the DDB
row has no `slack_react_always` attribute, `info.ReactAlways` is nil, and the bridge
falls back to the install-level default. This preserves the operator's ability to
flip the workspace-wide default via `km-config.yaml slack.react_always` without
recreating every sandbox.

## What shipped

### 1. Write path — `internal/app/cmd/create_slack_inbound.go`

`provisionSlackInboundQueue` now writes a second DDB attribute alongside
`slack_inbound_queue_url` — but ONLY when `cli.NotifySlackInboundReactAlways` is
non-nil. Absence of the attribute on the row is what carries the "use install default"
semantic.

Stored as a string (`"true"` / `"false"`) to match the existing `UpdateSandboxAttr`
interface signature. The bridge tolerates both string and native BOOL types
forward-compat (a future direct PutItem write could use BOOL without breaking the
reader).

The write is best-effort non-fatal: if the DDB write fails the sandbox still works
correctly with the install-level default. Logged at WARN.

### 2. Read path — `pkg/slack/bridge/aws_adapters.go::DDBSandboxByChannel.FetchByChannel`

Extracts `slack_react_always` from the row alongside the existing `sandbox_id`,
`slack_inbound_queue_url`, and `state` attributes. Type-tolerant: handles both
`AttributeValueMemberBOOL` (forward-compat) and `AttributeValueMemberS` (current
write shape). Returns `nil` `*bool` when the attribute is absent.

### 3. Routing struct — `pkg/slack/bridge/events_interfaces.go::SandboxRoutingInfo`

New tri-state field `ReactAlways *bool`. Sits alongside the existing `SandboxID`,
`QueueURL`, `Paused` fields.

### 4. Handler — `pkg/slack/bridge/events_handler.go` step 10

```go
effectiveReactAlways := h.ReactAlways
if info.ReactAlways != nil {
    effectiveReactAlways = *info.ReactAlways
}
if !effectiveReactAlways && msg.ThreadTS != "" {
    // skip reaction (debug log)
} else {
    // post 👀
}
```

Per-sandbox value is preferred when set. Top-level messages still always react —
that's the engagement signal, and the per-sandbox override doesn't silence the
"I saw you" acknowledgement to humans.

Debug log line on skip now carries `sandbox` and `per_sandbox_override` fields
so CloudWatch tail shows which sandbox the bypass was for and whether it was the
install default or the per-sandbox attribute that drove the decision.

### 5. Tests

- `pkg/slack/bridge/events_handler_test.go::TestEventsHandler_ReactAlways_PerSandboxOverride`
  — 6-case truth table covering (install × per-sandbox × top-level/thread):
  - install=true, per-sb nil, thread → reacts (install wins)
  - install=false, per-sb nil, thread → silent (install wins)
  - install=true, per-sb=&false, thread → silent (per-sb overrides)
  - install=false, per-sb=&true, thread → reacts (per-sb overrides)
  - install=true, per-sb=&false, top-level → reacts (engagement always)
  - install=false, per-sb=&true, top-level → reacts (engagement)

- `pkg/slack/bridge/aws_adapters_test.go::TestDDBSandboxByChannel_ReactAlways_Override`
  — 4-case DDB extraction:
  - attribute absent → nil
  - S `"true"` → `&true`
  - S `"false"` → `&false`
  - BOOL true → `&true` (forward-compat)

- `internal/app/cmd/create_slack_inbound_test.go::TestCreate_SlackInboundReactAlwaysOverride`
  — 3-case write path:
  - nil → no write (install default applies)
  - `&true` → writes `slack_react_always="true"`
  - `&false` → writes `slack_react_always="false"`

13 new tests total. All green.

## Deployment

Sandbox-create code change (write path) + bridge Lambda code change (read path).

```sh
make build                            # km v0.3.776+
./km init --dry-run=false             # full terragrunt apply — bridge Lambda code
```

Existing sandboxes are unaffected at the bridge layer (their DDB rows don't have
the new attribute, so `info.ReactAlways == nil`, and they continue using the
install-level default). To opt an existing sandbox into per-sandbox override:
`km destroy --remote --yes && km create <profile>`. The DDB write fires on
create-time only — there is no `km update` path for this attribute yet.

## Operator config (test path)

Two sandboxes, different reactor behaviour:

```yaml
# profiles/learn.v2.polite.yaml — operator-forced first-only
spec:
  cli:
    notifySlackInboundReactAlways: false   # Phase 91.5: per-sandbox override

# profiles/learn.v2.yaml — default (uses km-config.yaml slack.react_always)
spec:
  cli:
    # no notifySlackInboundReactAlways → falls back to install default
```

Provision both:

```sh
./km create profiles/learn.v2.polite.yaml   # row gets slack_react_always="false"
./km create profiles/learn.v2.yaml          # row has no slack_react_always attribute
```

In Slack:
- Posting in the polite sandbox's `#sb-…` thread → silent dispatch (per-sb override)
- Posting in the default sandbox's `#sb-…` thread → 👀 (install-level chatty default)

## Edge cases worth knowing

- **DDB write failure is non-fatal.** Sandbox still provisions; bridge uses install
  default. Logged at WARN with sandbox_id + value. The operator can manually run
  `aws dynamodb update-item` to add the attribute later.

- **Type-tolerant read.** Today the write goes via `UpdateSandboxAttr` which writes
  strings. The reader handles both S and BOOL types. A future refactor could move
  the write to direct PutItem with native BOOL without breaking the bridge.

- **Profile change requires destroy/create.** No `km update` path; the attribute is
  only written at sandbox-create time. Documented in the docs/troubleshooting row.

- **Top-level engagement always reacts.** Even with `notifySlackInboundReactAlways:
  false`, the bot still 👀-reacts to top-level @-mentions. That's the load-bearing
  "I saw you" signal — silencing it would leave operators unsure whether a new
  task was acknowledged. The first-only semantic only suppresses 👀 on
  *follow-up* messages within an engaged thread.

## Architectural notes (carry-forward)

- `SandboxRoutingInfo` is now the natural place to add future per-sandbox bridge
  routing fields. Any future `notifySlackInbound{X}` profile field with bridge-side
  semantics can follow the same recipe: write via `UpdateSandboxAttr` at sandbox-
  create, extract in `FetchByChannel`, use in the relevant handler step.

- The same pattern could promote Phase 91's `notifySlackInboundMentionOnly` from
  install-level to per-sandbox. Phase 91.5 establishes the recipe; not done in this
  follow-up because nobody has asked for it yet. (`MentionOnly` runs at step 4b
  BEFORE the channel lookup at step 6, so the order would need a swap — minor
  refactor.)

## Why an in-place follow-up

Same rationale as 91.1, 91.3, 91.4. ~150 lines (counting tests), single subsystem,
direct extension of Phase 91 architecture using machinery (`SandboxRoutingInfo`,
`UpdateSandboxAttr`) that already existed for other per-sandbox bridge state
(`QueueURL`, `Paused`). The path was carved by Phase 67; Phase 91.5 walked it.
