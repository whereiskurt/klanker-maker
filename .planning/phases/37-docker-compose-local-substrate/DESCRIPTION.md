# Phase 37: Docker Compose Local Substrate (Connected Mode)

## Problem

EC2 works but provisions a full VM per sandbox — slow to start, expensive for quick iteration, and requires Terraform/Terragrunt round-trips. For rapid agent testing and development, a lighter-weight local substrate that still connects to the same AWS backend (SSM, SES, DynamoDB, S3) would give operators fast sandbox creation with full platform integration.

## Goal

`km create profiles/claude-dev.yaml --substrate docker` provisions a local sandbox using Docker Compose with the same 5-container topology (main + 4 sidecars), connected to the existing AWS platform — SSM Parameter Store for secrets, SES for email, DynamoDB for budget tracking, S3 for artifacts and OTEL. Same enforcement, same telemetry, different compute.

## Depends on

- Phase 36 (km-sandbox base image with entrypoint)
- Requires `km-config.yaml` with valid AWS credentials (connected mode only)

## Design

### New substrate: `docker`

```yaml
spec:
  runtime:
    substrate: docker
    # instanceType ignored for docker
    # spot ignored for docker
    # region used for AWS API calls (SSM, SES, DynamoDB, S3)
```

`km validate` accepts `docker`. The compiler has a third path: `compileDocker()`.

### What's the same as EC2

| Concern | How it works |
|---------|-------------|
| **SSM secrets** | Entrypoint fetches `KM_SECRET_PATHS` from Parameter Store at container start |
| **OTP secrets** | Entrypoint fetches and deletes from SSM |
| **SES email** | Sandbox gets its email address, mail poller syncs from S3 |
| **Budget tracking** | HTTP proxy meters AI tokens, writes to DynamoDB |
| **Budget enforcement** | Budget-enforcer Lambda fires on the same 1-minute schedule |
| **OTEL telemetry** | Tracing sidecar exports to S3 |
| **Audit logs** | Audit-log sidecar writes to CloudWatch |
| **GitHub credentials** | Token from SSM, GIT_ASKPASS wired by entrypoint |
| **Proxy CA trust** | CA cert downloaded from S3, installed in container trust store |
| **DNS filtering** | DNS proxy sidecar enforces allowedDNSSuffixes |
| **HTTP filtering** | HTTP proxy sidecar enforces allowedHosts + GitHub repo MITM |
| **initCommands** | Entrypoint executes profile-defined commands at boot |
| **Rsync snapshots** | Save/restore to S3 via entrypoint |
| **Sandbox identity** | Ed25519 key pair in DynamoDB, signed email |

### What's different from EC2

| Concern | EC2 | Docker |
|---------|-----|--------|
| **Compute** | Full VM (t3.medium) | Docker containers on operator's laptop |
| **Provisioning** | Terragrunt apply (~45-90s) | `docker compose up -d` (~5-10s) |
| **Teardown** | Terragrunt destroy (~30-60s) | `docker compose down -v` (~2s) |
| **Security Groups** | AWS VPC SG egress rules | No SG — proxy enforcement is the only wall |
| **iptables DNAT** | Transparent redirect to proxy | Not needed — containers use proxy env vars |
| **SSM Session Manager** | `km shell` via SSM | `km shell` via `docker exec` |
| **IAM role** | Per-sandbox assumed role (instance profile) | Per-sandbox assumed role (STS session creds via credential refresh sidecar) |
| **Audit logs** | CloudWatch + S3 export on destroy | S3 directly (continuous write, no CloudWatch) |
| **OTEL traces** | S3 via tracing sidecar | S3 via tracing sidecar (identical) |
| **Spot interruption** | Handled by spot handler | N/A — containers are persistent |
| **TTL auto-destroy** | EventBridge scheduler → Lambda | Background goroutine or host cron |
| **SCP containment** | Org-level deny | N/A — no org account boundary |
| **IMDSv2** | Enforced on instance | N/A — no IMDS |

### Scoped AWS credentials (two roles, not one)

Containers must never get the operator's full AWS credentials. Instead, `km create` provisions two scoped IAM roles and injects short-lived STS session credentials as environment variables:

**Sandbox role** (main container) — same as the EC2 role (`km-ec2spot-ssm-{id}-{region}`):
- Bedrock InvokeModel/InvokeModelWithResponseStream
- SSM GetParameter (for `KM_SECRET_PATHS`, GitHub token)
- SSM DeleteParameter (for OTP secrets)
- S3 GetObject on artifacts bucket (CA cert, rsync snapshots, mail inbox)
- S3 PutObject on artifacts bucket (artifact upload, rsync save)
- SES SendEmail/SendRawEmail (scoped to sandbox email address)
- EventBridge PutEvents (idle detection)
- Region-locked to `allowedRegions` from profile
- Session duration from `roleSessionDuration` (default 1h)

**Sidecar role** (http-proxy, audit-log, tracing containers) — narrower scope:
- DynamoDB GetItem/UpdateItem on budget table (http-proxy token metering)
- S3 PutObject on artifacts bucket audit prefix (audit-log sidecar)
- S3 PutObject on artifacts bucket OTEL prefix (tracing sidecar)
- Region-locked

The DNS proxy needs no AWS credentials (pure in-memory DNS filtering).

#### How it works

1. `km create` uses the **operator's credentials** (klanker-terraform profile) to:
   - Create the sandbox IAM role via Terraform (same module as EC2, but with `sts:AssumeRole` trust for the terraform account instead of `ec2.amazonaws.com`)
   - Call `sts:AssumeRole` on the sandbox role → get temporary credentials
   - Call `sts:AssumeRole` on a sidecar role → get separate temporary credentials
2. Credentials are written into the generated `docker-compose.yml` as env vars
3. Credentials are also written to `~/.km/sandboxes/{id}/.env` (compose `env_file`)
4. A **credential refresh sidecar** (or background process) calls AssumeRole every 45 minutes and updates the `.env` file. Containers pick up new credentials on next AWS SDK call via standard credential chain refresh.

#### Credential refresh

STS session credentials expire (default 1h, max from profile's `roleSessionDuration`). Options:

**Option A — Refresh sidecar container:** A lightweight container that calls AssumeRole periodically and writes new credentials to a shared Docker volume. Other containers read from the volume via AWS SDK's file-based credential provider.

**Option B — Longer session + recreate:** Use the max session duration (up to 12h for IAM roles). For sandboxes with TTL ≤ 12h, no refresh needed. `km extend` regenerates credentials.

**Option C — Shared credentials file:** Write credentials to a file on a shared tmpfs volume. AWS SDK's `AWS_SHARED_CREDENTIALS_FILE` env var points containers to it. Refresh process updates the file.

Option C is simplest — one tmpfs volume, one refresh process, all containers point to the same file:

```yaml
services:
  km-cred-refresh:
    image: km-sandbox:latest
    command: ["/opt/km/cred-refresh.sh"]
    volumes:
      - cred-vol:/creds
    environment:
      KM_SANDBOX_ROLE_ARN: arn:aws:iam::052251888500:role/km-sb-a1b2c3d4-use1
      KM_SIDECAR_ROLE_ARN: arn:aws:iam::052251888500:role/km-sidecar-a1b2c3d4-use1
      # Operator creds (only this container gets them, scoped to AssumeRole only)
      AWS_ACCESS_KEY_ID: <operator-key>
      AWS_SECRET_ACCESS_KEY: <operator-secret>
      AWS_SESSION_TOKEN: <operator-token>

  main:
    volumes:
      - cred-vol:/creds:ro
    environment:
      AWS_SHARED_CREDENTIALS_FILE: /creds/sandbox
      AWS_PROFILE: sandbox

volumes:
  cred-vol:
    driver_opts:
      type: tmpfs
      device: tmpfs
```

The `cred-refresh` container is the **only** container with operator credentials, and those are scoped via an inline IAM policy to only allow `sts:AssumeRole` on the two sandbox roles. The main container and sidecars never see operator credentials.

#### IAM trust policy change

The EC2 sandbox role trusts `ec2.amazonaws.com` as its principal. For Docker, the role also needs to trust the operator's account for `sts:AssumeRole`:

```json
{
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    },
    {
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::052251888500:root" },
      "Action": "sts:AssumeRole",
      "Condition": {
        "ArnLike": {
          "aws:PrincipalArn": [
            "arn:aws:iam::052251888500:role/aws-reserved/sso.amazonaws.com/AWSReservedSSO_*",
            "arn:aws:iam::052251888500:role/km-provisioner-*"
          ]
        }
      }
    }
  ]
}
```

This is the same trust policy pattern used by `km-org-admin` — trusted principals are limited to SSO and provisioner roles.

### DNS routing

Docker Compose doesn't support iptables DNAT. Instead:
- DNS proxy container gets a static IP (`172.20.0.10`) on the bridge network
- Main container uses `dns: [172.20.0.10]` in compose to route DNS through the proxy
- Sidecar DNS proxy forwards allowed queries to `8.8.8.8` (upstream, not kube-dns)

### Networking: proxy env vars vs bridge

On EC2, iptables DNAT transparently redirects all traffic through the proxy. On Docker, the main container uses explicit proxy env vars (`HTTP_PROXY=http://km-http-proxy:3128`). This means:
- Tools that respect proxy env vars (curl, pip, npm, git, Python requests) go through the proxy ✅
- Tools that ignore proxy env vars (raw socket connections) bypass the proxy ⚠️

This is the same limitation as the current EC2 HTTPS path — transparent MITM isn't fully wired yet (Phase 31 TODO). The proxy env vars cover the main use cases.

### TTL enforcement

No EventBridge scheduler locally. Options:
1. **`km create` spawns a background TTL watcher** — a goroutine or detached process that sleeps until TTL, then runs `docker compose down`
2. **Container healthcheck** — main container checks its own TTL annotation and exits
3. **Manual** — operator runs `km destroy` when done

Option 1 for v1: `km create` writes a `.km-ttl` file with the expiry timestamp. A lightweight `km-ttl-watcher` binary (or shell script) runs in the background and tears down at expiry.

### Sandbox directory layout

```
~/.km/sandboxes/sb-a1b2c3d4/
├── docker-compose.yml      # generated by compiler
├── .km-ttl                 # expiry timestamp for local TTL watcher
├── metadata.json           # same metadata as S3 (profile, timestamps, budget)
└── workspace/              # optional: bind-mount for persistence across restarts
```

Metadata is also written to S3 (same as EC2) so `km list`, `km status`, `km otel` work identically.
