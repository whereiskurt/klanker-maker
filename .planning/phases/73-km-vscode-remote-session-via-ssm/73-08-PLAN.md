## docs/vscode.md structure

```markdown
# VS Code Remote-SSH access to klankermaker sandboxes

Phase 73 adds `km vscode start | status` so operators can connect their **local desktop VS Code**
(via the Remote-SSH extension) to a sandbox over SSM port-forward.

## How it works
- One-line architecture diagram
- Per-sandbox ed25519 keypair, generated locally, never leaves operator's laptop
- SSM tunnel = security boundary; SSH adds key auth + encrypted file transfer

## One-time setup
1. Install VS Code's "Remote - SSH" extension (Microsoft, free)
2. Run `make build && km init --sidecars` if upgrading (schema-change requirement)

## Per-sandbox lifecycle
1. km create profile.yaml --alias my-poc
   → keypair written to ~/.km/keys/sb-<id>
2. km vscode start <sb>
   → opens SSM port-forward, prints "F1 → Remote-SSH → km-sb-<id>"
3. In VS Code: F1 → "Remote-SSH: Connect to Host..." → km-sb-<id>
   → vscode-server auto-deploys (~50MB on first connect), opens empty workspace
   → File → Open Folder → /workspace
4. Edit, save, etc.
5. Ctrl-C the km vscode start terminal when done (sshd stays up; reconnect any time)
6. km destroy <sb> --remote --yes
   → cleans up ssh-config block + ~/.km/keys/sb-<id>* automatically

## Profile field
`spec.cli.vscodeEnabled` (bool*, default true). Set false to skip sshd provisioning entirely.

## CLI commands
- `km vscode start <sb> [--local-port N]` — open foreground SSM port-forward
- `km vscode status <sb>` — check sshd state + authorized_keys presence

## Operator state on disk
| Path | Mode | Purpose |
|------|------|---------|
| `~/.km/keys/sb-<id>` | 0600 | Private key (OpenSSH PEM) |
| `~/.km/keys/sb-<id>.pub` | 0644 | Public key (`ssh-ed25519 ... km-sb-<id>`) |
| `~/.ssh/config` (managed block) | 0600 | Per-sandbox Host entries between markers |

## Sandbox-side state
Cloud-init userdata (gated on vscodeEnabled=true) runs at boot:
- systemctl enable --now sshd
- mkdir + chmod /home/sandbox/.ssh
- write authorized_keys with operator's pubkey
- chown sandbox:sandbox + restorecon (for AL2023 SELinux)

## Troubleshooting
- "Private key for sb-X not found" → cross-machine portability gap; copy `~/.km/keys/sb-X*` from creation machine
- "VS Code not enabled in this sandbox's profile" → profile had `vscodeEnabled: false`; recreate with default
- "Local port 2222 already in use" → use `--local-port 9222` (any free port)
- VS Code Remote-SSH hangs at "Setting up VS Code" → operator's network blocks GitHub release downloads; vscode-server downloads through the operator's local SSH tunnel which terminates at the sandbox

## Limitations (deferred)
- Per-machine keys (no export/import). Sandboxes created on machine A can't be `km vscode start`-ed from machine B today.
- One operator per sandbox (single authorized_keys entry).
- StrictHostKeyChecking no (defense-in-depth comes from SSM/IAM; SSH host-key trust deferred).
- No `km vscode stop` — Ctrl-C ends the foreground tunnel; sshd keeps running.

## Security model
- SSM tunnel = real security boundary (IAM + encrypted)
- Per-sandbox ed25519 = blast radius of 1
- Private key stays on operator laptop
- Pubkey lives only on operator + the one sandbox
- No public exposure of port 22 (SG remains egress-only)
```

   | Field | Type | Default | Purpose |
   |---|---|---|---|
   | `vscodeEnabled` | bool* | true | Provision sshd + authorized_keys at sandbox boot. Set false to skip. |

   | Path | Purpose |
   |---|---|
   | `~/.km/keys/sb-<id>` | Private key (mode 0600) generated by `km create` |
   | `~/.km/keys/sb-<id>.pub` | Public key (mode 0644) shipped via userdata |
   | `~/.ssh/config` (managed block) | Host entries between `# BEGIN km vscode hosts` markers |

   ### One-time setup

   ```bash
   # Install the Remote - SSH extension in VS Code (Microsoft, free)

   # Schema change → refresh management Lambda
   make build && km init --sidecars
   ```

   ### Per-sandbox workflow

   ```bash
   km create profiles/<your-profile>.yaml --alias my-poc   # keypair generated locally
   SB=$(km list | awk '/my-poc/ {print $1}')
   km vscode start $SB                                      # opens tunnel, blocks until Ctrl-C
   # In VS Code: F1 → "Remote-SSH: Connect to Host..." → km-$SB
   km destroy $SB --remote --yes                            # cleans up keys + ssh-config block
   ```

   ### Important workflow notes

   - **`km init --sidecars` is required** after Phase 73 ships so the management Lambda's km
     binary recognizes the new `VSCodeSSHPubKey` userdata field. Without it, `km create
     --remote` produces a sandbox with broken authorized_keys (silent SSH failure).
   - Existing sandboxes created before Phase 73 do NOT get sshd provisioning retroactively —
     `km destroy && km create` to provision.
   - Cross-machine portability: keys live on the creation machine only. Operators who want to
     `km vscode start` from a different laptop must manually copy `~/.km/keys/sb-<id>*`.
   - One operator per sandbox in v1 (single authorized_keys entry).

   See `docs/vscode.md` for the full operator guide.
