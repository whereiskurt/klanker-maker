    Create `73-UAT.md` with frontmatter:

        ---
        phase: 73
        slug: km-vscode-remote-session-via-ssm
        status: pending
        scenarios: 6
        ---

    Then a section per scenario, taken from the "Manual-Only Verifications" table in
    73-VALIDATION.md. Each scenario is a numbered list with:
    1. Pre-conditions (what must be set up first).
    2. Steps the operator runs (verbatim shell commands where possible).
    3. Expected outcome at each step.
    4. A checkbox `- [ ] Scenario passed` at the end.

    The six scenarios:

    **Scenario 1: End-to-end Remote-SSH from local desktop VS Code**
    1. `make build && km init --sidecars` (refresh Lambda)
    2. `km create profiles/learn.yaml --alias vscode-smoke`
    3. `km vscode start <sb>` (terminal blocks; copy printed Host alias)
    4. In VS Code: F1 to "Remote-SSH: Connect to Host..." pick `km-<sb-id>`
    5. Confirm vscode-server installs (~50MB on first connect)
    6. File to Open Folder to `/workspace`
    7. Edit a file, save
    8. Ctrl-C the `km vscode start` terminal
    9. `km shell <sb> -- cat /workspace/<edited-file>` confirms persistence
    Expected: edits persist; tunnel exits cleanly; sshd stays up on sandbox.

    **Scenario 2: Per-sandbox keypair lifecycle**
    1. `ls ~/.km/keys/` filtered for sb-<id> before create returns empty
    2. `km create profiles/learn.yaml --alias kp-test`
    3. `ls -la ~/.km/keys/sb-<id>*` shows private (mode 0600) and public (mode 0644)
    4. `km destroy <sb> --remote --yes`
    5. `ls ~/.km/keys/sb-<id>*` errors "no such file"
    Expected: keypair generated at create, deleted at destroy, mode bits correct.

    **Scenario 3: ssh-config managed-block lifecycle**
    1. Backup current `~/.ssh/config` (cp to .bak)
    2. `km vscode start <sb>` (Ctrl-C immediately after the host block prints)
    3. Inspect `~/.ssh/config`: block between markers exists with correct Host alias
    4. `km destroy <sb> --remote --yes`
    5. Inspect `~/.ssh/config`: Host entry removed; markers removed if no entries left;
       surrounding content unchanged
    Expected: managed block roundtrips cleanly; outside-markers content untouched.

    **Scenario 4: vscodeEnabled false produces clean error**
    1. Create profile YAML with `spec.cli.vscodeEnabled: false`
    2. `km create <profile>` and capture sb-id
    3. `km vscode start <sb>` should fail with "VS Code not enabled in this sandbox's profile
       (set spec.cli.vscodeEnabled: true and recreate)" — NOT a raw SSM error
    Expected: clean operator-actionable error; no port-forward opened.

    **Scenario 5: Local port collision behavior**
    1. `nc -l 2222 &` on operator laptop
    2. `km vscode start <sb>` expects bind error from `aws ssm start-session`
    3. `km vscode start <sb> --local-port 9222` succeeds
    4. Verify `~/.ssh/config` Host entry's Port line reads 9222
    5. `kill %1` cleanup
    Expected: --local-port override works; ssh-config Port line matches.

    **Scenario 6: Cross-machine portability gap is informative**
    1. `km create` on machine A (key files written to A's `~/.km/keys/`)
    2. Sync repo to machine B (or just clone)
    3. `km vscode start <sb>` from machine B expects helpful error pointing at the missing key
       file location and suggesting copy
    Expected: error tells operator EXACTLY which file is missing and what to do.

    Add a final section "Sign-off" with a checkbox `- [ ] All six scenarios passed` and a
    line for the operator to date+sign with their initials.
  </action>
