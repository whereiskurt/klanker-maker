    via `--connection-token "$TOKEN"`. The replacement block:

    ```
    - Generates fresh token: `TOKEN=$(openssl rand -hex 32)`
    - Writes `/etc/km/vscode/token` mode 0600, owned by `sandbox`
    - Execs `code serve-web --host 127.0.0.1 --port 8443 --connection-token "$TOKEN" --accept-server-license-terms`
    ```

    Also append a new bullet under the "Architecture (Q3: A …)" block
    (right after the Token bullet) that documents the choice and rationale:

    ```
    - Token-injection form: the wrapper passes `--connection-token "$TOKEN"`
      directly (NOT `--connection-token-file`). Rationale: the file-form
      had a Microsoft bug (issue #215537, fixed in PRs #219041/#223524 but
      stable-release timing unverified). Inlining the token is reliable on
      every released `code` CLI version, and the token is still written to
      `/etc/km/vscode/token` 0600 so `km vscode start` can read it back
      for the operator's URL. Drop `--without-connection-token=false`
      entirely — research found the flag is a standalone toggle (presence
      = disable token), and the `=false` form is invalid.
    ```

    **Edit 2 — docs/superpowers/specs/2026-05-06-km-vscode-design.md,
    lines 92-110 (the "Launch wrapper" code block):** Replace lines
    104-109 (the existing exec block) with:

    ```bash
    # Bind 127.0.0.1 only — SSM port-forward is the security boundary.
    exec /usr/local/bin/code serve-web \
        --host 127.0.0.1 \
        --port 8443 \
        --connection-token "$TOKEN" \
        --accept-server-license-terms
    ```
