<critical_research_findings>
1. **Template column-0 placement (Pitfall 3).** `{{ .VSCodeSSHPubKey }}` MUST be at column 0
   inside the heredoc. Any leading whitespace in the template source becomes leading whitespace
   in the rendered authorized_keys, which sshd silently rejects.

2. **Loud-fail validation (Pitfall 4).** If a stale Lambda renders userdata with
   VSCodeEnabled==true but VSCodeSSHPubKey=="" (because the Lambda's km binary doesn't know
   about the new field), the operator gets a sandbox with a broken authorized_keys (blank line)
   and silent SSH failure. Catch this at generateUserData time:
   ```go
   if params.VSCodeEnabled && params.VSCodeSSHPubKey == "" {
       return "", fmt.Errorf("VSCodeEnabled=true but VSCodeSSHPubKey is empty — run `km init --sidecars` to refresh the management Lambda")
   }
   ```

3. **restorecon defensive wrap (Pitfall 5).** Non-AL2023 AMIs (e.g., Ubuntu) lack
   `policycoreutils`. Wrap with `command -v restorecon` so the cloud-init script doesn't fail.

4. **NetworkConfig vs variadic.** Per finding #7 in the planner brief, add VSCodeSSHPubKey to
   NetworkConfig. ECS / Docker compileX paths pass an empty string by default (just don't
   populate the field); the `{{- if .VSCodeEnabled }}` template gate prevents usage when empty.
</critical_research_findings>

# Phase 73: VS Code Remote-SSH access (sshd + authorized_keys + SELinux context)

{{- end }}

the template even if surrounding bash is indented for readability — text/template emits
leading whitespace verbatim into the heredoc.
