# Phase 67.2: Slack ACK reaction bounded retry — Research

**Researched:** 2026-05-14
**Domain:** Go HTTP retry loop with classified error taxonomy inside an AWS Lambda
**Confidence:** HIGH

## Summary

Phase 67.2 is a tightly bounded, single-file, ~80 LoC modification to
`SlackReactorAdapter.Add` plus a one-line context-timeout bump in
`EventsHandler`. The locked design (CONTEXT.md) is complete and
implementable as-specified. Research adds three concrete refinements:

1. **Slack docs reveal a larger error vocabulary than CONTEXT.md
   enumerates.** Eight terminal codes (`access_denied`, `is_archived`,
   `token_expired`, `no_permission`, `ekm_access_denied`,
   `enterprise_is_restricted`, `org_login_required`,
   `two_factor_setup_required`) and three additional transient codes
   (`accesslimited`, `team_access_not_granted`, `team_added_to_org`) are
   missing from CONTEXT.md's taxonomy. The locked
   `default-unknown→transient` policy absorbs these safely, but
   explicitly enumerating them in `classifyReactionError` is one extra
   switch arm with no risk increase.
2. **An in-repo retry pattern already exists** at `pkg/slack/client.go:406-419`
   that uses `time.NewTimer + select { case <-ctx.Done(); case <-t.C }`
   for context-aware sleeps. The new retry loop should mirror this
   shape exactly — same idiom, same package family, near-zero review
   surface.
3. **No external retry library is needed.** Go stdlib (`math/rand` for
   jitter, `time.NewTimer` for sleeps, `select` for context-cancel)
   covers the entire requirement in ~30 LoC. No `github.com/cenkalti/backoff`
   in go.mod; adding one would be net-negative.

**Primary recommendation:** Hand-roll the retry loop in
`SlackReactorAdapter.Add` mirroring the `PostToBridge` pattern from
`pkg/slack/client.go`. Lift the HTTP-attempt body into a private helper
(`(s *SlackReactorAdapter) doOneAttempt(ctx, token, payload) (resp,
class, error)`) so the loop stays readable. Factor classification into
a pure function `classifyReactionError(httpStatus int, apiErr string,
netErr error) reactionErrorClass` for direct table-driven testability.
Inject `Rand io.Reader` and `Sleep func(time.Duration)` as nil-defaulting
struct fields so jitter tests are deterministic without affecting
production code.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Architecture:**
- Retry lives inside `SlackReactorAdapter.Add`, not at call site.
- `Reactor` interface signature unchanged (back-compat with existing
  fakes/tests: `TestEventsHandler_Reactor_FailureDoesNotBlock`,
  `TestEventsHandler_Reactor_BotLoopSkips`, `TestEventsHandler_Reactor_NilReactorIsNoop`,
  `TestEventsHandler_Reactor_HappyPath`).
- Handler goroutine context: 5s → 10s. Single line change at
  `pkg/slack/bridge/events_handler.go:235`.
- Bridge-only deploy. No sandbox redeploy. No SSM/DDB/profile schema
  changes. No new IAM permissions. No Slack scope additions.

**Error classification (three buckets, evaluated in order):**
- **Success → nil:** HTTP 200 + `ok:true`; HTTP 200 + `error:"already_reacted"`.
- **Terminal — do NOT retry — Error level:** `invalid_auth`, `not_authed`,
  `account_inactive`, `token_revoked`, `missing_scope`.
- **Terminal — do NOT retry — Warn level:** `bad_timestamp`,
  `message_not_found`, `channel_not_found`, `not_reactable`,
  `thread_locked`, `invalid_name`, `too_many_emoji`, `too_many_reactions`.
- **Transient — retry with backoff:** HTTP 429 with `Retry-After`
  (existing `ErrSlackRateLimited`); HTTP 5xx (any 500-599); network
  errors (non-nil `err` from `http.Client.Do`); Slack JSON errors
  `internal_error`, `service_unavailable`, `fatal_error`,
  `request_timeout`, `ratelimited`.
- **Default for unknown error strings:** transient. Cost of one extra
  retry on actually-terminal error is acceptable.

**Retry strategy:**
- Max 3 attempts total (1 initial + 2 retries).
- Backoff schedule: 200ms, then 600ms, each with ±25% jitter.
- `Retry-After` override: if `RetryAfterSeconds: N` and
  `N ≤ remaining context budget` → sleep `N`; if `N > remaining
  budget` → return error immediately.
- Context cancellation respected at every sleep via
  `select { case <-time.After(d): case <-ctx.Done(): }`.
- Total worst-case wall-clock: ~800ms of sleeps + 3 HTTP round-trips,
  comfortably under 10s.

**Log shape:**
- One Warn line on final give-up (preserves existing
  `events: reaction failed` line operators grep for).
- One Debug line per intermediate retry.
- Auth-class terminal errors logged at Error level.
- New structured field `attempt` (1-indexed) on every line.
- Existing fields preserved: `channel`, `ts`, `emoji`, `err`.

**Test strategy:**
- 10 new tests in `pkg/slack/bridge/aws_adapters_test.go` (full list
  in CONTEXT.md `<decisions>` § "Test strategy").
- Existing handler-side tests stay passing.
- Adapter gains a `Rand io.Reader` or `Sleep func(time.Duration)`
  injection point for deterministic jitter under test.

**Deployment:**
- `make build && km init --lambdas` only.
- No `km init --sidecars`. No sandbox redeploy. No state migration to
  undo.

### Pattern 1: Context-aware bounded retry loop (mirror `pkg/slack/client.go:404-419`)

**What:** Loop N attempts. Before each retry (not before attempt 1), sleep
on a `time.Timer` inside a `select` so ctx-cancel cuts retries cleanly.

**When to use:** Any time you need a bounded number of HTTP attempts with
context-aware sleeps in the bridge Lambda.

**Example:**
```go
// Source: pkg/slack/client.go:404-419 (in-repo, verified)
var lastErr error
attempts := len(BridgeBackoff) + 1
for attempt := 0; attempt < attempts; attempt++ {
    if ctx.Err() != nil {
        return nil, ctx.Err()
    }
    if attempt > 0 {
        t := time.NewTimer(BridgeBackoff[attempt-1])
        select {
        case <-ctx.Done():
            t.Stop()
            return nil, ctx.Err()
        case <-t.C:
        }
    }
    // ... do one HTTP attempt, set lastErr if needed, classify, continue or return
}
return nil, lastErr
```

### Pattern 2: Pure classifier function for table-driven tests

**What:** Extract response classification from the I/O path so it can be
tested without spinning up `httptest.Server` instances.

**Example:**
```go
// Recommended new helper in pkg/slack/bridge/aws_adapters.go
type reactionErrorClass int

const (
    classSuccess        reactionErrorClass = iota
    classTerminalAuth   // Error level log
    classTerminalBadInput // Warn level log
    classTransient
    classRateLimited
)

func classifyReactionError(httpStatus int, apiErr string, netErr error) reactionErrorClass {
    if netErr != nil {
        return classTransient
    }
    if httpStatus == http.StatusTooManyRequests {
        return classRateLimited
    }
    if httpStatus >= 500 && httpStatus < 600 {
        return classTransient
    }
    if httpStatus == 200 && (apiErr == "" || apiErr == "already_reacted") {
        return classSuccess
    }
    switch apiErr {
    case "invalid_auth", "not_authed", "account_inactive", "token_revoked",
         "missing_scope", "token_expired", "no_permission", "access_denied",
         "ekm_access_denied", "enterprise_is_restricted", "org_login_required",
         "two_factor_setup_required":
        return classTerminalAuth
    case "bad_timestamp", "message_not_found", "channel_not_found",
         "not_reactable", "thread_locked", "invalid_name",
         "too_many_emoji", "too_many_reactions", "is_archived",
         "deprecated_endpoint", "method_deprecated", "invalid_name",
         "invalid_arg_name", "invalid_arguments", "invalid_charset",
         "invalid_form_data", "invalid_post_type", "missing_post_type",
         "no_item_specified", "not_allowed_token_type", "no_access":
        return classTerminalBadInput
    case "internal_error", "service_unavailable", "fatal_error",
         "request_timeout", "ratelimited", "accesslimited",
         "team_access_not_granted", "team_added_to_org",
         "external_channel_migrating":
        return classTransient
    }
    // Default for unknown error strings — locked policy.
    return classTransient
}
```

### Pattern 3: Test-injectable Sleep and Rand

**What:** Two nil-defaulting struct fields on `SlackReactorAdapter` so
tests assert exact wall-clock with a fake clock and deterministic jitter.

**Example:**
```go
type SlackReactorAdapter struct {
    HTTPClient *http.Client
    BaseURL    string
    Tokens     BotTokenFetcher

    // Sleep, if non-nil, is called instead of <-time.After during backoff.
    // Tests set this to a stub that records calls without actually sleeping.
    // Note: tests still drive ctx cancellation via the real time package.
    Sleep func(d time.Duration)

    // Rand, if non-nil, supplies the jitter PRNG. Tests inject a *rand.Rand
    // with a fixed seed for deterministic backoff durations.
    // Production: nil → use math/rand/v2's default global source.
    Rand *rand.Rand
}
```

Tests inject by assigning these fields after struct construction —
zero impact on production callers who just `&SlackReactorAdapter{...}`.

### Anti-Patterns to Avoid

- **Custom retry library import.** A 30-line stdlib loop is more
  readable than a `backoff.WithContext(backoff.NewExponentialBackOff(),
  ctx)` chain that requires reading the lib's docs to follow.
- **Catching all errors as "transient or terminal" via reflection.**
  A switch on `apiErr` string is faster and clearer; the locked policy
  is `default → transient`, so the switch's default case carries the
  semantics.
- **Calling `crypto/rand` for jitter.** Adds io.Reader complexity, makes
  tests harder to seed.
- **Calling `time.Sleep` directly.** Not ctx-cancellable. Use
  `time.NewTimer` + `select` as `pkg/slack/client.go:412` does.
- **Modifying the `Reactor` interface signature** to surface attempt
  count or last-error-class. Breaks the existing 4 handler-side fakes
  and tests.

## Common Pitfalls

### Pitfall 1: `Retry-After` larger than budget burns the deadline

**What goes wrong:** Slack returns 429 with `Retry-After: 30` and the
loop sleeps 30s, exceeding the 10s handler context. Lambda goroutine
gets killed without ever returning.

**Why it happens:** Naïve `time.Sleep(time.Duration(rl.RetryAfterSeconds)
* time.Second)` ignores `ctx.Deadline()`.

**How to avoid:** Compute remaining budget via
`deadline, ok := ctx.Deadline(); remaining := time.Until(deadline)`.
If `time.Duration(rl.RetryAfterSeconds)*time.Second > remaining`, return
the `ErrSlackRateLimited` immediately without sleeping. This is locked
in CONTEXT.md as `TestReactor_429_RetryAfterExceedsBudget`.

**Warning signs:** A `TestReactor_429_RetryAfterExceedsBudget` failure
where the test takes >2s instead of returning immediately.

### Pitfall 2: First-attempt `ctx.Err()` check misses

**What goes wrong:** If the goroutine context was already canceled
(e.g. handler shutdown), the first attempt still fires an HTTP call
that may even succeed but the response gets discarded after ctx
deadline.

**Why it happens:** `pkg/slack/client.go:407` shows `if ctx.Err() != nil
{ return nil, ctx.Err() }` as the loop's first action **on every
iteration including attempt 0**. Forgetting to check on attempt 0
fires an unnecessary HTTP request.

**How to avoid:** Check `ctx.Err()` at the **top** of the for-loop, not
inside the `if attempt > 0` branch.

**Warning signs:** `TestReactor_ContextCanceled_StopsRetrying` shows
1 HTTP call when 0 is expected.

### Pitfall 3: Jitter PRNG seeded once at init flakes on parallel tests

**What goes wrong:** Tests inject `*rand.Rand` per-test for
determinism. If the adapter has a package-level `var jitter =
rand.New(rand.NewSource(1))`, parallel tests share that state and order
flakes the assertions.

**Why it happens:** Package-level mutable state.

**How to avoid:** Make `Rand` a per-adapter struct field. Production
default: nil → use `math/rand/v2`'s global source (already
goroutine-safe). Tests: assign `adapter.Rand = rand.New(rand.NewSource(42))`
on the local instance only.

**Warning signs:** A jitter-assertion test passes alone but fails with
`-race` or `-count=10`.

### Pitfall 4: `defer resp.Body.Close()` inside the for-loop accumulates defers

**What goes wrong:** `defer` runs at function return, not loop
iteration. Three attempts → three deferred Close calls all firing
together at function return. On a 4xx-Terminal short-circuit, only
the first defer fires; the others never get registered. But on the
3-retry exhaustion path, all three defers stack up.

**Why it happens:** Reusing the existing line `defer resp.Body.Close()`
at `aws_adapters.go:491` inside a loop.

**How to avoid:** Either close explicitly inside the loop body
(`_ = resp.Body.Close()` after `io.ReadAll`) or extract the one-attempt
body into a helper function so each call's defer runs at function
return. CONTEXT.md `<decisions>` lists the helper-function refactor
as Claude's discretion; this pitfall pushes the decision toward "yes,
extract a helper" — `doOneAttempt(ctx, token, body) (*slackAPIResponse,
int, string, error)` returning (response, http-status, retry-after,
network-error).

**Warning signs:** `go vet` doesn't catch this directly; review-time
inspection or a memory-leak test under `httptest.NewServer` with
`Server.Client().CloseIdleConnections()` failing to release sockets.

### Pitfall 5: `slog` log level filtering vs `KM_LOG_LEVEL`

**What goes wrong:** CONTEXT.md says Debug-level intermediate-retry
lines are "silent at default Lambda log level; visible if
`KM_LOG_LEVEL=debug`". Verify the bridge actually wires
`KM_LOG_LEVEL`. If the package-level logger at `handler.go:20` is
`slog.LevelDebug` and the EventsHandler's `Logger` is `slog.Default()`
or a `nil`-fallback, Debug lines either always emit or never emit
depending on which logger the goroutine uses.

**Why it happens:** Two loggers in the same package
(`handler.go:20` package-level; `events_handler.go:55-60` per-handler
with `slog.Default()` fallback) may have different levels.

**How to avoid:** Verify in implementation that
`h.Reactor.Add(...)` uses **the same logger the rest of the EventsHandler
uses**. The retry loop is inside `Add`, which has access only to the
package-level `logger` variable unless we pass one in. **Recommendation:**
The adapter should call the package-level `logger` from `handler.go:20`
(same as other adapters) rather than introduce a `Logger` field on
`SlackReactorAdapter`. This matches the existing adapter pattern
(`SlackPosterAdapter` does not have a `Logger` field).

**Warning signs:** `TestReactor_AuthError_NoRetry_LogsError` cannot
assert log content via the handler's logger; needs `bridge.SetLogger`
override (as `handler_logging_test.go:29` demonstrates).

## Code Examples

Verified patterns from in-repo and Slack docs:

### Single-attempt HTTP body (extracted helper)

```go
// Recommended helper inside aws_adapters.go
// Source: refactor of existing Add() body at aws_adapters.go:475-512
func (s *SlackReactorAdapter) doOneAttempt(
    ctx context.Context, token string, body []byte,
) (apiResp *slackAPIResponse, httpStatus int, retryAfter string, err error) {
    req, err := http.NewRequestWithContext(ctx, "POST",
        s.getBaseURL()+"/reactions.add", bytes.NewReader(body))
    if err != nil {
        return nil, 0, "", err
    }
    req.Header.Set("Authorization", "Bearer "+token)
    req.Header.Set("Content-Type", "application/json; charset=utf-8")

    hc := s.HTTPClient
    if hc == nil {
        hc = http.DefaultClient
    }
    resp, err := hc.Do(req)
    if err != nil {
        return nil, 0, "", err
    }
    defer resp.Body.Close()

    retryAfter = resp.Header.Get("Retry-After")
    b, err := io.ReadAll(resp.Body)
    if err != nil {
        return nil, resp.StatusCode, retryAfter,
            fmt.Errorf("bridge: read reactions.add response: %w", err)
    }
    var r slackAPIResponse
    if err := json.Unmarshal(b, &r); err != nil {
        return nil, resp.StatusCode, retryAfter,
            fmt.Errorf("bridge: decode reactions.add: %w", err)
    }
    return &r, resp.StatusCode, retryAfter, nil
}
```

### Retry loop (replaces current `Add` body)

```go
// Source: composition of pkg/slack/client.go:404-419 + Phase 67.2 design
func (s *SlackReactorAdapter) Add(ctx context.Context, channel, ts, emoji string) error {
    token, err := s.Tokens.Fetch(ctx)
    if err != nil {
        return fmt.Errorf("bridge: get bot token for reactions.add: %w", err)
    }
    body, err := json.Marshal(map[string]any{
        "channel": channel, "timestamp": ts, "name": emoji,
    })
    if err != nil {
        return fmt.Errorf("bridge: marshal reactions.add: %w", err)
    }

    const maxAttempts = 3
    baseDelays := []time.Duration{200 * time.Millisecond, 600 * time.Millisecond}
    var lastErr error

    for attempt := 1; attempt <= maxAttempts; attempt++ {
        if ctx.Err() != nil {
            return ctx.Err()
        }

        resp, status, retryAfter, netErr := s.doOneAttempt(ctx, token, body)
        var apiErr string
        if resp != nil {
            apiErr = resp.Error
        }
        class := classifyReactionError(status, apiErr, netErr)

        switch class {
        case classSuccess:
            return nil
        case classTerminalAuth:
            lastErr = fmt.Errorf("bridge: reactions.add: %s", apiErr)
            logger.Error("events: reaction failed (auth)",
                "channel", channel, "ts", ts, "emoji", emoji,
                "attempt", attempt, "err", lastErr)
            return lastErr
        case classTerminalBadInput:
            lastErr = fmt.Errorf("bridge: reactions.add: %s", apiErr)
            return lastErr
        case classTransient:
            if netErr != nil {
                lastErr = netErr
            } else {
                lastErr = fmt.Errorf("bridge: reactions.add: %s", apiErr)
            }
        case classRateLimited:
            // Reuse the typed error checkRateLimit() builds; we already
            // have status+retryAfter, but to stay consistent with the
            // existing pattern, build the typed error here.
            ra := 1
            if n, e := strconv.Atoi(retryAfter); e == nil && n > 0 {
                ra = n
            }
            rl := &ErrSlackRateLimited{RetryAfterSeconds: ra, Method: "reactions.add"}
            lastErr = rl
            // If RetryAfter exceeds remaining budget, give up immediately.
            if dl, ok := ctx.Deadline(); ok {
                if time.Duration(ra)*time.Second > time.Until(dl) {
                    return rl
                }
            }
            // Otherwise sleep exactly RetryAfter (skip backoff schedule).
            if attempt < maxAttempts {
                if err := s.sleepWithCtx(ctx, time.Duration(ra)*time.Second); err != nil {
                    return lastErr
                }
                continue
            }
        }

        // For Transient (non-RateLimited): sleep next backoff with jitter.
        if class == classTransient && attempt < maxAttempts {
            d := baseDelays[attempt-1]
            d = withJitter(d, s.Rand) // ±25%
            logger.Debug("events: reaction retry",
                "channel", channel, "ts", ts, "emoji", emoji,
                "attempt", attempt, "err", lastErr, "next_delay_ms", d.Milliseconds())
            if err := s.sleepWithCtx(ctx, d); err != nil {
                return lastErr
            }
        }
    }
    // Exhausted.
    logger.Warn("events: reaction failed",
        "channel", channel, "ts", ts, "emoji", emoji,
        "attempt", maxAttempts, "err", lastErr)
    return fmt.Errorf("%w (attempt=%d exhausted)", lastErr, maxAttempts)
}

func (s *SlackReactorAdapter) sleepWithCtx(ctx context.Context, d time.Duration) error {
    if s.Sleep != nil {
        s.Sleep(d)
        return ctx.Err()
    }
    t := time.NewTimer(d)
    defer t.Stop()
    select {
    case <-ctx.Done():
        return ctx.Err()
    case <-t.C:
        return nil
    }
}
```

### Jitter helper (deterministic under test)

```go
// Source: stdlib math/rand/v2 idiom
import randv2 "math/rand/v2"

// withJitter returns d ± 25%. If r is nil, uses math/rand/v2 global source.
func withJitter(d time.Duration, r *rand.Rand) time.Duration {
    var f float64
    if r != nil {
        f = r.Float64() // [0.0, 1.0)
    } else {
        f = randv2.Float64()
    }
    // Map [0.0, 1.0) → [-0.25, 0.25) → d * (1 ± 0.25)
    factor := 1.0 + (f-0.5)*0.5
    return time.Duration(float64(d) * factor)
}
```

## Open Questions

1. **Should `classifyReactionError` also fold in the additional terminal codes Slack docs list?** Research lists ~10 extra codes (`access_denied`, `is_archived`, `token_expired`, `no_permission`, `ekm_access_denied`, `enterprise_is_restricted`, `org_login_required`, `two_factor_setup_required`, etc.) and 3 extra transient (`accesslimited`, `team_access_not_granted`, `team_added_to_org`). CONTEXT.md's locked taxonomy is "complete enough" because the unknown→transient default catches everything; but explicitly enumerating these:
   - What we know: The default-transient policy is safe — an extra retry on a terminal error is cheap.
   - What's unclear: Whether the plan should enumerate (more code, but better grep / observability when a new error type appears) or rely on default-transient (smaller diff).
   - Recommendation: **Enumerate them in the classifier.** The cost is ~10 extra lines in one switch statement; the benefit is that when an operator greps for `auth.test`-style auth failures in CloudWatch, all auth-class codes (including the ones added in research) log at Error, not Warn. Specifically add `token_expired` → terminal-auth: it's an actionable operator condition that bot-token rotation fixes, and burying it as Warn would mask the real root cause.

2. **Should we honor `Retry-After` if it appears in the JSON body rather than the HTTP header?** Slack's `internal_error` / `ratelimited` JSON responses occasionally include a `retry_after` field in the body even on HTTP 200.
   - What we know: `slackAPIResponse` at `aws_adapters.go:260-265` does not parse a `retry_after` field today. `checkRateLimit()` only reads the HTTP header.
   - What's unclear: Whether real-world Slack 200+`ratelimited`-body responses include a retry-after value.
   - Recommendation: **Out of scope.** The HTTP-header path covers true 429s; for `ratelimited` arriving as a 200-body, treat as transient via the default 200ms/600ms backoff. If production data shows this is a real signal, a follow-up phase can extend `slackAPIResponse`.

3. **Does Lambda cold-start actually justify the 5s→10s context bump?**
   - What we know: The bridge's `SSMBotTokenFetcher` has a 15-min in-process cache (`aws_adapters.go:202`). On cold start, `Fetch()` makes one SSM GetParameter call (typically 50-200ms in `us-east-1`). After that, all calls within 15 minutes hit the cache.
   - What's unclear: Whether the cold-start path is the dominant failure mode for the original 5s budget, or whether normal Slack p99 latency + a single retry already fits in 5s.
   - Recommendation: The 10s bump is **insurance for the retry budget**, not for cold-start. Worst case after retries: 800ms sleeps + 3×(token fetch 200ms cached/0ms warm + HTTP ~300ms typical) = ~1.7s. The 5s budget would technically fit, but the bump to 10s adds headroom for genuine Slack-incident slowness (a real Slack 500ms p99 jumping to 3s under incident is plausible). **The 10s budget is correct; cold-start is not the load-bearing reason — it's incident-mode latency.**
