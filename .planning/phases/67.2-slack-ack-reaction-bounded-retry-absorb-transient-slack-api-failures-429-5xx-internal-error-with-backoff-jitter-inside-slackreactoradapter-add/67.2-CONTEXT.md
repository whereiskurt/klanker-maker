# Phase 67.2: Slack ACK reaction bounded retry — Context

**Gathered:** 2026-05-14
**Status:** Ready for planning
**Source:** PRD Express Path (`docs/superpowers/specs/2026-05-14-slack-ack-reaction-bounded-retry-design.md`)

<domain>
## Phase Boundary

Make the Phase 67.1 ACK reaction robust to transient Slack API failures.
Today's single-attempt fire-and-forget in the bridge Lambda
(`pkg/slack/bridge/events_handler.go:228-241`) logs and discards
recoverable failures: HTTP 429 (despite already producing a typed
`ErrSlackRateLimited{RetryAfter}`), HTTP 5xx, network errors, and Slack
JSON errors `internal_error` / `service_unavailable` / `fatal_error` /
`request_timeout`.

The phase adds bounded retry with backoff **inside
`SlackReactorAdapter.Add`** (`pkg/slack/bridge/aws_adapters.go:459`) and
classifies error codes into three buckets (success / terminal-no-retry /
transient-retry). The handler-side `Reactor` interface signature
(`pkg/slack/bridge/events_interfaces.go:73-79`) is unchanged — existing
fakes and tests keep working. The handler-side goroutine context timeout
bumps from 5s → 10s to fit the retry budget; that is the only
handler-side change.

**In scope:**
- Retry loop in `SlackReactorAdapter.Add` (max 3 attempts, exponential
  backoff with jitter, `Retry-After` honoring)
- Error-code classification taxonomy (success / terminal-auth /
  terminal-bad-input / transient / unknown→transient)
- Structured logging at three levels (Error for auth-class, Warn for
  final give-up, Debug for intermediate retries) with new `attempt`
  field
- Handler goroutine timeout bump 5s → 10s
- Test moat covering retry exhaustion, terminal-no-retry, `Retry-After`
  honoring, context cancellation, network errors, jitter determinism
- Bridge-only deploy (`make build && km init --lambdas`)

**Out of scope** (explicitly deferred — see `<deferred>`):
- Retry for `chat.postMessage` and `conversations.archive`
- CloudWatch metric filters and `km doctor` check on ACK failure rate
- Reaction TTL / cleanup
- Slack scope changes, SSM/DDB schema changes, profile-field changes
- Sandbox-side redeploy

</domain>

<decisions>
## Implementation Decisions

### Architecture (locked)

- **Retry lives inside `SlackReactorAdapter.Add`**, not at the call site.
  Reason: classification is intimate with the API response parsing
  (`already_reacted`, `ErrSlackRateLimited`, Slack `error` strings). The
  adapter is already the place that distinguishes these. The handler
  stays a one-line `Reactor.Add(...)` call inside the existing goroutine.

- **`Reactor` interface signature unchanged.** Back-compat with all
  existing fakes/tests (`TestEventsHandler_Reactor_FailureDoesNotBlock`,
  `TestEventsHandler_Reactor_BotLoopSkips`, etc.).

- **Handler goroutine context: 5s → 10s.** Single line change at
  `pkg/slack/bridge/events_handler.go:235`. The goroutine does not block
  the 200 response, so the extra wall-clock is free.

- **Bridge-only deploy.** No sandbox redeploy, no SSM/DDB/profile schema
  changes, no new IAM permissions, no Slack scope additions.

### Error classification (locked)

Three buckets, evaluated in this order inside `Add`:

**Success → return nil:**
- HTTP 200 + `ok:true`
- HTTP 200 + `error:"already_reacted"` (Slack delivered the event
  twice; we already reacted on first delivery). Already implemented in
  Phase 67.1; preserved unchanged.

**Terminal — do NOT retry:**

| Bucket | Slack `error` codes | Log level |
|---|---|---|
| Operator-actionable auth | `invalid_auth`, `not_authed`, `account_inactive`, `token_revoked`, `missing_scope` | **Error** |
| Bad input / unrecoverable | `bad_timestamp`, `message_not_found`, `channel_not_found`, `not_reactable`, `thread_locked`, `invalid_name`, `too_many_emoji`, `too_many_reactions` | **Warn** |

**Transient — retry with backoff:**
- HTTP 429 with `Retry-After` (the existing `ErrSlackRateLimited`)
- HTTP 5xx (any 500-599)
- Network errors: `net.Error` timeout, `io.EOF`, connection reset, DNS
  failure — anything `http.Client.Do` returns as a non-nil `err`
- Slack JSON errors: `internal_error`, `service_unavailable`,
  `fatal_error`, `request_timeout`, `ratelimited`

**Default for unknown error strings:** treat as **transient** (retry).
Safer than hard-failing on a string Slack adds tomorrow. The cost of one
extra retry on an actually-terminal error is acceptable; the cost of
silently ignoring a new transient signal is not.

### Retry strategy (locked)

- **Max 3 attempts total** (1 initial + 2 retries).
- **Backoff schedule:** 200ms, then 600ms, each with **±25% jitter**.
  Jitter de-correlates retries across many sandboxes during a Slack
  incident, avoiding a thundering herd hitting Slack's API in lockstep.
- **`Retry-After` override:** if response is `ErrSlackRateLimited` with
  `RetryAfterSeconds: N`:
  - If `N ≤ remaining context budget`: sleep `N` instead of the backoff
    schedule.
  - If `N > remaining context budget`: return the error immediately
    without sleeping. No point burning the deadline.
- **Context cancellation respected at every sleep.** Each backoff sleeps
  inside `select { case <-time.After(d): case <-ctx.Done(): }`. If the
  10s wall-clock budget or Lambda shutdown fires, return the last error
  immediately.
- **Total worst-case wall-clock:** ~800ms of sleeps + 3× HTTP round-trip
  time, comfortably under 10s for normal Slack latency.

### Log shape (locked)

- One **Warn** line on final give-up. Preserves the existing
  `events: reaction failed` line operators already grep for in
  CloudWatch.
- One **Debug** line per intermediate retry. Silent at default Lambda
  log level; visible if `KM_LOG_LEVEL=debug`.
- Auth-class terminal errors logged at **Error** so they stand out
  against the existing Warn baseline. These require operator action
  (rotate token, re-install app for missing scope) and should not be
  buried.
- **New structured field on every line: `attempt` (1-indexed).** Lets
  future grep queries count retry exhaustion vs first-attempt success.
- Existing structured fields (`channel`, `ts`, `emoji`, `err`) preserved.

### Test strategy (locked)

New tests in `pkg/slack/bridge/aws_adapters_test.go`:

1. `TestReactor_Retries_OnInternalError_ThenSucceeds` — fake server
   returns `internal_error` twice, then `ok:true`. Expect: 3 calls, nil
   error.
2. `TestReactor_RetriesExhausted_ReturnsError` — fake returns
   `internal_error` 4 times. Expect: 3 calls, wrapped error.
3. `TestReactor_TerminalError_NoRetry` — `message_not_found`. Expect: 1
   call, wrapped error.
4. `TestReactor_AuthError_NoRetry_LogsError` — `invalid_auth`. Expect: 1
   call, error log level (assert via captured logger).
5. `TestReactor_429_HonorsRetryAfter` — first response 429 with
   `Retry-After: 1`, second 200. Expect: 2 calls, ~1s wall-clock, nil
   error.
6. `TestReactor_429_RetryAfterExceedsBudget` — `Retry-After: 30` with a
   tightened 2s context. Expect: 1 call, `ErrSlackRateLimited` returned
   (no sleep, no second attempt).
7. `TestReactor_ContextCanceled_StopsRetrying` — cancel ctx during first
   backoff. Expect: 1 call, `ctx.Err()` returned promptly.
8. `TestReactor_NetworkError_Retries` — `httptest` server that closes
   the connection. Expect: 3 calls, then give up.
9. `TestReactor_AlreadyReacted_NoRetry_NoError` — regression guard for
   Phase 67.1 idempotency behavior.
10. `TestReactor_UnknownErrorString_TreatedTransient` — Slack returns
    `error:"some_new_thing"`. Expect: 3 calls (default-transient
    behavior).

Existing handler-side tests stay passing. **Jitter is not
deterministic**, so the adapter gains a `Rand io.Reader` or `Sleep
func(time.Duration)` injection point (today the adapter has no time
injection). Tests seed deterministically.

### Deployment (locked)

Bridge-only change. Mirrors Phase 67.1 deployment pattern exactly:

```bash
make build
km init --lambdas
```

No `km init --sidecars`, no sandbox redeploy. Rollback: revert PR and
`km init --lambdas`. No state migration to undo.

### Claude's Discretion

These were not pinned in the spec — the planner may decide:

- **Where the `Rand` / `Sleep` injection points live** (struct field on
  `SlackReactorAdapter` vs constructor function vs package-level var).
  Recommendation: struct fields nil-defaulting to `crypto/rand`-seeded
  jitter and `time.Sleep` respectively, set in tests only.
- **Whether to factor classification into a helper function or inline.**
  Recommendation: helper function `classifyReactionError(httpStatus int,
  apiErr string, netErr error) reactionErrorClass` for testability.
- **Exact error-wrapping format** for the final give-up. Recommendation:
  preserve the existing `fmt.Errorf("bridge: reactions.add: %s",
  apiResp.Error)` shape so operators grepping for the existing string
  still find it; add `attempt=N exhausted` suffix when retries were
  attempted.
- **Test fixture helpers.** Recommendation: small `recordingTransport`
  RoundTripper that counts calls + sequences responses, used across all
  10 new tests.

<specifics>
## Specific Ideas

### Files modified

- `pkg/slack/bridge/aws_adapters.go` — `SlackReactorAdapter.Add` adds
  retry loop; new helper `classifyReactionError` (if Claude's discretion
  picks that factoring); two new struct fields (`Sleep`, `Rand`) with
  nil-defaulting.
- `pkg/slack/bridge/aws_adapters_test.go` — 10 new tests + shared
  fixture helpers.
- `pkg/slack/bridge/events_handler.go` — single line change at line 235
  (`5*time.Second` → `10*time.Second`).
- `pkg/slack/bridge/events_handler_test.go` — verify existing tests
  still pass; no new tests required at this layer.

### Interface contracts preserved

- `Reactor` interface (`pkg/slack/bridge/events_interfaces.go:73-79`):
  unchanged.
- `ErrSlackRateLimited` (`pkg/slack/bridge/interfaces.go:18-29`):
  unchanged; the retry loop reads its `RetryAfterSeconds` field.
- `checkRateLimit` (`pkg/slack/bridge/aws_adapters.go:333`): unchanged;
  still produces `ErrSlackRateLimited` for 429s.

### Reference patterns

- **Existing fire-and-forget shape:** `PauseHinter` goroutine at
  `events_handler.go:212-221` is the exact pattern this code mirrors.
  Phase 67.1's existing Reactor goroutine (228-241) mirrors PauseHinter
  too. No new concurrency model.
- **Existing retry-style logic:** `checkRateLimit` already returns a
  typed error with retry-after semantics; the retry loop consumes that
  contract, doesn't reinvent it.

### Documentation

- Update `docs/slack-notifications.md` § ACK reaction (Phase 67.1)
  subsection to mention the retry behavior — one paragraph addition.
- Update `CLAUDE.md` § Slack Notifications → ACK reaction subsection
  (currently at line ~215-223) to reference the retry behavior.
- The spec at
  `docs/superpowers/specs/2026-05-14-slack-ack-reaction-bounded-retry-design.md`
  is the authoritative design reference; do not duplicate its content
  in user-facing docs.

<deferred>
## Deferred Ideas

These were discussed during brainstorming and explicitly cut for scope:

- **Retry for `chat.postMessage` and `conversations.archive`.**
  Different blast radius — `chat.postMessage` failures are already
  visible to the user (their reply doesn't land), and `archive` is a
  one-shot lifecycle path triggered by `km destroy` where the operator
  already sees the error. The existing pre-Phase-74
  `TestKmSlackPost_BridgeReturns503ThenSuccess_Exit0` deferred item
  (see Phase 74 `deferred-items.md`) hints at related work. A future
  phase can revisit if production data shows recurring failures.

- **CloudWatch metric filters + `km doctor` check on ACK failure rate.**
  Discussed and cut. A follow-up could add
  `slack_ack_reaction_failure_rate` reading CloudWatch Metric Filters,
  if recurring failures justify the observability investment. Today's
  Warn-line grep is sufficient to start.

- **Reaction TTL / cleanup.** The 👀 stays on the message forever today;
  not a robustness concern. Slack itself does not expose reaction-aging
  APIs.

- **Feature flag for old behavior.** Not added. The retry behavior is
  strictly more reliable than the current single-attempt; there is no
  caller that would prefer the old behavior. Rollback is via PR revert
  + `km init --lambdas`.

</deferred>
