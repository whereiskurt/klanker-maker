## Deviations from Plan

### Deviation 1 — Bridge Lambda deploy requires full `km init`, not `km init --lambdas`

- **Found during:** Task 3 operator UAT (2026-05-15 deploy cycle).
- **Issue:** The plan, the design spec, and the Phase 67.2 paragraphs Tasks 1-2 just wrote into `docs/slack-notifications.md` and `CLAUDE.md` all say the deploy step is `make build && km init --lambdas`. In practice the `--lambdas` partial path does NOT upload the bridge zip — it only builds zips locally and pokes the create-handler env. The bridge zip is deployed via the `terragrunt apply` against the `lambda-slack-bridge` module that runs as part of full `km init`. Until full `km init` runs, the retry loop sits unused in S3 / locally and the deployed bridge keeps the Phase 67.1 single-attempt code.
- **Operator workaround:** ran full `km init` (after the Terraform lock-file remediation noted below). The bridge zip uploaded, the redeployed Lambda now contains the retry loop, smoke-test 👀 reaction confirmed live.
- **Impact:** Phase 67.2 deploy succeeded — the retry loop IS in production. But the documentation accuracy issue persists in the just-shipped docs. Worth a one-line follow-up to correct the deploy guidance in both `docs/slack-notifications.md` § Phase 67.2 retry behavior and `CLAUDE.md` § ACK reaction (Phase 67.1) → Phase 67.2 paragraph. Not blocking — the retry loop landed and works.
- **Disposition:** Documentation accuracy issue noted for a future doc-tightening pass. Not auto-fixing in this plan because (a) the deliverable (live 👀 reaction with retry loop behind it) was achieved, (b) the operator already approved the UAT outcome, and (c) the fix is a clean swap of `km init --lambdas` → `km init` in two paragraphs and one design-spec paragraph — small enough to defer to whoever next touches the surrounding docs.

### Deviation 2 — Terraform lock-file drift blocked the initial deploy attempt

- **Found during:** Task 3 operator UAT (2026-05-15 deploy cycle, first workstation).
- **Issue:** Phase 80 (commit `c578db1`) added `hashicorp/tls` to `root.hcl`'s `required_providers`, but pre-Phase-80 modules still had `.terraform.lock.hcl` files that pre-date the addition. Running `km init --lambdas` (and full `km init`) on the first workstation failed with a Terraform provider-lock mismatch on the `tls` provider.
- **Operator workaround:** moved to a separate workstation, ran `terragrunt run --all --queue-exclude-dir 'sandboxes/**' init -- -upgrade` to refresh the lock files, then re-ran full `km init` cleanly. The lambda-slack-bridge module uploaded the new zip with the retry loop, and the live Slack smoke test confirmed 👀.
- **Impact:** No Phase 67.2 code is affected. This is a Phase 80 follow-up issue (every operator running `km init` on a workstation initialized before Phase 80 will hit this).
- **Disposition:** logged to `deferred-items.md` as a Phase 80 follow-up — `km init` should pass `-upgrade` to terraform automatically (or detect lock drift and remediate) so adding a new provider to `root.hcl` doesn't break the next operator's init. **Out of Phase 67.2 scope.**
