<objective>
Inject Claude Code OpenTelemetry environment variables into both EC2 user-data and ECS container definitions, driven by the profile's claudeTelemetry settings.

Purpose: This is the core integration — making Claude Code inside sandboxes actually emit telemetry to the local OTel Collector. The env vars tell Claude Code to enable telemetry, use gRPC OTLP to localhost:4317, and optionally log prompts and tool details. Resource attributes tag every signal with sandbox_id, profile_name, and substrate for downstream filtering.

Output: Updated compiler templates that conditionally inject OTEL env vars, plus test coverage proving the env vars appear (or don't) based on profile settings.
</objective>

2. Add a new template section in the user-data bash script, after the profile env vars section (section 2.8) and before secrets (section 3). Add as section "2.9. Claude Code OTEL environment":

```bash
# ============================================================
# 2.9. Claude Code OpenTelemetry environment
# ============================================================
{{- if .ClaudeTelemetryEnabled }}
cat >> /etc/profile.d/km-profile-env.sh << 'CLAUDE_OTEL'
export CLAUDE_CODE_ENABLE_TELEMETRY=1
export OTEL_METRICS_EXPORTER=otlp
export OTEL_LOGS_EXPORTER=otlp
export OTEL_EXPORTER_OTLP_PROTOCOL=grpc
export OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4317
{{- if .ClaudeTelemetryLogPrompts }}
export OTEL_LOG_USER_PROMPTS=1
{{- end }}
{{- if .ClaudeTelemetryLogToolDetails }}
export OTEL_LOG_TOOL_DETAILS=1
{{- end }}
CLAUDE_OTEL
cat >> /etc/profile.d/km-profile-env.sh << EOF
export OTEL_RESOURCE_ATTRIBUTES="sandbox_id={{ .SandboxID }},profile_name={{ .ProfileName }},substrate={{ .Substrate }}"
EOF
echo "[km-bootstrap] Claude Code OTEL telemetry configured"
{{- end }}
```

Note: OTEL_RESOURCE_ATTRIBUTES uses the non-quoted heredoc (EOF not 'EOF') because it needs template interpolation for SandboxID, ProfileName, Substrate. The static env vars use 'CLAUDE_OTEL' (quoted) since they have no templates.
