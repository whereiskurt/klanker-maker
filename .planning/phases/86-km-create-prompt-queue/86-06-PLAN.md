       **R1 — Regression: km create without --prompt produces an idle-but-installed queue:**

       Per CONTEXT.md (revised 2026-05-19) the runner script + systemd unit are installed UNCONDITIONALLY on every EC2 sandbox via inline `userdata.go` heredoc seeding (the `configFiles` alternative was rejected because it `chown -R sandbox:sandbox`s system directories). The unit IS `enabled` and auto-starts on every boot; the runner's main loop sleep-polls for `pending` queue entries that — without `--prompt` — never arrive. "Regression" therefore means: NO queue directory + NO real prompt activity (no `runs/<ts>/output.json`), runner sleep-loop is negligible CPU. It does NOT mean "unit absent."

       ```bash
       ./km create profiles/learn.yaml
       SB=<sandbox-id>
       ./km shell $SB
       # Inside sandbox:
       ls -la /workspace/.km-agent/queue/ 2>&1                  # ENOENT — no queue directory (R1 critical)
       systemctl is-enabled km-queue.service                    # "enabled" (unit installed via userdata.go heredoc)
       systemctl is-active km-queue.service                     # "active" (running) — runner is sleep-polling for pending entries
       systemctl status km-queue.service --no-pager | head -20  # main PID exists; idle in sleep loop
       ls -la /workspace/.km-agent/runs/ 2>&1                   # empty or ENOENT — no real prompt activity (R1 critical)
       tail /workspace/.km-agent/km-queue.log 2>&1              # may show "no pending entries" probe messages; no runs
       top -bn1 -p $(systemctl show -p MainPID --value km-queue.service) | tail -2  # %CPU near 0
       ```
       Expected:
       - `/workspace/.km-agent/queue/` directory is ABSENT (R1 critical — operator-side push never ran).
       - `/workspace/.km-agent/runs/` is empty or absent (R1 critical — no real prompt activity).
       - `systemctl is-enabled km-queue.service` returns `enabled` (unit installed via userdata.go heredoc — Phase 86 design).
       - `systemctl is-active km-queue.service` returns `active` (running) — the runner main loop is idle-polling for `pending` entries that never arrive. This is the expected idle state per the locked CONTEXT.md decision (`Restart=on-failure` + runner sleep-loop). The OLD R1 expectation of `inactive (dead)` is OBSOLETE — that would only hold if the unit were not enabled, which would defeat the pause/resume reconcile design.
       - Runner process exists with %CPU near 0 — bash `sleep $PROBE_INTERVAL` loop, negligible footprint.
       - Otherwise the sandbox is functionally identical to pre-Phase-86: existing test suites, `km agent run`, `km shell` all behave unchanged.
