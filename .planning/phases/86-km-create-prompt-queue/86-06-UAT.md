## Scenario PQ-12 — Pause/resume reconcile

**Goal:** Verify that a `running` queue entry is reconciled back to `pending` after a pause+resume cycle, and the runner restarts cleanly to continue the queue.

### Commands

```bash
./km create profiles/learn.yaml --prompt "sleep 300; echo done"
SB=<sandbox-id>

# Wait 30 seconds for the runner to pick up entry 001 and mark it running
sleep 30
./km agent list $SB --queue   # Expect: 001 status=running

# Pause (hibernate) the sandbox
./km pause $SB
./km list | grep $SB          # Expect: status=stopped or paused

# Resume the sandbox
./km resume $SB
sleep 60                      # Wait for systemd to start km-queue.service + reconcile

./km agent list $SB --queue   # Expect: 001 reconciled to pending (or running again)

# Wait for sleep 300 to complete (second execution after reconcile, ~5 minutes from resume)
# Then re-check:
sleep 360
./km agent list $SB --queue   # Expect: 001 status=done
```

### Expected

- Before pause: `km agent list $SB --queue` shows **001 status=running**.
- After `km pause $SB`: `km list` shows sandbox as `stopped`/`paused`.
- After `km resume $SB` + 60s: **001 status resets to `pending`** (reconcile step fired on runner restart) or is already `running` again.
- Eventually (after `sleep 300` completes its second execution): **001 status=done**.
- The runner auto-starts on resume via `systemctl enable km-queue` (no manual `systemctl start` needed).

## Scenario PQ-13 — Direct-API indefinite auth wait

**Goal:** Verify that with `--no-bedrock`, the queue stays `pending` indefinitely until `claude auth login` is completed inside the sandbox, then drains immediately.

### Commands

```bash
# Create sandbox with --no-bedrock (no Bedrock credentials; direct Claude API)
./km create profiles/learn.yaml --no-bedrock --prompt "What model are you running?"
SB=<sandbox-id>

# Wait 30 seconds — runner should be in auth-probe loop (waiting for credentials.json)
sleep 30
./km agent list $SB --queue   # Expect: 001 status=pending (auth probe looping)

# Open an interactive shell into the sandbox
./km shell $SB

# --- Inside the sandbox shell: ---
ls -la /home/sandbox/.claude/.credentials.json   # Expect: file absent
tail /workspace/.km-agent/km-queue.log           # Expect: "waiting for auth" probe messages
claude auth login                                # Complete the OAuth browser flow
exit
# --- Back on the operator machine: ---

sleep 10
./km agent list $SB --queue   # Expect: 001 status=done (queue drained immediately after auth)
```

- Inside the sandbox: `/home/sandbox/.claude/.credentials.json` is **absent** before `claude auth login`.
- `km-queue.log` shows periodic probe messages (e.g., "waiting for auth" or equivalent log lines every ~5 minutes).
- After `claude auth login` completes the OAuth flow and `credentials.json` is written: within ~10 seconds, the runner detects credentials and proceeds.

## Scenario R1 — Regression: km create without --prompt produces idle-but-installed queue

**Goal:** Verify that `km create` WITHOUT `--prompt` provisions a sandbox that is functionally identical to pre-Phase-86: no queue directory, no run activity, but `km-queue.service` IS installed and enabled (idle-polling, no CPU cost).

### Commands

```bash
./km create profiles/learn.yaml
SB=<sandbox-id>

./km shell $SB

# --- Inside the sandbox shell: ---
ls -la /workspace/.km-agent/queue/ 2>&1              # Expect: ENOENT (no queue dir)
ls -la /workspace/.km-agent/runs/ 2>&1               # Expect: empty or ENOENT (no run activity)
systemctl is-enabled km-queue.service                # Expect: "enabled"
systemctl is-active km-queue.service                 # Expect: "active" (runner main loop is idle-polling)
systemctl status km-queue.service --no-pager | head -20  # Expect: active (running), main PID shown
tail /workspace/.km-agent/km-queue.log 2>&1          # Expect: probe messages or no file (no entries to run)
top -bn1 -p $(systemctl show -p MainPID --value km-queue.service) | tail -2  # Expect: %CPU near 0
exit
# --- Back on the operator machine: ---
```

### Expected

- `/workspace/.km-agent/queue/` directory is **ABSENT** — the operator-side push path was never invoked (R1 critical pass criterion).
- `/workspace/.km-agent/runs/` is **empty or absent** — no real prompt activity (R1 critical pass criterion).
- `systemctl is-enabled km-queue.service` returns **`enabled`** — unit is installed unconditionally via userdata.go heredoc (Phase 86 design; unit installed on every EC2 sandbox).
- `systemctl is-active km-queue.service` returns **`active`** — runner main loop is running, idle-polling for `pending` entries that will never arrive.
- `systemctl status km-queue.service` shows a main PID (process exists), `Active: active (running)`.
- Runner process CPU usage is **near 0%** — bash `sleep $PROBE_INTERVAL` loop; negligible footprint.
- `km agent list $SB`, `km shell $SB`, and other pre-Phase-86 commands all work normally — no behavior change from the operator's perspective.

**Note on expected active state:** `systemctl is-active km-queue.service` returning `active` is CORRECT per the Phase 86 design. The runner uses `Restart=on-failure` and an internal sleep-poll loop — it stays running indefinitely when there are no queue entries. An `inactive (dead)` result would indicate the unit is not properly enabled/installed, which would be a FAIL.
