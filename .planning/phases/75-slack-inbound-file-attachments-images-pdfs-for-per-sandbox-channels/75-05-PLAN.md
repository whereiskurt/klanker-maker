<objective>
Wire the Plan 02 `S3FileDownloader` into the bridge Lambda's cold-start initialization in `cmd/km-slack-bridge/main.go`. This is the small but critical glue step that takes the in-package downloader implementation from Plan 02 and makes it actually consumed at runtime by the deployed Lambda.

Purpose: Without this wiring, `EventsHandler.FileDownloader` is nil at runtime, so the Plan 02 fork in `Handle` falls through to the (intentional) "text-only fallback with Warn log" — i.e., file_share events would reach the bridge, get past `isBotLoop`, and silently dispatch as text-only with no attachments. The unit tests in Plan 02 use injected mocks; this plan does the production injection.

Output: ~15-25 LoC added to `cmd/km-slack-bridge/main.go` cold-start block. No new file. Verify via `make build` (the bridge zip rebuilds cleanly; ldflags-stamped). The actual deployment happens in Plan 06 when the operator runs `km init`.

Why a separate plan: Plan 02 builds the downloader and its test moat against a mock S3 + mock HTTP. Plan 04 provisions the IAM and memory the runtime needs. This plan is the connector and depends on both. Splitting it out keeps Plan 02 focused on testable in-package behavior and keeps Plan 04 focused on infrastructure. The wiring task is small enough to be one task in one plan.
</objective>
