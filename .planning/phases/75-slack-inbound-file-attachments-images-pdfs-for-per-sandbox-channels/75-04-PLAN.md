<objective>
Wire the platform-side prerequisites that make the bridge actually able to PUT files into S3 and let `km slack init` / `km doctor` enforce the new scope. Three parallel sub-changes:
1. **Bridge IAM + memory bump**: extend `infra/modules/lambda-slack-bridge/v1.0.0/main.tf` with a new IAM policy granting `s3:PutObject` on `slack-inbound/*` AND bump `memory_size` from 256 → 1024 (Pitfall 2: PutObject buffers full file in memory for retry-rewindability)
2. **S3 lifecycle**: new tiny Terraform module `infra/modules/s3-artifacts-lifecycle/v1.0.0/` providing a 30-day expiration rule on the `slack-inbound/` prefix; wired into `infra/live/use1/` like other regional modules
3. **Scope checks**: one-line additive change to `VerifyEventsAPIScopes` and `checkSlackAppEventsScopes` to require `files:read`; one new unit test for each

Purpose: Without IAM, Plan 02's `s3.PutObject` calls in the bridge return 403. Without the memory bump, 100MB file downloads OOM the 256MB Lambda. Without lifecycle, S3 storage grows indefinitely (CONTEXT.md locks 30-day matching DDB TTL). Without scope checks, `km slack init` accepts a stale token that's missing `files:read` and the bridge silently fails at runtime with 401s from files.slack.com.

These three sub-changes are independent and can be done in parallel WITHIN this plan (single executor, but no internal dependency ordering). They're grouped into one plan because each is small and they share a single deployment path (operator runs `make build && km init`) — Plan 06's UAT exercises all three.
</objective>
