<decisions>
## Implementation Decisions (LOCKED 2026-06-14)

### Link anchors
- **Native links only.** The `markdown` block renders `[label](url)` as clickable anchors natively;
  bare `https://` URLs use Slack's default auto-link (full URL shown). 
- Do **NOT** auto-anchor bare URLs with derived text. Do **NOT** build a Sources footer in v1.
- This satisfies the operator's "leverage anchors for clicking" requirement for free.

### Renderer architecture
- New `renderRich(input) -> (blocksJSON, fallbackText, ok)` in `pkg/slack/blocks.go`, selected by
  `KM_SLACK_RENDER=blocks-rich`.
- **Prose segments â†’ `markdown` blocks** carrying verbatim GFM (native bold/italic/headings/lists/code
  + native link anchors). Honor the **12,000-char cumulative** markdown-block cap by chunking across
  multiple `markdown` blocks.
- **Leading H1 still promoted to a real `header` block** (markdown-block headings all render at the same
  size, so hierarchy is otherwise lost). Keep this from today's renderer.
- **đź”§ tool lines stay `context` blocks** (unchanged from today's Tier-2).
- **Plain `text` fallback unchanged** â€” still required for email notifications / search / push. Keep the
  existing `stripForFallback`/`buildFallback` logic.

### Table transformer
- **GFM pipe tables â†’ dedicated `table` blocks** (NOT pipe-tables-inside-a-markdown-block, which is
  unreliable across surfaces). New GFM-tableâ†’table-block transformer.
- Column alignment derived from the delimiter row (`:--` left, `:-:` center, `--:` right).
- Header row â†’ bold `rich_text` cells (the v1 `table` block has no explicit header-row flag).
- Body cells: `raw_number` (pure numeric) / `rich_text` (inline formatting present) / `raw_text` (plain).
- **Guards:** â‰¤20 columns, â‰¤100 rows, pad ragged rows to the column count. Table cells are NOT markdown â€”
  code spans / nested lists / multi-line content inside a cell degrade to `raw_text`.
- **Table-size policy:** cap at **one** `table` block; tables exceeding 100 rows / 20 cols **fall back to
  today's monospace reflow** (existing `fencePipeTables` path). Claude tables are almost always tiny.
- Reuse the existing pipe-table **detection** (we already detect+reflow tables today); only the output
  representation changes.

### AI-disclaimer footer
- **No footer by default.** Expose `KM_SLACK_AI_FOOTER` flag (default off / `"false"`), settable per-profile
  via `/etc/km/notify.env`. When true, append a "Generated by AI â€” verify before sharing" `context` block.
- Not on by default for anyone (operator-initiated status posts stay clean).

### Rollout / fallback
- **Opt-in first.** Default render stays `blocks` (today's Tier-2). `blocks-rich` is reached only by
  setting `KM_SLACK_RENDER=blocks-rich`. Default-path golden fixtures do NOT move in this phase.
- **Fallback chain intact:** `renderRich` â†’ on `ok=false` â†’ existing Tier-2 `RenderBlocks` â†’ Tier-1
  `Mrkdwnify`. Existing tiers are **DEMOTED to fallback, NOT deleted.**
- Keep the fail-soft `recover()` wrapper so a transformer panic falls back rather than dropping the post.

### Streaming
- **Deferred.** Keep today's per-turn discrete-post model. No `chat.startStream`/feedback-buttons work.

### Tests
- Unit coverage for `renderRich` + the table transformer (alignment, cell-type selection, guard/overflow
  fallback, 12K chunking, fail-soft recover).
- New golden fixtures for the **`blocks-rich` tier ONLY**. Do not modify default-path (`blocks`) goldens.

### Docs
- `docs/slack-notifications.md` Â§ Phase 111 (render modes, `markdown`/`table` block adoption,
  `KM_SLACK_AI_FOOTER`, surface caveats).
- `klanker:slack` skill render-mode section update + plugin version bump (per the plugin-version cache gate).

### Deploy
- Ship the `km-slack` sidecar binary via `km init --sidecars`. Existing sandboxes pick up the new renderer
  on `km destroy && km create` (or sidecar refresh). No Lambda/TF/DDB deploy needed.
