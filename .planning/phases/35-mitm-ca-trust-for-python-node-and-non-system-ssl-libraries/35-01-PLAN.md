# Phase 35-01: MITM CA Trust for Python, Node, and Non-System SSL Libraries

## Problem

The HTTP proxy sidecar does MITM interception for Bedrock/Anthropic token metering and GitHub repo filtering. The platform CA cert is installed into the system trust store (`update-ca-trust`), which covers tools using system OpenSSL (curl, wget, git). But several languages and tools bundle their own CA stores and ignore the system trust store:

- **Python** (`requests`, `urllib3`, `httpx`): Uses `certifi` which ships its own `cacert.pem`
- **pip**: Uses `certifi` internally for package downloads
- **Node.js**: Uses a compiled-in CA store, ignores system CAs
- **Go**: Uses system CAs on Linux by default (works), but `GOPROXY` traffic may fail depending on build
- **Rust** (`reqwest`, `ureq`): Uses `webpki-roots` or `native-tls` depending on build

These tools will reject MITM-intercepted HTTPS connections with `SSLCertVerificationError` or equivalent.

## Solution

Set environment variables that override the bundled CA stores, pointing them to the system's merged CA bundle (which includes the km proxy CA after `update-ca-trust`):

| Variable | Consumed By |
|----------|-------------|
| `SSL_CERT_FILE` | OpenSSL convention (Python `ssl` module, many Rust crates) |
| `REQUESTS_CA_BUNDLE` | Python `requests` library (overrides certifi) |
| `CURL_CA_BUNDLE` | curl (redundant with system store, but explicit) |
| `NODE_EXTRA_CA_CERTS` | Node.js (appends to compiled-in store) |

The merged bundle path depends on the distro:
- Amazon Linux 2023 / RHEL: `/etc/pki/tls/certs/ca-bundle.crt`
- Debian / Ubuntu: `/etc/ssl/certs/ca-certificates.crt`

## Changes

### File: `pkg/compiler/userdata.go`

**Change 1:** After the proxy CA is installed in the system trust store (the `update-ca-trust` / `update-ca-certificates` block), detect which bundle path exists and export the CA env vars in `/etc/profile.d/km-audit.sh`.

**Change 2:** Also set the vars in the systemd environment for sidecar services, so child processes spawned by the sandbox user pick them up regardless of shell profile sourcing.
