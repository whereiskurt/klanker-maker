# Phase 36: km-sandbox Base Container Image - Context

**Gathered:** 2026-03-30
**Status:** Ready for planning

<domain>
## Phase Boundary

A `km-sandbox` container image + entrypoint script that replaces EC2 user-data, serving as the main container for Docker (Phase 37) and EKS (Phase 38) substrates. The image provides the same sandbox environment as EC2 — proxy CA trust, secret injection, GitHub credentials, initCommands, rsync restore, OTEL telemetry, mail polling, and artifact upload — all driven by environment variables via a container entrypoint.

</domain>

<decisions>
## Implementation Decisions

### Base image & distro
- Amazon Linux 2023 minimal (`amazonlinux:2023-minimal`) — matches EC2 AMI exactly, same package manager (dnf), same CA trust paths (`/etc/pki/`), same package names
- Minimal base + initCommands model — image has OS essentials (git, jq, tar, python3, AWS CLI v2, gosu, ca-certificates, sandbox user). Profile-specific tools (Claude Code, Goose, Node.js) installed by initCommands at container start
- No SSM agent — `km shell` uses `docker exec` (Docker) or `kubectl exec` (EKS). No SSM IAM permissions needed. Port forwarding via compose ports / kubectl port-forward
- Shell script helpers — mail poller and artifact upload as bash scripts, not Go binaries. AWS CLI does the heavy lifting. Simple and debuggable

### Entrypoint design
- Bash shell script (`entrypoint.sh`) — direct port of EC2 user-data logic. Each section is a function. Easy to read, debug (`bash -x`), and modify
- Categorized failure handling — critical steps (CA trust, secrets) abort on failure; optional steps (rsync, mail poller, GitHub) warn and continue. Sandbox boots in degraded mode rather than failing completely
- gosu for user drop — `exec gosu sandbox /bin/bash` at the end. Proper PID 1 handoff, no TTY issues, no signal forwarding problems
- SIGTERM trap for graceful shutdown — entrypoint traps SIGTERM/SIGINT, runs artifact upload, then exits. Docker compose sends SIGTERM on `docker compose down` with 30s grace period

### Build & distribution
- Source lives in `containers/sandbox/` — new top-level `containers/` directory alongside `sidecars/`. Room for future images (e.g., `containers/reaper/` for EKS)
- `make sandbox-image` Makefile target + integrated into `make ecr-push` — same pipeline as sidecar images, pushed to ECR alongside them
- Same VERSION file as km CLI — `km-sandbox:v0.0.55` matches `km` CLI and sidecar versions. One version number for the whole platform. `latest` tag always points to most recent
- Auto-build on first use — `km create --substrate docker` checks for `km-sandbox:latest` locally; if missing, builds it automatically from `containers/sandbox/`

### Credential delivery
- Credential refresh sidecar + shared tmpfs volume (Docker substrate) — a `km-cred-refresh` container is the ONLY container with operator credentials. It calls AssumeRole on the sandbox/sidecar roles every 45 min, writes session creds to a tmpfs volume. Other containers read via `AWS_SHARED_CREDENTIALS_FILE`
- IRSA on EKS (Phase 38) — Kubernetes handles token rotation via projected service account tokens. No refresh sidecar needed on EKS
- Two IAM roles, not one:
  - **Sandbox role** (main container): Bedrock invoke, SSM secrets, SES email, S3 artifacts, EventBridge. Region-locked
  - **Sidecar role** (http-proxy, audit-log, tracing): DynamoDB budget table, S3 audit/OTEL prefixes. Region-locked
- Entrypoint is substrate-agnostic — reads AWS_* env vars regardless of where they came from (refresh sidecar, IRSA, or direct injection)

### Claude's Discretion
- Exact entrypoint function ordering and error messages
- AWS CLI install method in Dockerfile (zip vs pip)
- gosu version selection
- Dockerfile layer optimization and caching strategy
- tmpfs volume size for credential file

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `sidecars/*/Dockerfile` — existing multi-stage Go build pattern (builder + scratch). Sandbox image follows similar structure but uses AL2023 instead of scratch
- `Makefile` — `ecr-push`, `ecr-login`, `ecr-repos` targets. Sandbox image adds to these
- `pkg/compiler/userdata.go` (lines 26-666) — the EC2 user-data shell script. Entrypoint.sh is a direct port of this logic, section by section
- `pkg/compiler/service_hcl.go` — ECS template with `MAIN_IMAGE_PLACEHOLDER` that will be replaced with the real ECR URI

### Established Patterns
- Sidecar images built via `docker buildx --platform linux/amd64` and pushed to `{ACCOUNT}.dkr.ecr.{REGION}.amazonaws.com/km-{name}:{version}`
- VERSION file at repo root controls all version tags
- `km init` builds and pushes sidecars to ECR as part of regional initialization
- CA cert generated by `km init` and stored in S3 (`sidecars/km-proxy-ca.crt`)

### Integration Points
- `pkg/compiler/service_hcl.go:132` — `MAIN_IMAGE_PLACEHOLDER` needs to be replaced with `km-sandbox:{version}` ECR URI
- `pkg/compiler/compiler.go:221` — `UserData: ""` for ECS path — needs entrypoint env vars instead
- `internal/app/cmd/init.go` — add sandbox image push alongside sidecar pushes
- `Makefile:ecr-repos` — add `km-sandbox` to the repository list

</code_context>

<specifics>
## Specific Ideas

- Entrypoint should mirror EC2 user-data section numbering and comments for easy cross-reference
- The `[km-entrypoint]` log prefix (matching `[km-bootstrap]` on EC2) for all entrypoint output
- Critical vs optional step categorization: CA trust and secrets are critical (agent can't function without Bedrock access or API keys); rsync, mail, GitHub are optional (sandbox still usable without them)

</specifics>

<deferred>
## Deferred Ideas

- Fat image variant with pre-installed agents (Claude Code, Goose, Node.js) for faster boot — potential future optimization after minimal image proves out
- SSM agent inclusion for unified `km shell` experience — revisit if docker exec / kubectl exec proves insufficient
- Go binary entrypoint for better error handling — revisit if shell script becomes unwieldy

</deferred>
