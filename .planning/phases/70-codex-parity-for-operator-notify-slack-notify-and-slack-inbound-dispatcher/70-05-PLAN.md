<objective>
Extend the Phase 67 Slack inbound poller (`km-slack-inbound-poller` bash heredoc at `pkg/compiler/userdata.go` lines ~1279-1585) to dispatch either `claude -p` (existing path, unchanged) or `codex exec` (new path) based on the effective agent, AND extend the DDB write-back to always carry `agent_type` and `last_assistant_msg`.

Per CONTEXT.md locked decisions:
- Boot-time read of `KM_AGENT` from `/etc/km/notify.env` (default `claude`)
- Claude dispatch UNCHANGED: `claude -p "$PROMPT" --output-format json --dangerously-skip-permissions [--resume $SESSION]`
- Codex first-turn: `codex exec --json --dangerously-bypass-approvals-and-sandbox "$PROMPT"`
- Codex resume: `codex exec resume $SESSION "$PROMPT" --json --dangerously-bypass-approvals-and-sandbox` (SUBCOMMAND form, NOT `--resume` flag — per Plan 70-00 spike)
- Codex session-ID extraction via hook-file: km-notify-hook's Stop branch writes session_id to `/tmp/km-codex-session.$KM_CODEX_RUN_ID` when `KM_CODEX_RUN_ID` is non-empty
- DDB writer ALWAYS sets `agent_type` + `last_assistant_msg` (truncated to 2000 chars); reader defaults `agent_type=claude` when absent (backward compat)

**Critical gotcha to enforce (70-RESEARCH.md Pitfall 2):** `last_assistant_msg` content may contain literal `"`, `\`, and `\n` characters from agent replies. Embedding it directly into `aws dynamodb put-item --item '{...}'` JSON via shell-string interpolation will produce malformed JSON or data corruption. The implementation MUST use `jq -Rs .` to JSON-encode the truncated string, and the surrounding DDB JSON must NOT add extra quotes around the jq result. This plan's Task 2 includes an explicit `TestPoller_JQEscaping_RoundTrip` test that injects a synthetic reply containing all three characters and asserts the DDB round-trip works.

**Prefix routing and cross-agent thread switching are NOT in this plan** — Plan 70-06 handles those. This plan delivers just the dispatch fork and the writeback. Plan 70-06 builds on this plan's `EFFECTIVE_AGENT` variable.

This plan does require Plan 70-03's `KM_CODEX_RUN_ID` handling in the km-notify-hook bash heredoc. That edit can land here (since the hook script is in the same file as the poller), but to avoid two-plan ownership of the hook script:
- **Plan 70-03 (already in Wave 2) implements the hook-side write** to `/tmp/km-codex-session.$KM_CODEX_RUN_ID` if `KM_CODEX_RUN_ID` is set in the hook's env. Add this as a small addendum to Plan 70-03's Task 2 if not already there.
- **Plan 70-05 (this plan) implements the poller-side read** and the `KM_CODEX_RUN_ID` inline-export pattern.

If Plan 70-03 did NOT include the hook-side write, this plan's Task 2 adds it as a focused 5-line bash edit to the Stop branch of the hook script.
