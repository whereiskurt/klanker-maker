# Plan 70-11 — Path B agent-run JSONL parse + synthetic Stop hook

## Why

Plan 70-09 UAT discovered that `km agent run --codex` does NOT notify (no Slack post, no email) under Path B. Reason: Plan 70-03 added Codex-aware branches to `/opt/km/bin/km-notify-hook` (PermissionRequest + Stop fast-path with `last_assistant_message`), but Codex 0.121/0.133 do NOT fire user-defined hooks from `~/.codex/config.toml` (Plan 70-00 spike v1/v2/v3 confirmed). Plan 70-10's Path B fix only addressed the POLLER path (Slack inbound dispatch). The operator-side agent-run path was left depending on hooks that don't fire.

This plan closes the gap by parsing the JSONL stream `codex exec --json` writes to `output.json` and synthesizing the Stop hook payload that Plan 70-03 already knows how to consume.

## Goal

After `km agent run --codex --prompt "..." --wait` completes on a Codex sandbox, an operator email AND a Slack post in `#sb-{sandbox}` land within seconds — same as what would happen for Claude via Claude Code's actual Stop hook.

## Change scope

`internal/app/cmd/agent.go` `BuildAgentShellCommands` — append a small bash block to the **codex branch** of the agent script (NOT to the claude branch — Claude's Stop hook fires natively).

The new block:
```bash
if [ -s "$RUN_DIR/output.json" ]; then
  KM_LAST_MSG=$(jq -rs 'map(select(.type=="item.completed" and .item.type=="agent_message")) | last | .item.text // ""' "$RUN_DIR/output.json")
  KM_SID=$(jq -r 'select(.type=="thread.started") | .thread_id // empty' "$RUN_DIR/output.json" | head -1)
  if [ -n "$KM_LAST_MSG" ]; then
    jq -n --arg msg "$KM_LAST_MSG" --arg sid "$KM_SID" \
      '{hook_event_name:"Stop", last_assistant_message:$msg, session_id:$sid}' \
      | /opt/km/bin/km-notify-hook Stop || true
  fi
fi
```

Plan 70-03's `km-notify-hook` Stop branch already prefers `last_assistant_message` (the Codex JSONL field name) when present in the stdin payload. So with this synthesized payload, the existing hook code does the email + Slack work.

`KM_SLACK_THREAD_TS` is NOT set by the operator-side path (only the poller sets it), so the hook's Slack branch fires unconditionally — matching SC-6's "when env var is not set, Stop hook posts as usual" requirement.


- Hook log shows km-notify-hook ran with Stop event

## Out of scope

- Claude-side changes (Claude hooks fire natively).
- The poller path (Plan 70-10 already covers it).
- The "no last_assistant_message in stream" edge case is handled by the `if [ -n "$KM_LAST_MSG" ]` guard — no hook invocation, no spurious Slack post.
- Failure logging beyond `2>>"$RUN_DIR/stderr.log" || true` — keep the hook invocation non-fatal so a hook bug doesn't break the agent run itself.
