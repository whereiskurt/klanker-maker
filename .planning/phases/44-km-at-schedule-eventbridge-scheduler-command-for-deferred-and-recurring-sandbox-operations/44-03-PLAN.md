<objective>
Wire the km at command -- connect time parsing, EventBridge Scheduler, and DynamoDB into a Cobra command with list/cancel subcommands and a km schedule alias. Includes a CLI-side max-sandbox guardrail for recurring create schedules.

Purpose: This is the user-facing CLI that ties together the parser (Plan 01) and AWS infrastructure (Plan 02). Operators use `km at '<time>' <command> [args...]` to schedule sandbox operations, `km at list` to view schedules, and `km at cancel <name>` to remove them. For recurring creates, the CLI warns if the sandbox count is already at the configured max_sandboxes limit (the Lambda also enforces at fire-time, but this gives immediate operator feedback).

Output: `internal/app/cmd/at.go`, `internal/app/cmd/at_test.go`, updated `internal/app/cmd/root.go`
</objective>

  <files>internal/app/cmd/at.go, internal/app/cmd/at_test.go</files>
  <behavior>
    --- km at '<time-expr>' <command> [args...] ---
    - Parses time expression via at.Parse (or validates --cron flag via at.ValidateCron)
    - Generates schedule name (auto or --name override)
    - For "create" command: builds CreateScheduleInput targeting CreateHandlerLambdaARN with SandboxCreate detail JSON as Target.Input
    - For lifecycle commands (destroy/kill/stop/pause/resume/extend): builds CreateScheduleInput targeting TTLLambdaARN with SandboxIdle event JSON as Target.Input
    - One-time schedules: ActionAfterCompletion = DELETE
    - Recurring schedules: ActionAfterCompletion = NONE
    - Calls CreateAtSchedule, then PutSchedule to DynamoDB
    - Prints confirmation: "Scheduled: <name> -- <command> <args> at <expression>"

    --- Recurring create guardrail (SCHED-GUARDRAIL) ---
    - When command is "create" AND schedule is recurring (spec.IsRecurring):
      a. Call countActiveSandboxes (uses S3ListAPI) to get current active sandbox count
      b. If count >= cfg.MaxSandboxes (and MaxSandboxes > 0): print warning to stderr and return nil (skip scheduling)
         Warning: "Warning: active sandbox count (%d) is at or above limit (%d). Recurring create schedule not created. Reduce active sandboxes or increase max_sandboxes."
      c. If count check fails (S3 error): log warning, proceed with scheduling (non-blocking best-effort)
      d. If count < limit or MaxSandboxes == 0 (unlimited): proceed normally
    - NOTE: The create-handler Lambda also enforces checkSandboxLimit at fire-time (Pitfall 8 in RESEARCH.md), so this CLI check is advisory/early-warning only

    --- km at list ---
    - Calls ListScheduleRecords from DynamoDB
    - Prints table: NAME | COMMAND | TARGET | SCHEDULE | STATUS | CREATED
    - Empty list prints "No scheduled operations."

    --- km at cancel <name> ---
    - Calls DeleteAtSchedule (EventBridge) + DeleteScheduleRecord (DynamoDB)
    - Prints "Cancelled: <name>"
    - Idempotent: no error if already cancelled/missing

    --- Flags ---
    - --cron '<expr>' bypasses NL parsing, uses raw cron expression
    - --name '<name>' overrides auto-generated schedule name
    - --group '<group>' defaults to "km-at" schedule group

    --- Error cases ---
    - Missing time expression: "Usage: km at '<time-expr>' <command> [args...]"
    - Unparseable time expression: error from at.Parse
    - Missing TTLLambdaARN or CreateHandlerLambdaARN: "Lambda ARN not configured -- run 'km init' first"
    - Unsupported command (e.g., "km at '...' validate"): "Command 'validate' is not schedulable -- only create, destroy, kill, stop, pause, resume, extend are supported"
  </behavior>
