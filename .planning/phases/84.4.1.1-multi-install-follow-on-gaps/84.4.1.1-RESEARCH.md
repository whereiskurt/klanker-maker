## Summary

Phase 84.4.1.1 closes five operator experience and correctness gaps surfaced during Phase 84.4.1 UAT-2. Four are code fixes with precise file:line targets; one is an investigation. All five were confirmed non-blocking for the 84.4.1 PASS but represent real DX holes that would bite a third-install operator.

The technical scope is narrow and self-contained: three Go source files take the bulk of the changes (`init.go`, `configure.go`, `doctor.go`), one new Organizations API interface method extends `unbootstrap.go`, and the regex in `validateArtifactsBucket` is a one-liner. All supporting patterns (DI interface-per-service, narrow API interfaces, `RunXWithDeps` testable cores, package-level fake implementations in `*_test.go` files) are already established and must be followed exactly.

The `checkSCP` function in `doctor.go` already has the `km-org-admin` assume-role pattern and `OrgsListPoliciesAPI` interface. The orphan SCP check (Gap #3a) extends that with a `ListPolicies` call. The uninit `--include-scp` flag (Gap #3b) is a new code path that mirrors the `--include-zone` pattern already in `unbootstrap.go` and follows the same `UnbootstrapDeps` struct extension pattern that Gap #4 (DDB lock table) used in Phase 84.4.1.

**Primary recommendation:** Implement the four code gaps as four sequential waves (each GREEN-only, no RED scaffolding needed since all insert points are identified), with Gap #5 investigation running as an exploratory task that produces either a concrete bug plan or a deferred session — not a blocking gate.

### Pattern 3: validateArtifactsBucket in config.Load() (Gap #2)

**What:** `config.Load()` reads `km-config.yaml` into a `Config` struct. After loading, call `validateArtifactsBucket(cfg.ArtifactsBucket)` if the value is non-empty. Return the error from `Load()` so callers hard-fail before reaching bootstrap.

**Note:** `validateArtifactsBucket` lives in `internal/app/cmd/configure.go` (package `cmd`), while `config.Load()` lives in `internal/app/config/config.go` (package `config`). The packages cannot import each other without creating a cycle. Two options:

1. Move `validateArtifactsBucket` (and its regex) to `internal/app/config/config.go` where `Load()` lives.
2. Call `validateArtifactsBucket` from callers of `config.Load()` (e.g., the root cobra command setup, or in each command's `RunE`).

**Recommended:** Move or duplicate the validation logic into `config` package as `ValidateArtifactsBucket`. This is the cleanest form — avoids import cycles and matches the `GetResourcePrefix()` / `GetSandboxSessionDocumentName()` pattern already in `config.go`.

```go
// internal/app/config/config.go
var canonicalBucketRE = regexp.MustCompile(`^[a-z][a-z0-9-]*-artifacts-[0-9]{12}$`)

func ValidateArtifactsBucket(name string) error {
    if name == "" {
        return fmt.Errorf("artifacts_bucket is empty; re-run `km configure`")
    }
    if lt := strings.Index(name, "<"); lt >= 0 {
        if gt := strings.Index(name[lt:], ">"); gt >= 0 {
            return fmt.Errorf("artifacts_bucket=%q is a placeholder", name)
        }
    }
    if !canonicalBucketRE.MatchString(name) {
        return fmt.Errorf("artifacts_bucket=%q does not match canonical shape ${prefix}-artifacts-${12-digit-account-id}", name)
    }
    return nil
}
```

Then in `config.Load()`:
```go
if cfg.ArtifactsBucket != "" {
    if err := ValidateArtifactsBucket(cfg.ArtifactsBucket); err != nil {
        return nil, err
    }
}
```

The existing `validateArtifactsBucket` in `configure.go` (package `cmd`) can remain for backward compat with existing tests — or be made to delegate to the config package version. The table-driven tests in `configure_84_3_test.go` cover the configure-side; the config-side gets its own test in `config_test.go`.

### Pattern 4: validateArtifactsBucket canonical regex (Gap #4)

**What:** Replace the sentinel-only check with a canonical shape regex.

**Current function (configure.go:218-234):**
```go
func validateArtifactsBucket(name string) error {
    if name == "" { return fmt.Errorf("...") }
    if lt := strings.Index(name, "<"); lt >= 0 { ... }
    if name == "km-artifacts-12345" { return fmt.Errorf("...") }
    return nil
}
```

**Strengthened form:**
```go
var canonicalBucketRE = regexp.MustCompile(`^[a-z][a-z0-9-]*-artifacts-[0-9]{12}$`)

func validateArtifactsBucket(name string) error {
    if name == "" {
        return fmt.Errorf("artifacts_bucket is empty; re-run `km configure` to derive ${prefix}-artifacts-${account_id} automatically")
    }
    if lt := strings.Index(name, "<"); lt >= 0 {
        if gt := strings.Index(name[lt:], ">"); gt >= 0 {
            return fmt.Errorf("artifacts_bucket=%q is a placeholder; ...", name)
        }
    }
    if !canonicalBucketRE.MatchString(name) {
        return fmt.Errorf("artifacts_bucket=%q does not match canonical shape ${prefix}-artifacts-${12-digit-account-id}; re-run `km configure` or pass --allow-non-canonical-bucket", name)
    }
    return nil
}
```

**Test cases (table-driven):**
| Input | Expected |
|-------|----------|
| `km-artifacts-052251888500` | nil (canonical) |
| `tg-artifacts-052251888500` | nil (canonical, different prefix) |
| `km-artifacts-12345` | error (too-short account ID — regex fails) |
| `tg-km-artifacts-use1-abcd0123` | error (non-canonical — regex fails) |
| `<prefix>-artifacts-<account-id>` | error (angle-bracket sentinel — caught first) |
| `` (empty) | error (empty sentinel — caught first) |
| `my-bucket` | error (no `-artifacts-` suffix) |

**Note on escape hatch:** The `--allow-non-canonical-bucket` flag should only be added to `km configure` if there is operator demand. Defer to implementation time; if operator feedback at UAT-3 indicates a legitimate need, add a `--allow-non-canonical-bucket bool` Cobra flag that suppresses the validation call.

## Common Pitfalls

### Pitfall 1: Import cycle between `config` and `cmd` packages
**What goes wrong:** `validateArtifactsBucket` is in `internal/app/cmd` (package `cmd`). If `config.Load()` calls it, you get a circular import.
**Why it happens:** The packages share a project but have a strict dependency direction: `cmd` imports `config`, not vice versa.
**How to avoid:** Move the validation function and regex to `internal/app/config/config.go`. Or: call `validateArtifactsBucket` from each cobra `RunE` that reads config, not from `Load()`.
**Warning signs:** `go build` reports "import cycle not allowed".

### Pitfall 2: `km-org-admin` role name is hardcoded in existing doctor.go
**What goes wrong:** `doctor.go:2904` hardcodes `"km-org-admin"` as the role name when assuming for the Organizations client, making it work only for the `km` prefix install.
**Why it happens:** Phase 84.4.1 fixed the SCP policy content to be prefix-agnostic, but missed the role name in doctor.
**How to avoid:** When implementing Gap #3a (orphan SCP check) and Gap #3b (uninit --include-scp), use `cfg.GetResourcePrefix() + "-org-admin"` for the role name. Do NOT fix doctor.go:2904 in the existing `checkSCP` path if it would break the existing test — file a note and fix only the new code paths.

### Pitfall 3: `validateArtifactsBucket` regex breaks the legacy `km-artifacts-12345` sentinel test
**What goes wrong:** Phase 84.4.1 test `TestIsPlaceholderBucket_AcceptsLegacyKmArtifacts12345` (referenced in verification.md) passes `km-artifacts-12345` and expects it to be rejected. After the regex change, `km-artifacts-12345` still matches the sentinel check (early return) so the behavior is unchanged. But `km-artifacts-052251888500` must now PASS.
**Why it happens:** Regex `^[a-z][a-z0-9-]*-artifacts-[0-9]{12}$` correctly rejects `km-artifacts-12345` (5 digits, not 12) — so the sentinel check and the regex check both catch it. No conflict.
**How to avoid:** Run `go test ./internal/app/cmd/ -run IsPlaceholder` after the change to confirm no regressions.

### Pitfall 4: SCP detach order matters
**What goes wrong:** If `km uninit` destroys the terraform state for the SCP module BEFORE detaching the SCP, the SCP remains attached to the account with no cleanup path via terraform.
**Why it happens:** `km uninit` calls `km unbootstrap`'s separate flow, not uninit's own module destroy loop. The SCP lives in the management account (Organizations), not in the application account's terraform state — so terraform destroy does NOT clean it up. This is why it orphans.
**How to avoid:** The `--include-scp` path must call Organizations detach+delete BEFORE the terragrunt destroy loop begins (Step 3). The confirmation prompt should make clear this detaches the SCP from the management account.

### Pitfall 5: `RunUninitWithDeps` signature change breaks uninit_test.go
**What goes wrong:** If `RunUninitWithDeps` gains a new `opts UninitOpts` parameter, all existing `uninit_test.go` callers fail to compile.
**Why it happens:** Go's strict type system; callers must update.
**How to avoid:** Either (a) use a `UninitOpts` struct from the start so the struct is easily extended, or (b) add the parameter as `opts UninitOpts` after the `force bool` parameter if force is the only existing one. Check `uninit_test.go` callers before changing the signature.

### Pitfall 6: `ListPolicies` returns all SCPs, including FullAWSAccess
**What goes wrong:** The orphan SCP check lists ALL SCPs including AWS-managed ones (e.g. `FullAWSAccess`). Filtering only for names ending in `-sandbox-containment` avoids false positives.
**Why it happens:** Organizations ListPolicies returns the full account SCP list.
**How to avoid:** Filter with `strings.HasSuffix(name, "-sandbox-containment")` before checking for orphan status.
