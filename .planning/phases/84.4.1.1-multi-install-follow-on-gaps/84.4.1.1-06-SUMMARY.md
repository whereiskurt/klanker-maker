## Investigation Finding (Gap #5 — UAT-2 "TTL/in-use" concern)

**Operator's concern (paraphrased):** "may be a bug related to determine TTL and if system is being used" during UAT-2 teardown.

**Investigation questions answered:**

1. **Are sandboxes in "stopping"/"stopped"/"creating" counted as active?**
   No. The filter is `r.Status == "running"` (exact equality). Sandboxes in transitional states are intentionally excluded. A sandbox that is stopping does NOT block uninit. **This is correct behaviour** — verified by sub-test D.

2. **Are TTL-expired but not-yet-destroyed sandboxes showing as "running"?**
   Yes — and this is the most likely root cause of the operator's concern. A TTL-expired sandbox whose record still reads `Status="running"` in DynamoDB (because the ttl-handler Lambda has not yet run its cleanup cycle) will block uninit. The status filter does NOT consult `TTLExpiry`. **No bug in the filter** — it relies on the Lambda having already updated the record status.
   
   **Documented mitigation:** Wait for the ttl-handler Lambda's next execution interval, then retry `km uninit`. If immediate teardown is required, use `km uninit --force`.

3. **Does ListSandboxes return records from other regions that contaminate the count?**
   No. The `r.Region == region` filter prevents cross-region contamination. Verified by sub-test C.

**Conclusion:** No concrete bug reproduced. The operator's concern is explained by the TTL-expired/Lambda-not-yet-run scenario. The new structured logging will surface `total_records` and `active_running` on every future `km uninit` run (at DEBUG level for per-record, INFO level for counts), enabling diagnosis from log output without code changes.

## Decisions Made
- Investigation-only — no changes to active-sandbox logic
- TTL-expired scenario documented as design note in uninit.go comment rather than as a code change; any future work (e.g. consulting TTLExpiry directly) should be a separate plan requiring operator approval
