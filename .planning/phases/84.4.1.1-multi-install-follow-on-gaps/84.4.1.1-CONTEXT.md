## Implementation Decisions

### Gap #1 — `km init --plan` missing buildLambdaZips on fresh clone — LOCKED

**Problem:** `runInitPlan` (init.go:1208+) skips the `buildLambdaZips` call that `runInit` (full apply, init.go:494) makes automatically. On a fresh clone with no `build/` dir, the create-handler module's terraform plan fails at `filebase64sha256(build/create-handler.zip)`.

**Decision:** Add `buildLambdaZips(repoRoot)` call to `runInitPlan` immediately before the module loop at init.go:~1211 (after `ensureRegionHCL`). Skip when verbose=false to avoid double-builds during `km init --dry-run=false` after a recent `--plan`. Same warn-on-failure semantics as runInit:494-496.

**Test:** Extend `init_test.go` with a fixture that captures a fresh-clone scenario (no `build/` dir) + asserts `buildLambdaZips` was invoked.

**Doc:** OPERATOR-GUIDE.md prereq note (commit `341a2e7`) STAYS as defense in depth even after code fix — it documents the asymmetry.

### Gap #2 — `km configure` artifacts_bucket prompt doesn't auto-derive default — LOCKED

**Problem:** `deriveArtifactsBucket(prefix, accountID)` exists at configure.go:206 (returns `${prefix}-artifacts-${accountID}`) but isn't called from the prompt path at configure.go:592. Operator can type any string; Phase 84.3 partial-pass gap re-confirmed live during UAT-2 (operator entered `tg-km-artifacts-use1-abcd0123`).

**Decision:** Wire `deriveArtifactsBucket(resourcePrefix, applicationAcct)` as the prompt default at configure.go:592 when `artifactsBucket == ""`. Also wire `validateArtifactsBucket` into `config.Load()` so silent placeholder pass-throughs hard-fail before reaching bootstrap.

**Test:** `configure_84_3_test.go` — extend the existing test to cover the new derive-and-prompt path. `config_test.go` — add `TestConfigLoad_RejectsPlaceholderArtifactsBucket`.

### Gap #3 — Orphan SCPs not auto-detached by km uninit/unbootstrap — LOCKED

**Problem:** Both rg (Phase 84.4 UAT-1) and tg (Phase 84.4.1 UAT-2) installs leaked their `${prefix}-sandbox-containment` SCP attached to the application account after teardown. The rg orphan ACTIVELY BLOCKED tg-create-handler from `ec2:CreateSecurityGroup` during Phase 84.4.1 UAT-2 Step 7 (operator+I had to chain-assume km-org-admin and manually detach+delete `p-tbyi97x6`).

**Decision (split into two work items):**

**3a (this phase):** `km doctor` warns on orphan SCPs. Cross-reference `aws organizations list-policies` against the local `km-config.yaml` `resource_prefix` AND against any other `km-config.yaml` files in sibling working directories. SCPs whose name does not match any known prefix are reported as orphans with a recovery command (chain-assume + detach + delete). WARN level, not FAIL. Composes with Phase 85's doctor work.

**3b (this phase):** `km uninit` adds a SCP detach + delete step BEFORE state-bucket deletion, gated on `--include-scp` flag (default false). Default-false is intentional — operators may want to keep the SCP attached during a partial teardown. Confirmation prompt required even with the flag. Reuses Phase 84.3 `km bootstrap`'s direct AWS Organizations API code path (BuildSCPPolicyFromPrefix renderer doesn't matter for delete; just need `organizations:DetachPolicy` + `DeletePolicy`).

**Test:** `unbootstrap_test.go` extends UnbootstrapAPI mock to include OrganizationsAPI. Asserts the detach+delete sequence is invoked when `--include-scp` is set AND skipped (with WARN) when not set.

### Gap #4 — `validateArtifactsBucket` too permissive — LOCKED

**Problem:** Today rejects `<angle-brackets>` + literal `km-artifacts-12345` sentinel. Accepted `tg-km-artifacts-use1-abcd0123` (clearly a hand-edited placeholder) live during UAT-2.

**Decision:** Strengthen `validateArtifactsBucket` to enforce the canonical Phase 84.3 shape `^[a-z][a-z0-9-]*-artifacts-[0-9]{12}$`. Hard error for non-matching values. This composes with Gap #2's derive-and-validate-on-load wiring.

Edge case: operator may legitimately want a non-canonical bucket name (e.g. carrying an existing bucket from a prior install). Provide `--allow-non-canonical-bucket` escape hatch on `km configure` if needed; default-strict.

**Test:** `configure_84_3_test.go` — table-driven test covering canonical/non-canonical/placeholder/empty inputs.

### Gap #5 — Suspected TTL / "system in use" detection bug during km uninit — DISCRETION

**Problem:** Operator flagged during 2026-05-18 tg UAT-2 teardown that there "may be a bug related to determine TTL and if system is being used." No log capture or specific symptom yet.

**Decision (Claude's Discretion):** Open a `/gsd:debug` session for this gap rather than wedging it into this phase as a code fix. The phase commits to:
- Reading `km uninit` + `km destroy` code paths for TTL / "in use" / "active sandbox" checks
- Adding structured logging at any decision-point that determines "is the install in use"
- Writing a test fixture that exercises uninit when the install has 0 / 1 / N active sandboxes
- If a specific bug is reproduced from logs, file as a separate plan within this phase OR defer to Phase 84.4.1.2 if scope grows

This keeps the phase tractable. The operator wanted "5 gaps tracked" not "TTL bug fixed by hour 4."

### Closure criterion

Phase 84.4.1.1 is complete when:
1. Gap #1 code fix shipped + tested + OPERATOR-GUIDE.md note stays
2. Gap #2 derive+validate wired into both prompt-default and config.Load()
3. Gap #3a doctor orphan-SCP warn ships + 3b km uninit --include-scp flag ships
4. Gap #4 validator hard-enforces canonical shape
5. Gap #5 investigation produces either (a) a concrete repro + fix plan in this phase, or (b) a debug session that defers the fix to a follow-on
6. UAT-3 (lightweight): fresh tu prefix install on shared account WITHOUT manual workarounds. No new fresh-prefix UAT-2-scale exercise needed — just verify the 4 code fixes interact correctly.
