## Part 3 — Deploy Runbook (pre-E2E steps)

Execute in order on the **front-door / github-bridge install**:

```bash
# Step 1 — Build Lambda zips (bridge + create-handler + all others)
make build-lambdas

# Step 2 — Full terragrunt apply (bridge env block + code; NOT --sidecars)
km init --dry-run=false

# Step 3 — Verify the bridge Lambda updated
km github status    # confirm bridge-url resolves

# Step 4 — Create (or recreate) the E2E sandbox(es)
# Option A: new sandbox from a Codex-capable profile
km create profiles/learn.v2.desktop.yaml --alias gh-agent-e2e

# Option B: existing sandbox needs recreate to pick up new poller
km destroy <existing-sandbox-id> --remote --yes
km create <same-profile.yaml> --alias gh-agent-e2e

# Step 5 — Confirm the sandbox is ready
km list
# Wait for status=running
```

**Note on profile selection:**

- **Codex-capable sandbox** (for steps a–d): Any profile with `spec.agent.default: codex` OR with `codex` installed. The `profiles/learn.v2.desktop.yaml` profile with `notification.github.inbound.enabled: true` works. The operator may prefer a lightweight profile; the critical requirement is `notification.github.inbound.enabled: true` + codex binary on PATH.
- **Claude-only sandbox** (for step e): `profiles/github-review.yaml` — the lean built-in profile. This is the profile that should produce the "no Codex here" helpful-error comment when `/codex` is used.

- An open PR or issue in the test repo with an `@<bot>` mention already working (Phase 97 smoke test passing)

### DDB table name

```bash
GITHUB_THREADS_TABLE=$(km info 2>/dev/null | grep github-threads | awk '{print $NF}')

```

### Step a — `/codex` dispatch + agent_type write-back

**Action:** Post in the test PR (on a repo configured for SANDBOX-A):

```
@<bot> /codex /patch fix the flaky test in auth_test.go
```

**Expected observable signals:**

1. **Bridge** — 👀 reaction on the comment (Phase 97 ACK). Envelope carries `"agent":"codex"`.

2. **Journald on SANDBOX-A:**
   ```bash
   journalctl -u km-github-inbound-poller -n 50 --no-pager
   ```
   Look for:
   - `AGENT_OVERRIDE=codex`
   - `EFFECTIVE_AGENT=codex`
   - Codex session capture: `thread.started` + `thread_id` captured
   - No `--resume` flag (fresh first turn)
   - Successful DDB write-back line

3. **DDB row:**
   ```bash
   aws dynamodb get-item \
     --table-name "$GITHUB_THREADS_TABLE" \
     --key "{\"repo\":{\"S\":\"owner/repo\"},\"number\":{\"N\":\"<PR-number>\"}}" \
     --region us-east-1
   ```
   Expected: `agent_type.S = "codex"` and `agent_session_id.S = <thread_id>`.

4. **GitHub:** Bot posts a PR review or comment from Codex (the agent's reply to the task).

**Record result:** PASS / FAIL + observed DDB attribute value + journald snippet.

### Step b — No-verb follow-up continues Codex

**Action:** Post a follow-up comment in the **same thread** (same PR, no reply-thread nesting — GitHub does not thread PR comments):

```
@<bot> Can you also check the timeout constant?
```

(No `/codex` or `/claude` verb.)

**Expected observable signals:**

1. **Journald on SANDBOX-A:**
   - `AGENT_OVERRIDE=` (empty — no verb in comment)
   - `THREAD_AGENT_TYPE=codex` (read from DDB row written in step a)
   - `EFFECTIVE_AGENT=codex`
   - `--resume <thread_id>` present (resuming the Codex thread from step a)
   - Codex runs again with same thread_id

2. **DDB row:** `agent_type.S = "codex"` unchanged; `agent_session_id.S` = same thread_id (or updated if Codex rotates it — check journald for session capture).

**Record result:** PASS / FAIL + journald evidence of THREAD_AGENT_TYPE=codex.

### Step c — `/claude` cross-agent switch

**Action:** Post in the same PR:

```
@<bot> /claude Actually, let's have Claude review the whole auth module instead.
```

**Expected observable signals:**

1. **Journald on SANDBOX-A:**
   - `AGENT_OVERRIDE=claude`
   - `THREAD_AGENT_TYPE=codex` (stored from previous turns)
   - Cross-agent switch log: `cross-agent switch codex->claude; resetting session`
   - `GITHUB_SESSION=""` and `RESUME_ARG=""` (session cleared)
   - Claude runs without `--resume` (fresh session for Claude)
   - New `agent_session_id` captured (Claude session_id format)

2. **DDB row after turn:**
   - `agent_type.S = "claude"`
   - `agent_session_id.S = <new-claude-session-id>` (different from the codex thread_id)

3. **GitHub:** Claude posts a review on the PR (different agent than Codex's response).

**Negative signal (must NOT appear):** `--resume <old-codex-thread-id>` in journald — that would indicate the cross-agent switch did not clear the session.

**Record result:** PASS / FAIL + DDB agent_type before and after + journald cross-agent switch log line.
