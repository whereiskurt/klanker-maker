    - Test 3: When Runner.Verbose is false and command fails, stderr content is always printed to os.Stderr
    - Test 4: When Runner.Verbose is false, Destroy also captures output (same behavior as Apply)
    - Test 5: DestroyWithStderr still captures stderr to the provided buffer regardless of Verbose (used for lock detection)
  </behavior>
  <action>
    Add a `Verbose bool` field to the Runner struct. Default is false (quiet mode).

    Modify Apply():
    - When Verbose is true: keep current behavior (cmd.Stdout = os.Stdout, cmd.Stderr = os.Stderr)
    - When Verbose is false:
      - Capture stdout and stderr in bytes.Buffer instances
      - Run the command
      - If command succeeds: discard captured output (the caller already printed "Applying X..." before calling Apply)
      - If command fails: write the captured stderr to os.Stderr so errors are always visible, then return the error
      - Also scan captured stderr for lines containing "Warning" or "warn" and print those to os.Stderr regardless of success

    Modify Destroy():
    - Same logic as Apply() for quiet mode

    Modify DestroyWithStderr():
    - When Verbose is true: keep current behavior (os.Stdout + MultiWriter(os.Stderr, stderrBuf))
    - When Verbose is false: capture stdout in buffer, stderr goes to both buffer and stderrBuf (for lock detection), print stderr on failure

    Modify DestroyForceUnlock():
    - Same logic as Destroy() for quiet mode

    Do NOT change Output() — it already captures output programmatically.
    Do NOT change buildCommand() — it does not set stdout/stderr.
    Do NOT change NewRunner() — Verbose defaults to false (zero value).

    The runner_test.go tests should use exec.Command("echo", ...) or a test helper binary to verify output capture without requiring terragrunt. Use the existing test patterns in runner_test.go.
