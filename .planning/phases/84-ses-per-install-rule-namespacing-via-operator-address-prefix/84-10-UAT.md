### Gaps discovered (Phase 84.1 input)

1. **`km bootstrap` does not export `KM_ROUTE53_ZONE_ID`** — Fatal when `register_domain_identity=true`. `internal/app/cmd/bootstrap.go::ExportConfigEnvVars` exports most KM_* vars from `km-config.yaml` but skips `route53_zone_id`. The first bootstrap succeeded only because `register_domain_identity=false` made `var.hosted_zone_id` unused.

2. **`km bootstrap` is not idempotent** — Foundation module's `aws_ses_receipt_rule_set.shared` has `lifecycle.prevent_destroy = true` AND `count = var.register_shared_rule_set ? 1 : 0`. Second bootstrap invocation: auto-detect sees rule set already exists → `KM_REGISTER_SHARED_RULESET=false` → count flips 1→0 → terraform plans destroy → `prevent_destroy` blocks → bootstrap errors. The register-flag semantics are incompatible with prevent_destroy. Likely fix: register flags should mean "manage this" not "create-only-first-time"; once foundation has the resource in state, the flag should stay true.

3. **Foundation auto-detect operates on AWS reality, not foundation state ownership** — On in-place upgrade from Phase 82.x, `sandboxes.klankermaker.ai` domain identity existed in AWS (owned by regional v1.0.0 state). Auto-detect saw it and set `register_domain_identity=false` → foundation skipped creating → regional v1.0.0 destroy then nuked it → no module owned it after cutover. **Net data loss: domain identity, DKIM, MX, verification TXT, active rule set pointer all destroyed.** Recovery required direct terragrunt invocation with forced env vars.

4. **`km init` swallows ses-module errors silently** — The Verbose=false runner captures stderr and only prints on failure exit, but the wedged-terraform scenario produces no stderr because the process never returned. Operator-visible state: indefinite `Applying ses...` with no progress indicator. Lasted 10+ minutes before manual Ctrl-C.

5. **No timeout on regional module applies** — `runner.Apply` has no upper bound. Combined with Gap 4, a wedged terraform child process blocks `km init` indefinitely. Future similar conditions should surface as a timeout-with-context error.

6. **Foundation module does not import existing v1.0.0 resources** on upgrade — The bootstrap auto-detect's "exists in AWS → skip create" decision is correct ONLY for fresh installs and for installs where foundation already manages the resources. For in-place upgrades from Phase 82.x, foundation should `terraform import` ownership from regional v1.0.0 state before the regional cutover destroys those resources. Alternatives: (a) regional v2.0.0 keeps shared resources, foundation owns only the empty rule set; (b) `removed` blocks in regional v2.0.0 suppress destroy. This is the highest-impact gap — any operator upgrading from Phase 82.x will hit data loss.

7. **`km bootstrap` does not export `KM_ARTIFACTS_BUCKET`** — During recovery, direct `terragrunt plan` against regional ses produced malformed S3 ARNs (`arn:aws:s3:::/mail/km/*` — no bucket name) because the variable was empty. `km init` itself sets the env var (init.go:590) but `km bootstrap` doesn't, and any direct-terragrunt operator workflow needs it too. Suggest a single `ExportTerragruntEnvVars(cfg)` helper that exports the full set: `KM_RESOURCE_PREFIX`, `KM_DOMAIN`, `KM_EMAIL_SUBDOMAIN`, `KM_REGION`, `KM_REGION_LABEL`, `KM_ROUTE53_ZONE_ID`, `KM_ARTIFACTS_BUCKET`, `KM_ACCOUNTS_*`.

8. **State digest mismatch recovery is undocumented** — Ctrl-C of a mid-flight terragrunt apply leaves S3 state and DynamoDB lock-table digest out of sync. Recovery required: download S3 object, compute MD5 locally, `aws dynamodb update-item` to overwrite the stale `Digest` field in `tf-km-locks-use1`. OPERATOR-GUIDE.md should document this or `km doctor` should detect and offer to repair.
