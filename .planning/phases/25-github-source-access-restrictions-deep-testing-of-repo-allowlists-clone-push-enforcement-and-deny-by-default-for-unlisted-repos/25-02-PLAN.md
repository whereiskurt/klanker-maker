From 25-RESEARCH.md - recommended hook template:
```bash
#!/bin/bash
# /opt/km/hooks/pre-push
ALLOWED_REFS="${KM_ALLOWED_REFS:-}"
if [ -z "$ALLOWED_REFS" ]; then exit 0; fi
while read local_ref local_sha remote_ref remote_sha; do
  branch="${remote_ref#refs/heads/}"
  allowed=false
  IFS=: read -ra PATTERNS <<< "$ALLOWED_REFS"
  for pattern in "${PATTERNS[@]}"; do
    if [[ "$branch" == $pattern ]]; then allowed=true; break; fi
  done
  if [ "$allowed" = false ]; then
    echo "[km] Push to '$branch' denied -- not in allowedRefs" >&2; exit 1
  fi
done
exit 0
```

    1. Read `docs/security-model.md` and find the sections about `allowedRefs` and GitHub source access.

    2. Update the security model documentation to accurately describe:

    a) **AllowedRefs enforcement approach**: Enforced via git `pre-push` hook installed at `/opt/km/hooks/pre-push` on EC2 sandboxes. The compiler sets `git config --system core.hooksPath /opt/km/hooks` and exports `KM_ALLOWED_REFS` as a colon-separated pattern list. The hook uses bash glob matching (`[[ "$branch" == $pattern ]]`), supporting wildcards like `feature/*`.

    b) **Limitations**:
       - EC2 only -- ECS sandboxes do not have ref enforcement (no user-data bootstrap)
       - Can be bypassed with `git push --no-verify` if sandbox policy allows shell escape
       - Primary enforcement layer remains GitHub App token scoping (by repository, not by ref)
       - AllowedRefs is defense-in-depth, not the primary control

    c) **ECS GitHub credential gap**: Document that ECS sandboxes deploy the token refresh Lambda/EventBridge infrastructure but do NOT inject a credential helper into the container. `git clone` inside an ECS sandbox will fail with "Authentication failed" unless the operator manually injects the token. This is a known v1 limitation. The `github_token_inputs` block is emitted in ECS service.hcl for the Lambda infrastructure, but the container itself has no GIT_ASKPASS equivalent.

    d) **Deny-by-default contract**: Document that both nil `sourceAccess.github` AND empty `allowedRepos: []` result in zero GitHub token infrastructure being provisioned. This is the primary access control -- no token means no access.
