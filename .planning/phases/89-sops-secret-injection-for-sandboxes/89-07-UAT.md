   - ZERO replaces.

     ```bash
     km bootstrap --shared-secrets-key
     ```

   - Note: `--i-accept-destroys` is owned by Phase 84.2 and does NOT auto-apply — it only surfaces the diff for human review.

4. Apply (after Step 2 passes, or after Step 3 is reviewed-safe):

## Scenario 1: Encrypt a bundle with the new shared KMS alias

**Goal:** Verify the operator-side encryption workflow works against the per-install KMS key.

**Steps:**

1. Locate the KMS alias ARN:
   ```bash
   eval $(km env)
   aws kms list-aliases \
     --query "Aliases[?starts_with(AliasName, \`alias/${KM_RESOURCE_PREFIX:-km}-sandbox-secrets\`)].AliasArn" \
     --output text
   # Record the output — use it as <alias-arn> below
   ```

2. Create plaintext bundle (NEVER commit this file):
   ```bash
   cat > /tmp/codex-plain.yaml <<EOF
   OPENAI_API_KEY: sk-<your-real-openai-api-key>
   EOF
   ```

3. Encrypt with the KMS alias:
   ```bash
   sops --encrypt \
        --kms <alias-arn-from-step-1> \
        --input-type yaml \
        --output-type yaml \
        /tmp/codex-plain.yaml > secrets/codex.enc.yaml
   ```

4. Verify the bundle looks correct:
   ```bash
   head -5 secrets/codex.enc.yaml
   # Expected: lines containing ENC[AES256_GCM,...] ciphertext
   tail -20 secrets/codex.enc.yaml
   # Expected: a `sops:` metadata block with `kms:` entry referencing the alias ARN
   ```

5. Remove the plaintext (DO NOT commit it):
   ```bash
   find /tmp -name "codex-plain.yaml" -delete
   ```

6. Validate the profile (now that the bundle file exists):
   ```bash
   km validate profiles/codex.yaml
   # Expected: validates cleanly — profile declares spec.secrets.sopsFile: ./secrets/codex.enc.yaml
   ```

**PASS criterion:** `secrets/codex.enc.yaml` exists, has `sops:` block, `km validate profiles/codex.yaml` exits 0.

## Observed Results

Executed 2026-05-28 against the real `km` install (resource_prefix `km`,
application account 052251888500, us-east-1). UAT surfaced several real bugs
across Phase 89, Phase 88, and Phase 70 surfaces; all Phase-89 bugs were
hot-fixed in-session and re-verified. Bugs + commits listed under "Bugs found".

### Scenario A — fresh-install bootstrap

First `km bootstrap --shared-secrets-key --dry-run=false` FAILED:
`MalformedPolicyDocumentException: These condition keys are not permitted in a
key policy: aws:ResourceTag, kms:ResourceTag, kms:ResourceAliases`. The KMS
key policy used `kms:ResourceAliases` (valid only in IAM identity policies, not
key policies). Fixed (commit e91871d — dropped Statement 2; root-admin
Statement 1 + ec2spot identity policy is the real enforcement boundary).
After fix: bootstrap applied; `km doctor` → `✓ Shared secrets KMS key
alias alias/km-sandbox-secrets healthy`. (A stale PendingDeletion key from a
prior session briefly masked the alias in ListAliases; terraform refresh
detected it cleanly — 2 to add, 0 destroy.)

### Scenario 1 — encrypt bundle

`sops --encrypt --kms <alias-arn>` initially errored `no matching creation
rules found` due to a pre-existing `~/.sops.yaml`; resolved with
`--config /dev/null`. Bundle written to `secrets/codex.enc.yaml` with real
ENC[AES256_GCM,...] ciphertext + a `sops:` block referencing the alias ARN.
`km validate profiles/codex.yaml` → valid.

### Scenario 2 — km create (remote dispatch)

Multiple failures, each fixed:
- `additional properties 'secrets' not allowed` — Lambda km predated the schema; resolved by `km init` redeploy.
- `read sops bundle /tmp/secrets/codex.enc.yaml: no such file` — remote-create dispatch never bridged the bundle to the Lambda. Fixed (commit a6548e3: operator uploads bundle to remote-create prefix; create-handler `patchProfileForSops` downloads it + rewrites profile to abs path).
- `open /tmp/tmp/...-secrets.enc.yaml` — `uploadSopsBundleIfPresent` joined an absolute path against profileDir. Fixed (commit ed3c5fa: honor `filepath.IsAbs`).
After fixes: bundle present at `s3://.../sandboxes/<sb>/secrets.enc.yaml`;
cloud-init showed `Decrypting SOPS bundle...` + `SOPS bundle decrypted`.

### Scenario 3 — env exposure

Initial boot: `/etc/sandbox-secrets.env` was `0400 root:root`, so the
sandbox-user login shell's `[ -r ]` guard in `/etc/profile.d/zz-sandbox-secrets.sh`
silently failed → `$OPENAI_API_KEY` empty. Fixed (commit b8b3c6b: `0440
root:sandbox`). After fix (fresh create, no manual patch):
`-r--r----- root sandbox /etc/sandbox-secrets.env`; `su - sandbox -c 'echo
${OPENAI_API_KEY:0:11}'` → `sk-svcacct-`; KEYLEN=167. The `enforcement: both`
learn profile behaved identically.

### Scenario 4 — codex → OpenAI → Phase 88 metering

PASS (with codex-transport fix). Findings:
- `km agent run` (no `--codex`) ran claude → `claude: command not found`; it ignores `spec.cli.agent` (agent.go:215). Worked around with explicit `--codex`. (Phase 70 defect — see Open items.)
- agent-run script ran in a non-login shell and did NOT source `zz-sandbox-secrets.sh` → 401 Missing bearer. Fixed (commit 961fe3d: source the secrets file in the agent script).
- codex 0.133 defaults to a WebSocket transport (`wss://.../v1/responses`) that pins certs + rejects the MITM CA ("Attack attempt detected"); its HTTP fallback dropped auth. A direct authenticated `curl` to `/v1/responses` and `/v1/chat/completions` through the proxy both returned 200 — proving the MITM forwards auth and the injected key is valid; the 401 was codex-client-side. codex 0.133 also removed `wire_api="chat"`.
- FIX: codex `config.toml` with a custom provider (`openai` is reserved) + `supports_websockets = false` → codex uses HTTP POST to `/v1/responses`. Baked into both profiles (commit f3f6a16).

Result — codex turn via the SOPS-injected key, through the MITM:
```
$ km agent run learn-679ad69e --codex --prompt "Reply with exactly: phase 89 works" --wait
... codex output: "phase 89 works"  (9,347 tokens, gpt-4o-mini)

$ aws dynamodb query --table-name km-budgets PK=SANDBOX#learn-679ad69e
BUDGET#ai#gpt-4o-mini-2024-07-18   spentUSD = 0.0014127   ← Phase 88 metered the codex call
BUDGET#compute                     spentUSD = 0.011772
```
PASS criterion met: `BUDGET#ai#gpt-*` row with spentUSD > 0, written by the
http-proxy MITM metering a real codex `api.openai.com` call — no out-of-band
operator wiring beyond `km create`.

### Scenario 5 — km destroy S3 cleanup

Two gaps found + fixed:
- `km destroy --remote` (TTL/idle/destroy) is handled by the ttl-handler Lambda running `terraform destroy`. The SOPS bundle is uploaded via PutObject (outside terraform), so terraform destroy never removed it — the bundle was orphaned in S3. `deleteSopsBundleNonFatal` (89-05) only ran on the *local* destroy path. Fixed (commit 68b9188: ttl-handler handleDestroy Step 7 deletes `sandboxes/{id}/secrets.enc.yaml`).
- The `sandbox-secrets-7day` lifecycle backstop wasn't applied — `get-bucket-lifecycle-configuration` returned `NoSuchLifecycleConfiguration` because the live wiring still pointed at s3-artifacts-lifecycle/v1.0.0. Fixed (commit 5fc7aca: bump live source to v1.1.0).

Verified: local `km destroy` deletes the bundle immediately (89-05 path, working);
the orphaned codex-e7b18c00 bundle was removed manually pending the ttl-handler +
lifecycle deploys. Both fixes require `km init` to land on infrastructure.
