## Scenario A (pre-bootstrap): Fresh-install `km bootstrap --shared-secrets-key --plan` expectation

**Goal:** Confirm the Phase 84.2 destroy-class gate does NOT trip on a fresh install (Pitfall 7 from RESEARCH.md — `prevent_destroy = true` on a brand-new resource must not surface as a replace diff).

**Steps:**

1. Run:
   ```bash
   km bootstrap --shared-secrets-key --plan
   ```

2. Inspect output. **Expected:**
   - ~3 creates: `aws_kms_key.secrets[0]`, `aws_kms_alias.secrets[0]`, possibly a data-source lookup for account ID.
   - ZERO destroys.
   - ZERO replaces.
   - NO "destroy-class change detected" trip messages from the Phase 84.2 gate.

3. **If the gate trips unexpectedly** (unlikely on fresh install — indicates a schema misconfiguration or plugin-cache anomaly):
   - Run:
     ```bash
     km bootstrap --shared-secrets-key --i-accept-destroys --plan
     ```
   - Paste the resulting diff into Observed Results below, Scenario A section.
   - Determine if the diff is truly destructive (true positive). For a fresh install, the only true positive would be a schema misconfiguration in `infra/modules/sandbox-secrets-key/v1.0.0/`.
   - Once reviewed and confirmed safe, re-run WITHOUT `--plan` to actually apply:
     ```bash
     km bootstrap --shared-secrets-key
     ```
   - Note: `--i-accept-destroys` is owned by Phase 84.2 and does NOT auto-apply — it only surfaces the diff for human review.

4. Apply (after Step 2 passes, or after Step 3 is reviewed-safe):
   ```bash
   km bootstrap --shared-secrets-key
   ```

5. Verify:
   ```bash
   km doctor 2>&1 | grep -i secret
   # Expected: ✓ Shared secrets KMS key healthy
   ```

**PASS criterion:** Step 2 shows ~3 creates / 0 destroys / 0 replaces; Step 5 prints `✓ Shared secrets KMS key healthy`. If Step 3 fired, document the escape-hatch usage and the diff in Observed Results.

## Scenario 1: Encrypt a bundle with the new shared KMS alias

**Goal:** Verify the operator-side encryption workflow works against the per-install KMS key.

**Steps:**

1. Locate the KMS alias ARN:
   ```bash
   eval $(km env)
   aws kms list-aliases \
     --query "Aliases[?starts_with(AliasName, \`alias/${KM_RESOURCE_PREFIX:-km}-sandbox-secrets\`)].AliasArn" \
     --output text
   # Record the output — use it as <alias-arn> below
   ```

2. Create plaintext bundle (NEVER commit this file):
   ```bash
   cat > /tmp/codex-plain.yaml <<EOF
   OPENAI_API_KEY: sk-<your-real-openai-api-key>
   EOF
   ```

3. Encrypt with the KMS alias:
   ```bash
   sops --encrypt \
        --kms <alias-arn-from-step-1> \
        --input-type yaml \
        --output-type yaml \
        /tmp/codex-plain.yaml > secrets/codex.enc.yaml
   ```

4. Verify the bundle looks correct:
   ```bash
   head -5 secrets/codex.enc.yaml
   # Expected: lines containing ENC[AES256_GCM,...] ciphertext
   tail -20 secrets/codex.enc.yaml
   # Expected: a `sops:` metadata block with `kms:` entry referencing the alias ARN
   ```

5. Remove the plaintext (DO NOT commit it):
   ```bash
   find /tmp -name "codex-plain.yaml" -delete
   ```

6. Validate the profile (now that the bundle file exists):
   ```bash
   km validate profiles/codex.yaml
   # Expected: validates cleanly — profile declares spec.secrets.sopsFile: ./secrets/codex.enc.yaml
   ```

**PASS criterion:** `secrets/codex.enc.yaml` exists, has `sops:` block, `km validate profiles/codex.yaml` exits 0.

## Scenario 2: `km create` the codex sandbox

**Goal:** Full creation flow — bundle upload to S3, ec2spot v1.2.0 IAM attachment (`kms:Decrypt`), cloud-init decrypt at boot.

**Steps:**

1. Create the sandbox and wait for it to reach running state:
   ```bash
   km create profiles/codex.yaml --wait 2>&1 | tee /tmp/sb-create.log
   ```

2. Capture the sandbox ID:
   ```bash
   SB=$(grep -oE 'sb-[a-f0-9]+' /tmp/sb-create.log | head -1)
   echo "Sandbox: $SB"
   ```

3. Verify the encrypted bundle was uploaded to S3:
   ```bash
   eval $(km env)
   aws s3 ls "s3://${KM_ARTIFACTS_BUCKET}/sandboxes/${SB}/secrets.enc.yaml"
   # Expected: a size > 0 entry
   ```

4. Inspect cloud-init output once the sandbox is running:
   ```bash
   km shell ${SB} -- sudo grep -A5 'SOPS bundle\|sops decrypt\|sandbox-secrets' /var/log/cloud-init-output.log
   # Expected: lines like:
   #   [km-bootstrap] Decrypting SOPS bundle...
   #   [km-bootstrap] SOPS bundle decrypted to /etc/sandbox-secrets.env
   ```

**PASS criterion:** `km create` exits 0, bundle is in S3, cloud-init log shows `Decrypting SOPS bundle...` and `SOPS bundle decrypted to /etc/sandbox-secrets.env`.

     ```bash
     km shell ${SB} -- sudo tail -50 /var/log/cloud-init-output.log
     ```

**PASS criterion:** `/etc/sandbox-secrets.env` is root:root 0400; `$OPENAI_API_KEY` prints `sk-` prefix (not empty) in the sandbox login shell.

## Scenario 4: Codex inside sandbox accrues `BUDGET#ai#gpt-*` via Phase 88 meter

**Goal:** Prove Phase 88 OpenAI metering and Phase 89 secret injection compose end-to-end — Codex picks up `OPENAI_API_KEY` from the decrypted env, makes a real `api.openai.com` call, and the http-proxy MITM writes a `BUDGET#ai#gpt-*` row to DynamoDB — WITHOUT any out-of-band operator action post-create.

**Steps:**

1. Trigger a Codex call through the proxy:
   ```bash
   km agent run ${SB} --prompt "Say 'phase 89 end-to-end works' once, nothing else" --wait \
     2>&1 | tee /tmp/sb-agent.log
   ```
   - Codex picks up `OPENAI_API_KEY` from the login-shell env (via `/etc/profile.d/zz-sandbox-secrets.sh`).
   - The http-proxy MITM intercepts the `api.openai.com` call and meters it.

2. Wait ~30 seconds for the DynamoDB write to settle.

3. Query DynamoDB for the budget row:
   ```bash
   eval $(km env)
   aws dynamodb query \
     --table-name "${KM_RESOURCE_PREFIX:-km}-budgets" \
     --key-condition-expression 'PK = :pk' \
     --expression-attribute-values "{\":pk\":{\"S\":\"SANDBOX#${SB}\"}}" \
     --output json | jq '.Items[] | select(.SK.S | startswith("BUDGET#ai#gpt-"))'
   ```
   - Expected: ≥1 item with `spentUSD` > 0 and `SK` matching `BUDGET#ai#gpt-*` (e.g., `gpt-4o`, `gpt-5`, etc.)

4. Cross-check with km status:
   ```bash
   km status ${SB} | grep -A5 -i 'gpt\|openai\|AI'
   # Expected: per-model breakdown shows the gpt-* entry with non-zero spend
   ```

**PASS criterion:** A `BUDGET#ai#gpt-*` row exists in DynamoDB with `spentUSD > 0` — proving the full chain (sops decrypt → env exposure → Codex picks up key → real OpenAI call → http-proxy MITM meters → DynamoDB write) worked WITHOUT any operator post-create wiring.

**Phase 88 regression check:** Compare this result to Phase 88 plan 07 UAT — Phase 88 required out-of-band SSM SecureString + temp file + SSM SendCommand. Phase 89 closes that gap: `km create` is the only operator action needed.

## Scenario 5: `km destroy` cleans up the S3 bundle

**Goal:** Confirm Phase 89's destroy path deletes the S3 bundle object (Phase 89-05 destroy.go cleanup), and the 7-day S3 lifecycle rule is present as belt-and-suspenders for orphaned objects.

**Steps:**

1. Destroy the sandbox:
   ```bash
   km destroy ${SB} --remote --yes
   ```

2. Verify the bundle was deleted from S3:
   ```bash
   eval $(km env)
   aws s3 ls "s3://${KM_ARTIFACTS_BUCKET}/sandboxes/${SB}/secrets.enc.yaml"
   # Expected: NoSuchKey / empty output (object deleted by km destroy path)
   ```

3. Verify the S3 lifecycle rule is present (belt-and-suspenders):
   ```bash
   aws s3api get-bucket-lifecycle-configuration \
     --bucket "${KM_ARTIFACTS_BUCKET}" \
     --query 'Rules[?ID==`sandbox-secrets-7day`]' \
     --output json
   # Expected: non-empty JSON array containing the lifecycle rule
   ```

**PASS criterion:** S3 object no longer present post-destroy; lifecycle rule `sandbox-secrets-7day` exists on the artifacts bucket.

## Scenario 6 (optional): `km uninit` symmetric cleanup

**WARNING: DO NOT run on a production install. ONLY safe on a throwaway install where losing the KMS key is acceptable.**

**Goal:** Verify `km uninit` removes only this install's KMS alias and schedules key deletion (7-day window), leaving sibling installs untouched.

**Steps:**

1. Note the key ID before uninit (to verify PendingDeletion state after):
   ```bash
   eval $(km env)
   aws kms list-aliases \
     --query "Aliases[?AliasName==\`alias/${KM_RESOURCE_PREFIX:-km}-sandbox-secrets\`].TargetKeyId" \
     --output text
   # Record as KEY_ID
   ```

2. Run uninit (throwaway install only):
   ```bash
   km uninit --yes
   ```

3. Verify alias is removed:
   ```bash
   aws kms list-aliases \
     --query "Aliases[?starts_with(AliasName, \`alias/${KM_RESOURCE_PREFIX:-km}-sandbox-secrets\`)]" \
     --output json
   # Expected: empty array []
   ```

4. Verify the key is scheduled for deletion (not hard-deleted yet — 7-day window):
   ```bash
   aws kms describe-key --key-id <KEY_ID-from-step-1> \
     --query 'KeyMetadata.{KeyState:KeyState,DeletionDate:DeletionDate}'
   # Expected: KeyState=PendingDeletion, DeletionDate approx 7 days from now
   ```

5. If a sibling install was set up: verify its alias still exists and is unaffected:
   ```bash
   aws kms list-aliases \
     --query "Aliases[?starts_with(AliasName, \`alias/<sibling-prefix>-sandbox-secrets\`)]"
   # Expected: sibling alias still present
   ```

**PASS criterion:** Own alias removed, own key has `KeyState=PendingDeletion` with DeletionDate ~7 days out; sibling installs untouched.
