## Diagnostic Finding (Task 2.5 — code inspection instead of live km destroy)

Task 2.5 was a `checkpoint:human-verify` requiring the operator to run a live `km destroy --remote --yes` to observe which Case letter fired in stderr. Instead, the orchestrator confirmed the root cause via code inspection:

- `internal/app/cmd/destroy.go` lines 106-112: when `--remote` is set (the default per `BoolVar(&remote, "remote", true, ...)`), the command early-returns with `publisher.PublishSandboxCommand(ctx, sandboxID, "destroy")` BEFORE reaching the `destroySlackChannel(...)` call at line 474.
- `cmd/ttl-handler/`: `grep -rn "slack" cmd/ttl-handler/` returns nothing. The Lambda runs terraform destroy directly with zero Slack code.

This is architecturally unambiguous — no Case letter fires for `--remote` destroys because the function is never reached. The Task 2 visibility commit (all 6 Cases instrumented) was correct, but the instrumentation never fires on the default path.

**Why Option A:**
- No Lambda redeploy, no new IAM, no new toolchain upload
- Operator workstation already has the signing key in SSM (`/sandbox/operator/signing-key`), bridge URL, and IAM perms
- Matches local-path trust model exactly
- Smallest blast radius

## Decisions Made

1. **Option A over Option B/C**: Adding Slack teardown to the Lambda would require a Lambda redeploy, new IAM (Lambda needs SSM read + Slack bridge call perms), toolchain upload, and a `km init --lambdas` cycle. Option A (run locally before dispatch) has identical trust model to the local path and zero Lambda changes.

2. **Extract shared helper**: The inline SSM/key setup block at destroy.go:474 was 10 lines. Extracting `runSlackTeardown()` eliminates the risk of the remote and local paths diverging if someone modifies one without updating the other.

3. **Task 2.5 resolved via code inspection**: The checkpoint was designed for a live `km destroy` to reveal which Case letter fires. Code inspection revealed the architectural root cause (function is never called on --remote path) unambiguously. No live destroy was needed to select the fix branch.

4. **Pre-existing test failures are out of scope**: Failures in `at_test.go`, `create_test.go`, `shell_test.go`, etc. are not caused by Plan 02 changes and are logged to deferred-items per deviation scope boundary rules.
