# Phase 63.1 Plan 03: SLCK-13 Rotation Hardening + Observability Summary

JWT auth with Ed25519 rotation: single `km slack rotate-token` command, bridge structured logging via slog, and fail-fast 5xx policy to surface real Slack errors instead of masking them as `replayed_nonce`.

## Objective

Close SLCK-13: harden the bot-token rotation flow end-to-end. Plans 01 and 02 stabilized step-11d runtime injection (SLCK-11) and the destroy-archive auto-trigger (SLCK-12). Plan 03 adds the operator-facing rotation command, regression tests, operator runbook, live UAT, and Task 5 bridge observability hardening (added after UAT exposed the masked-error bug).

**Part A — Bridge handler structured logging (commits 87df706, 8beb22c):**

Added `log/slog` to `pkg/slack/bridge/handler.go`:

- INFO on every request: `action`, `sender_id`, `channel`, `nonce_prefix`

**Part B — PostToBridge fail-fast on 5xx (commits d61b718, e69c56f):**

Chose **B1** (fail-fast on 5xx) over B2 (rebuild-envelope-per-retry):

- The bridge reserves the nonce in DynamoDB at step 3, before calling Slack (step 7). Retrying the same envelope after a 5xx causes 401 `replayed_nonce`, masking the real error.
- B1: treat 5xx as fail-fast (2 lines: collapse the 5xx branch into the 4xx branch). Network errors (pre-HTTP) are still retried — they're pre-nonce-reserve and safe.
- B2 would require passing an `EnvelopeBuilder` to every caller — larger API surface, same benefit.

Caller audit (all pass; no caller relied on 5xx retry):
- `cmd/km-slack/main.go` — single call; benefits from honest errors
- `RunSlackTest` / `RunSlackRotateToken` — single call; now see real bridge error body
- `doctor_slack.go` / `destroy_slack.go` — single call; no behaviour change

Updated client tests: removed `TestPostToBridge_RetryOn5xx_ThenSucceed` and `TestPostToBridge_PersistentRetryFail`; added `TestPostToBridge_5xxNotRetried` and `TestPostToBridge_503NotRetried`.

## B1 vs B2 Decision Rationale

B1 chosen because:
1. The nonce-replay protection makes same-envelope 5xx retry harmful (not just wasteful): it turns a diagnostic `not_in_channel` into a misleading `replayed_nonce`.
2. All callers do a single `PostToBridge` call with no retry wrapper — none rely on the 5xx retry behaviour.
3. B1 is a minimal change (collapse the 5xx branch into the 4xx branch). B2 would require changing the function signature and all callers.
4. Callers that need idempotent retry can build a fresh envelope (new nonce) and call again. This is the correct pattern.
