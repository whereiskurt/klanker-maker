## Implementation Decisions

### SLCK-11 — Step 11d runtime injection visibility (Lambda subprocess)

**Locked decisions:**
- Each step 11d failure branch in `internal/app/cmd/create.go:790-825` MUST emit `fmt.Fprintf(os.Stderr, ...)` matching the existing successful-injection pattern at line 820. Five visible outcomes:
  1. `✓ Slack: channel %s wired into sandbox env\n` (success — already there)
  2. `⚠ Slack: /km/slack/bridge-url not configured — env not injected (run km slack init)\n`
  3. `⚠ Slack: failed to read terraform outputs — env not injected (non-fatal): %v\n`
  4. `⚠ Slack: no EC2 instance ID in outputs — env not injected (docker/non-EC2 substrate)\n`
  5. `⚠ Slack: SSM SendCommand failed — env not injected (non-fatal): %v\n`
- The visibility fix is the FIRST commit of this requirement. After it ships, the next `km create --remote` reveals which branch fires — that informs the second commit (root-cause fix).
- Root cause is ONE of: (a) SSM SendCommand timing — sandbox SSM agent not reachable when `runner.Output()` returns; (b) Lambda IAM missing `ssm:SendCommand` for sandbox instance ARNs; (c) `extractOutputInstanceID()` returning empty for our terraform module outputs. The plan must include a probe step that distinguishes these.
- Where the root cause requires a retry loop (most likely (a)), the loop is bounded: max 6 attempts × 5s sleep (30s total), warns on final failure with the same stderr pattern. Sandbox is provisioned regardless — this is non-fatal.

**Why:** UAT confirmed `KM_SLACK_CHANNEL_ID` and `KM_SLACK_BRIDGE_URL` are missing from `/etc/profile.d/km-notify-env.sh` on remote-created sandboxes (uat3, dc34). The compile-time `KM_NOTIFY_SLACK_ENABLED=1` IS set, proving the profile reaches the compiler. Step 11d's success message is silent in Lambda CloudWatch, AND no warning is visible — meaning either step 11d is being skipped (slackChannelID empty) OR running with all warns suppressed by `create.go:189-193 log.Logger = zerolog.New(io.Discard)`. We need the failure branches to bypass zerolog and write directly to stderr.

### Claude's Discretion (SLCK-11)
- Exact retry / backoff / sleep durations within the bounds above
- Whether to add a structured log line (zerolog) IN ADDITION to the stderr print, for log aggregation
- Test coverage approach for the retry loop (table-driven test with mocked SSM client)

### SLCK-12 — km destroy Slack archive auto-trigger

**Locked decisions:**
- `destroy_slack.go` already has visible-stderr logging for the archive outcome (commit `377b588`). Build on it: add the SAME visible-stderr emission for the FINAL-POST step (Case H at `destroy_slack.go:105-110`), each early-return branch (Case A no channel, Case B shared mode, Case F no bridge URL, Case G key load failed), and the `shouldArchive=false` skip path. Without this, the `Case H` "skip archive" branch fires silently and we never know.
- After visibility ships, the next `km destroy` of a per-sandbox sandbox reveals which branch fires. The plan must explicitly diagnose:
  - If Case G fires (key load) → operator key access from `klanker-terraform` profile. Likely OK; verify.
  - If Case H fires (final-post bridge call failed) → diagnose the bridge response: was it `bad_signature` (operator drift recurring), `channel_mismatch` (operator-signed posts shouldn't hit channel-ownership check — bridge bug?), or network/timeout (Lambda cold start, retries needed)?
- The operator-signed final-post path goes through the bridge with `slack.SenderOperator`. The bridge handler's channel-ownership check at `handler.go:128-142` correctly skips for `isOperator`. So `channel_mismatch` should NOT fire. If logs reveal it does, that's a Phase 63 bridge bug.
- End state: a `km destroy <sandbox-id>` of a per-sandbox sandbox with `slackArchiveOnDestroy != false` reliably:
  1. Posts the "Sandbox destroyed" message to `#sb-{alias}`
  2. Archives the channel
  3. Emits both outcomes on operator stderr: `✓ Slack: posted teardown message to %s` and `✓ Slack: archived channel %s`

**Why:** UAT Scen 4b confirmed the channel was NOT archived after `km destroy` (manual `conversations.archive` returned `{"ok":true}`, meaning the bot-archive succeeded only when called manually). The destroy flow runs but the Slack effect is missing. We need to instrument and diagnose.

### Claude's Discretion (SLCK-12)
- Whether to make the `final-post → archive` sequence resilient to a final-post failure (i.e., still attempt archive even if final-post fails). Current behavior (`Case H`) skips archive entirely, which arguably loses the cleanup. Plan should evaluate.
- Whether to add a "dry run" or `--no-slack` flag to `km destroy` for testing or operator override

### SLCK-13 — Bot token rotation full E2E

**Locked decisions:**
- This is primarily UAT validation, not new code (the rotation path already exists end-to-end). The plan must cover:
  1. Pre-flight: confirm `SSMBotTokenFetcher` cache TTL is 15 minutes (per Phase 63 design at `aws_adapters.go`)
  2. Revoke the current bot token in Slack App admin → confirm bridge starts returning `invalid_auth` from `auth.test` for new posts (note: bridge caches token for 15 min, so failures may take up to 15 min to surface)
  3. Either wait 15 min OR force Lambda cold start (`forceLambdaColdStart` already exists in `init.go`) to invalidate cache
  4. Reissue new token in Slack App admin
  5. `km slack init --force --bot-token <new>` — should show `shared channel "km-notifications" already exists (...); reusing` (per `1ad765c`)
  6. `km slack test` — must succeed within 15 min (or after second forceLambdaColdStart)
  7. `km doctor` — `slack-token` check must pass
- The plan should ship an operator runbook section in `docs/slack-notifications.md` with the explicit revoke-and-rotate cycle steps and timing expectations.
- Where feasible, the plan should add an automated unit test that proves `--force` works with a stub bot token (already partially covered by Plan 63-07 tests).

**Why:** UAT Scen 7 only validated the idempotent `--force` path with the same token. The full revoke-and-rotate cycle is the actual security-relevant operation operators will perform when responding to a leaked or compromised token (which has already happened multiple times in this codebase's life — see CLAUDE.md auto-memory).

### Claude's Discretion (SLCK-13)
- Whether to add a one-off `km slack rotate-token` subcommand that bundles "force cold start + put new token in SSM + smoke test" into one command for operator UX. (Probably yes — operator-friendly.)
- Whether to add bridge-side TTL configuration so operators can lower the 15-min cache for emergency rotations
</decisions>
