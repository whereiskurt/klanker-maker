## Summary

Phase 63.1 addresses three instrumentation and reliability gaps discovered during Phase 63 UAT (2026-04-30). None of the gaps are security regressions — the underlying mechanics all work when invoked correctly. The failures are all instrumentation failures: zerolog is silenced by `log.Logger = zerolog.New(io.Discard)` in both `create.go:189-193` and `destroy.go:138`, so every `log.Warn()` call in the affected code paths goes to `/dev/null`. The fix pattern is uniform: replace `log.Warn()` with `fmt.Fprintf(os.Stderr, ...)` in the affected branches.

The three gaps require a two-commit sequencing discipline for SLCK-11 and SLCK-12: **visibility commit first, root-cause fix second**. The visibility commit reveals which specific branch fires on the next live run, and that observation directly drives the root-cause fix. Shipping the root-cause fix without first establishing observability is guessing. The CONTEXT.md mandates this sequence explicitly.

SLCK-13 is primarily a runbook + validation exercise. The cache TTL, Lambda cold-start trigger, and `--force` path all exist and are tested. The gap is that the full revoke-and-reissue cycle was never exercised end-to-end during Phase 63 UAT — only the idempotent `--force` path with the same token was tested. The plan needs to exercise the full cycle and document the operator procedure.

**Primary recommendation:** Sequence three waves — (1) visibility instrumentation for both SLCK-11 and SLCK-12, (2) root-cause fixes driven by observation, (3) SLCK-13 runbook + validation + optional `km slack rotate-token` command.

## Architecture Patterns

### Pattern 1: stderr-bypass for operator-visible output

**What:** `fmt.Fprintf(os.Stderr, "  ⚠ Slack: ...\n", ...)` bypasses zerolog and appears on the operator terminal regardless of `--verbose` mode.

**When to use:** Any step 11d failure branch in `create.go` and any early-return branch in `destroy_slack.go`.

**Established by:** The codebase already uses this pattern at `create.go:820` (the success branch), `destroy_slack.go:140` (Case I, archive fail), and `destroy_slack.go:144` (archive success). The convention is two-space indent, `✓` for success, `⚠` for warning.

```go
// Source: internal/app/cmd/create.go:820 (existing success pattern)
fmt.Fprintf(os.Stderr, "  ✓ Slack: channel %s wired into sandbox env\n", slackChannelID)

// Source: internal/app/cmd/destroy_slack.go:140-144 (existing warn + success patterns)
fmt.Fprintf(os.Stderr, "  ⚠ Slack: archive of %s failed (non-fatal): %v\n", meta.SlackChannelID, archErr)
fmt.Fprintf(os.Stderr, "  ✓ Slack: archived channel %s\n", meta.SlackChannelID)
```

### Pattern 2: Bounded SSM SendCommand retry loop

**What:** `productionSSMRunner.RunShell` calls `ssm.SendCommand` with `TimeoutSeconds: 30`. This is the document timeout (how long SSM waits for the command to execute), NOT how long the caller waits for the SSM agent to become available. If the SSM agent hasn't registered yet, `SendCommand` returns an error immediately (`InvalidInstanceId` or similar).

**When to use:** SLCK-11 root cause (a) — SSM agent not yet registered at the time step 11d fires.

**Pattern:** Wrap `injectSlackEnvIntoSandbox` in a retry loop. The loop should:
1. Call `injectSlackEnvIntoSandbox` and check the error
2. On `InvalidInstanceId`-type error (or any non-nil error): sleep 5s, retry
3. After max attempts: emit the `⚠ Slack: SSM SendCommand failed` stderr line and return

```go
// Source: pattern derived from existing retry in pkg/slack/client.go (3 retries, 1s/2s/4s backoff)
// Bounds from CONTEXT.md: max 6 attempts × 5s = 30s total
const (
    ssmRetryMax   = 6
    ssmRetryDelay = 5 * time.Second
)
var injectErr error
for attempt := 1; attempt <= ssmRetryMax; attempt++ {
    injectErr = injectSlackEnvIntoSandbox(ctx, ssmRunner, instanceID, slackChannelID, bridgeURL)
    if injectErr == nil {
        break
    }
    if attempt < ssmRetryMax {
        time.Sleep(ssmRetryDelay)
    }
}
if injectErr != nil {
    fmt.Fprintf(os.Stderr, "  ⚠ Slack: SSM SendCommand failed — env not injected (non-fatal): %v\n", injectErr)
} else {
    fmt.Fprintf(os.Stderr, "  ✓ Slack: channel %s wired into sandbox env\n", slackChannelID)
}
```

### Pattern 3: Lambda cold-start trigger for cache invalidation

**What:** `forceLambdaColdStart` in `init.go:1642` calls `lambda.UpdateFunctionConfiguration` with a new `TOOLCHAIN_VERSION` env var. This forces Lambda to create a new execution environment, discarding the in-process `SSMBotTokenFetcher` cache.

**When to use:** SLCK-13 — when operator cannot wait 15 minutes for the `SSMBotTokenFetcher` cache to expire after revoking and reissuing a bot token.

**Note:** `forceLambdaColdStart` targets `km-create-handler`. The bridge Lambda is a separate function (`km-slack-bridge`). A new or adapted call must target the bridge Lambda by function name. The pattern is identical.

```go
// Source: internal/app/cmd/init.go:1642
func forceLambdaColdStart(ctx context.Context, cfg aws.Config) error {
    client := lambda.NewFromConfig(cfg)
    _, err := client.UpdateFunctionConfiguration(ctx, &lambda.UpdateFunctionConfigurationInput{
        FunctionName: aws.String("km-create-handler"), // change to "km-slack-bridge" for SLCK-13
        Environment: &lambdatypes.Environment{
            Variables: map[string]string{
                "TOOLCHAIN_VERSION": version.String(),
            },
        },
    })
    return err
}
```

### Pattern 4: `km slack init --force` with name_taken handling

**What:** `slack.go:213-237` handles the `name_taken` error when `--force` is passed: if the channel already exists, it reuses the existing stored channel ID rather than failing. This was the fix from commit `1ad765c`.

**When to use:** SLCK-13 — the `--force --bot-token <new>` path exercises this code. After token rotation, `CreateChannel` will return `name_taken` because the channel was never deleted; the fix correctly reuses the existing ID.

**Key invariant:** `km slack init --force` does NOT delete the existing channel. It only: (a) validates the new token via `auth.test`, (b) updates SSM `/km/slack/bot-token`, (c) re-applies the bridge Lambda Terraform, (d) reuses the existing channel if `name_taken`.

### Pattern 5: destroy_slack.go case instrumentation

**What:** Each early-return branch in `destroySlackChannel` currently returns `nil` silently (after `log.Warn()` which is discarded). Adding `fmt.Fprintf(os.Stderr, ...)` before each `return nil` follows the established pattern.

**Full case map:**
| Case | Current behavior | Visibility fix |
|------|-----------------|----------------|
| A | `return nil` (no-op, no log) | `fmt.Fprintf(os.Stderr, "  Slack: no channel configured — teardown notification skipped\n")` |
| B | `log.Warn()` + `return nil` | `fmt.Fprintf(os.Stderr, "  Slack: shared/override mode — teardown notification skipped\n")` |
| E | `return nil` (no-op, no log after shouldArchive check) | `fmt.Fprintf(os.Stderr, "  Slack: archive disabled (slackArchiveOnDestroy=false) — channel %s kept\n", meta.SlackChannelID)` |
| F | `log.Warn()` + `return nil` | `fmt.Fprintf(os.Stderr, "  ⚠ Slack: /km/slack/bridge-url not configured — teardown notification skipped\n")` |
| G | `log.Warn()` + `return nil` | `fmt.Fprintf(os.Stderr, "  ⚠ Slack: failed to load operator signing key — teardown notification skipped: %v\n", err)` |
| H | `log.Warn()` + `return nil` | `fmt.Fprintf(os.Stderr, "  ⚠ Slack: final-post bridge call failed — archive skipped: %v\n", postErr)` |

Cases A and B are informational (no warning warranted — correct non-action). Cases E is informational. Cases F, G, H warrant `⚠` because they represent unexpected failures.

### Anti-Patterns to Avoid

- **Replacing `log.Warn()` calls entirely:** Keep the `log.Warn()` calls. They are useful in `--verbose` mode (when zerolog is NOT discarded) and for future log aggregation. ADD `fmt.Fprintf` alongside them; don't remove the existing calls.
- **Non-bounded retry loops:** The SLCK-11 retry must have a hard maximum (6 attempts). An unbounded loop on SSM errors would hang `km create` indefinitely.
- **Retrying on all SSM errors:** Only retry on errors consistent with "agent not yet registered." A permission error (IAM missing `ssm:SendCommand`) will never succeed and should fail fast after the first attempt. In practice, since all SSM errors return as `error` without type discrimination here, the bounded retry is acceptable — 6 × 5s = 30s is the worst-case for a permission error too, which is acceptable for a non-fatal step.

## Common Pitfalls

### Pitfall 1: `TimeoutSeconds` in SendCommand ≠ agent availability wait

**What goes wrong:** `productionSSMRunner.RunShell` sets `TimeoutSeconds: 30` on `SendCommand`. Developers assume this means "wait 30 seconds for the agent." It does not. `TimeoutSeconds` is the timeout for the shell command to execute once the agent receives it. If the agent is not registered, `SendCommand` returns an error immediately (typically `InvalidInstanceId` or a validation error). The 30s timeout never starts if the agent is absent.

**Why it happens:** AWS SSM documentation conflates two different timeouts. `TimeoutSeconds` in `SendCommand` controls command execution time. Agent availability is a separate concern.

**How to avoid:** Wrap `injectSlackEnvIntoSandbox` in the bounded retry loop described in Pattern 2. The retry loop is the correct mechanism for agent availability.

**Warning signs:** Step 11d always fires the `⚠ SSM SendCommand failed` branch on fresh creates, but succeeds if `km create` is re-run on the same sandbox.

### Pitfall 2: `destroySlackChannel` silent success-path on Cases A and B

**What goes wrong:** Cases A (`meta.SlackChannelID == ""`) and B (`!meta.SlackPerSandbox`) return `nil` with zero output. This is correct behavior, but without instrumentation it's indistinguishable from a swallowed error. The operator cannot tell whether `destroySlackChannel` ran and correctly skipped, or ran and silently failed.

**Why it happens:** The cases were designed as "correct non-actions" that don't need logging. But the zerolog discard means even the implicit "no-op" is invisible.

**How to avoid:** Add informational (non-warning) `fmt.Fprintf(os.Stderr, ...)` for Cases A and B. These are not `⚠` — they're expected paths. They provide audit confirmation that the destroy loop ran and correctly classified the sandbox.

### Pitfall 3: `SSMBotTokenFetcher` 15-min cache is per-Lambda-instance

**What goes wrong:** After revoking a bot token in the Slack App admin console, operators expect the bridge to immediately start failing. Instead, the bridge continues to work for up to 15 minutes because `SSMBotTokenFetcher.cache` is in-process memory on the Lambda execution environment. Slack accepts previously-fetched tokens until they are actually invalidated on Slack's side, but `auth.test` on a revoked token returns `invalid_auth` immediately. The bridge does NOT re-validate on every call — it only calls SSM on cache miss.

**Why it happens:** The 15-min in-process cache (`aws_adapters.go:187-238`) is a performance optimization that trades freshness for SSM API cost. On token revocation, the cache holds the old (now-revoked) token until TTL expires or the execution environment is recycled.

**How to avoid:** For SLCK-13, use `forceLambdaColdStart` targeting `km-slack-bridge` to force a new Lambda execution environment, which clears the in-process cache. Document in the operator runbook: "After revoking a token, either wait 15 minutes OR run `km slack rotate-token` (or the equivalent Lambda cold-start command) to force immediate invalidation."

**Important nuance:** There may be multiple Lambda execution environments (concurrent invocations). `UpdateFunctionConfiguration` marks the function for update, but in-flight execution environments may persist briefly. In practice, all environments recycle within 1-2 minutes of a configuration update.

### Pitfall 4: Case H diagnosis — `postErr` from `bridgePoster` is the canonical diagnostic

**What goes wrong:** When Case H fires (final-post bridge call failed), the current code discards `postErr` silently. The error contains the HTTP status code and response body from the bridge, which is the primary diagnostic. Without it, we cannot distinguish:
- `bad_signature` (operator key drift — unlikely post-`701a4cb` fix, but possible)
- `channel_mismatch` (should be impossible for operator-signed posts — bridge bug if seen)
- Network/timeout (Lambda cold start, DNS resolution)
- Bridge Lambda returning 500 (internal bridge error)

**How to avoid:** The SLCK-12 visibility fix specifically needs to include `postErr` in the stderr output for Case H: `fmt.Fprintf(os.Stderr, "  ⚠ Slack: final-post bridge call failed — archive skipped: %v\n", postErr)`. The `%v` of a `PostToBridge` error includes the HTTP status and response body.

### Pitfall 5: `loadSlackOperatorKey` uses `klanker-terraform` AWS profile implicitly

**What goes wrong:** `loadSlackOperatorKey` at `slack.go:525` calls `ssmClient.GetParameter` on `/sandbox/operator/signing-key`. The `ssmClient` is created from `awsCfg` in `destroy.go`, which is loaded with the operator's configured AWS profile (defaulting to `klankermaker` or whatever `--aws-profile` specifies). If the operator profile doesn't have SSM GetParameter permissions on this path, Case G fires.

**Why it matters for SLCK-12 diagnosis:** If Case G is firing, the diagnosis is an IAM permission gap, not a code bug. The fix is `km init` (which calls `EnsureSandboxIdentity` to re-provision the operator key) or correcting IAM.

**How to verify:** After SLCK-12 visibility ships, if Case G fires: run `aws ssm get-parameter --name /sandbox/operator/signing-key --with-decryption` manually with the operator profile. If it fails, IAM is the issue. If it succeeds, something in `loadSlackOperatorKey`'s decoding logic failed (unlikely).
