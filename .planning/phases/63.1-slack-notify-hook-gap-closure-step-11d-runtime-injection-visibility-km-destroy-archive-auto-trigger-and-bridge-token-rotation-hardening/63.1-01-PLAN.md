<objective>
Close SLCK-11: make `km create` step 11d Slack env injection visible AND fix the silent-failure root cause that prevented `KM_SLACK_CHANNEL_ID` / `KM_SLACK_BRIDGE_URL` from reaching `/etc/profile.d/km-notify-env.sh` on remote-created sandboxes (UAT Scen 3 finding, see `63-10-UAT.md` line 32).

The root mechanism is documented in `63.1-RESEARCH.md`: zerolog is silenced in non-verbose Lambda subprocess runs (`create.go:189-193`), so every `log.Warn()` in the step 11d branches at `create.go:790-825` writes to `io.Discard`. The likely root cause is SSM SendCommand timing — the sandbox SSM agent may not be reachable when `runner.Output` returns, so `injectSlackEnvIntoSandbox` returns an `InvalidInstanceId`-style error that gets discarded.

This plan ships TWO atomic commits (per CONTEXT.md sequencing — visibility FIRST, root-cause fix SECOND):
1. **Commit 1 (visibility):** Replace each silent `log.Warn()` in step 11d with `fmt.Fprintf(os.Stderr, ...)` matching the existing success line at `create.go:820`. 2. **Commit 2 (root-cause fix):** Wrap `injectSlackEnvIntoSandbox` in a bounded retry loop (6 attempts × 5s = 30s max) to handle the SSM-agent-readiness timing issue. The same stderr print pattern applies — final failure surfaces a single ⚠ line.
