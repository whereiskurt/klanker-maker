    Template for the new section:

    ```markdown
    ## Phase 87 — additionalSnapshots (snapshot-backed EBS volumes)

    Operators can declare a list of EBS snapshots in `spec.runtime.additionalSnapshots`. Each entry materialises a fresh `aws_ebs_volume` from the snapshot at sandbox creation, attaches it on `/dev/sd[f-p]`, and mounts it at the configured mountpoint using whatever filesystem is on the snapshot (detected via `blkid` at boot).

    ### Schema

    ```yaml
    spec:
      runtime:
        substrate: ec2-spot
        additionalSnapshots:
          - snapshotId: snap-0123abcdef0123456   # required, regex ^snap-[0-9a-f]{8,17}$
            mountPoint: /opt/models              # required, absolute path, not in reserved list
            device: /dev/sdh                     # optional, pinned device (else auto-picked from /dev/sd[f-p])
            encrypted: true                      # optional, omitted = inherit from snapshot
            size: 200                            # optional GiB, omitted/0 = inherit, must be >= snapshot.VolumeSize
    ```

    ### Validation layers

    **Layer 1 (`km validate`, no AWS calls):**
    - EC2-only substrate (docker / future ECS rejected)
    - snapshotId regex `^snap-[0-9a-f]{8,17}$`
    - mountPoint absolute + not in reserved blocklist (`/`, `/shared`, `/workspace`, `/proc`, `/sys`, `/dev`, `/etc`, `/usr`, `/var`, `/root`, `/home`, `/boot`, `/tmp`, `/run`, `/opt` top-level)
    - mountPoint must not collide with `additionalVolume.mountPoint` or other snapshot entries
    - device (if explicit) regex `^/dev/sd[f-p]$`, unique across entries
    - size (when set) >= 1

    **Layer 2 (`km create` pre-flight, AWS call):**
    - Single batched `DescribeSnapshots` covering all snapshotIds in the profile
    - Each snapshot must be in State=completed (pending/error rejected)
    - Each snapshot must be in the sandbox's target region
    - If size override is explicit, must be >= snapshot.VolumeSize
    - If caller lacks `ec2:DescribeSnapshots`, WARN and skip (terragrunt apply becomes the fallback failure path)

    ### Coexistence with additionalVolume

    Both fields can be set on the same profile. They live in different render fields; the compiler allocates `/dev/sd[f-p]` devices across both (additionalVolume first if set, then additionalSnapshots in declaration order). Pool exhaustion (>11 total entries minus AMI BDM volumes) → compile error.

    ### Lifecycle

    - Materialised volumes are created and destroyed with the sandbox (`km destroy` cleans them up).
    - The source snapshot is reference-only — terraform never touches it.
    - On `km destroy`, source snapshots SURVIVE; only the materialised volumes are deleted.

    ### Example

    See `profiles/example-additional-snapshots.yaml`. Replace the `snap-*` placeholder IDs with your own (`aws ec2 create-snapshot ...`).

    ### Module version

    Phase 87 ships as `infra/modules/ec2spot/v1.1.0/` (additive minor version per Phase 80 module-immutability convention). Existing sandboxes pinned to `v1.0.0` are unchanged; new sandboxes use `v1.1.0` via the bumped `infra/templates/sandbox/terragrunt.hcl` source path.
