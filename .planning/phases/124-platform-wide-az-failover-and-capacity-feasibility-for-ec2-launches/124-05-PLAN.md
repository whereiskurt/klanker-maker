  <name>Task 1: km create --wait-for-capacity outer backoff</name>
  <files>internal/app/cmd/create.go, internal/app/cmd/create_wait_capacity_test.go</files>
  <behavior>
    - `--wait-for-capacity` registered on the create cobra command; bare flag => default 30m, `=Nm` overrides; absent => off
    - When set and the AZ sweep exhausts all iterate-class AZs (no success, no fail-fast), wait ~5min and re-sweep until the deadline
    - A fail-fast class (quota/auth/invalid) breaks out immediately even with --wait-for-capacity (no point waiting on a quota wall)
    - The flag is NEVER added to the args passed to the cold-create Lambda km create subprocess
  </behavior>
  <action>
    Register a `--wait-for-capacity` flag on the create command (use a string/duration flag with
    NoOptDefVal "30m" so `--wait-for-capacity` alone means 30m and `--wait-for-capacity=15m` overrides;
    empty/absent => disabled). Follow the existing flag-registration style in NewCreateCmd.

    Wrap the existing AZ-sweep loop in an outer backoff: when --wait-for-capacity is set and the inner
    sweep returns "exhausted all iterate-class AZs without success", sleep ~5min (respect ctx
    cancellation) and re-run the sweep, until the deadline elapses; then fail. A fail-fast classification
    (ClassQuota/ClassAuth/ClassInvalid) must short-circuit the outer loop too — never wait on a quota wall
    (124-RESEARCH Open Question 3, Pitfall 5).

    CRITICAL: the cold-create Lambda runs `km create` as a subprocess (cmd/create-handler). Ensure
    --wait-for-capacity is NOT forwarded into that subprocess arg list — it is operator-path only
    (124-RESEARCH Open Question 3; design spec § Surface summary). Grep the create-handler arg assembly
    to confirm it is never appended.

    Add internal/app/cmd/create_wait_capacity_test.go: `TestNewCreateCmd_WaitForCapacityFlag` asserts the
    flag is registered with NoOptDefVal "30m" (bare => 30m, =Nm parses), and a unit assertion that the
    outer-backoff helper stops on a fail-fast class and on deadline. (Live re-sweep timing is a 124-06 gate.)
  </action>

  </verify>
