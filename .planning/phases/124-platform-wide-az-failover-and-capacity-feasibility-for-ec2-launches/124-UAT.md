## G4 — fail-fast on a GPU vCPU quota wall naming L-DB2E81BA — PASSED (2026-06-29, observed live)

Captured opportunistically: a recreate dispatched ~2 min after destroying the first g6.12xlarge hit a
**quota wall** because AWS vCPU-quota accounting lags termination (48 lingering + 48 new = 96 > 64
limit). The new `toolchain/km` classified `VcpuLimitExceeded` as `ClassQuota` and **fail-fasted in 47s
naming L-DB2E81BA + the increase URL — NOT a 4-AZ sweep** (create-handler Lambda log, sb-62226f75):

```
api error VcpuLimitExceeded: ... current vCPU limit of 64 ...
  ✗ Provisioning failed: GPU/vCPU quota limit reached.
  Quota code: L-DB2E81BA (Running On-Demand G and VT instances)
  Request an increase: https://console.aws.amazon.com/servicequotas/.../L-DB2E81BA
```
This is the exact fail-fast-vs-sweep behavior REQ-124-SWEEP/CLASSIFY built (a quota wall must not
masquerade as no-capacity and burn a sweep). A quota fail-fast correctly writes nothing to the store.

## Deploy-surface lessons (captured live)

1. **A partially-failed `km init` can leave binaries stale while the infra looks fully deployed.**
   The first deploy's S3 multipart upload of `sidecars/km` (and the run that should have refreshed
   `toolchain/km`) died on a flaky-network "Connection was closed" mid-upload, but the overall run
   carried on — terragrunt-applied resources (km-capacity table, 124-07 IAM grant) landed, yet
   `toolchain/km` stayed at the pre-124-07 build. `km doctor` showed the table green while the Lambda
   binary wielding it was old → the cold path silently ran without the write-back. **Verify binary
   freshness (S3 LastModified vs commit time), not just `km doctor` infra checks, after a deploy that
   errored mid-run.** Fix: rebuild `toolchain/km` (linux/arm64, stripped) + `sidecars/km` (linux/amd64)
   and upload directly with a resilient `aws s3 cp` (4MB chunks, low concurrency, `AWS_MAX_ATTEMPTS=10`).

2. **GPU vCPU quota accounting lags instance termination by minutes** (`L-DB2E81BA`), and **terraform's
   RunInstances can retry after km has already reported the create failed, orphaning a running GPU**
   (DDB row `failed`, instance `running`, nobody managing it). Destroy-then-immediately-recreate of a
   same-size GPU will hit the quota wall; wait for the quota to release, and after a failed GPU create
   sweep for orphaned instances by `tag:km:sandbox-id`. (Mirrors `project_gpu_capacity_az_and_l4_fallback`.)
