<objective>
Thread the new prefix and email_subdomain through Terragrunt and Terraform: extend site.hcl with two new site-level vars driven by KM_RESOURCE_PREFIX and KM_EMAIL_SUBDOMAIN; add var.resource_prefix to the five hardcoded Lambda modules and route every name attribute through it; **fix the lambda-slack-bridge module's hardcoded `function_name = "km-slack-bridge"` at main.tf:5** (Phase 67 drift — RESEARCH Pattern 5); update the seven DynamoDB live configs (5 existing + 2 new from Phase 67/68: dynamodb-slack-threads, dynamodb-slack-stream-messages) to use site.label in their table_name inputs; wire env-var-driven names into the Lambda module env blocks; **fix the lambda-slack-bridge live config (Pitfall 9 from RESEARCH)** by adding the missing `dependency "slack_threads"` block and four inputs (resource_prefix, signing_secret_path, slack_threads_table_name, sandboxes_table_name); and migrate the deferred HCL-template-emitting case in `pkg/compiler/budget_enforcer_hcl.go`.

Critical safety constraint: TF logical resource names (aws_dynamodb_table.budget, aws_iam_role.create_handler) MUST stay constant. Only `name` attribute values are parameterized. This avoids destroy/create on stateful resources (Pitfall 1).

**Lambda function rename caveat (RESEARCH Tertiary source #1):** Changing `function_name` on `aws_lambda_function` triggers destroy+create on the Lambda function resource. For the **default prefix** (`km`) on existing installs, this resolves to the same name (`km-slack-bridge`) so no replace happens. For custom-prefix installs migrating from default, manual `terraform state mv` may be needed — document explicitly in SUMMARY.

Purpose: Plans 02 and 03 prepared the Go code to consume env-var-driven names with safe fallbacks. Plan 04 makes the wires real on the AWS side: Lambda env blocks set the env vars, TF resources are created with prefix-aware names, and DynamoDB tables are renamed via the existing var.table_name pattern (no replace risk because table_name change is an in-place update for tables already using `name = var.table_name`).

</objective>
