# Plan 98-06 (gap closure) — fix GH-X-RESUME deployment defects found in live UAT

## Source of gaps

Live UAT on 2026-06-07 (paused `github-bot` sandbox `learn-566b57b5`, real GitHub PR #11).
The bridge correctly detected `status=paused`, took the resume branch, durably enqueued
the prompt, and the poller drained it on boot — but two real defects surfaced. (A third
finding, a missing per-sandbox github-token, is a **profile-config issue**, not a code
gap — see "Out of scope" below.)

### Gap A — ec2:StartInstances IAM condition never matches (the blocker)
`infra/modules/lambda-github-bridge/v1.1.0/main.tf` scoped `ec2:StartInstances` with
`Condition StringEquals aws:ResourceTag/km:managed = "true"`. **No km sandbox EC2 instance
carries a `km:managed` tag** — they carry `km:resource-prefix=<prefix>`, `km:sandbox-id`,
`km:label`. So every auto-resume `StartInstances` was denied:

```
ERROR github-bridge: auto-resume failed ... StartInstances ... StatusCode: 403
UnauthorizedOperation: ... not authorized to perform: ec2:StartInstances ...
because no identity-based policy allows the ec2:StartInstances action
```

The unit tests asserted the action string but never that a real instance satisfies the
condition — classic IAM-vs-runtime gap (`feedback_verify_deploy_surface_not_just_code`).

**Initial fix already applied** in this branch: condition changed to
`aws:ResourceTag/km:resource-prefix = var.resource_prefix`. This task FORMALIZES it with a
regression test + redeploy.

### Gap B — resume does not write DDB status → running
The bridge resumer (`EC2Resumer.StartSandbox`) only calls `StartInstances`; nothing writes
the `km-sandboxes` `status` back to `running`. Observed: instance state `running` while the
km-tracked `status` stayed `paused`. Consequences: `km list`/`km resume` show stale state,
and a follow-up @-mention re-reads `status=paused` and fires a redundant `StartInstances`.

## Out of scope (recorded, not fixed here)

- **Gap C — missing per-sandbox github-token (config, not code).** `learn.v2.yaml` declares
  `sourceAccess.github.allowedRepos: ["*"]`. With two App installations
  (`kurt-hundeck-greenhouse`, `whereiskurt`), wildcard-only repos make the installation
  ambiguous, so `generateAndStoreGitHubToken` skips the mint (loud stderr warning at create).
  `km-github` then gets `ParameterNotFound` and cannot post. Fix is operator-side: use an
  owner-scoped `allowedRepos` (`whereiskurt/*`) or pin `installation-id`. Tracked as the
  Task-3 prerequisite above. (Optional future hardening, not in this plan: have the bridge or
  poller surface "no per-sandbox token" more visibly than relying on the agent's prose.)
