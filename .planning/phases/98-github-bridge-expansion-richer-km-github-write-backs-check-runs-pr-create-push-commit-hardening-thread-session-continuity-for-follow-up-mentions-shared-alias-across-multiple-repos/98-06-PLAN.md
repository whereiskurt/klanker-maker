# Plan 98-06 (gap closure) — fix GH-X-RESUME deployment defects found in live UAT

## Source of gaps

Live UAT on 2026-06-07 (`github-bot` alias, boxes `learn-566b57b5` then `learn-57d26b4b`,
real GitHub PR #11). Across the session the full chain was eventually proven end-to-end —
auto-resume → status→running → poller drain → review → **`km-github` posted the review to
PR #11** (`klanker-maker[bot]` at 19:11:34Z). Getting there surfaced **five** defects. Two
(A, B) are already fixed + deployed + live-validated; three more (C, D, E) are folded into
this plan as Tasks 3–5. Full session detail in `98-UAT.md`.

### Gap A — ec2:StartInstances IAM condition never matches (the blocker)
`infra/modules/lambda-github-bridge/v1.1.0/main.tf` scoped `ec2:StartInstances` with
`Condition StringEquals aws:ResourceTag/km:managed = "true"`. **No km sandbox EC2 instance
carries a `km:managed` tag** — they carry `km:resource-prefix=<prefix>`, `km:sandbox-id`,
`km:label`. So every auto-resume `StartInstances` was denied:

```
ERROR github-bridge: auto-resume failed ... StartInstances ... StatusCode: 403
UnauthorizedOperation: ... not authorized to perform: ec2:StartInstances ...
because no identity-based policy allows the ec2:StartInstances action
```

The unit tests asserted the action string but never that a real instance satisfies the
condition — classic IAM-vs-runtime gap (`feedback_verify_deploy_surface_not_just_code`).

**Initial fix already applied** in this branch: condition changed to
`aws:ResourceTag/km:resource-prefix = var.resource_prefix`. This task FORMALIZES it with a
regression test + redeploy.

### Gap B — resume does not write DDB status → running
The bridge resumer (`EC2Resumer.StartSandbox`) only calls `StartInstances`; nothing writes
the `km-sandboxes` `status` back to `running`. Observed: instance state `running` while the
km-tracked `status` stayed `paused`. Consequences: `km list`/`km resume` show stale state,
and a follow-up @-mention re-reads `status=paused` and fires a redundant `StartInstances`.

### Gap C — resume timing race: resumer only matches `stopped`, no-ops on `stopping` (Task 3)
`EC2Resumer.StartSandbox` filters `DescribeInstances` with `instance-state-name=stopped`. A
quick pause-then-mention (pause at 18:07:55, bridge at 18:08:30 — 35s later) hit the box
mid-transition while it was still **`stopping`**, so the filter found nothing →
`no stopped EC2 instances found` → non-fatal give-up → the prompt enqueued but the box never
started (still paused). The resumer should tolerate `stopping` (include it in the filter and
wait briefly for `stopped` before `StartInstances`, or retry), so a fast pause→mention isn't
stranded. Confirmed NOT a fix regression — the same comment, reposted once the box was fully
stopped, resumed cleanly with no 403.

### Gap D — token mint robustness: contents:write demand + empty refresher input (Task 4)
Two coupled defects left every per-sandbox github-token absent (`ParameterNotFound`), so
`km-github` could not post until manually unblocked:
  1. **Over-broad permission request.** `create.go` hardcodes
     `githubWriteVerbs := {"comment","review","push","checks"}`; the `push` verb compiles to
     `contents:write`. GitHub's installation-token API is all-or-nothing — requesting any
     level the installation does not grant **422s the entire mint**. The `whereiskurt`
     installation grants `contents:read` (not write), so even a comment-only token died on
     the `contents:write` demand. Fix: request only the permissions the installation grants
     (intersect) and/or make the verb set profile-driven — a broader-than-granted ask must
     not nuke the token. (Operator is separately adding `Contents: write` to the App, which
     unblocks `push`/`pr create`; the robustness fix lands regardless so a read-contents
     install still mints a comment/review token.)
  2. **Empty refresher input.** When create hits `ErrAmbiguousInstallation` (wildcard-only
     `allowedRepos` + >1 installation), it bakes `installation_id:""` and `permissions:{}`
     into the static `km-github-token-refresher-{id}` schedule input, so the refresher can
     **never** mint — and the SSM `installation-id` pin doesn't help because the schedule
     uses its frozen input, not the live pin. Fix: resolve installation at create from the
     pinned SSM `installation-id` when repos are wildcard-only, and carry a non-empty
     `installation_id` + compiled `permissions` into the schedule input.

### Gap E — stale/cross-box session strands the FIFO (Task 5)
The sandbox-side poller reads `agent_session_id` from `km-github-threads` for `(repo,
number)` and runs `claude -p --resume <id>`. The PR #11 row pointed at a session from the
**first, destroyed** box (`learn-566b57b5`); on the new box that session doesn't exist →
`No conversation found with session ID …` → claude **exit 1** → poller returns the message
to the FIFO → **head-of-line block** + the poller wedged (spun 1m36s CPU, stopped polling).
Two fixes: (a) the poller must treat a resume failure (`No conversation found` / non-zero
under `--resume`) as recoverable — retry the turn **without** `--resume` (fresh session)
rather than failing the message; (b) a continuity row whose stored `sandbox_id` differs from
the current sandbox (box recreated) must be invalidated (drop the stale `agent_session_id`)
so it never drives a cross-box `--resume`.
