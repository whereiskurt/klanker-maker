<objective>
Add `var.activate_rule_set` (bool, default `true`) to the SES Terraform module to make the `aws_ses_active_receipt_rule_set` resource opt-in. The AWS SES API allows only ONE active receipt rule set per account/region. When a second km install runs `km init`, Terraform currently plans to steal activation from the first install, silently cutting inbound email for existing sandboxes.

The chosen design is **Option 1 — operator opt-in flag**: `activate_rule_set` defaults to `true` for backward compatibility (existing single-install `km init` plans are zero-diff). Second installs set `KM_SES_ACTIVATE_RULESET=false` before running `km init`, preventing the activation flip. The operator manually activates the correct rule set via AWS CLI only when they intend to hand off inbound email. This matches Phase 82's "isolation when operators want it, default when they don't" philosophy and avoids the complexity of Option 3 (shared rule set refactor).

Purpose: Close Gap 3 from Phase 82 VERIFICATION.md. Without this fix, any operator who runs `km init` for a second install (e.g., `resource_prefix: rg`) will inadvertently deactivate the first install's inbound email pipeline.

Output: Modified SES module (variables.tf + main.tf), updated live terragrunt.hcl, operator documentation in CLAUDE.md and OPERATOR-GUIDE.md. Followed by an operator checkpoint to verify the plan shows zero unexpected changes.
</objective>
