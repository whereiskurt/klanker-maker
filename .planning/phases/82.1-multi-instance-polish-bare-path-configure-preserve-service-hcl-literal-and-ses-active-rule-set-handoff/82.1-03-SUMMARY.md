## Decisions Made
- **Option 1 chosen (operator opt-in flag):** Default `true` preserves zero-diff for existing installs. Second install operators explicitly set `KM_SES_ACTIVATE_RULESET=false` before `km init`. Manual `aws ses set-active-receipt-rule-set` for deliberate handoff. This matches Phase 82's "isolation when operators want it, default when they don't" philosophy.
- **No `moved {}` block required:** Terraform 1.x with AWS provider v6.45.0 auto-detected the `aws_ses_active_receipt_rule_set.km_sandbox` → `aws_ses_active_receipt_rule_set.km_sandbox[0]` address change and handled it as a state move. The operator-run terragrunt plan confirmed: `Plan: 0 to add, 0 to change, 0 to destroy.`
- **`depends_on` lines left unchanged:** Non-indexed references to count-gated resources in `depends_on` are no-ops when count=0; Terraform 1.x handles them without error.

## Deviations from Plan

None - plan executed exactly as written.

The only observation worth noting: the plan's "IMPORTANT CAVEAT" warned that adding `count` to an already-deployed resource might cause a destroy+recreate requiring a `moved {}` block. Terraform 1.x handled this automatically — the caveat was correct to flag it, and the operator verification checkpoint was the right gate to catch this before any apply. The checkpoint outcome (auto-move, zero diff) validated that no remediation was needed.

## Issues Encountered
- None. The operator checkpoint verified the Terraform plan was safe before any `--dry-run=false` apply was attempted. Terraform's automatic state address migration handled the count-index change cleanly.

## User Setup Required

Second-install operators must set `KM_SES_ACTIVATE_RULESET=false` before running `km init` to prevent stealing SES activation from the primary install. See OPERATOR-GUIDE.md § SES activation handoff for the full runbook.

No configuration required for existing single-install deployments — the default `activate_rule_set=true` is a zero-diff change.
