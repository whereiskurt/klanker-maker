# Phase 80.1: Auto-detect existing OIDC provider — Context

**Gathered:** 2026-05-11
**Status:** Ready for planning
**Source:** In-conversation design after Phase 80 UAT discovery (no `/gsd:discuss-phase` ceremony — design is small enough that the locked decisions below were settled in a single design round-trip with the operator)

<domain>
## Phase Boundary

Extends Phase 80's `cluster-irsa` Terraform module and `km cluster add` CLI to support the case where an `aws_iam_openid_connect_provider` for the cluster's issuer URL **already exists** in the target AWS account.

**In scope:**
- Module variable + count-based conditional on `aws_iam_openid_connect_provider` resource (create vs reference-only)
- `data "aws_iam_openid_connect_provider"` data source for the reference-only branch
- CLI auto-detection via `aws iam list-open-id-connect-providers` before generating terragrunt.hcl
- `--register-oidc-provider=auto|true|false` operator override flag (default `auto`)
- README update + CLAUDE.md update reflecting the new behavior
- Test coverage: existing TestGenerateClusterHCL must still pass; new test for the false branch of the template; new test for auto-detection logic in the CLI

**Out of scope:**
- Changing the trust policy structure (Phase 80's local-OIDC-provider design is preserved for the true branch)
- Changing the `--oidc-provider-arn` flag (still required; still names the cluster's issuer for URL derivation regardless of who registered the provider)
- Splitting OIDC provider lifecycle into a separate `km cluster add-issuer` subcommand (option 3 from the design discussion — explicitly rejected in favor of option 2)
- Importing existing providers into Terraform state (we only *reference* the existing provider via data source; never adopt it)
- Refactoring the existing Phase 80 module shape (no `moved {}` blocks, no breaking variable renames)

</domain>

<decisions>
## Implementation Decisions (Locked)

### Module changes (`infra/modules/cluster-irsa/v1.0.0/`)

- New variable `register_oidc_provider` with type `bool` and default `true`. Default of `true` preserves Phase 80's behavior for any existing cluster-irsa stack that doesn't pass the new variable.
- `aws_iam_openid_connect_provider.this` becomes `count = var.register_oidc_provider ? 1 : 0`. The TLS data source becomes `count` on the same condition (no point fetching the thumbprint if we're not creating a provider).
- New `data "aws_iam_openid_connect_provider" "existing"` with `count = var.register_oidc_provider ? 0 : 1` and `url = local.oidc_provider_url`. AWS provider supports URL-based lookup.
- New local `oidc_provider_arn_local` = `var.register_oidc_provider ? aws_iam_openid_connect_provider.this[0].arn : data.aws_iam_openid_connect_provider.existing[0].arn`. Trust policy `principals.identifiers` becomes `[local.oidc_provider_arn_local]`.
- `outputs.tf` `oidc_provider_arn` returns `local.oidc_provider_arn_local` (operator-facing — they get the ARN that's in the trust policy, regardless of who registered it).

### CLI changes (`internal/app/cmd/cluster.go`)

- New flag on `km cluster add`: `--register-oidc-provider <auto|true|false>` (default `auto`).
- Auto-detection logic in `RunClusterAdd` BEFORE `generateClusterHCL` is called:
  - When flag is `auto` (default): call `aws iam list-open-id-connect-providers` (already-loaded SDK client; uses the same `klanker-application` profile as the rest of `km cluster add`). Find a provider whose URL matches the URL derived from `--oidc-provider-arn`. If found → set `register_oidc_provider=false` in the generated terragrunt.hcl. If not found → set `register_oidc_provider=true`. Log which mode was chosen and why.
  - When flag is `true`: force `register_oidc_provider=true`. Skip the IAM list call.
  - When flag is `false`: force `register_oidc_provider=false`. Skip the IAM list call.
- Operator-visible log line at INFO level: `"OIDC provider auto-detected: [creating | reusing existing arn:aws:iam::...]"` so the operator knows which path was taken without having to read the generated HCL.

### Template changes (`clusterTerragruntHCLTemplate` in cluster.go)

- New placeholder `{REGISTER_OIDC_PROVIDER}` substituted into a new `register_oidc_provider = {REGISTER_OIDC_PROVIDER}` line in the `inputs = { ... }` block.
- Substitution uses Go-side computed boolean → "true" / "false" string literal (HCL bool, not quoted).

### IAM permission

- `iam:ListOpenIDConnectProviders` is already covered by `km-operator-policy`'s IAM-related statements (verify in research). If not, this phase EXTENDS km-operator-policy/v1.0.0/ — note: that means both the create-handler Lambda role AND the cluster-irsa role get the new perm. Pre-flight `sts.GetCallerIdentity` in `km cluster add` may need IAM ListOpenIDConnectProviders permission on the *operator profile* — not the Lambda role — but operator typically has Admin so this is unlikely to matter.

### Backwards compatibility

- Existing cluster-irsa stacks (provisioned by Phase 80 before this phase ships) re-applying with the new module: `register_oidc_provider` defaults to `true`, which matches old behavior. **No `moved {}` blocks needed.** The `data` source has `count = 0` so it's not evaluated; the resource keeps its existing state address `aws_iam_openid_connect_provider.this[0]` (note: `[0]` index appears because count is now used, but Terraform's auto-migration handles `aws_iam_openid_connect_provider.this` → `aws_iam_openid_connect_provider.this[0]` via the address change — verify with `terraform plan` on a Phase-80-provisioned stack as part of UAT).
- If `terraform plan` shows a destroy+create on the OIDC provider in a re-applied old stack (because `[0]` indexing trips replacement), add a `moved {}` block: `moved { from = aws_iam_openid_connect_provider.this; to = aws_iam_openid_connect_provider.this[0] }`. This is a likely-needed change; researcher should confirm.

### Documentation updates

- `CLAUDE.md` Phase 80 section: replace the "One stack per issuer URL" limitation note with a description of the new behavior. Note the auto-detect is the default and the override flag exists.
- `docs/k8s/README.md`: add a "Same-account installation" subsection that walks through the case where the EKS cluster IS the klanker account. Note: the README currently uses the cross-account framing throughout — same-account is a one-paragraph addition, not a rewrite.

### Tests

- New unit test `TestGenerateClusterHCL_RegisterOidcProviderFalse`: regex-asserts the generated HCL contains `register_oidc_provider = false` when the placeholder is substituted with the false value.
- New unit test `TestAutoDetectOidcProvider`: mocks the IAM client, asserts the auto-detection logic correctly picks "register=false" when the mock returns a matching provider URL, and "register=true" otherwise. Add a seam to `RunClusterAdd` (or a new helper) for the IAM client interface so it's mockable. Same shape as the existing `ClusterRunner` interface from Plan 80-05.
- Existing 6 cluster tests from Phase 80 must continue to pass without modification.

### Integration test (phase-close gate)

- Run `km cluster add` against `klanker-application` with an OIDC provider URL whose provider is ALREADY registered in the application account (the residue from Phase 80's smoke test — `SMOKE-TEST-80` — was destroyed, so this phase's integration test must pre-register one manually OR target a real EKS cluster's issuer URL that the operator has already mirrored once).
- Expected: `km cluster add` logs "reusing existing OIDC provider", `terragrunt apply` succeeds, role provisioned with the existing provider as Principal, idempotency check passes, `km cluster rm` cleans up the role + skips destroying the (out-of-scope) pre-existing OIDC provider.

</decisions>

<specifics>
## Specific Ideas / References

- **Option 2 from design discussion (2026-05-11):** "auto-detect in the CLI" was chosen over option 1 (explicit flag only) and option 3 (data-source-only + separate `km cluster add-issuer` subcommand). The override flag `--register-oidc-provider` is the option-1 escape hatch retained on top of option 2.
- **Phase 80 module's existing `oidc_provider_host` local** and the host-derivation regex are unchanged — auto-detect uses the same derived URL.
- **Reference for URL-based data source lookup:** `aws_iam_openid_connect_provider` data source documentation. URL form: `https://oidc.eks.us-east-1.amazonaws.com/id/ABC123`.
- **Reference for IAM ListOpenIDConnectProviders SDK call:** `pkg/aws/` already wraps several IAM clients (verify in research). Same pattern.
- **Pitfall:** Terraform `count` indexing — Phase 80 module has unindexed `aws_iam_openid_connect_provider.this`. Adding `count` to it will rename the address to `[0]`. Verify whether Phase 80's existing stack survives re-apply or needs a `moved {}` block.
- **Pitfall:** `data.aws_iam_openid_connect_provider` returns an error if no provider matches. The `count = 0` branch must NOT evaluate the data source — confirm Terraform short-circuits data source evaluation when `count = 0`. (It does, but the planner/researcher should double-check the exact semantics.)

</specifics>

<deferred>
## Deferred Ideas

- **Importing an existing provider into Terraform state** (rather than referencing it via data source). Deferred — adds complexity for no benefit; the data source approach is read-only and lifecycle-safe.
- **Separate `km cluster add-issuer <url>` subcommand for explicit provider lifecycle.** Rejected in favor of option 2 (auto-detect). Could revisit if operators report confusion.
- **`km cluster rm` cleaning up the local OIDC provider if no other cluster-irsa stack references it.** Deferred — `terragrunt destroy` already destroys the provider in the register=true case; in the register=false case the provider was never owned by this stack so we correctly don't touch it. Multi-stack ref-counting is out of scope for this phase.
- **`km doctor` checks for OIDC providers without any cluster-irsa stack referencing them.** Deferred — separate observability work, not blocking the auto-detect feature.

</deferred>
