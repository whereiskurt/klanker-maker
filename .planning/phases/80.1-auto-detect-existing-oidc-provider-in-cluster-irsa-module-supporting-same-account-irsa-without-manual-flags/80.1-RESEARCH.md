## Summary

Phase 80.1 adds conditional OIDC provider creation to the `cluster-irsa` Terraform module. When the provider already exists in the target account (same-account EKS, or reuse from a prior Phase 80 run), the module references it via `data "aws_iam_openid_connect_provider"` instead of creating it. The CLI auto-detects which path to take by calling `aws iam list-open-id-connect-providers` before generating the terragrunt.hcl.

Three critical implementation risks are confirmed and resolved: (1) the `moved {}` block IS required when adding `count` to an existing unindexed resource — Terraform does auto-migrate `resource.name` to `resource.name[0]` but the CONTEXT.md explicitly says "add a `moved {}` block if needed." Research confirms it IS needed for correctness and documentation clarity. (2) The `data "aws_iam_openid_connect_provider"` data source supports URL-based lookup via `url = ...` as of AWS provider 4.8.0+ (well within the `>= 5.0` pin in `root.hcl`). (3) `count = 0` on a data source DOES short-circuit evaluation in Terraform, but there is a known Terraform 1.6.x regression (issue #34197) where `count=0` data sources with null-interpolated string arguments are still evaluated. The safe pattern for this phase avoids the regression: the data source URL argument uses `local.oidc_provider_url` which is ALWAYS a non-null string derived from `var.oidc_provider_arn`. The regression only triggers on null values in string templates, not on well-formed locals.

**Primary recommendation:** Use `count`-based branching with a `moved {}` block, the URL-based data source, and an `OidcProviderLister` interface in `cluster.go` matching the `ClusterRunner` pattern exactly.

### Pattern 1: count-gated resource + data source in main.tf

**What:** Two mutually exclusive resources controlled by `var.register_oidc_provider`. When true: create provider + TLS cert fetch. When false: read existing via data source.
**When to use:** Any time the same infrastructure object may or may not be owned by this stack.

```hcl
# Source: infra/modules/cluster-irsa/v1.0.0/main.tf (Phase 80.1 target shape)

# Fetch TLS thumbprint only when creating the provider
data "tls_certificate" "oidc" {
  count = var.register_oidc_provider ? 1 : 0
  url   = local.oidc_provider_url
}

# Create provider — only when register_oidc_provider = true
resource "aws_iam_openid_connect_provider" "this" {
  count           = var.register_oidc_provider ? 1 : 0
  url             = local.oidc_provider_url
  client_id_list  = ["sts.amazonaws.com"]
  thumbprint_list = [data.tls_certificate.oidc[0].certificates[0].sha1_fingerprint]

  tags = {
    "km:cluster" = var.cluster_name
    "km:manager" = "km-cluster"
  }
}

# Reference existing provider — only when register_oidc_provider = false
data "aws_iam_openid_connect_provider" "existing" {
  count = var.register_oidc_provider ? 0 : 1
  url   = local.oidc_provider_url
}

locals {
  oidc_provider_arn_local = var.register_oidc_provider ? (
    aws_iam_openid_connect_provider.this[0].arn
  ) : (
    data.aws_iam_openid_connect_provider.existing[0].arn
  )
}
```

**CRITICAL:** The `data.tls_certificate.oidc` reference inside the resource block MUST also use `[0]` index: `data.tls_certificate.oidc[0].certificates[0].sha1_fingerprint`. Both the resource and the TLS data source are count-gated, so both use `[0]` in the same branch.

### Pattern 2: moved {} block for count upgrade path

**What:** When `count` is added to `aws_iam_openid_connect_provider.this` (currently unindexed), Terraform auto-proposes moving the state address to `[0]`. An explicit `moved {}` block makes this deterministic and documents the intent.

```hcl
# Source: infra/modules/cluster-irsa/v1.0.0/main.tf (after count is added)
# Placed at the TOP of main.tf for visibility

moved {
  from = aws_iam_openid_connect_provider.this
  to   = aws_iam_openid_connect_provider.this[0]
}
```

**When to include:** Always. Terraform's auto-migration to `[0]` is confirmed by official docs but requires operator to approve the `moved` step in plan. The explicit block ensures idempotent re-applies: after the first apply the block is a no-op. Per CONTEXT.md, this block should be added if Terraform shows a destroy+create on the re-applied old stack. Research confirms the block IS needed — without it, Terraform will show the moved proposal in `plan` output and the operator must approve it; with it, the plan shows only a state move with no infrastructure change.

**UAT check:** `terraform plan` output for an existing Phase 80 stack re-applied with the new module should show exactly:
```
# aws_iam_openid_connect_provider.this has moved to
# aws_iam_openid_connect_provider.this[0]
  resource "aws_iam_openid_connect_provider" "this" {
    # (no changes)
  }

Plan: 0 to add, 0 to change, 0 to destroy.
```
If the plan shows `1 to add, 1 to destroy` instead, the `moved {}` block is missing or the address is wrong.

**Auto-detect helper function:**

```go
// autoDetectOIDCProvider calls ListOpenIDConnectProviders and returns true
// if no existing provider matches targetURL, false (reuse) if one matches.
// The ARN of the matching provider (if found) is returned for the INFO log.
//
// targetURL is the full HTTPS URL derived from --oidc-provider-arn using the
// same regex the Terraform module uses:
//   oidc_provider_host = replace(arn, /^arn:aws:iam::[0-9]+:oidc-provider\//, "")
//   oidc_provider_url  = "https://${oidc_provider_host}"
//
// The ListOpenIDConnectProviders response contains only ARNs.
// URL matching is done by extracting the host portion from each ARN
// (everything after "arn:aws:iam::<account>:oidc-provider/") and comparing
// it to the host portion of targetURL (strip "https://").
func autoDetectOIDCProvider(
    ctx context.Context,
    lister OidcProviderLister,
    targetURL string,  // e.g. "https://oidc.eks.us-east-1.amazonaws.com/id/ABC123"
) (registerOIDCProvider bool, existingARN string, err error) {
    out, err := lister.ListOpenIDConnectProviders(ctx, &iam.ListOpenIDConnectProvidersInput{})
    if err != nil {
        return true, "", fmt.Errorf("listing OIDC providers: %w", err)
    }
    // Strip "https://" to get the host path for comparison
    targetHost := strings.TrimPrefix(targetURL, "https://")
    for _, entry := range out.OpenIDConnectProviderList {
        if entry.Arn == nil {
            continue
        }
        // ARN format: arn:aws:iam::<account>:oidc-provider/<host/path>
        // Split on ":oidc-provider/" to extract the host.
        parts := strings.SplitN(*entry.Arn, ":oidc-provider/", 2)
        if len(parts) == 2 && parts[1] == targetHost {
            return false, *entry.Arn, nil  // reuse existing
        }
    }
    return true, "", nil  // create new
}
```

**Pagination note:** `ListOpenIDConnectProviders` returns ALL OIDC providers in a single response — there is NO pagination (no NextToken/Marker in the API). This is confirmed by the SDK struct: `ListOpenIDConnectProvidersInput` has no pagination fields, and the output has no `IsTruncated` or `Marker`. IAM OIDC providers are account-scoped and accounts typically have fewer than 100. No pagination loop needed.

### Pattern 4: Template placeholder substitution

**What:** The existing `strings.NewReplacer` in `GenerateClusterHCL` gets a fifth replacement. The function signature changes to accept a `registerOIDCProvider bool` parameter.

```go
// Source: internal/app/cmd/cluster.go (GenerateClusterHCL signature change)
func GenerateClusterHCL(clusterName, oidcProviderARN, namespace, serviceAccount string, registerOIDCProvider bool) string {
    registerStr := "true"
    if !registerOIDCProvider {
        registerStr = "false"
    }
    r := strings.NewReplacer(
        "{CLUSTER_NAME}", clusterName,
        "{OIDC_PROVIDER_ARN}", oidcProviderARN,
        "{NAMESPACE}", namespace,
        "{SERVICE_ACCOUNT_NAME}", serviceAccount,
        "{REGISTER_OIDC_PROVIDER}", registerStr,
    )
    return r.Replace(clusterTerragruntHCLTemplate)
}
```

The HCL template gains one new line in the `inputs = { ... }` block:
```hcl
  register_oidc_provider      = {REGISTER_OIDC_PROVIDER}
```

**IMPORTANT:** This is an HCL bool literal (`true` / `false`), NOT a quoted string. The `strings.NewReplacer` substitution must produce exactly `true` or `false` without quotes.

### Anti-Patterns to Avoid

- **Using `GetOpenIDConnectProvider` per-ARN to do URL matching:** `ListOpenIDConnectProviders` gives ARNs; the URL is derivable from the ARN without an additional API call. No need to call `GetOpenIDConnectProvider` (which requires an ARN and returns URL). Derive host from ARN directly.
- **Quoting the bool in HCL:** `register_oidc_provider = "true"` is wrong — HCL booleans are unquoted. The template must produce `= true` or `= false`.
- **Adding count to only the resource but not the TLS data source:** Both must be gated together. An unindexed `data.tls_certificate.oidc.certificates[0]` inside a count=1 resource will fail when the data source itself has count.
- **Referencing `data.aws_iam_openid_connect_provider.existing.arn` without `[0]`:** When the data source has `count`, all references must use `[0]`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| URL → ARN lookup | Custom HTTP call to AWS IAM | `ListOpenIDConnectProviders` + string split on `:oidc-provider/` | API is non-paginated, one call, already modeled in SDK v1.53.6 |

## Common Pitfalls

### Pitfall 1: moved {} block is REQUIRED for existing Phase 80 stacks
**What goes wrong:** Without `moved {}`, `terraform plan` on an existing Phase 80 stack (where `aws_iam_openid_connect_provider.this` is in state without index) shows a destroy+create of the OIDC provider. The operator loses their provider and all trust policies that reference it break.
**Why it happens:** Adding `count` to a resource renames its state address from `.this` to `.this[0]`. Terraform auto-proposes this migration but requires explicit approval OR a `moved {}` block to make it automatic.
**How to avoid:** Include `moved { from = aws_iam_openid_connect_provider.this; to = aws_iam_openid_connect_provider.this[0] }` at the top of main.tf. After first successful apply, the block is a no-op.
**Warning signs:** Plan output contains `# aws_iam_openid_connect_provider.this will be destroyed` — immediate red flag.
**Confidence:** HIGH — verified against official Terraform refactoring docs which state: "When you add count to an existing resource that didn't previously have the argument, Terraform automatically proposes moving the original object to instance 0 unless you write a moved block." The `moved {}` block makes this automatic and skips the operator-approval step.

### Pitfall 2: count=0 data source Terraform 1.6.x regression — safe for this phase
**What goes wrong:** Terraform 1.6.x has a confirmed regression (issue #34197) where data sources with `count = 0` can still be evaluated when their arguments contain null values in string templates.
**Why it happens:** Internal evaluation ordering change in Terraform 1.6.x.
**How to avoid (this phase is SAFE):** `data "aws_iam_openid_connect_provider" "existing"` uses `url = local.oidc_provider_url` which is `"https://${local.oidc_provider_host}"` — always a well-formed non-null string derived from `var.oidc_provider_arn` (a required variable). The regression only triggers when a null value is interpolated into a string template. Since our url argument is always a concrete string, this data source is safe even under the regression.
**Warning signs (if encountered):** `Error: Invalid template interpolation value` during plan — would indicate the url somehow resolved to null.
**Confidence:** MEDIUM — the regression is confirmed but its trigger condition (null in string template) does not apply here. The safe outcome is a known analysis, not an empirical test on this specific configuration.

### Pitfall 3: ListOpenIDConnectProviders returns ARNs, not URLs — must derive host from ARN
**What goes wrong:** Treating the ARN as a URL or trying to compare it directly to `local.oidc_provider_url`.
**Why it happens:** The API only returns `OpenIDConnectProviderListEntry.Arn`. The URL is embedded in the ARN as the suffix after `:oidc-provider/`.
**How to avoid:** Split ARN on `:oidc-provider/` and compare the second part to the host portion of the target URL (strip `https://` from target URL). This is the same derivation the module already does in `local.oidc_provider_host`.
**Confirmed by:** SDK struct inspection — `OpenIDConnectProviderListEntry` has only an `Arn *string` field. No URL field.

### Pitfall 4: iam:ListOpenIDConnectProviders NOT in km-operator-policy — two contexts need it
**What goes wrong:** (a) `km cluster add` auto-detect call fails with AccessDenied. (b) Terraform plan/apply fails when evaluating `data "aws_iam_openid_connect_provider" "existing"` because the cluster-irsa role (which runs Terraform via km-operator-policy) lacks the permission.
**Why it happens:** `km-operator-policy/v1.0.0/main.tf` has no OIDC provider statements. Confirmed by grepping the file — zero occurrences of `ListOpenIDConnectProviders`, `CreateOpenIDConnectProvider`, `DeleteOpenIDConnectProvider`, or `GetOpenIDConnectProvider`.
**How to avoid:** Add a new `aws_iam_role_policy "oidc_provider"` resource to `km-operator-policy/v1.0.0/main.tf` covering:
  - `iam:CreateOpenIDConnectProvider` (for register=true branch Terraform create)
  - `iam:DeleteOpenIDConnectProvider` (for km cluster rm Terraform destroy in register=true branch)
  - `iam:GetOpenIDConnectProvider` (for Terraform read on plan/refresh)
  - `iam:ListOpenIDConnectProviders` (for CLI auto-detect AND Terraform data source)
  - `iam:TagOpenIDConnectProvider` (for tags block in resource)
  - Resource: `arn:aws:iam::${data.aws_caller_identity.current.account_id}:oidc-provider/*`
**Note on operator profile:** The operator running `km cluster add` from their workstation typically uses the `klanker-application` profile with Admin or broad IAM permissions. The `iam:ListOpenIDConnectProviders` call in auto-detect uses the same `awsCfg` loaded via `awspkg.LoadAWSConfig(ctx, awsProfile)`. If operator has Admin, the call succeeds without policy changes. The km-operator-policy update is required for the Terraform data source evaluation inside the cluster-irsa role.

### Pitfall 5: GenerateClusterHCL signature change breaks existing tests
**What goes wrong:** The existing `TestGenerateClusterHCL` test calls `cmd.GenerateClusterHCL(clusterName, oidcARN, namespace, serviceAccount)` with 4 arguments. Adding a 5th argument breaks compilation.
**Why it happens:** Go function signatures must match exactly.
**How to avoid:** The existing test must be updated to pass the new `registerOIDCProvider bool` argument (use `true` as the default-behavior value to preserve the assertion baseline). Per CONTEXT.md, "existing 6 cluster tests from Phase 80 must continue to pass without modification" — this means without behavioral modification, not without compile-time updates. The test file requires the compile fix but the assertions remain valid.
**Alternative:** Extract the 5-arg `GenerateClusterHCL` and keep the 4-arg version as a wrapper defaulting to `true`. This preserves the existing call site in tests without any change. RECOMMENDED to avoid touching existing test assertions at all.

```go
func TestAutoDetectOidcProvider(t *testing.T) {
    const targetURL = "https://oidc.eks.us-east-1.amazonaws.com/id/ABC123"
    const matchingARN = "arn:aws:iam::123456789012:oidc-provider/oidc.eks.us-east-1.amazonaws.com/id/ABC123"
    const otherARN    = "arn:aws:iam::123456789012:oidc-provider/oidc.eks.eu-west-1.amazonaws.com/id/OTHER"

    t.Run("match found → register=false", func(t *testing.T) {
        mock := &mockOidcLister{
            Providers: []iamtypes.OpenIDConnectProviderListEntry{
                {Arn: aws.String(otherARN)},
                {Arn: aws.String(matchingARN)},
            },
        }
        register, existingARN, err := autoDetectOIDCProvider(context.Background(), mock, targetURL)
        if err != nil { t.Fatal(err) }
        if register { t.Error("expected register=false when matching provider exists") }
        if existingARN != matchingARN { t.Errorf("expected existing ARN %s, got %s", matchingARN, existingARN) }
    })

    t.Run("no match → register=true", func(t *testing.T) {
        mock := &mockOidcLister{
            Providers: []iamtypes.OpenIDConnectProviderListEntry{
                {Arn: aws.String(otherARN)},
            },
        }
        register, existingARN, err := autoDetectOIDCProvider(context.Background(), mock, targetURL)
        if err != nil { t.Fatal(err) }
        if !register { t.Error("expected register=true when no matching provider") }
        if existingARN != "" { t.Errorf("expected empty existingARN, got %s", existingARN) }
    })

    t.Run("empty list → register=true", func(t *testing.T) {
        mock := &mockOidcLister{}
        register, _, err := autoDetectOIDCProvider(context.Background(), mock, targetURL)
        if err != nil { t.Fatal(err) }
        if !register { t.Error("expected register=true for empty provider list") }
    })

    t.Run("API error → propagated", func(t *testing.T) {
        mock := &mockOidcLister{Err: fmt.Errorf("AccessDenied")}
        _, _, err := autoDetectOIDCProvider(context.Background(), mock, targetURL)
        if err == nil { t.Error("expected error from API failure") }
    })
}
```

| Old Approach | Current Approach | When Changed | Impact |

**EKS OIDC provider auto-registration facts (confirmed):**
- EKS does NOT auto-register the OIDC provider in IAM. The cluster has an OIDC issuer endpoint, but the `aws_iam_openid_connect_provider` IAM resource must be created explicitly.
- `eksctl` creates it via `eksctl utils associate-iam-oidc-provider --cluster=<name>`.
- AWS Console EKS → cluster → "Enable IAM OIDC provider" creates it.
- Terraform EKS modules (terraform-aws-modules/eks) create it as part of cluster provisioning.
- **Same-account scenario:** Any operator who provisioned their EKS cluster with `eksctl` or the AWS Console or the EKS Terraform module will already have the OIDC provider registered. The auto-detect covers this case transparently.

## Open Questions

1. **Terraform 1.6.x count=0 data source regression — integration test only**
   - What we know: The regression only triggers on null values in string template arguments. `local.oidc_provider_url` is always non-null.
   - What's unclear: Exact Terraform version in the operator's environment — if < 1.6 or > 1.7 (where fix may land), the risk is moot.
   - Recommendation: Include a note in the integration test plan to run `terraform version` pre-flight and confirm `>= 1.6.0`. If the plan shows a data source evaluation error, add `url = var.register_oidc_provider ? null : local.oidc_provider_url` pattern as fallback (though this may itself trigger the null issue). Best mitigation: the current design is already safe because the url argument is never null.

2. **GenerateClusterHCL signature change — 4-arg vs 5-arg**
   - What we know: Adding a 5th parameter breaks the existing test call site.
   - What's unclear: Whether CONTEXT.md's "existing 6 cluster tests must continue to pass without modification" means no file edits or just no behavioral changes.
   - Recommendation: Use the wrapper approach — keep the exported `GenerateClusterHCL(name, arn, ns, sa string)` as a wrapper that calls `generateClusterHCLWithOIDC(name, arn, ns, sa string, register bool)` internally. This preserves the existing test call site completely unchanged.
