# Phase 33.1 Plan 02: HCL Wiring and Terraform Module Branching for Raw AMI ID Summary

**One-liner:** Compiler emits discriminated-union `ami_slug`/`ami_id` HCL fields; ec2spot Terraform module branches on `use_slug_lookup` to skip `data.aws_ami` and resolve `local.effective_ami_id` for both instance resource types.

## What Was Built

### Task 1: Wire AMIID through ec2HCLParams, HCL template, and populate closures

Updated `pkg/compiler/service_hcl.go`:
- Added `log` import
- Extended `ec2HCLParams` struct with `AMIID string` field; updated comment to document the discriminated-union semantics (exactly one of AMISlug/AMIID is non-empty)
- Updated `ec2ServiceHCLTemplate` AMI section header to `Phase 33 / 33.1` and added `ami_id = "{{ .AMIID }}"` line with matching column alignment
- Replaced the `AMISlug` populate closure: returns `""` when `isRawAMIID()` is true; added new `AMIID` closure that returns the raw ID when `isRawAMIID()` is true
- Added hibernation+raw-AMI advisory via `log.Printf` in `validateEC2StorageFields` (not a hard error — Phase 56 workflows may use pre-encrypted private AMIs)

Updated `pkg/compiler/ec2_storage_test.go`:
- Added `bytes`, `log`, `os` imports
- Added `TestHibernationRawAMIWarning`: verifies compilation succeeds (no error), `ami_id` passthrough in HCL, and log output contains "hibernation" and "encrypted" keywords

Result: Plan 01's two intentionally-red tests (`TestAMIRawIDInHCL`, `TestAMISlugInHCLEmitsEmptyAMIID`) flipped from RED to GREEN.

### Task 2: Add var.ami_id and effective_ami_id to ec2spot Terraform module

Updated `infra/modules/ec2spot/v1.0.0/variables.tf`:
- Added `variable "ami_id"` (type=string, default="") immediately after `variable "ami_slug"`

Updated `infra/modules/ec2spot/v1.0.0/main.tf`:
- Added `use_slug_lookup = var.ami_slug != "" && local.total_ec2spot_count > 0` to the second locals block (where `total_ec2spot_count` is defined — required for forward reference to work)
- Changed `data "aws_ami" "base_ami"` count from `local.total_ec2spot_count > 0 ? 1 : 0` to `local.use_slug_lookup ? 1 : 0`
- Added new `locals` block immediately after the `data "aws_ami"` closing brace with `effective_ami_id` computed using a `length()` guard: raw-ID path uses `var.ami_id` directly; slug path uses `data.aws_ami.base_ami[0].image_id` only when `length > 0`
- Rewired `aws_spot_instance_request.ec2spot.ami` from `data.aws_ami.base_ami[0].image_id` to `local.effective_ami_id`
- Rewired `aws_instance.ec2_ondemand.ami` from `data.aws_ami.base_ami[0].image_id` to `local.effective_ami_id`
- Ran `terraform fmt` to resolve pre-existing column alignment drift

## Phase 56 Unblocked

A profile with `spec.runtime.ami: ami-0abcdef1234567890` now flows through the complete datapath:

1. `profile.ValidateSchema` accepts it (Plan 01 schema `oneOf`)
2. `generateEC2ServiceHCL` emits `ami_id = "ami-0abcdef1234567890"` and `ami_slug = ""`
3. Terraform `var.ami_id` receives the raw ID; `local.use_slug_lookup = false`; `data.aws_ami.base_ami` count = 0 (no AWS API call)
4. `local.effective_ami_id = "ami-0abcdef1234567890"` (direct passthrough)
5. `aws_instance.ami = local.effective_ami_id` → EC2 instance launches from the exact AMI

## OPERATOR ACTION REQUIRED: km init --sidecars

This phase changes two layers that are bundled into the management Lambda:

1. **JSON schema** (Plan 01): The `ami` field now accepts `ami-xxxxxxxx` IDs via `oneOf`. The Lambda's bundled km binary validates profiles using the old schema and will reject raw AMI IDs until refreshed.
2. **Terraform module** (this plan): `infra/modules/ec2spot/v1.0.0/main.tf` and `variables.tf` now include `var.ami_id`, `local.use_slug_lookup`, and `local.effective_ami_id`. Remote `km create` calls run against the Lambda's cached module copy.

**After merging Phase 33.1, the operator must run:**

```bash
km init --sidecars
```

This refreshes the Lambda's bundled km binary and Terraform module copies. Without this step, `km create` on a profile with `ami: ami-xxxxxxxx` will fail at the Lambda's schema validation stage (see memory: `project_schema_change_requires_km_init.md`).
