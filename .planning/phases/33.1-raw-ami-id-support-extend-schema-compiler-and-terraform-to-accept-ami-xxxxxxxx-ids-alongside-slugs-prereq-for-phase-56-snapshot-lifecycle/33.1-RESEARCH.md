## Summary

Phase 33 shipped `spec.runtime.ami` as a three-value enum (plus empty string fallback). The JSON schema enforces this via `enum: [...]` and the Terraform module resolves all AMI IDs indirectly through a `locals.ami_filters` map keyed by `var.ami_slug`. A raw AMI ID (e.g. `ami-0abcdef1234567890`) fails the schema enum check immediately and would also fail the `ami_filters[local.resolved_ami_slug]` map lookup at Terraform plan time.

Phase 56 (snapshot lifecycle) will write `ami-xxxxxxxx` strings produced by EC2 `create-image` calls into generated profiles. This makes Phase 33.1 a hard prerequisite: the entire schema → compiler → Terraform dataflow must be taught to distinguish "this is a slug, resolve via filter" from "this is a raw ID, pass directly to the instance resource."

The change is deliberately tight in scope. The three-slug path must be byte-identical in behavior before and after the change (verified by the existing 20-truth test suite). The raw-ID path adds a new code branch at each of three layers: JSON schema (`oneOf` replacing `enum`), Go compiler (`isRawAMIID()` helper + new HCL field), and Terraform (count-conditional `data.aws_ami` + `coalesce`-style locals).

**Primary recommendation:** Detect raw vs slug in Go (single source of truth), emit two distinct HCL fields (`ami_slug` and `ami_id`), and use a count-conditional `data.aws_ami` in Terraform that fires only when `var.ami_slug != ""`.

### Pattern 1: Schema — `oneOf` to accept slug enum OR raw ID pattern

**What:** Replace the current `enum` constraint with a `oneOf` that accepts either the existing slug enum values or a string matching `^ami-[0-9a-f]{8,17}$`.

**When to use:** When a field must accept a closed set of named values AND an open set of values with a known format.

**JSON Schema Draft 2020-12 form:**

```json
"ami": {
  "type": "string",
  "description": "AMI slug resolved per-region, or a raw AMI ID (ami-xxxxxxxx). Empty or omitted defaults to amazon-linux-2023.",
  "oneOf": [
    {
      "enum": ["amazon-linux-2023", "ubuntu-24.04", "ubuntu-22.04", ""]
    },
    {
      "pattern": "^ami-[0-9a-f]{8,17}$"
    }
  ]
}
```

**Why `oneOf` not `anyOf`:** `oneOf` enforces exactly-one branch matches, which produces a better error message when neither branch matches (e.g. `"ami-ZZZZ"` would say "must match one of: enum or pattern" rather than "at least one of failed"). Either works functionally but `oneOf` is more explicit. Note: `""` (empty string) must stay in the enum branch because it is the "use default" sentinel.

**Confidence:** HIGH — santhosh-tekuri/jsonschema v6 fully implements Draft 2020-12 `oneOf`, and this project already uses `pattern` at metadata.prefix and `enum` at multiple fields. No behavioral risk.

**Important:** The existing `type: "string"` keyword at the top level of the `ami` property can coexist with `oneOf` on the inner branches. The outer `type` still constrains the raw value to be a string; the `oneOf` provides further constraints. Tested pattern in this schema: `metadata.prefix` uses top-level `type: string` + `pattern` together.

### Pattern 2: Go compiler — `isRawAMIID()` helper + two-field emission

**What:** Add a small pure function that detects raw AMI IDs, then emit two separate HCL fields from the template: `ami_slug` (empty string when raw) and `ami_id` (empty string when slug).

**When to use:** When the same profile field controls a branching decision in downstream infrastructure but the downstream system (Terraform) cannot easily introspect string format.

```go
// Source: pkg/compiler/service_hcl.go (new helper)
// isRawAMIID returns true if s looks like a raw EC2 AMI ID (ami-[8..17 hex chars]).
// This is the single source of truth for slug vs raw-ID detection in this repo.
var rawAMIIDPattern = regexp.MustCompile(`^ami-[0-9a-f]{8,17}$`)

func isRawAMIID(s string) bool {
    return rawAMIIDPattern.MatchString(s)
}
```

**ec2HCLParams additions:**

```go
// Phase 33.1: raw AMI ID support
// Exactly one of AMISlug and AMIID will be non-empty at render time.
// AMISlug is set when spec.runtime.ami is a slug (or empty → default slug).
// AMIID  is set when spec.runtime.ami is a raw ami-xxxxxxxx ID.
AMIID string // raw EC2 AMI ID; empty when AMISlug is used
```

**Populate in `generateEC2ServiceHCL`:**

```go
// Phase 33.1: distinguish raw AMI ID from slug
AMISlug: func() string {
    if p.Spec.Runtime.AMI != "" && !isRawAMIID(p.Spec.Runtime.AMI) {
        return p.Spec.Runtime.AMI
    }
    if p.Spec.Runtime.AMI == "" {
        return "amazon-linux-2023"
    }
    return "" // raw ID: AMISlug intentionally empty
}(),
AMIID: func() string {
    if isRawAMIID(p.Spec.Runtime.AMI) {
        return p.Spec.Runtime.AMI
    }
    return ""
}(),
```

**HCL template change** — replace the existing single `ami_slug` line with two lines:

```
    ami_slug               = "{{ .AMISlug }}"
    ami_id                 = "{{ .AMIID }}"
```

**Why two fields, not one?** The Terraform module needs to distinguish "perform a `data.aws_ami` lookup" from "use this ID directly." A single `ami` field with mixed semantics would require Terraform to do its own `startswith(var.ami, "ami-")` detection — possible but less explicit. Two fields match the discriminated-union pattern used elsewhere in the codebase (e.g. `use_spot` + spot-specific fields).

**Backward compatibility:** When `AMISlug` is a known slug and `AMIID` is `""`, the behavior is byte-identical to the current code path. The only change to the generated HCL is the addition of `ami_id = ""` which is a no-op in the Terraform module.

### Pattern 3: Terraform — count-conditional `data.aws_ami` + `effective_ami` local

**What:** Make the `data.aws_ami` lookup conditional on whether `var.ami_slug` is non-empty (slug path), and compute `local.effective_ami_id` as the resolved ID regardless of which path was taken.

**When to use:** Conditional data source reads in Terraform where the count must be a deterministic integer known at plan time.

```hcl
# Phase 33.1: branch between slug lookup and raw ID passthrough.
# When ami_slug is non-empty, run the data.aws_ami lookup.
# When ami_id is provided directly, skip the lookup.
locals {
  # ... existing ami_filters map unchanged ...
  resolved_ami_slug = var.ami_slug != "" ? var.ami_slug : ""
  use_slug_lookup   = var.ami_slug != "" && local.total_ec2spot_count > 0
}

data "aws_ami" "base_ami" {
  count = local.use_slug_lookup ? 1 : 0   # was: local.total_ec2spot_count > 0 ? 1 : 0

  most_recent = true
  owners      = [local.ami_filters[local.resolved_ami_slug].owner]
  # ... rest unchanged ...
}

locals {
  # effective_ami_id resolves to the correct ID regardless of which branch was used.
  # Slug path: data.aws_ami lookup result.
  # Raw ID path: var.ami_id passed through directly.
  effective_ami_id = var.ami_id != "" ? var.ami_id : data.aws_ami.base_ami[0].image_id
}
```

**Use `effective_ami_id` everywhere `data.aws_ami.base_ami[0].image_id` was previously used:**

```hcl
resource "aws_spot_instance_request" "ec2spot" {
  ami = local.effective_ami_id
  # ...
}

resource "aws_instance" "ec2_ondemand" {
  ami = local.effective_ami_id
  # ...
}
```

**Important caveat on the locals block split:** Terraform requires `data.aws_ami.base_ami[0].image_id` to be referenced only when `count = 1`. The `effective_ami_id` local uses a conditional that avoids evaluating `data.aws_ami.base_ami[0].image_id` when `count = 0`. However, Terraform's evaluation model evaluates both branches of a ternary regardless — this means `data.aws_ami.base_ami[0].image_id` will still be evaluated even when count=0, causing a plan-time error. **The correct pattern is to use `try()`:**

```hcl
effective_ami_id = var.ami_id != "" ? var.ami_id : try(data.aws_ami.base_ami[0].image_id, "")
```

But `try()` with an empty fallback creates a silent failure if both are empty. **The cleanest production-safe pattern is:**

```hcl
effective_ami_id = (
  var.ami_id != ""
  ? var.ami_id
  : (local.use_slug_lookup ? data.aws_ami.base_ami[0].image_id : "")
)
```

This is valid because `data.aws_ami.base_ami[0].image_id` is only reached in the branch where `local.use_slug_lookup = true`, which is exactly when `count = 1`. Terraform evaluates the ternary lazily in this form — the inner conditional on `use_slug_lookup` guards the count-0 case.

**Confidence:** HIGH — this count-conditional pattern is the established idiom in this repo (see `aws_ebs_volume.additional` at line 549, `aws_iam_role.ec2spot_ssm` at line 209 in `main.tf`). The nested ternary with a `local.flag` guard is the correct way to avoid plan-time errors on count-0 data sources.

**New Terraform variable** (`variables.tf`):

```hcl
variable "ami_id" {
  type        = string
  description = "Raw EC2 AMI ID (ami-xxxxxxxx). When set, ami_slug is ignored and no data.aws_ami lookup is performed."
  default     = ""
}
```

### Anti-Patterns to Avoid

- **Doing raw-ID detection in Terraform via `startswith(var.ami, "ami-")`:** Puts detection logic in two places (Go + HCL). Single source of truth belongs in Go.
- **Combining `ami_slug` and `ami_id` into a single `ami` Terraform variable:** The module's internal branching becomes harder to read and test. Two variables with clear semantics are better.
- **Using `try(data.aws_ami.base_ami[0].image_id, "")` as the count-0 guard:** `try()` swallows errors silently. The nested ternary with `local.use_slug_lookup` guard is explicit.
- **Removing `ami_slug` from the HCL template:** Existing sandbox profiles that have `ami: ubuntu-24.04` in their stored metadata would need no recompilation if the compiler always emits both fields. Dropping `ami_slug` would be a breaking change.
- **Validating the AMI ID by calling AWS API from the Go compiler:** Out of scope for this phase; Terraform plan will surface invalid AMI IDs. The compiler's job is structural validation only.

## Common Pitfalls

### Pitfall 1: Terraform plan-time error on count-0 data source reference

**What goes wrong:** `local.effective_ami_id = ... data.aws_ami.base_ami[0].image_id ...` — even the unselected branch of a ternary is evaluated during plan. When `count = 0`, `data.aws_ami.base_ami[0]` is an empty tuple and `.image_id` panics.

**Why it happens:** Terraform evaluates both sides of a ternary to determine the result type, even if one branch is never taken at apply time.

**How to avoid:** Gate the reference behind a local boolean that matches the count: `local.use_slug_lookup ? data.aws_ami.base_ami[0].image_id : ""` — Terraform's evaluation model does evaluate both sides, BUT when `use_slug_lookup = false` and `count = 0`, the tuple is empty and the right-side `""` is selected. However, accessing index `[0]` on an empty tuple still raises an error. The safe form is to match the count guard exactly: only reference `data.aws_ami.base_ami[0].image_id` inside the same conditional that controls `count`. In practice: put `effective_ami_id` in a second `locals` block after the `data` block, and use `length(data.aws_ami.base_ami) > 0 ? data.aws_ami.base_ami[0].image_id : var.ami_id`.

**Warning signs:** `Error: Invalid index — The given key does not identify an element in this collection value.` during `terraform plan` with `ami_slug = ""` and `ami_id = "ami-xxx"`.

### Pitfall 2: `oneOf` produces confusing validation errors for partially-matching values

**What goes wrong:** A value like `"ubuntu-25.04"` (a new slug not yet in the enum) passes the `pattern` branch because it does NOT match `^ami-[0-9a-f]{8,17}$`, and it does NOT match the enum. `oneOf` reports "matched 0 of 1 required" — the error is "one of these must match" rather than "value not in allowed list." This is acceptable behavior (the error still tells the user what's valid) but test expectations must account for it.

**How to avoid:** Accept this as intentional. Document in the schema description field what values are allowed. The validator's error message surface is controlled by santhosh-tekuri/jsonschema — no custom error formatting needed.

### Pitfall 3: AMI ID format — 8 vs 17 hex characters

**What goes wrong:** Using `^ami-[0-9a-f]{17}$` (17-only) rejects the old 8-character format. Legacy snapshots or public AMIs created before June 2018 use the 8-character format.

**Why it happens:** AWS migrated from 8-char to 17-char resource IDs in 2018. Both formats remain valid and are returned by `aws ec2 describe-images`.

**How to avoid:** Pattern must be `^ami-[0-9a-f]{8,17}$` (8 to 17 inclusive). The Phase 56 snapshot workflow will produce 17-char IDs (all new AWS resources), but accepting 8-char IDs costs nothing and avoids a foot-gun.

**Note on case:** AWS AMI IDs are lowercase hex. The pattern `[0-9a-f]` is correct. Do NOT use `[0-9a-fA-F]` — AWS APIs always return lowercase, and accepting uppercase would allow non-canonical forms that would fail `aws ec2 describe-images`.

### Pitfall 4: Hibernation + raw AMI ID requires encrypted root — compile-time validation gap

**What goes wrong:** A profile with `hibernation: true` and `ami: ami-0abc...` (unencrypted AMI) compiles fine, but `terraform apply` fails with `InvalidParameterCombination: Hibernation is not supported for instances with unencrypted root EBS volumes.`

**Why it happens:** The existing `validateEC2StorageFields` enforces that `hibernation: true` requires `spot: false`, but it does not know whether a raw AMI has an encrypted root volume — that requires an AWS API call.

**How to avoid (recommended for this phase):** Emit a compiler-level warning (not error) when `Hibernation = true && isRawAMIID(ami)`, indicating the operator must ensure the AMI has an encrypted root volume. Phase 33 already handles the slug path by setting `encrypted = var.hibernation_enabled` on the `root_block_device`. For a raw-ID path, the AMI's own root volume encryption state is what Terraform uses; there is no `root_block_device` override that can retroactively encrypt an unencrypted AMI's root. Document this as a known limitation. Do NOT make it a hard error because the raw AMI might already have an encrypted root.

**Warning signs at apply time:** `Error: Error modifying EC2 Instance ... InvalidParameterCombination` with hibernation message.

### Pitfall 5: Cross-region constraint — raw AMI IDs are region-scoped

**What goes wrong:** A profile specifies `ami: ami-0abc123...` which exists in `us-east-1`, but `spec.runtime.region: us-west-2`. Terraform applies in `us-west-2` and fails with `InvalidAMIID.NotFound`.

**Why it happens:** AMI IDs are not global — they are per-region. The operator or the Phase 56 snapshot exporter must ensure the AMI exists in the target region.

**How to avoid:** Add a compiler-level comment (not error) noting that raw AMI IDs are region-scoped. Hard rejection would be too aggressive (Phase 56 ensures region alignment). Document in the AMI field's schema description. Phase 56 is responsible for ensuring the snapshot-generated profile specifies the same region. This phase just passes the ID through.

### Pitfall 6: Schema `additionalProperties: false` on `runtime` — need to ensure `ami` property remains the only property being changed

**What goes wrong:** The `runtime` object in the JSON schema has `additionalProperties: false`. Any typo in the property name `ami` (e.g. `amiId`) during the schema edit would silently allow the field but it would never populate in Go since the YAML unmarshal key is `ami`.

**How to avoid:** Both the JSON schema property key and the Go struct `yaml:"ami"` tag must remain `ami`. No new Go field needed — the existing `AMI string yaml:"ami"` field accepts raw IDs after this change.

## Open Questions

1. **`locals.resolved_ami_slug` with empty `ami_slug`**
   - What we know: `main.tf` line 30: `resolved_ami_slug = var.ami_slug != "" ? var.ami_slug : "amazon-linux-2023"`. When `var.ami_slug = ""` (raw-ID path), `resolved_ami_slug` falls back to `"amazon-linux-2023"`.
   - What's unclear: If `data.aws_ami.base_ami` count = 0, `local.ami_filters[local.resolved_ami_slug]` is still referenced in the `data` block's `owners` and `filter` arguments. With `count = 0`, Terraform should not evaluate those arguments, but the behavior depends on provider version.
   - Recommendation: Keep `resolved_ami_slug` logic unchanged. The `count = 0` conditional on `data.aws_ami` blocks argument evaluation in modern Terraform (>= 0.13). If a plan-time error occurs, switch `owners` to `local.use_slug_lookup ? [local.ami_filters[local.resolved_ami_slug].owner] : ["self"]` as a fallback.

2. **Hibernation + raw AMI validation depth**
   - What we know: Hibernation requires an encrypted root volume. The slug path handles this via `encrypted = var.hibernation_enabled` in `root_block_device`. A raw AMI's root volume encryption state is fixed at AMI creation time.
   - What's unclear: Should the compiler emit a hard error or a warning?
   - Recommendation: Emit a **warning string** (returned alongside the compiled HCL, not an error) rather than blocking compilation. Hard error would prevent Phase 56 workflows where the operator knows their snapshot AMI has an encrypted root. Planner should add a warning emission path to `validateEC2StorageFields` if needed, or defer to Phase 56.
