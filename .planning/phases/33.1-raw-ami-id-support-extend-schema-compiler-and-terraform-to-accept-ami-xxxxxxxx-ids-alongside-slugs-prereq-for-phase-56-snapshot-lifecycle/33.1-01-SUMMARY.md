# Phase 33.1 Plan 01: Schema and Compiler Helper for Raw AMI ID Support Summary

**One-liner:** JSON schema `ami` field extended to `oneOf(slug enum, ^ami-[0-9a-f]{8,17}$ pattern)` with `isRawAMIID()` helper and Wave 0 red tests targeting Plan 02's HCL wiring.

## What Was Built

### Task 1: Schema oneOf and raw-ID schema tests

Updated `pkg/profile/schemas/sandbox_profile.schema.json` to replace the single `enum` constraint on the `ami` property with a `oneOf` containing two subschemas:
- Branch 1: `enum` of the 4 valid slug values (`amazon-linux-2023`, `ubuntu-24.04`, `ubuntu-22.04`, `""`)
- Branch 2: `pattern: ^ami-[0-9a-f]{8,17}$` matching 8-17 lowercase hex char AMI IDs

Added two test functions to `pkg/profile/schema_storage_test.go`:
- `TestSchemaAMIRawIDValid`: table-driven, covers 17-char and 8-char forms; asserts `ValidateSchema` returns no errors and `Parse` round-trips the value verbatim
- `TestSchemaAMIRawIDInvalid`: table-driven, covers 6 rejection cases (uppercase hex, too-short, too-long, no hex chars, unknown slug, uppercase prefix); asserts `ValidateSchema` returns errors

### Task 2: isRawAMIID() helper and Wave 0 test scaffold

Added to `pkg/compiler/service_hcl.go`:
- `regexp` import
- `rawAMIIDPattern = regexp.MustCompile(`^ami-[0-9a-f]{8,17}$`)` — package-level compile-once var
- `isRawAMIID(s string) bool` — single detection point for slug vs raw ID in the compiler

Added three test functions to `pkg/compiler/ec2_storage_test.go`:
- `TestIsRawAMIID`: 8-case table-driven test, all passing (helper exists and is correct)
- `TestAMIRawIDInHCL`: Wave 0 scaffold — asserts `ami_id = "ami-0abcdef1234567890"` and `ami_slug = ""` in HCL; intentionally RED (Plan 02 target)
- `TestAMISlugInHCLEmitsEmptyAMIID`: Wave 0 scaffold — asserts `ami_id = ""` alongside `ami_slug = "ubuntu-24.04"`; intentionally RED (Plan 02 target)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Used ValidateSchema instead of Parse for schema rejection tests**
- **Found during:** Task 1 TDD RED phase
- **Issue:** Plan said "profile.Parse returns error" for rejection tests, but `profile.Parse` only does YAML unmarshaling — it does not run JSON schema validation. Raw AMI IDs like `"ami-0abcdef1234567890"` unmarshal without error regardless of schema.
- **Fix:** `TestSchemaAMIRawIDValid` uses `profile.ValidateSchema` to assert no schema errors (plus `profile.Parse` to verify round-trip). `TestSchemaAMIRawIDInvalid` uses `profile.ValidateSchema` to assert schema errors. This matches the pattern used by `TestSchemaRootVolumeValidation` in the same file.
- **Files modified:** `pkg/profile/schema_storage_test.go`
- **Commit:** 16f8eb8
