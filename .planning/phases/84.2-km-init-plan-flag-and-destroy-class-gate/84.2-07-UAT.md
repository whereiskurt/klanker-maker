### Scenario 2: Synthetic protected destroy trips gate
**Result:** FAIL — gate did NOT trip. Filed as Phase 84.3 gap (see below).

Edit: commented out the `aws_ses_receipt_rule.sandbox_catchall` resource block in `infra/modules/ses/v2.0.0/main.tf` plus its dangling output in `outputs.tf` (the first run errored because the output still referenced the resource — captured as bonus evidence for Scenario 7).

Re-run output (after commenting the output too):

```
Planning ses... 0 to add, 0 to change, 1 to destroy ⚠

Total across 16 modules: 0 to add, 3 to change, 1 to destroy

Run 'km init --dry-run=false' to apply.

```

The per-module summary correctly shows `1 to destroy ⚠` for the `ses` module, but:
- Aggregate footer ends with the apply hint (no trip block)
- Exit code is 0 (gate did not trip)

**Root cause:** `pkg/terragrunt/planreport/protected.go` ProtectedTypes includes `aws_ses_receipt_rule_set` (the parent rule set) but **not** `aws_ses_receipt_rule` (individual rules). The Phase 82→84 cutover destroyed individual receipt rules — exactly the case the gate exists to catch — so this is a regression of the phase's stated requirement DESTROY-CLASS-GATE.

Working tree restored via `git checkout -- infra/modules/ses/v2.0.0/main.tf infra/modules/ses/v2.0.0/outputs.tf`.

## Phase 84.3 gap entries

### GAP-1 (blocking, regression-of-purpose) — `aws_ses_receipt_rule` missing from ProtectedTypes

**Requirement:** DESTROY-CLASS-GATE — Phase 84.2 must trip the gate on any destroy of resource types whose loss caused the Phase 82→84 incident.

**Observed:** `aws_ses_receipt_rule` destroys pass the gate (exit 0, no trip block). Only the parent `aws_ses_receipt_rule_set` is protected.

**Why this matters:** The Phase 82→84 cutover destroyed the legacy rule set's child rules (operator inbound + sandbox catchall) via the `removed { destroy = true }` orphan path. Those children are `aws_ses_receipt_rule` instances. Today's gate does not catch a re-occurrence of the exact incident it was built for.

**Fix sketch:**
- Add `aws_ses_receipt_rule` to `pkg/terragrunt/planreport/protected.go` with an incident-link comment.
- Add a gate_test.go fixture exercising a single `aws_ses_receipt_rule` destroy.
- Re-run Scenario 2 + 3 in Phase 84.3 UAT.

### GAP-2 (non-blocking, validation-doc drift) — Scenario 4b synthetic edit blocked by `prevent_destroy`

**Observed:** `aws_ses_receipt_rule_set.shared` has `lifecycle.prevent_destroy = true` (foundation module line 92-94), so any synthetic destroy edit causes terraform to error before the gate runs — different failure mode than the validation doc projects.

**Fix sketch:** Rewrite Scenario 4b to target a foundation-owned resource without `prevent_destroy` (e.g. `aws_ses_domain_dkim`, `aws_ses_active_receipt_rule_set`), or document explicitly that `prevent_destroy` is the active line of defense at the foundation level and the gate at the regional level is a second-tier net.

### GAP-3 (non-blocking, validation-doc drift) — Scenario 6 unset path

**Observed:** `unset KM_ROUTE53_ZONE_ID; km init --plan` does not skip the ses module on this install, because km loads `route53_zone_id` from `km-config.yaml` via `ExportTerragruntEnvVars`, not from the shell env.

**Fix sketch:** Update the validation doc to describe the skip-trigger as a `km-config.yaml` field absence, or convert Scenario 6 into a unit test against `runInitPlan` with a fake config that omits the field.

### Non-gap caveats

- Scenario 1 reports 3 in-place changes (ttl/email lambda env drift); gate correctly stays green at 0 destroys. Soften validation prose from "all-zero" to "0 destroys" if a stricter assertion is desired.
