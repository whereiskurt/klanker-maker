# Phase 84.2: `km init --plan` flag with destroy-class gate — Context

**Gathered:** 2026-05-16
**Status:** Ready for planning
**Source:** PRD Express Path (design spec dated 2026-05-16, approved)

<domain>
## Phase Boundary

**Delivers:** A `--plan` flag on `km init` and `km bootstrap --shared-ses` that runs a real `terragrunt plan` per module, parses the JSON, and trips a curated destroy-class gate before any apply can run. Adds a pure-logic `pkg/terragrunt/planreport/` package (Parse, Evaluate, ProtectedTypes) consumed by both commands.

**Does NOT deliver:**
- Changes to the existing `runInit` apply path (Phase 84.1 owns those edits — surfaces stay separate to avoid merge churn).
- Changes to `runInitDryRun` (the static documentation info-dump stays as-is).
- A top-level `km plan` command. The `--plan` flag is colocated on the commands whose apply behavior it guards.
- Operator-side YAML/JSON config for the protected list. The list is compiled-in; updates require PR review.
- Auto-apply after override. `--i-accept-destroys` only clears the `--plan` exit code; the operator must separately run `--dry-run=false`.
- Cross-module dependency analysis (terragrunt handles edges natively).
- Periodic drift detection via `km doctor` (possible follow-up phase).

</domain>

<decisions>
## Implementation Decisions

All decisions below are LOCKED — captured from the approved design spec.

### Flag shape (Decision 1)
- New `--plan` flag, independent of `--dry-run`.
- Combinable with `--dry-run=true` (shows both info dump and plan).
- `--plan` itself implies no apply ever runs.
- **Why:** The current `--dry-run=true` info dump surfaces wrapper-level steps (DNS, build, upload) not visible in `terragrunt plan`. Replacing its semantics would silently change default behavior.

### Scope (Decision 2)
- All regional modules planned, in `runInit` order (sequential).
- No selectable subset / per-module flag.
- **Why:** A real safety gate must cover everything `apply` would touch; a subset flag invites plan-missing the module about to drift.

### Safety gate (Decision 3)
- Destroy-class gate with curated compiled-in `ProtectedTypes` allowlist.
- Any destroy OR replace of a protected resource type halts with exit 1.
- Override: `--i-accept-destroys` (per-invocation only, never persisted, does NOT auto-apply — only clears the `--plan` exit code).
- **Why:** An all-destroys gate is too noisy (routine lambda zip-hash churn would trip it); no-gate is just `terragrunt plan` with extra steps.

### Bootstrap parity (Decision 4)
- `km bootstrap --shared-ses` gets a parallel `--plan` flag in this same phase.
- **Why:** Phase 84 Gaps 2, 3, 6 happened in the bootstrap path too. Symmetric coverage is cheap once the plumbing exists.

### Output model (Decision 5)
- Sequential execution (matches `runInit` order — never parallel).
- Default: per-module one-line summary (`ses: 2 to add, 0 to change, 1 to destroy ⚠`).
- `--verbose` streams the full plan text.
- Destroy-gate trip block ALWAYS prints in full regardless of verbose.
- **Why:** Terragrunt's own `run --all plan` parallelism interleaves output, complicates JSON aggregation for the gate, and bursts AWS API calls.

### No config file for protected list (Decision 6)
- `ProtectedTypes` lives compiled-in `pkg/terragrunt/planreport/protected.go`.
- Updates require PR review; each addition references the motivating incident in a comment.
- **Why:** An operator-side config file would let an organization silently disable the safety. The safety value comes from review pressure.

### Initial Protected Types list

### Data flow — `km init --plan`
1. Parse flags. Reject `--plan` combined with `--sidecars` or `--lambdas`.
2. Call `ExportTerragruntEnvVars(cfg)` (Phase 84.1-01 helper).
3. Validate AWS credentials (same path as `runInit`).
4. For each module in `regionalModules(regionDir)` order:
   - Skip with `[skip: <var> not set]` if any `m.envReqs` env var is empty AND not in `configEnv` — does NOT count toward gate.
   - Create temp file via `os.CreateTemp("", "km-plan-*.tfplan")`; defer remove.
   - Call `runner.PlanWithOutput(ctx, dir, planFile.Name())` (capture stdout for `--verbose`).
   - Call `runner.ShowPlanJSON(ctx, dir, planFile.Name())`.
   - `planreport.Parse(bytes)` → `Report{Module: m.name, ...}`.
   - Print one-line summary.
   - If `--verbose`, stream captured plan text.
   - Append report.
5. `result := planreport.Evaluate(reports, acceptDestroys)`.
   - If `result.Blocked`: print full trip block, exit 1.
   - Else: print aggregate summary + footer "Run `km init --dry-run=false` to apply", exit 0.

### Data flow — `km bootstrap --shared-ses --plan`
Same structure, single module (`infra/live/<region>/ses-shared-rule-set/`). One report, same gate.

### Gate algorithm (locked)
```go
trips := []Trip{}
for _, r := range reports {
    if r.ParseFailed {
        trips = append(trips, Trip{Module: r.Module, Reason: "plan JSON parse failed — conservative trip"})
        continue
    }
    for _, ch := range append(r.Destroys, r.Replaces...) {
        if slices.Contains(ProtectedTypes, ch.Type) {
            trips = append(trips, Trip{Module: r.Module, Type: ch.Type, Address: ch.Address, Action: ch.Action})
        }
    }
}
if len(trips) > 0 && !acceptDestroys {
    return GateResult{Blocked: true, Trips: trips}
}
return GateResult{Blocked: false, Trips: trips}  // trips listed even on override
```

JSON action codes: `["delete"]` = destroy; `["delete","create"]` or `["create","delete"]` = replace. Both count as trips.

### Trip block format (always full, regardless of verbose)
```
✗ km init --plan would destroy 3 protected resources:

  ses-shared-rule-set:
    - aws_ses_domain_identity.sandboxes      [DESTROY]
    - aws_ses_domain_dkim.sandboxes          [DESTROY]
    - aws_route53_record.dkim[0]             [DESTROY]

These resource types are on the protected list because past incidents
caused unrecoverable data loss (see .planning/phases/84-.../84-10-UAT.md).

To proceed anyway, re-run with --i-accept-destroys (you must understand
why each resource is destroying — terragrunt apply will not ask again).

Exit 1.
```

### Error handling matrix (locked)
| Failure | Behavior |
|---|---|
| `terragrunt plan` exits non-zero (auth/env/syntax) | Stop loop. Print module name + captured stderr. Exit non-zero with module-specific message. |
| `terraform show -json` fails on successful plan | Log warning, mark report `ParseFailed: true`, continue. Gate treats parse-failed as **trip** (conservative). |
| Context cancelled (Ctrl-C) | Propagate `context.Canceled` (Phase 84.1-02 timeout work). Cleanup temp plan files. Exit 130. |
| Temp file create fails | Fall back to ephemeral inline plan (no JSON parse, conservative-trip), log warning. |
| Module envReqs unsatisfied | Skip `[skip: <var> not set]`. Does NOT count toward gate. |
| AWS throttling during plan | Surfaces as `terragrunt plan` non-zero — same as row 1. |
| Plan succeeds with no changes | `0 to add, 0 to change, 0 to destroy ✓`. Clean. |

### Override semantics (locked)
`--i-accept-destroys`:
- Per-invocation flag only; never persisted to disk or env.
- Clears `--plan` exit code from 1 to 0 when only failures are protected trips.
- Still prints full trip list (visibility).
- Does NOT auto-apply.

### Phase exit criteria (from roadmap goal)
Phase closes when:
- (a) `km init --plan` runs cleanly against operator's already-Phase-84 account with all-zero counts.
- (b) Synthetic destroy injected into a regional module trips gate with correct address + module + action, exits 1.
- (c) Same scenario with `--i-accept-destroys` exits 0 but still prints trip list.
- (d) `km bootstrap --shared-ses --plan` exhibits the same three behaviors against the foundation module.
- (e) `planreport` package has table-driven unit tests covering clean / add-only / destroy-trip / replace-trip / parse-fail / override toggle.
- (f) `--plan` combined with `--sidecars` or `--lambdas` is rejected as mutually exclusive.
- (g) Skipped modules (missing env-vars) reported as `[skip: KM_FOO not set]` and do NOT count toward gate.
- (h) Plan failures (auth, syntax) stop the loop with module-named stderr and exit non-zero.

## Specific Ideas

### Concrete references in spec
- Existing apply entry: `internal/app/cmd/init.go:276` (`runInitDryRun` — DO NOT modify).
- Existing runner: `pkg/terragrunt/runner.go:104` (`runner.Plan` already exists — `PlanWithOutput` is a sibling, not a replacement).
- Phase 84.1-01 helper: `ExportTerragruntEnvVars(cfg)` — must be called once before terragrunt invocation (matches Phase 84.1 contract).
- Phase 84.1-02 timeouts: desirable, not strictly blocking. If 84.1-02 hasn't landed, `PlanWithOutput` can take its own `context.WithTimeout` locally.

### Test fixtures expected
- `testdata/ses-clean.json` — no changes
- `testdata/ses-rule-add.json` — 2 rules added, 0 destroyed
- `testdata/ses-82to84-destroy.json` — Phase 84 trip scenario (hand-craft to schema if no real capture)
- `testdata/lambda-replace.json` — zip hash → replace (non-protected type — should NOT trip)
- `testdata/malformed.json` — parse-fail path

### Operator UAT scenarios (manual, real AWS)
1. Clean re-apply against operator's already-Phase-84 account → all-zero counts.
2. Synthetic destroy (e.g., delete catchall rule from `infra/modules/ses/v2.0.0/main.tf`) → trip with correct address. Restore module after.
3. Override flow: repeat (2) with `--i-accept-destroys` → trip printed, exits 0, no apply.
4. Bootstrap parity: `km bootstrap --shared-ses --plan` clean + synthetic destroy trip.

### Open questions to resolve during planning
1. **JSON schema version pinning.** `terraform show -json` is schema-versioned. Pin minimum schema version and assert on parse. Open: which version ships with the terraform pinned by current terragrunt?
2. **Per-module timeout coordination with Phase 84.1-02.** Decide whether to thread a `--timeout` flag through `--plan` or inherit defaults.
3. **`km doctor` integration.** Should `km doctor` mention "consider running `km init --plan` to check for unexpected drift"? Probably yes, low-cost addition.

</specifics>
