**Phase Goal:** Ship `km init --plan` and `km bootstrap --shared-ses --plan` flags backed by a destroy-class gate over a curated allowlist of protected resource types. ## Gaps Summary

### GAP-1 (Blocking) — `aws_ses_receipt_rule` missing from ProtectedTypes

This is the central failure of Phase 84.2. The phase was built explicitly to prevent a recurrence of the Phase 82->84 incident. That incident destroyed `aws_ses_receipt_rule` instances (the operator inbound rule and the sandbox catchall rule) via a `removed { destroy = true }` orphan path. The gate does not protect `aws_ses_receipt_rule` — only `aws_ses_receipt_rule_set` (the parent container). A recurrence of the exact incident would pass the gate today with exit 0.

The omission propagates through three layers:

1. **protected.go** — `aws_ses_receipt_rule` not in `ProtectedTypes`
2. **protected_test.go** — `TestProtectedTypes_Contents` does not list it, so the test passes and locks in the incomplete allowlist
3. **gate_test.go** — no test exercises an `aws_ses_receipt_rule` destroy through `Evaluate()`

The fix is a three-line change to protected.go (add the entry with incident comment), one additional entry in the `TestProtectedTypes_Contents` expected list, and one new gate_test.go test function.

### GAP-2 (Non-blocking) — Validation doc drift: Scenario 4b synthetic edit blocked by `prevent_destroy`

`infra/modules/ses-shared-rule-set/v1.0.0/main.tf` line 92-94 has `lifecycle.prevent_destroy = true` on `aws_ses_receipt_rule_set.shared`. A synthetic destroy edit causes terraform to error at plan time before the gate evaluates. The validation doc's Scenario 4b projects a gate trip; the actual behavior is a terraform error. The foundation `prevent_destroy` is functioning as a first line of defense; the gate test scenario needs a different resource that doesn't carry `prevent_destroy`.

### GAP-3 (Non-blocking) — Validation doc drift: Scenario 6 skip path uses wrong trigger

The validation doc instructs `unset KM_ROUTE53_ZONE_ID; km init --plan` to trigger the ses module skip. km loads `route53_zone_id` from `km-config.yaml` via `ExportTerragruntEnvVars`, not from the shell env. The shell unset has no effect. The correct trigger is clearing the field in `km-config.yaml`. The underlying skip logic in production code may be correct; this is a prose defect in the validation doc.

## Verdict

Phase 84.2 ships functional `km init --plan` and `km bootstrap --shared-ses --plan` commands with a working destroy-class gate architecture. The output format, flag registration, mutual-exclusion enforcement, plan-failure error handling, and gate logic for the 9 listed types are all correctly implemented and verified.

However, the phase's load-bearing promise — that the Phase 82->84 incident type could not silently happen again — is not satisfied. The specific AWS resource type destroyed in that incident (`aws_ses_receipt_rule`) is absent from `ProtectedTypes`. An operator running `km init --plan` on a plan that would destroy SES receipt rules today would see exit 0 and no trip block, identical to the blind spot the phase was designed to eliminate.

This is a blocking gap requiring Phase 84.3 closure before the DESTROY-CLASS-GATE requirement can be marked satisfied.
