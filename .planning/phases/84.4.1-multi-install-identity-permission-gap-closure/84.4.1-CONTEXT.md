# Phase 84.4.1: Multi-install identity/permission gap closure - Context

**Gathered:** 2026-05-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the cross-install identity/permission/sharing gaps that downgraded Phase 84.4 to PARTIAL PASS during the 2026-05-18 fresh-prefix `rg` UAT, plus 6 fresh-clone DX rough edges surfaced in the same UAT. Re-verify the multi-install thesis via a fresh-prefix UAT-2 against a new probe install. <decisions>
## Implementation Decisions

### SCP cross-install composition (CO-PRIMARY gap closure)

- **Design: pattern-based allowlist** with strict suffix wildcards. Each install owns its own `${prefix}-sandbox-containment` SCP attached to its own application account, but the policy body's `trusted_arns_*` allowlists use `arn:aws:iam::*:role/*-create-handler`, `*-ec2spot-ssm-*`, `*-budget-enforcer-*`, `*-github-token-refresher-*`, `*-ttl-handler`, `*-ecs-spot-handler` — so any install's create-handler / ECS-spot / budget / GitHub-token / TTL / ECS-spot-handler role is trusted by every install's SCP via suffix matching.
- **Reuses** `BuildSCPPolicyFromPrefix(resourcePrefix)` slot pattern from `scp/v2.0.0/` — only the allowlist values change (the `km-` literal swaps for `*-`), the policy name itself stays `${var.resource_prefix}-sandbox-containment`.
- **Module versioning: in-place update to `scp/v2.0.0/`** (no v2.0.1 bump). v2.0.0 was shipped 2026-05-17 and the only deployer is the canonical km install (UAT was partial-pass) — the pattern-based allowlist is a logical refinement of v2.0.0's per-install-naming work and they ship together.
- **Canonical km install transition: single `terragrunt apply`** that swaps the SCP body in-place. Expected `terragrunt plan` output: ~6 `update in-place` lines on `aws_organizations_policy.sandbox_containment` Statement[].Condition.ArnNotLike values. Zero destroys. Sub-10-second window documented in OPERATOR-GUIDE.md.
- **Security trade-off accepted:** a role named `evil-create-handler` deployed by an attacker would be trusted by the SCP. Mitigated by: (1) only operators have IAM create permissions in the application account, (2) the SCP is one layer of defense not the only one, (3) any deployed role still needs the cross-account assume-role grant to actually use anything.
- **Opt-in OU upgrade path: deferred.** Document the per-install-OU isolation design in OPERATOR-GUIDE.md as a future hardening step for operators who want stricter isolation. Not implemented in 84.4.1.

### ssm-session-doc design (TIER-1 gap closure)

- **Design: per-install rename to `${var.resource_prefix}-Sandbox-Session`.** Bump to `ssm-session-doc/v2.0.0/` with `var.resource_prefix` plumbed through from live wiring. Document name becomes `${prefix}-Sandbox-Session` (lowercase prefix, preserving the existing `Sandbox-Session` casing). Each install owns its own SSM document. Matches the Phase 84.4 isolation thesis.
- **Canonical km install transition: `moved` block + documented destroy/create.** v2.0.0 main.tf declares `moved { from = aws_ssm_document.km_sandbox_session, to = aws_ssm_document.sandbox_session }`. Terraform tries to move state but AWS doesn't support SSM document rename, so apply plans destroy `KM-Sandbox-Session` + create `km-Sandbox-Session`. Active SSM sessions survive (sessions reference doc at start, not on every command); new sessions started during the 1-2 second gap fail with `InvalidDocument` and retry. Documented in OPERATOR-GUIDE.md.
- **NOT chosen:** conditional naming preserving `KM-Sandbox-Session` for the km prefix (rejected — ugly capitalization asymmetry, hairy conditional in tf).
- **NOT chosen:** keep-shared with explicit `ssm-session-doc-shared/v1.0.0/` foundation-style module (rejected — foundation-pattern is heavier; per-install isolation is the consistent direction).
- **km shell code changes:** `km shell` looks up the document name from config/site.hcl (already does — verify). No code change needed if the document-name reference is already prefix-derived.

### detectSharedSESState fix (PRIMARY gap closure)

- **Design: separate the DKIM-import gate from the registerID gate.** Keep `detectSharedSESState` (`bootstrap.go:283-`) as-is — it returns `registerID` based on state ownership ("this install's tfstate owns the identity"), which is semantically correct. Move the gate at `bootstrap.go:595` to a new independent condition: auto-import any Route53 record whose name matches `<sel>._domainkey.<domain>` / `<domain>.` MX / `_amazonses.<domain>` TXT — regardless of `registerID`. The two concerns (does this install own the identity? do these records already exist in Route53?) are independent.
- **NOT chosen:** probing AWS via `GetEmailIdentity` inside `detectSharedSESState` (rejected — conflates state-ownership and AWS-existence into one flag, subtly changes the meaning of `KM_REGISTER_DOMAIN_IDENTITY` env var exported at `bootstrap.go:576`).
- **Record set: DKIM[0..2] + MX + `_amazonses` TXT.** Same set Plan 06 already lists, just behind the corrected gate. Confirmed by 84.4-08 UAT as the exact records that collide on shared-domain second installs. No new records added in 84.4.1.
- **Failure mode: fail-fast with operator-visible error.** First import failure aborts auto-import, returns error to `runBootstrapSharedSES`. Bootstrap exits non-zero with: "Auto-import failed at <record>: <reason>. Recovery: run `terragrunt import ...` manually in <dir>, then re-run `km bootstrap --shared-ses`." Error message includes the exact `terragrunt import` command(s) the operator should run.
- **Integration test: live-AWS recording via `KM_TEST_RECORD=1`.** One-time live AWS recording captures HTTP fixtures (SES ListIdentities response, Route53 ListResourceRecordSets, terragrunt-import-equivalent state diff), CI replays the fixtures deterministically. Pattern already exists in `pkg/aws/ses/` tests.

### Audit test extension (regression prevention)

- **Extend `pkg/terragrunt/modulehygiene_test.go` in two ways:** (1) also walk `infra/modules/*/v1.0.0/` modules at WARN level (logs to test output, doesn't fail CI — surfaces candidates for future v2.0.0 migration); (2) make `km`/`KM` matching case-insensitive in the resource-attribute scan.
- **Migration scope: only ssm-session-doc in 84.4.1.** The extended audit will surface other v1.0.0 modules with hardcoded literals (likely `dynamodb-*`, `lambda-slack-bridge`, `create-handler`, `email-handler`, etc.). Phase 84.4.1 migrates only ssm-session-doc (the actual blocking gap). The WARN-flagged candidates land in a new `.planning/phases/84.4.1-.../deferred-items.md` inventory for a follow-up phase to triage per-install vs shared per module.

### Fresh-clone DX gaps (bundled)

- **One plan: "Fresh-clone DX hardening."** Single PLAN.md covering all 6 gaps:
  1. `make build-lambdas` Makefile target drifted from `init.go:buildLambdaZips()` — sync to the 6-zip list, OR delete the Makefile target and have `km init --plan` check for required zips and print a clear `./km init --lambdas` hint. (Default direction: sync Makefile, more discoverable.)
  2. `km bootstrap --shared-ses` requires `region.hcl` which is gitignored and only written by `km init` — extract the region.hcl write logic from `init.go` into a shared helper, call it from `runBootstrapSharedSES` and `runBootstrapAll`.
  3. `km configure` state_bucket prompt has no default and no HeadBucket retry — show computed default `[tf-${prefix}-state-${region}]`, HeadBucket-check it, offer `[Y/edit/abort]` retry on 403, write accepted value back to km-config.yaml in sync with site.hcl's computation.
  4. `km unbootstrap` leaves orphan DynamoDB lock table created by terragrunt's backend — add explicit delete call in unbootstrap.go's teardown path.
  5. Phase 84.3 closure-e `isPlaceholderBucket` regression — already fixed in commit d551bba; 84.4.1 adds a regression test to lock the behavior.
  6. `downloadTerraform` caches stale binary across tfVersion bumps — already fixed in commit d5d554b; 84.4.1 adds a regression test for the cache-invalidation logic.
- **Wave placement:** parallel with TIER-1 work (Wave 2 candidate) — these are wrapper-level changes with no overlap with the TIER-1 module work.

- **Prerequisite ordering: apply all 84.4.1 module changes to canonical km install BEFORE UAT-2 starts.** Sequence:

  4. Post-UAT verification: `km` install is byte-identical before-and-after (AWS-CLI inventory diff), `km shell` works, canonical install's sandbox lifecycle intact.
- **NOT chosen:** two-prefix UAT (`tg` then `td` simultaneously) — adds operator time without proportional signal; the pattern-based SCP design's correctness for N installs is unit-testable separately.

<specifics>
## Specific Ideas

- **SCP pattern model** comes from the 84.4-08 UAT workaround: "Manually edit the deployed SCP `p-cvd490xt` body to ADD `arn:aws:iam::052251888500:role/rg-create-handler` to the trusted-role allowlist" — that was the ad-hoc form of the union-by-suffix design now formalized as `*-create-handler`.
- **ssm-session-doc destruction story** is the load-bearing example for why "terragrunt import" is unsafe for hardcoded-name resources: rg's uninit destroyed the shared `KM-Sandbox-Session`, leaving km's state pointing at a missing AWS resource and `km shell` broken on km. The recovery `terragrunt apply` from `klankrmkr/infra/live/use1/ssm-session-doc/` re-created the document — but that's the kind of cross-install dependency 84.4 was supposed to eliminate.
- **detectSharedSESState's `registerID` semantics:** the variable name says "register this install's identity" — the bug is that `bootstrap.go:595` conflates that with "should DKIM records be auto-imported." Separating the gates is preserving original semantics while fixing the actual bug.
- **84.4-08 UAT verbatim error messages** that 84.4.1 must make impossible:
  - `Error: creating SSM Document (KM-Sandbox-Session): DocumentAlreadyExists`
  - `api error UnauthorizedOperation: You are not authorized to perform: ec2:CreateSecurityGroup ... with an explicit deny in a service control policy: arn:aws:organizations::481723467561:policy/o-om3mjz6hu8/service_control_policy/p-cvd490xt`
  - `InvalidChangeBatch: Tried to create resource record set ... but it already exists`
  - `read_terragrunt_config" failed: You attempted to run terragrunt in a folder that does not contain a terragrunt.hcl file. Path: "../region.hcl"`

<deferred>
## Deferred Ideas

- **Other v1.0.0 modules with hardcoded `KM-`/`km-` literals** that the extended audit surfaces (likely `dynamodb-*`, `lambda-slack-bridge`, `create-handler`, `email-handler`, ECS-related modules). Each needs per-install vs shared analysis. Filed as `.planning/phases/84.4.1-.../deferred-items.md` for a follow-up phase (suggested name: 84.4.2 or similar).
- **Per-install OU isolation as opt-in hardening.** Document the design in OPERATOR-GUIDE.md but don't implement. Future operators who want stricter SCP isolation can move their application account into a per-install OU manually.
- **`km doctor` SCP composition health check** — verify the deployed SCP body matches the v2.0.0 module's `BuildSCPPolicyFromPrefix` output. Useful drift-detection for operators who hand-edit policies. Out of scope for 84.4.1.
- **Multi-install registry/discovery mechanism** — explicit "which prefixes exist in this account" registry. The pattern-based SCP design eliminates the need for 84.4.1, but a future phase might want it for `km doctor` cross-install visibility (e.g., "list all installs in this account").
- **`km bootstrap --shared-ses --no-auto-import` flag** for operators who want explicit control. Default behavior is unconditional auto-import behind the new gate; explicit opt-out can be added if operators request it.
- **LocalStack-based integration test infrastructure** — broader CI capability not specific to 84.4.1. Recording-based pattern chosen for 84.4.1 keeps scope tight.
- **Two-step / parallel SCP transition strategies** — rejected for 84.4.1's single-apply approach. Documented here in case a future incident makes the sub-10s window untenable.

</deferred>
