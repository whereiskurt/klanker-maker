Wave 3 — operator-driven canonical km install transition.

Plans 01/02/03/04 landed the code. This plan time-windows the live `terragrunt
apply` that moves the canonical km install from Phase 84.4 deployed state to
Phase 84.4.1 deployed state. Two AWS-resource-mutating applies:

   (~5-10 update-in-place lines on aws_organizations_policy.sandbox_containment
   per RESEARCH.md Open Question 5). Zero destroys. Sub-10s window. After this,
   the canonical km install's SCP trusts any install's roles by suffix pattern.

   creates `km-Sandbox-Session` (note lowercase k). 1-2s window where new SSM
   sessions started against the destroyed doc fail with InvalidDocument. Active
   sessions started before destruction survive (sessions bind at StartSession
   time, not on every command).

After both applies, the canonical km install is at Phase 84.4.1 state and the
fresh-prefix UAT-2 (Wave 4 plan 84.4.1-06) can run against a fresh `tg` install
without any disagreement between the km install's deployed resources and the
84.4.1 code.

Operator must time-window the applies to avoid disrupting active sandboxes:
- Schedule during low-activity period
- Notify any active km shell sessions to disconnect
- Inventory-diff before AND after to lock the change scope (only SCP + ssm-session-doc
  may change; everything else byte-identical)

- Live AWS Organizations mutation (SCP policy body)
- Live SSM document destroy/create (1-2s outage window)
- Operator judgment call on inventory diff (what counts as volatile vs load-bearing)
- Operator credentials for management + application accounts required

Purpose: TIER-1 code-to-live alignment. Without this, Wave 4 UAT-2 would
inventory-diff against a Phase 84.4 km install state, not the 84.4.1 state, and
the assertion "fresh-prefix install leaves km install byte-identical" would be
non-meaningful.

terragrunt outputs + verdict.
