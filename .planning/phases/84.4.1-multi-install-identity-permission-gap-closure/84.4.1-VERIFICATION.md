### Gaps Summary

No goal-blocking gaps. The phase goal is achieved: a fresh-prefix install on a shared-domain account completed the full lifecycle without manual workarounds, and the canonical km install was unaffected.

Five non-blocking gaps surfaced during UAT-2 and are enumerated for a follow-on phase (e.g. 84.4.2):

1. **Gap #1 — `km init --plan` lambda-zip prereq missing:** `runInitPlan` does not call `buildLambdaZips`. Interim doc-note in OPERATOR-GUIDE.md (commit 341a2e7). Real fix: add `buildLambdaZips(repoRoot)` call at init.go:~1212.

2. **Gap #2 — `km configure` artifacts_bucket not auto-derived:** `deriveArtifactsBucket` exists at configure.go:206 but is not wired as the prompt default. Operator enters non-canonical values that pass validation. Fix: wire as default at configure.go:~592 + strengthen `validateArtifactsBucket` to enforce `${prefix}-artifacts-${account_id}` shape.

3. **Gap #3 — Orphan SCPs not auto-detached:** `km uninit`/`km unbootstrap` leave install SCP attached. Post-UAT-2 residual: `tg-sandbox-containment` (p-8in7702y). OPERATOR-GUIDE.md documents manual cleanup. Fix: `km uninit` detaches+deletes install's own SCP (with confirmation), or `km doctor` warns on orphan SCPs.

4. **Gap #4 — `validateArtifactsBucket` too permissive:** Accepts `tg-km-artifacts-use1-abcd0123` (obvious placeholder). Fix: enforce canonical `${prefix}-artifacts-${digit-account-id}` regex.

5. **Gap #5 — Suspected TTL/in-use detection during uninit:** Operator-flagged; needs reproduction + log capture. Investigation TBD.
