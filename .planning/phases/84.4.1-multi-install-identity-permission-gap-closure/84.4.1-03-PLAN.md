<objective>
PRIMARY gap closure: SES auto-import gate fix.

The Phase 84.4-08 UAT proved that Plan 06's auto-import logic never fires on
shared-domain second installs. `bootstrap.go:595` gates auto-import on
`!registerID`, but `detectSharedSESState` (bootstrap.go:283-361) returns
`registerID = true` even when the SES domain identity already exists in AWS
(per the Phase 84.1 GAP-2/GAP-3 semantic shift — registerID now means "this
install's tfstate owns this identity", which becomes true post-foundation-apply
regardless of whether the underlying AWS resource was created or imported).

The result: on the rg fresh install, bootstrap tried to CREATE the 3 DKIM CNAMEs
that km already owned in Route53; Route53 rejected them with
`InvalidChangeBatch: ... already exists`. Operator had to run `terragrunt import`
manually for each DKIM/MX/TXT record — exactly what Plan 06 was supposed to
eliminate.

CONTEXT.md's fix is **separate the DKIM-import gate from the registerID gate**:
- Keep detectSharedSESState as-is (registerID semantics preserved)
- Move the gate at bootstrap.go:595 to a new independent condition that fires
  whenever a real Route53/SES client pair is available — auto-import is
  state-independent because autoImportFoundationSESRecords (bootstrap.go:161-260)
  is already idempotent (checks stateReader.StateOwns before each import).

Plus a failure-mode improvement: when import fails, the error message includes
the exact `terragrunt import` recovery command (paths + resource addresses) that
the operator should copy-paste. The current error wrap at bootstrap.go:602 just
chains the inner error without recovery guidance.

This is the smallest implementation in 84.4.1 — a 1-line gate change + 1-line
error wrap enrichment + 2 Wave 0 tests unskipped. But it's the highest-impact
behavioral fix: without it, every multi-install second-install bootstrap on a
shared domain requires manual `terragrunt import` of 5 Route53 records.

`autonomous: true` — code-only change with full unit coverage.
