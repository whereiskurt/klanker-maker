### Locked Decisions

**SCP cross-install composition (CO-PRIMARY gap closure):**
- **Design: pattern-based allowlist** with strict suffix wildcards. Each install owns its own `${prefix}-sandbox-containment` SCP attached to its own application account, but the policy body's `trusted_arns_*` allowlists use `arn:aws:iam::*:role/*-create-handler`, `*-ec2spot-ssm-*`, `*-budget-enforcer-*`, `*-github-token-refresher-*`, `*-ttl-handler`, `*-ecs-spot-handler` — so any install's role is trusted by every install's SCP via suffix matching.
- **Module versioning: in-place update to `scp/v2.0.0/`** (no v2.0.1 bump). v2.0.0 was shipped 2026-05-17 and the only deployer is the canonical km install (UAT was partial-pass) — the pattern-based allowlist is a logical refinement of v2.0.0's per-install-naming work.
- **Canonical km install transition: single `terragrunt apply`** that swaps the SCP body in-place. Expected ~6 `update in-place` lines on `aws_organizations_policy.sandbox_containment` Statement[].Condition.ArnNotLike values. Zero destroys. Sub-10-second window.
- **Security trade-off accepted:** a role named `evil-create-handler` would be trusted. Mitigated by (1) operator-only IAM create perms, (2) SCP is one layer not the only one, (3) any deployed role still needs cross-account assume-role grant.
- **Reuses** `BuildSCPPolicyFromPrefix(resourcePrefix)` slot pattern — only the slot values change from `${prefix}-*` to `*-*`.
- **Opt-in OU upgrade path: deferred** — document only.

**ssm-session-doc design (TIER-1 gap closure):**
- **Design: per-install rename to `${var.resource_prefix}-Sandbox-Session`.** Bump to `ssm-session-doc/v2.0.0/`. Document name becomes `${prefix}-Sandbox-Session` (lowercase prefix, preserving `Sandbox-Session` casing). Each install owns its own SSM document.
- **Canonical km install transition: `moved` block + documented destroy/create.** v2.0.0 main.tf declares `moved { from = aws_ssm_document.km_sandbox_session, to = aws_ssm_document.sandbox_session }`. Terraform tries to move state but AWS doesn't support SSM document rename → apply plans destroy `KM-Sandbox-Session` + create `km-Sandbox-Session`. Active SSM sessions survive (sessions reference doc at start). New sessions during 1-2s gap fail with `InvalidDocument` and retry.
- **NOT chosen:** conditional naming preserving `KM-Sandbox-Session` for the km prefix (rejected).
- **NOT chosen:** keep-shared with explicit `ssm-session-doc-shared/v1.0.0/` (rejected — foundation pattern is heavier).
- **km shell code changes:** CONTEXT.md notes "km shell looks up the document name from config/site.hcl — verify" — **VERIFICATION RESULT: FALSE. The document name `KM-Sandbox-Session` is hardcoded in 5 Go callsites.** See `<phase_requirements>` for details.

**detectSharedSESState fix (PRIMARY gap closure):**
- **Design: separate the DKIM-import gate from the registerID gate.** Keep `detectSharedSESState` (`bootstrap.go:283-361`) as-is — returns `registerID` based on state ownership ("this install's tfstate owns the identity"), which is semantically correct per the Phase 84.1 GAP-2/GAP-3 semantic shift. Move the gate at `bootstrap.go:595` to a new independent condition: auto-import any Route53 record whose name matches DKIM/MX/`_amazonses` TXT patterns — regardless of `registerID`.
- **NOT chosen:** probing AWS via `GetEmailIdentity` inside `detectSharedSESState` (rejected — conflates state-ownership and AWS-existence into one flag, subtly changes meaning of `KM_REGISTER_DOMAIN_IDENTITY` env var exported at `bootstrap.go:576`).
- **Record set: DKIM[0..2] + MX + `_amazonses` TXT.** Same as Plan 06; just behind corrected gate. No new records.
- **Failure mode: fail-fast with operator-visible error.** First import failure aborts auto-import, returns error to `runBootstrapSharedSES`. Error includes exact `terragrunt import` commands the operator can copy-paste to recover.
- **Integration test: live-AWS recording via `KM_TEST_RECORD=1`.** Pattern already exists in `pkg/aws/ses/` tests (per CONTEXT.md — see Pitfall 6 in this doc for verification result).

## Summary

Phase 84.4.1 is implementation work behind locked architectural decisions. Three TIER-1/PRIMARY changes — SCP `*-*` pattern allowlist, ssm-session-doc v2.0.0 per-install rename, SES auto-import gate separation — plus six wrapper-level DX fixes bundled into one DX-hardening plan, plus a UAT-2 plan structurally similar to 84.4-08.

The most important research finding that contradicts CONTEXT.md's assumption: **`KM-Sandbox-Session` is hardcoded in 5 Go callsites** (`shell.go:500`, `agent.go:356,430,619`, `agent_auth.go:157,411`). CONTEXT.md said "verify km shell looks up the doc name from config/site.hcl — probably does." It does NOT. The ssm-session-doc v2.0.0 plan must include Go-side refactor: introduce a config-driven document-name helper (e.g., `cfg.GetSandboxSessionDocumentName()` returning `"${prefix}-Sandbox-Session"`) and update all 5 callsites + their tests.

Other surprises:
- `Runner.Import()` already exists at `runner.go:148` (Phase 84.4 Plan 00 added it) — no new method needed.
- `autoImportFoundationSESRecords` (`bootstrap.go:161-260`) is already idempotent — the implementation is correct; only the gate is wrong.
- `BuildSCPPolicyFromPrefix` (`bootstrap.go:743`) currently produces `${prefix}-*` patterns — the new design needs `*-*` patterns. Backward-compat behavior changes for `km` prefix: existing tests at `bootstrap_scp_test.go:50-56` assert `km-ecs-spot-handler` literal appears in policy output. Those tests must update to assert `*-ecs-spot-handler` (or be split into compat + new tests).
- d5d554b ONLY bumped the version constant `tfVersion := "1.9.8"` at init.go:1689; the cache invalidation logic was NOT added.
- d551bba changed `internal/app/config/config.go:isPlaceholderBucket` to only reject angle-bracket tokens; the `configure.go:228` `validateArtifactsBucket` still rejects literal `"km-artifacts-12345"` — two different validators, the configure-time one is unchanged.
- `infra/live/management/scp/terragrunt.hcl:73-86` passes operator-level hardcoded `km-provisioner-*`, `km-lifecycle-*`, `km-ecs-spot-handler`, `km-ttl-handler`, `km-create-handler` literals as `trusted_role_arns`. Those must change to `*-*` patterns OR be derived from `KM_RESOURCE_PREFIX`. The change matters even for the canonical km install — without it, the second install's roles aren't in this list, so the v2.0.0 module's `trusted_arns_base = var.trusted_role_arns` still denies them.

**Primary recommendation:** Planner should organize five waves: Wave 0 = pre-implementation BEFORE snapshot of canonical km install; Wave 1 (parallel) = ssm-session-doc v2.0.0 module + scp/v2.0.0 pattern update + bootstrap.go gate fix + audit extension; Wave 2 = live wiring flip + canonical km install operator apply (NOT autonomous); Wave 3 = DX hardening (parallel internal tasks); Wave 4 = UAT-2 on `tg` (NOT autonomous) + OPERATOR-GUIDE.md replacement.

```hcl
# Per-install document naming via ${var.resource_prefix}-Sandbox-Session,
# preserving the v1.0.0 "Sandbox-Session" suffix and Standard_Stream sessionType.
#
# Migration from v1.0.0: a moved {} block declares the rename, but AWS SSM
# does not support document rename — terraform's AWS provider falls back to
# destroy + create. Active SSM sessions (started before the rename) keep
# working; new sessions started during the ~2-second destroy/create gap fail
# with InvalidDocument and the operator retries.

resource "aws_ssm_document" "sandbox_session" {
  name            = "${var.resource_prefix}-Sandbox-Session"
  document_type   = "Session"
  document_format = "JSON"

  content = jsonencode({
```

```go
// New helper:
func getTerraformVersion(path string) (string, error) {
    cmd := exec.Command(path, "version", "--json")
    out, err := cmd.Output()
    if err != nil {
        return "", err
    }
    var v struct {
        TerraformVersion string `json:"terraform_version"`
    }
    if jErr := json.Unmarshal(out, &v); jErr != nil {
        return "", jErr
    }
    return v.TerraformVersion, nil
}
```

**Note:** `getTerraformVersion` needs to handle the linux/arm64 binary on a darwin/arm64 host. The cached binary won't execute natively on macOS during local `make build`. Possible solutions:
- Check binary size as a crude proxy (1.6.6 zip is ~70MB; 1.9.8 is ~80MB).
- Embed the desired version in a sidecar file (e.g., `build/terraform.version`).
- The `--json` `terraform version` command works on linux but exec'd on darwin will silently exec-fail.

**Recommendation: use a sidecar version file.** When downloadTerraform runs, write `tfDesiredVersion` to `build/terraform.version`. The check becomes:

```go

// After successful download:
```

This avoids the cross-arch exec problem entirely.

### Anti-Patterns to Avoid

- **Do NOT use `terragrunt import` as a workaround for hardcoded-name resources.** The 84.4-08 UAT proved this is unsafe — cross-install state ownership leads to teardown destruction. The ssm-session-doc migration to v2.0.0 with per-install naming is the correct fix.
- **Do NOT add a `KM_REGISTER_ROUTE53_RECORDS` env var to gate the auto-import.** The CONTEXT.md decision is clear: the auto-import gate at bootstrap.go:595 should be INDEPENDENT of any state-derived env var. Probe Route53 directly.
- **Do NOT change `detectSharedSESState`'s return type or signature.** It is called from multiple paths (`DetectSharedSESState`, `DetectSharedSESStateWithStateReader`, `runBootstrapSharedSES`, `runBootstrapSharedSESPlanWithWriter`). Keep the function and its return semantics unchanged.
- **Do NOT change the policy NAME in scp/v2.0.0 (stays `${var.resource_prefix}-sandbox-containment`).** Only the `trusted_arns_*` slot values change to `*-*` patterns. Naming-per-install is correct; the bug was only in the trust-relation patterns.
- **Do NOT shell out to `aws ssm describe-document` to verify the rename.** The terraform apply output IS the source of truth — if apply succeeds, the rename happened.

## Common Pitfalls

### Pitfall 1: `bootstrap_scp_test.go` Existing Tests Assert `km-*` Literals

The current tests at lines 50-56 + line 39 hardcode that `BuildSCPPolicyFromPrefix("km", ...)` produces strings containing `km-ecs-spot-handler`, `km-budget-enforcer-*`, etc. After 84.4.1, those will become `*-ecs-spot-handler`, etc. The tests MUST be rewritten:

```go
// BEFORE (will FAIL after 84.4.1):
expectedRoleSubstring := ":role/" + tc.prefix + "-ecs-spot-handler"
if !strings.Contains(got, expectedRoleSubstring) { ... }

// AFTER (84.4.1 contract):
expectedRoleSubstring := ":role/*-ecs-spot-handler"
if !strings.Contains(got, expectedRoleSubstring) { ... }
// AND assert NO prefix-bound substring appears:
if strings.Contains(got, ":role/" + tc.prefix + "-ecs-spot-handler") {
    t.Errorf("policy should use *-pattern, not %s-prefix", tc.prefix)
}
```

**Warning sign:** Plan executor sees `TestBuildSCPPolicyPrefix` failing — that's expected during 84.4.1 implementation; the test contract changes.

### Pitfall 2: `KM_RESOURCE_PREFIX` Not Exported in All Operator-Driven Apply Paths

The canonical km install transition for scp/v2.0.0 requires `KM_RESOURCE_PREFIX=km` set in the shell when the operator runs `terragrunt apply` from `infra/live/management/scp/`. If unset, `get_env("KM_RESOURCE_PREFIX", "km")` falls back to `"km"` (default) — works. But if the operator runs from a wrong working dir or with stale env, behavior is undefined.

**Prevention:** UAT-2 plan must include an explicit `km env` invocation before any manual `terragrunt apply` (CONTEXT.md notes `km env` was added in Phase 84.3 closure-g).

### Pitfall 3: SSM Session Document Destroy/Create Race During Apply

When v2.0.0 applies on canonical km install:
1. `aws_ssm_document.km_sandbox_session` (`KM-Sandbox-Session`) is destroyed.
2. `aws_ssm_document.sandbox_session` (`km-Sandbox-Session`) is created.

`lifecycle.create_before_destroy = true` is set in BOTH v1.0.0 and v2.0.0. But the AWS SSM API's document-rename limitation means create-before-destroy may not apply here (the AWS provider may force destroy-then-create when the underlying API rejects update). Active sessions started against the old document name survive (sessions reference the doc at start, not on every command). NEW sessions started during the ~1-2 second gap will fail with `InvalidDocument`.

**Prevention:** The OPERATOR-GUIDE.md update must explicitly document this window. Operators should avoid running `km shell` during the canonical km install's `terragrunt apply` for the ssm-session-doc module.

### Pitfall 4: SCP `*-*` Pattern is BROADER Than Intended

`arn:aws:iam::*:role/*-create-handler` would match a hypothetical role named `evil-create-handler` in ANY account. CONTEXT.md accepts this trade-off, but documentation must explain WHY:
1. The application account's IAM:CreateRole is denied to everyone except operators (the existing SCP enforces this).
2. The trusted_arns_base still requires roles to be in the application account (`arn:aws:iam::${var.application_account_id}:role/...`) — wait, no, the new pattern is `arn:aws:iam::*:role/...` with wildcarded account.
3. The protection layers (operator-only IAM creation, SCP, role assume-role grants) compose to make abuse impractical.

**Prevention:** OPERATOR-GUIDE.md Multi-install runbook must include a "Security trade-off" subsection explaining the layered defenses.

### Pitfall 5: km uninit Destroys ssm-session-doc Even with v2.0.0

After 84.4.1, when a second install (e.g., `tg`) is torn down:
1. Its `ssm-session-doc` module destroys `tg-Sandbox-Session`.
2. The canonical km install's `km-Sandbox-Session` is UNTOUCHED (different document name).

This is the entire point of v2.0.0. But the UAT-2 plan must explicitly verify this — `km shell` on the canonical install MUST work immediately after `tg` uninit, with no manual recovery step.

**Prevention:** UAT-2 Step 11 (post-teardown verification) MUST include:

```bash
# Verify canonical km install's km shell works (the load-bearing 84.4.1 closure)
cd /Users/khundeck/working/klankrmkr
./bin/km shell kmsmoke   # must open SSM session, no InvalidDocument error
```

### Pitfall 6: `pkg/aws/ses/` Recording Pattern — CONTEXT.md May Be Wrong

CONTEXT.md (line 39, "Integration test" subsection of detectSharedSESState fix) says: "Pattern already exists in `pkg/aws/ses/` tests." Verification: `pkg/aws/ses.go` and `pkg/aws/ses_test.go` exist, but `pkg/aws/ses/` (subdirectory) does NOT. The grep for `KM_TEST_RECORD` in `pkg/aws/` returned only `pkg/aws/budget_test.go` — no SES recording infrastructure found.

**Implication:** The "integration test via KM_TEST_RECORD recording" approach may be aspirational, not pre-existing. The planner has two options:
1. Build the recording-replay infrastructure as part of 84.4.1 (significant scope creep).
2. Use the existing fake-injection pattern in `bootstrap_dkim_import_test.go` (lines 31-111: `fakeSesDkimGetter`, `fakeRoute53RecordLister`, `fakeStateOwner`, `fakeImporter`) — extend with a new test that exercises the new gate logic.

**Recommendation: option 2.** The existing fake pattern is sufficient. The new test asserts that `runBootstrapSharedSES` (with stateReader showing identity owned by foundation) STILL invokes `autoImportFoundationSESRecords`. Use the cmd-package-internal test pattern (not `cmd_test`) so the unexported interfaces are visible.

### Pitfall 7: `runBootstrapAll` Plan Mode Doesn't Call `runBootstrapPlan`

`runBootstrapAll` (bootstrap.go:868-901) has a bug: in plan mode (`plan = true`), both subflows call `RunBootstrapSharedSESPlanFunc` (lines 881-882 and 891-893). This means subflow 1 (which should be the foundation SCP+KMS+artifacts plan) actually runs the SES plan twice. This bug pre-dates 84.4.1 but the region.hcl prereq fix needs to call `ensureRegionHCL` early in BOTH subflows; if the bug is in scope to fix, the planner should note it. CONTEXT.md doesn't list it explicitly — likely out of scope for 84.4.1 (but document as a follow-up).

### Pitfall 8: `terraform_data.scp_size_guard` Won't Trip After `*-*` Substitution

The `*-*` pattern is shorter than `${prefix}-*` (no account_id substitution), so the rendered SCP policy will be SMALLER after 84.4.1. The 5,000-byte threshold continues to be a meaningful guard for FUTURE additions. But the unit test `TestBuildSCPPolicySize` (bootstrap_scp_test.go:20-45) currently asserts `maxBytes: 5000` for all three test prefixes. After 84.4.1, all three will produce identical output (prefix is ignored in the slot values), so the parametrized test becomes a single-case test.
