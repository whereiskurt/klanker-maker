<objective>
TIER-1 gap closure: SCP cross-install composition.

The Phase 84.4-08 UAT proved that per-install SCP NAMES are not enough — when two
installs share an application account, each install's SCP body still hardcodes
${prefix}-* role patterns, causing the canonical km install's SCP to actively DENY
the rg install's create-handler. The workaround was manual policy-body editing,
which destroys the per-install isolation design.

This plan ships the pattern-based allowlist design from CONTEXT.md §"SCP
cross-install composition": swap the trusted_arns_* slot values from
${prefix}-* to *-* in BOTH the HCL module (scp/v2.0.0/main.tf) AND the Go-side
mirror (bootstrap.go:743 BuildSCPPolicyFromPrefix) AND the operator-level
trusted_role_arns list (infra/live/management/scp/terragrunt.hcl:73-86). Also
rewrites bootstrap_scp_test.go's existing tests, which assert ${prefix}-${role}
substrings that will break.

The change is in-place to scp/v2.0.0 (no version bump) per CONTEXT.md decision —
v2.0.0 was shipped 2026-05-17 and the only deployer is the canonical km install,
which transitions via a single terragrunt apply (~6 in-place updates on
aws_organizations_policy.sandbox_containment, zero destroys, sub-10s window).
That live apply happens in Wave 3 plan 84.4.1-05 (canonical km install
transition); this plan only lands the code.

Security trade-off accepted: an attacker-deployed role named `evil-create-handler`
would be trusted by the SCP. Mitigations: operator-only IAM:CreateRole in the
application account, SCP is one defense-in-depth layer, deployed roles still need
cross-account assume-role grant. Documented in OPERATOR-GUIDE.md in Wave 4
plan 84.4.1-06.

`autonomous: true` because all changes are code-side; the live apply is a separate
Wave 3 plan.

Purpose: SCP-CROSS-INSTALL-COMPOSITION requirement. Closure criterion (b) in
ROADMAP §Phase 84.4.1.

Output: 4 edited files + 1 rewritten test.
</objective>
