# Phase 84.4.1 Plan 03: SES Auto-Import Gate Fix Summary

One-liner: Drop `!registerID` from bootstrap.go:595 gate so shared-domain second installs auto-import DKIM/MX/TXT without manual `terragrunt import`.

## What Was Built

### Gate fix (bootstrap.go:595)

The old gate `!registerID && sesv1Client != nil && ...` prevented auto-import whenever
`detectSharedSESState` returned `registerID = true`. Under Phase 84.1's import-and-manage
semantic, `registerID = true` means "foundation tfstate owns the identity" â€” which is true
EVEN on a second install that inherited the identity. So the gate blocked auto-import on
exactly the shared-domain second-install scenario it was meant to serve.

New gate: `sesv1Client != nil && r53ImportClient != nil && stateReader != nil` â€” fires any
time real production clients are available. `autoImportFoundationSESRecords` is already
idempotent (calls `stateReader.StateOwns` before each import and skips owned records), so
unconditional invocation is safe. `registerID` retains its two other consumers (log output
at line 568-572 and `KM_REGISTER_DOMAIN_IDENTITY` env export at line 576) â€” no unused
variable issue.

### wrapAutoImportError helper (bootstrap.go:144-154)

Extracted from the inline `fmt.Errorf` at the old line 602. Production code at line 617
now calls `wrapAutoImportError(err, sesDir, emailDomain)`. Recovery message:

```
auto-import SES Route53 records: <inner error>

Recovery: manually import the failed record then re-run `km bootstrap --shared-ses --dry-run=false`. Example:
  cd <sesDir>
  terragrunt import 'aws_route53_record.dkim[0]' '<zone-id>_<token>._domainkey.<emailDomain>_CNAME'
```

## Runtime behavior change

| Scenario | Old (84.4) | New (84.4.1) |
|---|---|---|
| First km install (fresh domain) | registerID=true â†’ gate blocked â†’ auto-import skipped | gate fires â†’ autoImport runs â†’ StateOwns=false for all â†’ 5 imports attempted â†’ all succeed (records don't exist yet in Route53 â€” none to skip; apply creates them) |
| Second km install (shared domain, registerID=true) | gate blocked â†’ operator must run `terragrunt import` manually | gate fires â†’ autoImport runs â†’ StateOwns=false for unimported records â†’ imports them â†’ apply succeeds |
| Re-run bootstrap (all records already in state) | n/a (gate blocked) | gate fires â†’ autoImport runs â†’ StateOwns=true for all â†’ 0 imports â†’ no-op |

Wait â€” first-install scenario: when the domain is genuinely fresh, `autoImportFoundationSESRecords` calls `sesClient.GetIdentityDkimAttributes`. If DKIM tokens are empty (domain not yet verified), it logs a warning and returns nil. If tokens exist, it probes Route53 via `ListResourceRecordSets` â€” those records won't exist yet, so `fakeRoute53RecordLister.hasMX=false`/`hasTXT=false` path applies and zero imports occur. Net effect for first install: no change from before (auto-import is a no-op because records don't exist in Route53 yet).
