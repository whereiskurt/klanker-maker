# Phase 79.1: audit-pipe FIFO Recreation on Resumed Sandboxes — Research

**Researched:** 2026-05-16
**Domain:** systemd-tmpfiles.d, Linux FIFO semantics, Go sidecar self-heal, EC2 stop+resume boot sequencing
**Confidence:** HIGH

---

## Summary

Phase 79 shipped `km-presence.service` (root-owned daemon, 60s tick) that emits heartbeats to `/run/km/audit-pipe` via `printf ... | timeout 0.1 tee /run/km/audit-pipe`. On a fresh `km create` the FIFO is created by the cloud-init userdata script before any sidecar starts — everything works. On `km resume` (EC2 stop+start), `/run` is a tmpfs that is wiped clean by the kernel at every boot. cloud-init does NOT re-run on second boot (it uses the instance-id semaphore: same instance-id → scripts already ran → skip). `km-bootstrap.service` (Phase 56.2) does re-create `/run/km` directory at every boot but deliberately does NOT create the FIFO (a Phase 56.1 design decision: "let the sidecar FIFO-retry loop own the FIFO"). After resume, `km-presence.service` starts (after `km-audit-log.service` per its unit ordering) and on its first tick runs `tee /run/km/audit-pipe`. Since the FIFO is absent, `tee` creates a **regular file** at that path, owned by `root:root 0644`. `km-audit-log` (running as `km-sidecar`) then tries to open the path with O_RDWR and gets `EACCES` because it cannot write a root-owned 0644 file. After 10 retries it gives up, the audit pipeline is dead, and `km doctor presence_daemon_healthy` falsely WARNs.

The fix is two independent layers. Layer 1 (root cause) installs a `systemd-tmpfiles.d` drop-in that recreates the FIFO with correct ownership on every boot — before either `km-audit-log` or `km-presence` can race to `/run/km/audit-pipe`. Layer 2 (defense in depth) adds a self-heal branch inside `openAuditPipeWithRetry` that detects a non-FIFO path and replaces it before attempting the open, so the system degrades gracefully even if the FIFO is still absent or wrong-typed when audit-log starts.

**Primary recommendation:** Implement both layers. Layer 1 is the root-cause fix (correct on resume, correct on fresh boot). Layer 2 is defense-in-depth that makes the sidecar resilient to future sequencing surprises. Rollout: `make build && make sidecars && km init --sidecars`; existing sandboxes need `km destroy && km create` for either layer to take effect.

## Research Q&A

### Q1: systemd-tmpfiles `p` vs `p+` syntax

**Source:** Debian Unstable tmpfiles.d(5) man page — HIGH confidence.

The `p` type creates a named pipe (FIFO) **only if the path does not yet exist**. If a regular file already exists at the path, `p` is a no-op — it will not replace the existing file.

The `p+` type creates the FIFO AND, if anything else (a regular file, symlink, etc.) already occupies the path, removes it first and creates a fresh FIFO.

**Exact line format (seven whitespace-separated columns):**

```
#Type  Path                     Mode  User       Group      Age  Argument
p+     /run/km/audit-pipe       0666  km-sidecar km-sidecar -    -
```

Meaning: on every `systemd-tmpfiles --create` invocation, if `/run/km/audit-pipe` is absent → create FIFO; if it exists as any non-FIFO type → remove and create FIFO; if it exists as a correctly-typed FIFO → leave it alone.

**Use `p+`, not `p`.** `p` alone would be a no-op when km-presence has already written a regular file at the path (the bug scenario on a second resume after a failed recovery).

**Drop-in file location:** `/usr/lib/tmpfiles.d/km.conf` (vendor path, not `/etc/tmpfiles.d/` which is for admin overrides). The directory `/usr/lib/tmpfiles.d/` is the correct machine-shipped location.

**Parent directory creation:** tmpfiles.d implicitly creates leading directories (owned root:root 0755) for all line types that create file nodes. However, `/run/km` ownership must be `km-sidecar:km-sidecar` — so km-bootstrap's `mkdir -p /run/km && chown km-sidecar:km-sidecar /run/km` remains necessary. A second `d` line can be added to the same `km.conf` to ensure the directory exists with correct ownership even before km-bootstrap runs:

```
#Type  Path      Mode  User       Group      Age  Argument
d      /run/km   0755  km-sidecar km-sidecar -    -
p+     /run/km/audit-pipe  0666  km-sidecar km-sidecar -  -
```

### Q2: When does systemd-tmpfiles-setup.service run on AL2023?

**Source:** man8/systemd-tmpfiles.8, putorius.net article — HIGH confidence.

`systemd-tmpfiles-setup.service` is a core systemd unit (not optional, always enabled). It runs as one of the **earliest units in the boot sequence**: after `local-fs.target` (filesystems mounted) and before `sysinit.target`. Its unit ordering:

```
After=local-fs.target systemd-sysusers.service
Before=sysinit.target shutdown.target
```

On an EC2 stop+start (which is a full hardware reboot — power cycle, BIOS/EFI, kernel init, systemd), the full boot sequence runs from scratch. `systemd-tmpfiles-setup.service` runs on **every** such boot, including stop+resume cycles. The `/run` tmpfs is freshly mounted by the kernel before systemd starts, so `systemd-tmpfiles-setup.service` sees an empty `/run` and creates whatever entries are declared in `tmpfiles.d/*.conf`.

**Important ordering consequence:** `systemd-tmpfiles-setup.service` completes before `sysinit.target`, which completes before `basic.target`, which completes before `network.target`. Both `km-audit-log.service` and `km-presence.service` have `After=network.target`. This means the FIFO is guaranteed to be created by the time either sidecar attempts to open it — no `After=systemd-tmpfiles-setup.service` directive is needed in the km-audit-log or km-presence unit files. The ordering chain provides this transitively.

### Q3: AL2023 cloud-init re-run behavior on stop+resume

**Source:** AWS EC2 User Guide (docs.aws.amazon.com/AWSEC2/latest/UserGuide/user-data.html), cloud-init docs — HIGH confidence (confirmed by the pre-existing km-bootstrap.service Phase 56.2 comments in userdata.go:1064).

cloud-init tracks which modules have run using semaphore files under `/var/lib/cloud/instances/<instance-id>/sem/`. When an EC2 instance is stopped and started, the instance-id does not change. cloud-init sees the same instance-id, finds the semaphore files, and skips all modules that have already run. The userdata script (`scripts-user` module) does not re-execute.

This is the root cause: the initial `mkfifo /run/km/audit-pipe` in userdata.go:973 runs once at first boot but never again. The FIFO lives in tmpfs (`/run`), so it vanishes on every boot after the first.

The existing km-bootstrap.service (Phase 56.2) was aware of this but deliberately chose NOT to recreate the FIFO ("do NOT create the FIFO here to avoid an ownership/permission race" — userdata.go:1093–1094). That design decision was correct for the Phase 56.1 era (before km-presence), but Phase 79 introduced a writer (km-presence, running as root) that races the sidecar's retry loop.

### Q5: `openAuditPipeWithRetry` self-heal — exact analysis

**File:** `sidecars/audit-log/cmd/main.go`, lines 226–255.

**Current logic (verbatim):**
```go
func openAuditPipeWithRetry(pipePath string, maxAttempts int, baseBackoff time.Duration) (*os.File, error) {
    // ...
    for attempt := 1; attempt <= maxAttempts; attempt++ {
        if mkErr := os.MkdirAll(filepath.Dir(pipePath), 0755); mkErr != nil {
            lastErr = mkErr
            // log warn, continue
        } else {
            // Create FIFO if missing (ignore EEXIST).
            if _, statErr := os.Stat(pipePath); os.IsNotExist(statErr) {
                if mkfErr := syscall.Mkfifo(pipePath, 0666); mkfErr != nil && !os.IsExist(mkfErr) {
                    lastErr = mkfErr
                    // log warn
                }
            }
            f, openErr := os.OpenFile(pipePath, os.O_RDWR, 0)
            if openErr == nil {
                return f, nil
            }
            lastErr = openErr
            log.Warn()...
        }
        // sleep backoff
    }
    return nil, lastErr
}
```

**Bug:** The `os.IsNotExist(statErr)` guard only runs `Mkfifo` when the path is absent. When the path EXISTS as a regular file (stat succeeds, `IsNotExist` is false), the code falls through directly to `os.OpenFile(..., O_RDWR, 0)`. Opening a regular file `root:root 0644` as `km-sidecar` with O_RDWR requires write permission that km-sidecar doesn't have → `EACCES`. This fails on all 10 attempts with the same error.

**(a) What error:** `os.OpenFile(path, os.O_RDWR, 0)` on a `root:root 0644` regular file as km-sidecar → syscall returns `EACCES` (permission denied). The log shows `audit-log: retrying pipe open` with `err="open /run/km/audit-pipe: permission denied"`.

**(b) Where to insert the self-heal check:** Replace the simple `if _, statErr := os.Stat(pipePath); os.IsNotExist(statErr)` block with a two-branch check:

```go
if info, statErr := os.Stat(pipePath); os.IsNotExist(statErr) {
    // Path absent — create fresh FIFO.
    if mkfErr := syscall.Mkfifo(pipePath, 0666); mkfErr != nil && !os.IsExist(mkfErr) {
        lastErr = mkfErr
        log.Warn().Err(mkfErr).Int("attempt", attempt).Msg("audit-log: mkfifo failed")
    }
} else if statErr == nil && info.Mode()&os.ModeNamedPipe == 0 {
    // Path exists but is NOT a named pipe (e.g., regular file left by km-presence
    // when FIFO was absent on resume). Remove it and create a fresh FIFO.
    log.Warn().Str("pipe", pipePath).Str("mode", info.Mode().String()).
        Int("attempt", attempt).Msg("audit-log: path exists as non-FIFO, replacing")
    if rmErr := os.Remove(pipePath); rmErr != nil {
        lastErr = rmErr
        log.Warn().Err(rmErr).Int("attempt", attempt).Msg("audit-log: remove non-FIFO failed")
    } else if mkfErr := syscall.Mkfifo(pipePath, 0666); mkfErr != nil && !os.IsExist(mkfErr) {
        lastErr = mkfErr
        log.Warn().Err(mkfErr).Int("attempt", attempt).Msg("audit-log: mkfifo after remove failed")
    }
}
// FIFO now exists (or mkfifo above failed — fall through to open attempt).
f, openErr := os.OpenFile(pipePath, os.O_RDWR, 0)
// ...
```

**(c) Does km-sidecar have permission to `os.Remove("/run/km/audit-pipe")` when the file is owned by root?**

Yes. Linux file removal requires write+execute permission on the **containing directory**, not on the file itself (unless the sticky bit `+t` is set on the directory). `/run/km` is owned by `km-sidecar:km-sidecar` with default mode 0755 and NO sticky bit (see userdata.go:971–972: `mkdir -p /run/km && chown km-sidecar:km-sidecar /run/km`). km-sidecar owns the directory and has `rwx` on it, so it can unlink any file within `/run/km` regardless of that file's ownership. The `os.Remove` will succeed even on a `root:root 0644` regular file.

**No new imports needed:** `os.ModeNamedPipe` is a constant exported from the `os` package (Go 1.16+, confirmed for Go 1.25.5 used in this module). No `io/fs` import is required.

### Q8: Rollout pattern

**Layer 1 (userdata template change only — no new sidecar binary):**

The userdata template is baked into the `km` binary itself (it is a Go template compiled into `pkg/compiler/userdata.go`). When `km init --sidecars` is run, it (a) builds the km binary for linux/amd64, (b) uploads it to S3, and (c) the management Lambda downloads a fresh km binary and uses it to render userdata for new `km create` calls.

So Layer 1 requires: `make build && km init --sidecars`
- `make build` compiles the km binary with the new template.
- `km init --sidecars` uploads the new km binary + all sidecar binaries to S3 and refreshes the Lambda.

**Layer 2 (audit-log sidecar code change):**

The audit-log binary (`sidecars/audit-log/cmd/`) must be rebuilt and uploaded to S3. New sandboxes created after the upload will fetch the new binary. Existing sandboxes have the old binary; they need `km destroy && km create` or manual binary replacement.

Layer 2 requires: `make sidecars` (rebuilds and uploads all sidecar binaries including audit-log).

**Combined rollout for Phase 79.1:**

```bash
make build           # rebuild km binary (template change in pkg/compiler/userdata.go)
make sidecars        # rebuild + upload all sidecars including audit-log (Layer 2)
km init --sidecars   # upload new km binary to S3 + refresh Lambda
```

This is the same three-step pattern as Phase 79 and is consistent with the `project_schema_change_requires_km_init.md` memory.

**Is `km init --lambdas` needed?** No. Phase 79.1 makes no changes to the bridge Lambda, the create-handler Lambda, or any Terraform modules. `km init --lambdas` is for Lambda function code changes only.

**Is `terragrunt apply` needed?** No. No Terraform resources change.

### Q9: Retroactivity

**Neither layer is retroactive.** This matches the Phase 79 pattern.

- **Layer 1:** The `/usr/lib/tmpfiles.d/km.conf` file is written during cloud-init first boot. Existing sandboxes provisioned before Phase 79.1 do not have this file; their `/run/km` directory is recreated by km-bootstrap, but the FIFO is never created on resume. The manual recovery snippet works for existing sandboxes:
  ```bash
  rm /run/km/audit-pipe && mkfifo /run/km/audit-pipe && \
    chown km-sidecar:km-sidecar /run/km/audit-pipe && \
    chmod 666 /run/km/audit-pipe && \
    systemctl restart km-audit-log
  ```
- **Layer 2:** The self-heal logic lives inside the audit-log binary. Existing sandboxes have the old binary without the self-heal. The manual recovery above bypasses the need for Layer 2 on existing sandboxes.

**Conclusion:** Both layers require `km destroy && km create` for existing sandboxes. The manual recovery snippet is the operator workaround for existing sandboxes until they are reprovisioned.

### Q12: Pitfalls

**Pitfall 1: `p` vs `p+` confusion**
Using `p` instead of `p+` in the tmpfiles.d entry would be a silent no-op when a regular file already exists (e.g., after a failed first-boot or a second resume cycle before Phase 79.1 healing takes effect). Always use `p+`.

**Pitfall 2: Does systemd-tmpfiles-setup.service break the FIFO if audit-log is mid-read?**
Not a concern. `systemd-tmpfiles-setup.service` runs very early in boot — before `network.target` and therefore before `km-audit-log.service` starts. By the time km-audit-log opens the FIFO, tmpfiles has already finished. On a running system between boots, tmpfiles-setup runs exactly once per boot cycle; it does not re-execute unless triggered manually (`systemd-tmpfiles --create`). If someone manually runs `systemd-tmpfiles --create` while audit-log is reading the FIFO, `p+` would remove and recreate the FIFO — which would break the open file descriptor in audit-log. This is an operator error scenario, not a routine risk. Systemd runs this automatically only at boot.

**Pitfall 3: Race between `systemd-tmpfiles-setup.service` and `km-audit-log.service`**
The ordering chain `tmpfiles-setup → sysinit.target → basic.target → network.target → km-audit-log` ensures no race. The FIFO exists before km-audit-log starts. No `After=systemd-tmpfiles-setup.service` directive is needed in the km-audit-log unit file. The existing `After=network.target` is sufficient.

**Pitfall 4: km-presence.service ordering — could km-presence's first tick beat tmpfiles?**
No. km-presence has `After=network.target km-audit-log.service` — it starts after km-audit-log, which starts after network.target, which is long after tmpfiles-setup. km-presence can never be the first to write `/run/km/audit-pipe` on a post-Phase-79.1 sandbox.

**Pitfall 5: Layer 2 unlink+mkfifo concurrent with a writer**
`openAuditPipeWithRetry` runs at startup only, before the FIFO is opened for reading. At the moment the self-heal runs, no writer has the file open (km-presence hasn't started yet — it waits on km-audit-log per `After=km-audit-log.service`). There is no concurrency concern.

**Pitfall 6: `p+` sets mode/owner — will it fight km-bootstrap?**
The `d /run/km 0755 km-sidecar km-sidecar` line in km.conf and km-bootstrap's `chown km-sidecar:km-sidecar /run/km` are both setting the same ownership — no conflict. km-bootstrap runs as a `Type=oneshot` unit with `Before=amazon-ssm-agent.service`, which is after sysinit.target. By the time km-bootstrap runs, tmpfiles-setup has already created both entries. km-bootstrap's `mkdir -p /run/km && chown km-sidecar:km-sidecar /run/km` is idempotent and harmless.

**Pitfall 7: otelcol-contrib download in S3 — not relevant**
Phase 79.1 does not change the otelcol download path.

**Pitfall 8: The `km.conf` tmpfiles entry sets mode, user, group — but the file is created as `km-sidecar:km-sidecar`**
The `km-sidecar` user must exist before `systemd-tmpfiles-setup.service` runs. On AL2023, `systemd-sysusers.service` runs before `systemd-tmpfiles-setup.service` (the ordering: `After=systemd-sysusers.service`). km-sidecar is created by cloud-init userdata at first boot and is persisted in `/etc/passwd` on the root EBS volume across all stop+resume cycles. The user is available before tmpfiles-setup runs. No race.

### Pattern 1: systemd-tmpfiles.d `p+` for volatile FIFO recreation

**What:** Write a tmpfiles.d config file during first-boot cloud-init (persists on root EBS). systemd reads it at every subsequent boot and recreates the FIFO.

**When to use:** Any time a named pipe must exist in a tmpfs directory (`/run`) and multiple systemd services race to be the first writer or reader.

**Example:**
```
# /usr/lib/tmpfiles.d/km.conf
d  /run/km                 0755 km-sidecar km-sidecar - -
p+ /run/km/audit-pipe      0666 km-sidecar km-sidecar - -
```

Written to disk via the userdata template (Go heredoc → `cat > /usr/lib/tmpfiles.d/km.conf << 'TMPFILES'`).

### Pattern 2: Sidecar self-heal on wrong-type path

**What:** Before opening a path that must be a FIFO, stat it and check the mode. If it exists but is the wrong type, remove and recreate before attempting the open.

**When to use:** Any sidecar that opens a named pipe from a path shared with a writer that runs as a different user (and could inadvertently create a regular file at the path if the FIFO is absent).

**Example (Go):**
```go
if info, statErr := os.Stat(pipePath); os.IsNotExist(statErr) {
    _ = syscall.Mkfifo(pipePath, 0666) // create if absent
} else if statErr == nil && info.Mode()&os.ModeNamedPipe == 0 {
    // Wrong type: unlink and recreate
    log.Warn().Str("pipe", pipePath).Str("mode", info.Mode().String()).
        Int("attempt", attempt).Msg("audit-log: path exists as non-FIFO, replacing")
    if rmErr := os.Remove(pipePath); rmErr == nil {
        _ = syscall.Mkfifo(pipePath, 0666)
    }
}
```

### Anti-Patterns to Avoid

- **Using `p` instead of `p+`:** Silent no-op when a regular file exists. Use `p+`.
- **Adding tmpfiles.d entry to km-bootstrap instead of /usr/lib/tmpfiles.d/:** km-bootstrap runs after sysinit.target; tmpfiles-setup runs before it. Use the proper tmpfiles.d location.
- **Adding `After=systemd-tmpfiles-setup.service` to km-audit-log.service:** Unnecessary — the ordering chain through `network.target` already guarantees tmpfiles runs first. Adding an explicit dependency could create ordering cycles.

## Common Pitfalls

### Pitfall 1: `p` vs `p+` — silent no-op on wrong type

**What goes wrong:** Using `p` when a regular file exists at the path. `p` only creates; it does not replace. The FIFO remains absent, Layer 1 provides no benefit.

**Why it happens:** Easy to pick the simple type.

**How to avoid:** Always use `p+` for FIFO entries in `/run` (tmpfs directories where race conditions can leave a regular file).

**Warning signs:** On a sandbox where km-presence already wrote a regular file, `systemd-tmpfiles --create /usr/lib/tmpfiles.d/km.conf` runs but the path is still a regular file afterward.

### Pitfall 2: km-audit-log opens path before tmpfiles completes

**What goes wrong:** If km-audit-log.service somehow starts before `systemd-tmpfiles-setup.service` completes, it would see an empty `/run/km` and its retry loop would create the FIFO — which is fine. The concern is km-presence winning the race first.

**Why it happens:** Misconfigured unit ordering.

**How to avoid:** Both km-audit-log and km-presence have `After=network.target`. network.target is After=basic.target which is After=sysinit.target. tmpfiles-setup is Before=sysinit.target. The chain is guaranteed. No manual ordering fix needed.

### Pitfall 3: Self-heal `os.Remove` fails due to sticky bit

**What goes wrong:** If `/run/km` had the sticky bit set (`+t`), km-sidecar could not remove root-owned files even as directory owner.

**Why it doesn't apply here:** `/run/km` is created with `mkdir -p` (default 0755 no sticky bit) and `chown km-sidecar`. No sticky bit. Verified by inspecting userdata.go:971–972 and km-bootstrap:1095–1096. No `chmod +t` is ever applied to `/run/km`.

### Layer 2: Self-heal in `openAuditPipeWithRetry` (sidecars/audit-log/cmd/main.go)

Replace the existing stat+mkfifo block (lines 237–242) with:

```go
// Stat the path. Three cases:
// (a) absent: create fresh FIFO.
// (b) exists as FIFO: do nothing, fall through to OpenFile.
// (c) exists as wrong type (regular file left by km-presence on
//     resume before Phase 79.1 fix): unlink + create FIFO.
if info, statErr := os.Stat(pipePath); os.IsNotExist(statErr) {
    if mkfErr := syscall.Mkfifo(pipePath, 0666); mkfErr != nil && !os.IsExist(mkfErr) {
        lastErr = mkfErr
        log.Warn().Err(mkfErr).Int("attempt", attempt).Msg("audit-log: mkfifo failed")
    }
} else if statErr == nil && info.Mode()&os.ModeNamedPipe == 0 {
    // Layer 2 self-heal: path exists but is not a named pipe.
    log.Warn().Str("pipe", pipePath).Str("mode", info.Mode().String()).
        Int("attempt", attempt).Msg("audit-log: path exists as non-FIFO, replacing")
    if rmErr := os.Remove(pipePath); rmErr != nil {
        lastErr = rmErr
        log.Warn().Err(rmErr).Int("attempt", attempt).Msg("audit-log: remove non-FIFO failed")
    } else if mkfErr := syscall.Mkfifo(pipePath, 0666); mkfErr != nil && !os.IsExist(mkfErr) {
        lastErr = mkfErr
        log.Warn().Err(mkfErr).Int("attempt", attempt).Msg("audit-log: mkfifo after remove failed")
    }
}
// Fall through to OpenFile attempt in all cases.
```

No new imports — `os.ModeNamedPipe` is part of the `os` package already imported.
