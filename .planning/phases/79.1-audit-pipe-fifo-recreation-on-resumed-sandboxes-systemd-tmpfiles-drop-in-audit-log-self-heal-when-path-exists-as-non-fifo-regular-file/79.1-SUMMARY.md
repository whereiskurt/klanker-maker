**Phase ID:** 79.1

## Bug Summary

**Symptom:** After `km pause` + `km resume` on a sandbox provisioned with Phase 79 (km-presence daemon), `km doctor` reported `WARN: presence_daemon_healthy` indefinitely. `journalctl -u km-audit-log` showed `open /run/km/audit-pipe: permission denied` repeating across all 10 retry attempts.

**Discovered on:** sandbox `learn-14f484c7` (2026-05-16).

**Root cause — race on resumed boot:**

1. `/run` is a tmpfs. The kernel wipes it clean on every boot (including EC2 stop+start).
2. cloud-init does NOT re-run on second boot: it uses the instance-id semaphore (`/var/lib/cloud/instances/<id>/sem/`). Same instance-id → scripts already ran → skip. The initial `mkfifo /run/km/audit-pipe` from the userdata script never re-executes.
3. `km-bootstrap.service` (Phase 56.2) recreates `/run/km` directory at every boot but deliberately does NOT recreate the FIFO (a Phase 56.1 design decision: "let the sidecar retry loop own it"). After Phase 79 shipped km-presence, this assumption was no longer safe.
4. `km-presence.service` (root) starts after network.target. On its first tick (within 60s), it runs `printf ... | timeout 0.1 tee /run/km/audit-pipe`. Since the FIFO is absent, `tee` creates a **regular file** at that path, owned `root:root 0644`.
5. `km-audit-log.service` (running as `km-sidecar`) tries `os.OpenFile(path, os.O_RDWR, 0)`. Opening a `root:root 0644` regular file as km-sidecar requires write permission km-sidecar does not have. `EACCES` is returned on every one of the 10 retry attempts.
6. The audit pipeline is dead. No `source:"presence"` events reach CloudWatch. `km doctor presence_daemon_healthy` false-WARNs.

**Why Phase 79 didn't catch this:** Phase 79's UAT only tested a fresh `km create` — never a `km pause` + `km resume` cycle.

## Two-Layer Fix

| Layer | Description | File | Line range |
|---|---|---|---|
| **1 — root cause** | `systemd-tmpfiles.d` drop-in writes `/usr/lib/tmpfiles.d/km.conf` to the EBS root volume during cloud-init first-boot. `systemd-tmpfiles-setup.service` reads it at **every** subsequent boot (before `sysinit.target`, before either sidecar starts) and recreates the FIFO with `km-sidecar:km-sidecar 0666` ownership. The `p+` type replaces any wrong-type path atomically; `p` alone would be a silent no-op. | `pkg/compiler/userdata.go` | +15 lines between existing mkfifo block and km-audit-log.service heredoc |
| **2 — defense in depth** | `openAuditPipeWithRetry` now runs a three-branch stat check before each open attempt: (a) path absent → create FIFO via `syscall.Mkfifo`; (b) path exists as FIFO → no-op; (c) path exists as wrong type → `os.Remove` + `syscall.Mkfifo` recovery, log warn. No new imports (os.ModeNamedPipe, os.Remove, syscall.Mkfifo all already imported). km-sidecar can unlink root-owned files in `/run/km` because km-sidecar owns the directory and no sticky bit is set. | `sidecars/audit-log/cmd/main.go` | ~+20/-2 lines in `openAuditPipeWithRetry` |

**Key correctness detail — `p+` vs `p`:** Using `p` instead of `p+` in tmpfiles.d is a silent no-op when a regular file already occupies the path. `p+` clobbers wrong-type entries. Future tmpfiles.d work in this codebase must always use `p+` for FIFO entries in `/run`.

## Retroactivity

**Neither layer is retroactive.** This matches the Phase 79 / 75 / 73 / 68 / 67 / 63 pattern.

- **Existing sandboxes provisioned before Phase 79.1** do NOT have `/usr/lib/tmpfiles.d/km.conf` on their EBS root volume and run the old audit-log binary without the self-heal branch.
- `km destroy && km create` picks up both layers automatically for any sandbox reprovisioned after the rollout.
- Operator workaround for pre-Phase-79.1 sandboxes that have already tripped the bug (audit pipe is a regular file, not a FIFO):

```bash
km shell <sandbox-id>
sudo rm /run/km/audit-pipe && \
  sudo mkfifo /run/km/audit-pipe && \
  sudo chown km-sidecar:km-sidecar /run/km/audit-pipe && \
  sudo chmod 666 /run/km/audit-pipe && \
  sudo systemctl restart km-audit-log
```

After recovery: `journalctl -u km-audit-log` shows `reading from audit pipe pipe=/run/km/audit-pipe`, and `km doctor` reports `✓ Presence daemon healthy` within ~5 minutes.

## Lessons Learned

### 1. Lifecycle gate missing from Phase 79 UAT

Phase 79 closed without testing `km pause` + `km resume`. Every new boot-time daemon or systemd unit that touches tmpfs paths (`/run/*`) must have a lifecycle-cycle test before the phase closes: fresh boot AND at least one stop+resume cycle. Add this as a standard UAT gate for future sidecar phases.

### 2. tmpfs + cloud-init-skip-on-resume is a recurring bug class

Any file in `/run` that is created by cloud-init first-boot userdata will vanish on every subsequent boot and will NOT be recreated by cloud-init (instance-id semaphore prevents re-execution). The correct pattern for `/run` state that must survive stop+resume:

- **Primary:** Pair the `/run` path with a `/usr/lib/tmpfiles.d/` drop-in written during first-boot cloud-init (persists on EBS root volume; `systemd-tmpfiles-setup.service` re-creates on every boot before any user-space daemon starts)
- **Secondary (defense in depth):** Make the consumer daemon resilient to wrong-type paths (stat + self-heal before open)

Future sandbox state in `/run/km/*` should follow this pattern. Operator tip: `grep -rn "/run/km" pkg/compiler/userdata.go` to audit for other paths that may need the same treatment.

### 3. `p+` vs `p` matters — document this for future tmpfiles.d work

`p` in tmpfiles.d creates a FIFO only if the path is absent. `p` is a silent no-op when any other file type occupies the path. `p+` replaces the wrong-type path atomically. For FIFOs in tmpfs directories where a race can leave a regular file, always use `p+`. This is the first tmpfiles.d drop-in in the km codebase; the `p+` pattern is now established.

### 4. Layer 1 makes Layer 2 a recovery path, not a primary path

The UAT confirmed that on a correctly provisioned post-Phase-79.1 sandbox, Layer 1 alone is sufficient: the FIFO is recreated by tmpfiles before km-audit-log starts, so the Layer 2 self-heal branch is never triggered. Layer 2 is defense-in-depth for any future scenario where tmpfiles.d runs but the FIFO is still wrong-typed (e.g., another daemon writes the path before tmpfiles completes — currently impossible per the boot ordering analysis, but worth having). This is the correct engineering tradeoff: eliminate the root cause (L1), then add a runtime safeguard (L2).
