Inserted lines:
```
# Layer 1 (Phase 79.1): systemd-tmpfiles drop-in for /run/km/audit-pipe.
# Ensures the FIFO is recreated with correct ownership on every boot
# (including EC2 stop+resume, when cloud-init does NOT re-run and the
# /run tmpfs is wiped). systemd-tmpfiles-setup.service runs before
# sysinit.target, so the FIFO exists before km-audit-log.service or
# km-presence.service start. The p+ type replaces a regular file
# (e.g., one left by km-presence tee racing the sidecar on a prior
# boot) -- p alone would be a silent no-op in that case.
cat > /usr/lib/tmpfiles.d/km.conf << 'TMPFILES'
# Klankrmkr sandbox: volatile runtime directory and audit-pipe FIFO.
# systemd-tmpfiles-setup.service reads this before sysinit.target on every boot.
d  /run/km                  0755 km-sidecar km-sidecar - -
p+ /run/km/audit-pipe       0666 km-sidecar km-sidecar - -
TMPFILES
```

Key correctness details:
- `TMPFILES` terminator is at column 0 (required for bash heredoc)
- `p+` (not `p`): replaces a regular file with a FIFO; `p` alone would be a no-op
- Ownership `km-sidecar km-sidecar` with mode `0666` matches km-audit-log.service User requirement
- No backticks in comment text (Go raw string literal `...` cannot contain backticks)

## Decisions Made

- Used `p+` not `p` for the audit-pipe entry: the bug is that km-presence's `tee` creates a regular file at the path; `p` alone would be a no-op in exactly that scenario
- Kept the existing `mkfifo /run/km/audit-pipe` bash block untouched: it handles first-boot before systemd-tmpfiles runs, while the tmpfiles.d entry handles every subsequent resume — belt-and-suspenders as specified in RESEARCH.md
- Stripped backticks from template comments: Go raw string literals use backtick as the delimiter, so any embedded backtick silently terminates the string and causes a parse error at the next identifier
