<objective>
Close UAT Gap B: when an operator accepts a Slack Connect invite to a per-sandbox `#sb-XXX` channel, Slack emits a `subtype: "channel_join"` system message. The current `isBotLoop` deny-list (only catches `bot_message`, `message_changed`, `message_deleted`) lets it through to SQS → poller runs a wasted Claude turn (~$0.05) → reply post fails with `cannot_reply_to_message` because system messages are un-threadable.

The fix: switch isBotLoop from deny-list to allow-list semantics. A "real human turn" is `subtype == ""` OR `subtype == "thread_broadcast"`. Reject everything else (channel_join, channel_leave, channel_topic, channel_purpose, channel_name, channel_archive, channel_unarchive, pinned_item, unpinned_item, file_share, channel_convert_to_private, channel_convert_to_public, ekm_access_denied, message_replied, plus any future Slack-added subtypes — allow-list is regression-proof). Keep the existing bot_id and bot-user-ID checks as second-line defence. Add a debug log line on drop so future surprises are forensically visible.

Purpose: Gap B is independent of Gap A but landed in same wave (different files, no merge conflict). Without this fix, every Slack Connect invite acceptance burns API spend and pollutes the channel with ghost activity. Allow-list semantics are also future-proof — Slack adds new subtypes over time and every one is currently a regression in deny-list mode.

Output: One surgical edit to `isBotLoop` (lines 216-233 of events_handler.go) and an extended test suite covering 10+ system subtypes plus the positive `thread_broadcast` case.

**Note on `me_message` (`/me waves`-style stylistic posts):** these carry user text but are filtered by the allow-list. Rationale: `me_message` posts are a stylistic Slack convention rarely used as a deliberate agent prompt — operators typing `/me waves` to a sandbox almost certainly do not intend to spend a Claude turn on it. The cost of the false negative (an /me post that was meant as a prompt is silently dropped) is far lower than the false positive (every casual /me post burns ~$0.05 of Bedrock spend). If a user reports the dropped /me as a bug we can move it onto the allow-list trivially; until then, filter and observe via the new `events: subtype filter dropped subtype=me_message ...` debug log.

## The bug — pkg/slack/bridge/events_handler.go:214-233

```go
// isBotLoop returns true if the message originated from our bot or is a
// non-user message subtype.
func (h *EventsHandler) isBotLoop(ctx context.Context, m slackMessageEvent) bool {
    if m.BotID != "" {
        return true
    }
    switch m.Subtype {
    case "bot_message", "message_changed", "message_deleted":
        return true
    }
    if m.User == "" {
        return true // no user, not a real human turn
    }
    botUID, err := h.BotUserID.Fetch(ctx)
    if err != nil {
        h.log().Warn("events: bot user_id fetch failed", "err", err)
        return false // fail open — let dedup catch loops
    }
    return botUID != "" && m.User == botUID
}
```

## The fix — switch to allow-list

```go
func (h *EventsHandler) isBotLoop(ctx context.Context, m slackMessageEvent) bool {
    // (1) Bot ID set — definitively a bot post.
    if m.BotID != "" {
        return true
    }
    // (2) Allow-list of subtypes that count as "real human turn".
    //     Slack defines many subtypes for system events (channel_join, file_share, etc.)
    //     and historically adds new ones (e.g. ekm_access_denied was added years after launch).
    //     A deny-list missed channel_join in Phase 67 UAT — Gap B. Allow-list is regression-proof.
    switch m.Subtype {
    case "", "thread_broadcast":
        // Real human turn — fall through to bot-user-ID check.
    default:
        h.log().Debug("events: subtype filter dropped", "subtype", m.Subtype, "channel", m.Channel, "ts", m.TS)
        return true
    }
    // (3) Empty user — not a real human turn (e.g. message_replied no-content).
    if m.User == "" {
        return true
    }
    // (4) Second-line defence: user equals our own bot's user_id.
    botUID, err := h.BotUserID.Fetch(ctx)
    if err != nil {
        h.log().Warn("events: bot user_id fetch failed", "err", err)
        return false // fail open — let dedup catch loops
    }
    return botUID != "" && m.User == botUID
}
```

```go
// isBotLoop returns true if the message should NOT be dispatched to the
// sandbox: bot self-messages, system events (channel_join, file_share, etc.),
// or messages with no user attribution.
//
// Slack subtype semantics: an empty subtype field means a regular human
// post. `thread_broadcast` means a user replied in a thread with "Also send
// to channel" ticked — also a real human turn. Every other subtype is a
// system event or alternate message form (file_share, channel_join,
// channel_topic, ekm_access_denied, ...) that is either un-threadable
// (system messages can't accept replies, breaks bot post) or carries no
// user prompt content. Use an allow-list rather than a deny-list because
// Slack has historically added new subtypes (e.g. ekm_access_denied came
// years after launch) and a deny-list silently regresses each time —
// Phase 67 UAT Gap B was a channel_join slip-through that burned a Claude
// turn on a Slack Connect invite acceptance.
func (h *EventsHandler) isBotLoop(ctx context.Context, m slackMessageEvent) bool {
	// (1) Bot ID set — definitively a bot post (covers all bot_message variants).
	if m.BotID != "" {
		return true
	}
	// (2) Allow-list: only these two subtypes count as a real human turn.
	switch m.Subtype {
	case "", "thread_broadcast":
		// fall through
	default:
		h.log().Debug("events: subtype filter dropped",
			"subtype", m.Subtype, "channel", m.Channel, "ts", m.TS)
		return true
	}
	// (3) No user attribution — not a real human turn.
	if m.User == "" {
		return true
	}
	// (4) Second-line defence: user is our own bot's user_id.
	botUID, err := h.BotUserID.Fetch(ctx)
	if err != nil {
		h.log().Warn("events: bot user_id fetch failed", "err", err)
		return false // fail open — let dedup catch loops
	}
	return botUID != "" && m.User == botUID
}
```

- `bot_message`, `message_changed`, `message_deleted` are still filtered (they hit the `default` branch now instead of an explicit case)
