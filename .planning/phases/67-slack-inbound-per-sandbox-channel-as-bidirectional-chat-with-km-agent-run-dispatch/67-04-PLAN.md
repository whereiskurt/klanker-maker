<objective>
Extend the EC2 user-data compiler to conditionally emit the sandbox-side bash poller, its systemd unit, and the new `KM_SLACK_INBOUND_QUEUE_URL` env var when the profile sets `notifySlackInboundEnabled: true`. Mirror the existing `km-mail-poller` pattern at the same three insertion points (RESEARCH.md verified line numbers: ~739/1107/1667). Also wire the Phase 63 `km-notify-hook` to forward `KM_SLACK_THREAD_TS` as `--thread` to `km-slack post` (the flag is already wired in `cmd/km-slack/main.go:47` but unused — Phase 67 finally consumes it for in-thread replies).

Purpose: This is the sandbox-side dispatch surface — without the poller, SQS messages would accumulate forever. The bash+systemd choice (vs Go binary) is a CONTEXT.md locked decision: matches km-mail-poller, no new sidecar binary, debuggable via journalctl. The `--thread` consumption is what makes Claude's replies land *inside* the user's Slack thread instead of as new top-level messages.

Output: Three insertion points in pkg/compiler/userdata.go (bash heredoc + systemd unit + systemctl enable + notify-hook hook), one new template-data field, four passing tests.
</objective>
