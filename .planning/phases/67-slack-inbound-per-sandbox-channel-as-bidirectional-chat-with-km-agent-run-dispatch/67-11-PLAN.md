**Two checker-flagged subtleties baked into this plan:**

1. **Env availability for the root-side poller block.** `KM_SLACK_CHANNEL_ID` and `KM_SLACK_BRIDGE_URL` are written by cloud-init to `/etc/profile.d/km-slack-runtime.sh` (userdata.go:185-211), but the systemd unit (userdata.go:1329) only points `EnvironmentFile=/etc/profile.d/km-notify-env.sh`. The new ROOT-side post block runs in the poller process, NOT inside the existing `sudo -u sandbox bash -c "..."` block (which re-sources `/etc/profile.d/*.sh` at userdata.go:1013). Therefore the new block cannot rely on those env vars being present. Fix: resolve both from SSM Parameter Store at poller startup (mirrors the existing `KM_SLACK_INBOUND_QUEUE_URL` SSM fallback at userdata.go:921-943) and cache them for the service lifetime — the values don't change between turns.

2. **Ack ordering.** Posting to Slack BEFORE `sqs delete-message` opens a host-crash window where the user sees a reply but the message gets redelivered → duplicate Claude run + duplicate Slack post on the next visibility-timeout expiry. Fix: move the post block INSIDE the existing `if [ -n "$NEW_SESSION" ]; then` branch, AFTER the SQS delete-message. Trade-off: on agent failure (non-zero exit OR empty .result) no fallback "(no recent assistant text)" message is posted to Slack; the operator sees silence and diagnoses via `journalctl -u km-slack-inbound-poller` (the existing "message returns to queue" WARN line). This is the safer default — a fake fallback message in production is indistinguishable from a real empty reply, while genuine silence is a clear signal to inspect logs.
