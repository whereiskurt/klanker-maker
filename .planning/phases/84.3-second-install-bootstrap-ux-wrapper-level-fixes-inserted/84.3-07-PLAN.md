<objective>
Fix gap 1: viper env-binding masks drift detection in ExportTerragruntEnvVars.

Root cause: `config.Load()` calls `v.AutomaticEnv()` which makes viper auto-bind `KM_REGION` → `cfg.PrimaryRegion`. By the time `ExportTerragruntEnvVars(cfg)` runs, `cfg.PrimaryRegion` already equals `os.Getenv("KM_REGION")`, so the comparison `envVal != cfgVal` is always false and no WARN fires.

Fix: Snapshot the raw yaml values from `v2` (the km-config.yaml viper instance) into a new `cfg.YAMLDefaults map[string]string` field BEFORE returning from `Load()`. `ExportTerragruntEnvVars` then compares `os.Getenv(envKey)` against `cfg.YAMLDefaults[yamlKey]` instead of against the viper-mutated `cfgVal`.

Affected keys (all env-bound via viper AutomaticEnv):
- KM_REGION → region
- KM_DOMAIN → domain
- KM_ARTIFACTS_BUCKET → artifacts_bucket
- KM_RESOURCE_PREFIX → resource_prefix
- KM_OPERATOR_EMAIL → operator_email
- KM_ROUTE53_ZONE_ID → route53_zone_id
- KM_SCHEDULER_ROLE_ARN → scheduler_role_arn
- KM_REGION_LABEL → region_label (derived, not a yaml key — see note in action)
- KM_EMAIL_SUBDOMAIN → email_subdomain
- KM_RESOURCE_PREFIX (special-cased in ExportTerragruntEnvVars)
- KM_EMAIL_SUBDOMAIN (special-cased in ExportTerragruntEnvVars)

Purpose: Make drift WARN fire when operator sets `KM_REGION=us-west-2` but km-config.yaml says `region: us-east-1`.

Output: config.go adds `YAMLDefaults` field and populates it; init.go's warnAndSetEnv reads from it. Plan 06 RED tests turn GREEN.
</objective>

MODIFIED (init.go) — update warnAndSetEnv to use YAMLDefaults:
```go
warnAndSetEnv := func(envKey, yamlKey, cfgVal string) {
    envVal := os.Getenv(envKey)
    if envVal == "" {
        // env not set — no drift possible; set from cfg if available
        if cfgVal != "" {
            os.Setenv(envKey, cfgVal)
        }
        return
    }
    // Use yaml snapshot for drift comparison; fall back to cfgVal if no snapshot.
    // After viper AutomaticEnv(), cfgVal == envVal for env-bound keys, so the
    // snapshot is required to detect real yaml↔env divergence (Phase 84.3 gap 1).
    yamlVal := cfgVal
    if cfg.YAMLDefaults != nil {
        if snap, ok := cfg.YAMLDefaults[yamlKey]; ok {
            yamlVal = snap
        }
    }
    if yamlVal != "" && envVal != yamlVal {
        fmt.Fprintf(os.Stderr, "WARN: %s=%s (env) overrides km-config.yaml %s=%s\n", envKey, envVal, yamlKey, yamlVal)
    }
    // env wins — do not override with cfg value
}
```

The special-cased KM_RESOURCE_PREFIX and KM_EMAIL_SUBDOMAIN blocks (~line 793 and ~line 800)

// KM_RESOURCE_PREFIX (special-cased — prefix can be "" legitimately)

NOTE on KM_REGION_LABEL: region_label is derived (compiler.RegionLabel(cfg.PrimaryRegion)) — it
has no yaml key. For this derived key, the drift comparison uses cfg.PrimaryRegion from YAMLDefaults["region"]
to derive the expected label. If the operator sets KM_REGION_LABEL directly (rare), warn if it
differs from the derived value. Current warnAndSetEnv("KM_REGION_LABEL", "region_label", regionLabel)
passes the derived value as cfgVal, which is fine — the snapshot for "region_label" will not be set
(no such yaml key), so yamlVal falls back to cfgVal (the derived value). This is the correct behavior
for this edge case.

NOTE on accounts.* keys: These are yaml-authoritative (Plan 02). Their cfg fields already reflect
yaml values (not env values) after Plan 02. So warnAndSetEnv for KM_ACCOUNTS_* already works correctly
— cfgVal IS the yaml value. YAMLDefaults snapshot is NOT needed for accounts.* keys.
