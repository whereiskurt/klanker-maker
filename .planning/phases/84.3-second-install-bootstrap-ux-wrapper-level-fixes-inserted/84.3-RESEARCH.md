## Summary

Phase 84.3 is a focused Go source patch across four files (`configure.go`, `pkg/config/config.go`, `bootstrap.go`, `init.go` + new `env.go`). All eight closures are wrapper-level: no Terraform module changes, no new AWS resources, no new dependencies. The research confirms that:

1. All code anchors in CONTEXT.md are accurate within ±5 lines. The only exception is that `ensureArtifactsBucket` is at line 1411 but the actual function definition starts at 1411 (caller at 1345 inside `runBootstrap`) — confirmed correct.
2. The viper bug for `accounts.*` is precisely understood: `isSetByEnv` at `config.go:197` returns `true` when `KM_ACCOUNTS_ORGANIZATION` is set (even to an unexpected value), preventing the yaml merge. The fix is to simply not include `accounts.organization`, `accounts.dns_parent`, `accounts.application` in the `isSetByEnv` guard — yaml always wins for those three keys.
3. `ExportTerragruntEnvVars` at `init.go:736` uses `if os.Getenv(key) == "" { os.Setenv(key, val) }` throughout — this silently skips setting when env is already set, even when the env value differs from cfg. Adding a drift WARN requires checking `os.Getenv(key) != "" && os.Getenv(key) != val` before the set.
4. The `runInitPlan` plan loop already has a `for _, envVar := range m.envReqs { skip }` pattern at line 1076-1085. The outputs.json probe inserts at the same level, before `planModule()`, using `os.Stat(outputsFile)`.
5. `km env` follows the exact same file-per-command pattern as `km info` (info.go). Registration is in root.go via `root.AddCommand(NewEnvCmd(cfg))`.
6. The mock seam for HeadBucket follows the `KMSEnsureAPI` variadic-last pattern: `func ensureArtifactsBucket(ctx, cfg, w io.Writer, s3client ...S3HeadBucketAPI)` — same as `ensureKMSPlatformKey`. Tests in `bootstrap_kms_test.go` show the exact mock structure to copy.
7. STS `GetCallerIdentity` is already used at `create.go:1355` and has an established mock pattern in `doctor_test.go`. The `STSCallerAPI` interface at `doctor.go:83` can be reused directly.

**Primary recommendation:** Implement wave-by-wave exactly as locked. The biggest risk is the viper accounts.* fix — verify via unit test that `KM_ACCOUNTS_ORGANIZATION=somevalue` in env no longer shadows yaml's `accounts.organization` after the fix.

## Common Pitfalls

### Pitfall 1: The viper accounts.* fix and `accounts.terraform`

**What goes wrong:** If the developer removes the `isSetByEnv` guard for ALL four `accounts.*` keys, operators who legitimately set `KM_ACCOUNTS_TERRAFORM` for one-off invocations will have their env var silently ignored. The CONTEXT.md locks that only `accounts.organization`, `accounts.dns_parent`, and `accounts.application` become yaml-authoritative.

**How to avoid:** In `config.go`, the merge loop (lines 278-313) should continue to check `!isSetByEnv(v, key)` for `"accounts.terraform"` and `"accounts.application"` — actually, re-reading CONTEXT.md: `accounts.application` IS in the locked set to remove env fallback. The three keys that lose env fallback are: `accounts.organization`, `accounts.dns_parent`, `accounts.application`. `accounts.terraform` keeps its current env-var-wins behavior.

**Warning signs:** `km bootstrap` status banner shows empty application/org account IDs even after the fix.

### Pitfall 2: ExportTerragruntEnvVars drift detection — only WARN when env differs from cfg

**What goes wrong:** The drift WARN must fire only when env is set AND the env value differs from what cfg has. If env matches cfg (e.g., operator did `eval $(km env)` and then re-ran a command), no WARN should fire. The check is:
```go
if envVal := os.Getenv(envKey); envVal != "" && envVal != cfgVal {
    fmt.Fprintf(os.Stderr, "WARN: %s=%s (env) overrides km-config.yaml %s=%s\n", envKey, envVal, yamlKey, cfgVal)
}
os.Setenv(envKey, cfgVal) // behavior unchanged — cfg always wins in the set path
```
Wait — the CONTEXT says top-level KM_* *keep* env-var-overrides-yaml semantics but emit a WARN. So the existing behavior (env wins, cfg skips) is preserved, but the WARN fires when they differ. The current code only sets from cfg when env is empty. The drift scenario is: env is set (non-empty) and differs from cfg. The WARN fires in that window. The set doesn't happen (env wins = no setenv call). The WARN fires before the `if os.Getenv(key) == "" { os.Setenv(...) }` block.

**Warning signs:** Tests pass when they shouldn't because a previously-set env var from another test leaked (use `t.Setenv` not `os.Setenv` in tests to auto-cleanup).

### Pitfall 3: HeadBucket 403 vs 404 distinction

**What goes wrong:** S3 HeadBucket returns 403 for two different cases: (a) bucket name taken by another account (globally unique namespace collision) and (b) operator has no permission to check. For closure (a), the intended behavior is specifically about the "name taken" case. A 404 means "name available, doesn't exist." The retry logic should only apply to 403, not 404.

**How to avoid:** Use `errors.As` to extract the HTTP status code from the AWS SDK error response:
```go
var httpErr *smithyhttp.ResponseError
if errors.As(err, &httpErr) && httpErr.Response.StatusCode == 403 { /* name taken */ }
```
Or check for `s3types.NotFound` vs the generic request failure. The project already handles `err == nil` (bucket exists) in `ensureArtifactsBucket:1430`.

**Warning signs:** Operator gets stuck in a retry loop because they have IAM permission issues, not bucket collision.

### Pitfall 4: configure.go HeadBucket needs STS to derive account ID FIRST

**What goes wrong:** The HeadBucket check needs an account ID to suggest the auto-suffixed name. The STS GetCallerIdentity call must happen before the HeadBucket check, not after. If STS fails (no credentials configured yet), the probe should skip gracefully rather than blocking the wizard.

**How to avoid:** Make the STS call best-effort: if it fails, skip the HeadBucket probe and the auto-derivation (proceed with what the operator typed). The wizard is not the place to enforce credentials — `km init --dry-run=false` will catch it later.

### Pitfall 5: `km bootstrap --all` flag vs `--shared-ses` interaction

**What goes wrong:** The current routing block in `bootstrap.go:797-812` checks `sharedSES && plan` → `sharedSES` → default. Adding `--all` needs to be checked BEFORE the existing `sharedSES` check, because `--all` with `--shared-ses` is an error (or `--all` takes precedence). The CONTEXT locked that `--all` chains BOTH subflows in order, making `--shared-ses` redundant when used with `--all`.

**How to avoid:** Insert `if all { return runBootstrapAll(ctx, cfg, dryRun, plan, acceptDestroys, w) }` at the top of the routing block, before the `sharedSES` check.

### Pitfall 6: outputs.json probe regex for efs module only vs generic

**What goes wrong:** The file-existence probe for outputs.json could be over-engineered into a general scanner of all terragrunt.hcl files. But the CONTEXT locked file-existence probe (not HCL parsing). The practical approach: for the `efs` module (the only one confirmed to depend on `network/outputs.json` in Phase 84.3's scope), stat the specific file. If future modules add similar dependencies, the pattern extends cleanly.

**Detection:** Check `infra/live/<regionLabel>/network/outputs.json` before planning `efs`. If missing, emit `[skip: depends on network/outputs.json — apply network first]`.
