<decisions>
## Implementation Decisions

### Closure (g): `km env` subcommand shape

- **Invocation:** Dedicated `km env` subcommand (NOT a `--export` flag on another command).
  - Why: future-proof for sub-flags (`--json`, `--unset`, `--aws-profile`, `--diff`); reads naturally; doesn't couple to doctor.
- **Default output:** `export KEY=value` lines on stdout — one per env var.
  - Why: survives `eval $(km env)` in bash/zsh; matches the canonical pattern operators already know; explicit `export` keyword (CI tooling expects it).
- **AWS_PROFILE:** EXCLUDED by default. Operators must pass `--aws-profile` to include it.
  - Why: AWS_PROFILE is operator-shell-local and may differ across operators; excluding by default prevents accidentally exporting one operator's profile into another's `eval`. KM_* vars (which terragrunt subprocesses actually read via `site.hcl get_env()`) are the default scope.
- **Scope:** Single output covers BOTH foundation (bootstrap) and regional (init) env vars — the full superset:
  - `KM_RESOURCE_PREFIX`, `KM_REGION`, `KM_REGION_LABEL`, `KM_DOMAIN`, `KM_EMAIL_SUBDOMAIN`, `KM_ROUTE53_ZONE_ID`, `KM_ACCOUNTS_ORGANIZATION`, `KM_ACCOUNTS_DNS_PARENT`, `KM_ACCOUNTS_APPLICATION`, `KM_ARTIFACTS_BUCKET`, `KM_OPERATOR_EMAIL`
  - Why: one `eval` covers any direct terragrunt invocation (foundation or regional); splitting via `--foundation/--regional` forces operators to know which layer they're poking.
- **Source of truth:** The same `ExportTerragruntEnvVars(cfg)` helper that internal commands use — `km env` is a stdout-print of the same env-var set, derived from the loaded km-config.yaml.

### Closure (f): Bootstrap workflow consolidation

- **Both fixes:** Add new `km bootstrap --all` AND make `km init` hard-fail (not warn) when artifacts bucket is missing.
  - Why: defense in depth. `--all` is the happy-path discoverability fix; hard-fail catches operators who still run subflows individually and skip one.
- **`km bootstrap --all` scope:** Chains plain `runBootstrap` then `runBootstrapSharedSES` in order. Does NOT chain `km init`.
  - Why: bootstrap and init are distinct lifecycle stages; chaining init couples failures (operator wouldn't know which step failed) and init's wave-based output drowns bootstrap's.
- **`--all` flag interactions:** Respects `--dry-run` and `--plan` symmetrically. `km bootstrap --all --plan` plans BOTH subflows in order (reuses Phase 84.2's planreport gate for the shared-ses subflow). `km bootstrap --all --dry-run=true` (default) prints both subflows' dry-run text.
  - Why: preserves the Phase 84.2 plan-before-apply discipline; no apply-only path that bypasses the gate.
- **Configure finale:** After writing km-config.yaml, print a "Next steps:" block on stdout AND write the same sequence as header comments at the top of the generated km-config.yaml.
  - Why: stdout block is visible immediately; yaml header comments survive re-reads and don't require a separate `NEXT-STEPS.md` file. Recommend `km bootstrap --all --dry-run=false` as the primary command; mention the granular flow as "or run individually: ...".
- **`km init` hard-fail message:** Specific error that names the missing step:
  ```
  ✗ artifacts bucket km-artifacts-123456789012 does not exist.

  Run: km bootstrap --all --dry-run=false
  (or `km bootstrap --dry-run=false` for just the SCP/artifacts subflow —
   `km bootstrap --shared-ses` alone does NOT create the artifacts bucket.)
  ```
  - Why: operator gets the fix in the error message; no diagnostic hop through `km doctor`; both the happy-path (`--all`) and granular-flow (`bootstrap` alone) are named so operators using either model can recover.

### Closure (h): Viper precedence for `accounts.*` + closure (c) WARN

- **`accounts.*` precedence — yaml authoritative:** REMOVE viper env-var fallback for nested `accounts.organization`, `accounts.dns_parent`, `accounts.application` keys.
  - `KM_ACCOUNTS_*` env vars are still EXPORTED to terragrunt subprocesses (site.hcl `get_env()` continues to read them) but they are NEVER read back into `cfg`.
  - Why: eliminates the precedence bug at the source. The current viper behavior silently shadows yaml values to empty when KM_ACCOUNTS_* is set to something viper interprets as unset — operators have no signal until they hit the "(not set)" banner.
- **Top-level `KM_*` precedence — keep env-precedence + WARN:** Top-level vars (KM_RESOURCE_PREFIX, KM_REGION, KM_DOMAIN, KM_EMAIL_SUBDOMAIN, KM_ROUTE53_ZONE_ID, KM_ARTIFACTS_BUCKET, KM_OPERATOR_EMAIL) keep current env-var-overrides-yaml semantics, but emit a single-line WARN per drifted var.
  - Why: operators legitimately set KM_REGION etc. for one-off invocations; preserving this preserves their workflow. The WARN closes the visibility gap (closure c) without changing behavior.
- **WARN line format:** Single line to stderr, one per drifted var, emitted once per command invocation:
  ```
  WARN: KM_REGION=us-west-2 (env) overrides km-config.yaml region=us-east-1
  ```
  - Why: stderr keeps stdout clean for `eval $(km env)` and other pipe consumers; one-line-per-var is greppable; quiet when no drift.
- **Configure interaction:** `km configure` warns (does NOT block) when KM_* env vars exported in the operator's shell would conflict with what's being written to yaml. One-line WARN per conflict matching the format above, but framed as "exported in shell will shadow km-config.yaml in current session".
  - Why: operator may have intentional override; blocking is invasive. WARN at configure time catches the issue at the earliest possible point.
- **Authoritative banner:** `km bootstrap`'s status banner reads from `cfg` post-load (which under the new precedence reflects the yaml values for accounts.*). If any required account ID is empty, emit a WARN naming the missing key and the config-load path.

### Closure (a): HeadBucket retry UX

- **On 403 (bucket name taken):** Print an inline suggestion using `${current_name}-${account_id}` and prompt for accept / edit / abort:
  ```
  ✗ tf-km-state-us-east-1 is taken (HeadBucket 403 — globally unique name)

  Suggestion: tf-km-state-us-east-1-123456789012  [Y / edit / abort]:
  ```
  - Why: account ID is always-available, deterministic, and collision-free for the second-account case. One-keystroke fix. Operator can override with `edit` or bail with `abort`.
- **Retry cap:** 1 retry then bail. If the account-ID-suffixed name ALSO returns 403, bail with `pick a unique state_bucket via --state-bucket`.
  - Why: account-ID-suffixed collisions are vanishingly rare (would imply duplicate AWS account IDs — not possible) or indicate a deeper auth issue (HeadBucket on operator's own account ID failing); deeper loops invite a stuck wizard.
- **Edit path:** If operator picks `edit`, prompt for a freeform name, HeadBucket-check that too, fail-bail if 403.

### Closure (e): Artifacts bucket derivation

- **Auto-derivation:** `km configure` derives `${prefix}-artifacts-${account_id}` using the same callerAccount lookup that powers the HeadBucket check.
  - Pattern matches the working `tf-${prefix}-state-${region}` derivation pattern (which is why state_bucket has worked all along — closure a is about cross-account name collision, not derivation).
- **Placeholder rejection:** Refuse to load `<prefix>-artifacts-12345678` (the literal example-yaml string) or any `<...>` angle-bracket-bracketed value. Hard error at load time:
  ```
  ✗ artifacts_bucket="<prefix>-artifacts-12345678" is the km-config.example.yaml placeholder.
  Re-run `km configure` to derive ${prefix}-artifacts-${account_id} automatically.
  ```
  - Why: hard rejection at load time means operators can't accidentally ship an install with the placeholder; the error tells them the fix.

### Closure (d): `km init --plan` skip detection mechanism

- **File-existence probe BEFORE plan invocation:** Before running `terragrunt plan -out=...` for a module, scan the module's `terragrunt.hcl` for references to `${get_terragrunt_dir()}/../<name>/outputs.json` (or `read_terragrunt_config(...)` pointing at the same). Stat each referenced file. If any are missing, emit `[skip: depends on <upstream>/outputs.json — apply <upstream> first]` and exit 0 for that module.
  - Why: cheap (no terragrunt invocation needed for the skip path); matches Phase 84.2's `[skip: KM_FOO not set]` pattern; doesn't couple to specific terragrunt error strings (which option B does); doesn't pull in an HCL parser dependency (which option C does).
- **Behavior in partial-coverage summary:** Plan summary lists skipped modules separately from planned modules: `network: 5 to add, 0 to change, 0 to destroy ✓ | efs: [skip: depends on network/outputs.json] | ...`. Final aggregate counts only the planned modules; skipped count reported separately.
- **Gate interaction:** Skipped modules do NOT count toward the destroy-class gate (same as Phase 84.2's `[skip: KM_FOO not set]`).
- **Exit code:** 0 when only skips, no destroys, gate not tripped — this is the "fresh install" success case.

### Closure (b): Dry-run banner text fix

- **Change:** `internal/app/cmd/bootstrap.go:407` (`fmt.Fprintf(w, "Dry run — would run: terragrunt plan %s\n", sesDir)`) and any sibling dry-run banner lines that incorrectly claim "would run: terragrunt plan" → change to "would run: terragrunt apply" (which is what the non-dry-run path actually invokes).
  - Why: it's a string fix to match reality. The Phase 84.2 `--plan` flag is the real plan path; `--dry-run=true` is the static documentation print and shouldn't claim to run plan.
- **Search target:** `grep -n "would run: terragrunt plan" internal/app/cmd/` to catch all instances.

<deferred>
## Deferred Ideas

- **`km bootstrap --all` chaining `km init`** — explicitly rejected; bootstrap and init are distinct lifecycle stages.
- **Apply-only `--all` (no `--plan` support)** — rejected; breaks Phase 84.2's plan-before-apply discipline.
- **Removing env-var fallback for ALL `KM_*` vars** — rejected; legitimate operator workflows depend on env-var overrides for KM_REGION, KM_DOMAIN, etc. Only `accounts.*` keys lose env-var fallback.
- **WARN summary block at top of every command** — rejected as too verbose; single line per drifted var is sufficient.
- **Suppress WARN on read-only commands** — rejected; complexity not worth the saved noise.
- **`km configure` blocking on shell-env conflicts** — rejected; operator may have intentional override.
- **Auto-replace state_bucket silently on 403** — rejected; operator may have wanted to bail rather than auto-rename.
- **Unlimited HeadBucket retry loop** — rejected; account-ID-suffixed collisions are vanishingly rare.
- **HCL AST parsing for plan-skip dependency graph** — rejected; file-existence probe is sufficient and avoids new dependency.
- **`NEXT-STEPS.md` file from `km configure`** — rejected; stdout + yaml header comments cover the need without leaving a stray file.
- **By-closure plan splitting (8 plans + UAT)** — rejected; by-file (4 waves) is the right granularity for cohesive surface changes.
- **Module-level `km-` hard-coded prefix fixes** — Phase 84.4 owns (SCP, EFS, S3-replication content fixes).
- **Foundation DKIM/MX/TXT auto-import** — Phase 84.4 owns (needs module-hardening decisions locked first).
- **Multi-install end-to-end runtime UAT** — Phase 84.4 owns (third-install UAT; 84.3 + 84.4 ship together for it).
