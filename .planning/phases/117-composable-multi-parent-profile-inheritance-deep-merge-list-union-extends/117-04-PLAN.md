<objective>
Author the `profiles/base/` fragment library from the overlap quantified in
RESEARCH, refactor `learn.v2.yaml` + `learn.v2.{chatty,polite,codex}` + `dc34.yaml`
onto those fragments, and PROVE compiled-userdata equivalence via byte-diff —
with the frozen `learn.v2.yaml` byte-identity golden as the hard gate.

This is the payoff plan and the riskiest. The byte-identity golden
(`TestUserdataLearnV2Phase92ByteIdentity`) currently calls `profile.Parse()` on
`learn.v2.yaml` directly. Once `learn.v2.yaml` gains `extends:`, Parse() reads
only the leaf's partial spec — the test would compile the WRONG (un-merged)
userdata. The helper MUST switch to `profile.Resolve()` so it compiles the merged
spec, and the resulting userdata must be byte-identical to the frozen baseline.

NOTE on the capture trap (memory: project_frozen_byte_identity_golden_capture_trap):
do NOT run `CAPTURE_PRE92_BASELINE=1` — that re-captures and folds the
post-baseline SubagentStop script into the frozen baseline, corrupting it. The
goal is for the resolved-then-compiled userdata to MATCH the existing frozen
golden, proving the refactor is byte-equivalent. If a genuine, reviewed diff
exists, it must be explicitly justified in the SUMMARY, not silently re-captured.

Purpose: collapse the ~80-line-per-profile duplication into shared fragments with
zero compiled-output change. Output: base library, refactored leaves, green gate.
</objective>
