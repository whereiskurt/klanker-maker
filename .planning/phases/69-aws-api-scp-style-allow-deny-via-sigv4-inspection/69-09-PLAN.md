Cover: SCP-style declarative control, why IAM/SCP alone is insufficient for per-sandbox pinning, what the gate inspects (SigV4 traffic on `*.amazonaws.com` and partition variants), what is out of scope (operation-level, region pinning, IMDS, VPC endpoints).
3. **Profile schema** — show the YAML block:
   ```yaml
   spec:
     sourceAccess:
       aws:
         inspection: off | observe | enforce
         allowlist:
           - sts
           - s3
           - dynamodb
           - bedrock-runtime
   ```
   Document each field, the enum values, the `["*"]` wildcard semantics, and the validation rules (Bedrock cross-check, wildcard-mixing, allowlist-required-when-not-off).
4. **The three modes** — table with columns: mode | allowlist | behavior — copied from 69-CONTEXT.md §Three modes. Plus prose elaborating "inventory before enforce" workflow.
5. **Platform-uid carve-out** — explain that calls from `uid < KM_AWS_PLATFORM_UID_MAX` (default 1000) bypass the gate and emit `aws_api_platform`. List the platform users (km-mail-poller, km-slack-inbound-poller, OTEL exporter, metadata sync) and reference the doctor check `aws_inspection_uid_map`.
6. **Audit events** — show the four JSON shapes from 69-CONTEXT.md verbatim. Document where they land (proxy stdout → audit-log sidecar → CloudWatch Logs/S3/stdout per `spec.observability.logDestination`).
7. **Learn mode workflow** — `km shell --learn` example, the generated profile shape (inspection: observe + alphabetized allowlist), the operator review-and-flip workflow.
8. **`km doctor` checks** — describe both new checks (`aws_inspection_uid_map`, `aws_allowlist_known_services`); note WARN-not-FAIL on unknown slugs and the `pkg/aws/sigv4_services.go` extensibility process.
9. **Demo storyboard** — copy the four-flow walkthrough from 69-CONTEXT.md §Demo storyboard. Include the exact AWS CLI commands.
10. **Troubleshooting matrix** — table: symptom | likely cause | resolution. Cover at minimum:
    - "AWS CLI fails with 403 KM_AWS_BLOCKED in body" → service not in allowlist; add it or switch to observe
    - "AWS CLI fails with TLS error" → cert pinning issue; check proxy MITM cert is trusted
    - "Audit log shows no aws_api_* events" → confirm KM_AWS_INSPECTION env var on http-proxy unit
    - "km email read still works after enforce + []" → expected; platform uid bypass
    - "aws_api_unsigned events spam the log" → anonymous S3 reads from a public bucket; document but no fix
11. **Security model** — what the gate prevents (per-sandbox AWS API access without IAM rewrites), what it doesn't prevent (IAM-bypass via legitimate signed calls to allowlisted services, IMDS access — out of scope).
12. **Operator workflow notes** — mirror the Slack docs:
    - `km destroy` + `km create` required to apply schema changes (no hot-reload)
    - `make build && km init --sidecars` after Phase 69 ships so management Lambdas pick up the schema additions
    - Rotation/changes: edit profile, destroy + recreate the affected sandboxes
13. **Deferred items** (forward compat) — operation-level entries, region pinning, IMDS gating, VPC endpoints, hot-reload. Reference `.planning/phases/69-...` for fix paths.
