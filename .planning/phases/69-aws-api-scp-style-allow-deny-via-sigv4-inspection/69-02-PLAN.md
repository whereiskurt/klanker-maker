<objective>
Add a new pinned BPF hash map `sock_to_uid` populated by the existing `cgroup/connect4` program. The map captures `bpf_get_current_uid_gid() & 0xFFFFFFFF` (the calling process's UID) keyed by the socket cookie, parallel to the existing `sock_to_original_ip` / `sock_to_original_port` maps.

This plan is the eBPF half of the platform-uid carve-out: the proxy needs to know "what UID owns this connection" so it can skip the AWS allowlist gate for system users (uid < 1000) and emit `aws_api_platform` events. Plan 69-06 wires the Go-side `GetCallerUID` lookup in transparent.go that consumes this map.

Per `69-RESEARCH.md` Topic 4 + Topic 11: this change is mechanically additive (one helper call + one map_update_elem) and the eBPF verifier risk is LOW. The kernel already calls `bpf_get_current_pid_tgid()` and `bpf_get_socket_cookie(ctx)` in this same program, so process context is available and `bpf_get_current_uid_gid()` is a peer helper that requires no extra capabilities.

Output:
- New `sock_to_uid` map in `pkg/ebpf/bpf.c` with `BPF_MAP_TYPE_HASH`, `__u64` key (socket cookie), `__u32` value (uid), `max_entries=262144` (matches existing maps)
- Three lines added to the `cgroup/connect4` program right after the existing `socket_pid_map` update (line ~198): get UID, mask, map_update_elem
- bpf2go regenerated bindings; `loader.go` pins/unpins the new map
- This plan is intentionally narrow so verifier risk is contained early in the wave structure (CONTEXT.md flags this as "exercise early with a small spike")
</objective>
