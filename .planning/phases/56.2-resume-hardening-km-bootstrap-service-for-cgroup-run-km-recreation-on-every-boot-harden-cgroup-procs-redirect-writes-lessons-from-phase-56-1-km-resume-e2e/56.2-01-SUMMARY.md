## Root Cause Documentation

### Bug A: Cgroup scope vanished after resume (userdata.go ~line 1441)

`/sys/fs/cgroup` is a kernel-managed pseudo-filesystem. Every entry under it is created at runtime — there is no on-disk persistence. When an EC2 instance is stopped and restarted, the kernel presents an empty cgroup hierarchy. The `mkdir -p /sys/fs/cgroup/km.slice/km-${SANDBOX_ID}.scope` at userdata.go:1441 ran during cloud-init on first boot only. After `km stop` + `km resume`, cloud-init does not re-run, so the scope directory is absent. The two `cgroup.procs` writes at the km-sandbox-shell and km-cgroup.sh sites both silently fail (the sandbox process is NOT placed in the eBPF cgroup, bypassing enforcement).

**Fix:** km-bootstrap.service runs on every boot via `systemctl enable`, recreating the cgroup scope before SSM accepts connections.

### Bug B: /run/km tmpfs wiped on resume (userdata.go ~line 419)

`/run` is a tmpfs. It is recreated empty at every boot (including resume from stop). The `mkdir -p /run/km` at userdata.go:419 and the `touch /run/km/learn-commands.log` at userdata.go:426 ran once during cloud-init. After resume, `/run/km/` does not exist. In learnMode, the `_km_learn` PROMPT_COMMAND append fails; the `/run/km/audit-pipe` FIFO is also absent (though Phase 56.1's FIFO-retry handles the FIFO re-creation for sidecars).

**Fix:** km-bootstrap.service recreates `/run/km/` and (when LearnMode=true) `touch`es `learn-commands.log` before SSM accepts connections.

### Bash redirect-open semantics (userdata.go ~lines 1455, 1487)

In bash, redirects are processed BEFORE the command runs. When a redirect target doesn't exist, bash performs `open(path, O_WRONLY|O_CREAT|O_TRUNC)` and gets ENOENT. This error is reported to the current stderr BEFORE `2>/dev/null` takes effect (because the stderr redirect is not yet active when the failed redirect-open error is generated). Result: `echo $$ > path 2>/dev/null` leaks the ENOENT to the user's terminal even though `2>/dev/null` is present.

**Fix:** Compound-command form `{ echo $$ > path; } 2>/dev/null || true` applies the stderr redirect to the ENTIRE compound command — including the inner redirect's `open()` call — so ENOENT is captured and discarded.

## km-bootstrap.service Unit Body

```ini
[Unit]
Description=Klankrmkr sandbox bootstrap (per-boot cgroup and /run/km setup)
Before=amazon-ssm-agent.service
[Service]
Type=oneshot
RemainAfterExit=yes
ExecStart=/usr/local/bin/km-bootstrap
[Install]
WantedBy=multi-user.target
```

**Why no `Wants=km.slice`:** `km.slice` is a cgroup directory, not a systemd unit file. `Wants=km.slice` would emit "unit not found" warnings on every boot. The km-bootstrap script creates the `km.slice` parent directory via `mkdir -p` as a side effect of creating the scope directory.

**Why no `DefaultDependencies=no`:** Keeping the implicit `After=sysinit.target` ensures the cgroup filesystem (`/sys/fs/cgroup`) is mounted before `mkdir -p` runs. Removing default dependencies creates ordering gap risk.

## km-bootstrap Script Body (Three Responsibilities)

1. **Source identity:** `. /etc/profile.d/km-identity.sh 2>/dev/null || true` — loads current sandbox's `KM_SANDBOX_ID`. Critical on baked-AMI relaunch where the AMI baked in sandbox A's identity; sourcing km-identity.sh picks up sandbox B's identity (Phase 56.1 Bug 4 lesson applied here).

2. **Recreate cgroup scope:** `mkdir -p /sys/fs/cgroup/km.slice/km-${KM_SANDBOX_ID}.scope` + `chown root:sandbox cgroup.procs 2>/dev/null || true` + `chmod 664 cgroup.procs 2>/dev/null || true`. Gated on `[ -n "${KM_SANDBOX_ID:-}" ]` to handle edge cases where identity file is absent.

3. **Recreate /run/km:** `mkdir -p /run/km` + `chown km-sidecar:km-sidecar /run/km 2>/dev/null || true`. When `LearnMode=true`, also `touch /run/km/learn-commands.log` + `chmod 666` — same `{{- if .LearnMode }}` template conditional used in the cloud-init block at userdata.go:424. The audit-pipe FIFO is intentionally NOT created here (owned by km-audit-log's FIFO-retry loop from Phase 56.1 P56.1-04).

## systemctl enable km-bootstrap Placement

```bash
systemctl daemon-reload
# km-bootstrap.service: persist enable so it runs on every boot (incl. resume).
systemctl enable km-bootstrap 2>/dev/null || true
```

The `systemctl enable` must appear AFTER `daemon-reload` so systemd has loaded the unit file before creating the enable symlink. The `2>/dev/null || true` is defense-in-depth for the edge case where daemon-reload has not yet processed the file.

## Open Question Outcomes

1. **Cgroup block gating:** Chose NOT to gate the cgroup creation block in km-bootstrap on enforcement mode. The script uses a runtime `[ -n "${KM_SANDBOX_ID:-}" ]` guard which is always true (KM_SANDBOX_ID is set by km-identity.sh for all profiles). This keeps the script simpler and avoids template conditionals inside the heredoc.

2. **Always emit km-bootstrap.service:** Yes — emitted for all enforcement modes. The `/run/km` parent dir is needed by km-audit-log on all enforcement modes (km-audit-log always uses the FIFO regardless of enforcement).

3. **km-sandbox-shell guard:** Dropped the `[ -w "$CGROUP_PROCS" ]` guard. The compound-command form handles the missing-file case gracefully without needing a pre-check, and after km-bootstrap runs the file is guaranteed to exist.

4. **chown on every boot:** Yes — `chown km-sidecar:km-sidecar /run/km 2>/dev/null || true` included to mirror the cloud-init ownership established at line 420.
