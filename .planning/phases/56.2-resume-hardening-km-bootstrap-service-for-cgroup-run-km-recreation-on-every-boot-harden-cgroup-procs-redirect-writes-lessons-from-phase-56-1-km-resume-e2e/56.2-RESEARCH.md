# Phase 56.2: Resume Hardening — Research

**Researched:** 2026-04-27
**Domain:** systemd unit authoring in cloud-init, cgroup-v2 pseudo-fs lifecycle, tmpfs semantics on EC2 stop+resume, bash redirect semantics
**Confidence:** HIGH (all findings from direct code inspection; no external library dependencies)

## Summary

Phase 56.2 fixes two root causes observed after `km stop` + `km resume` of a baked-AMI sandbox:

**Root cause 1 (cgroup scope gone on second boot):** `/sys/fs/cgroup` is a kernel-managed pseudo-filesystem. Every entry under it is created at runtime by the kernel or systemd — there is no on-disk persistence. When an EC2 instance is stopped and restarted, the kernel presents an empty cgroup hierarchy. The `mkdir -p /sys/fs/cgroup/km.slice/km-${SANDBOX_ID}.scope` at `userdata.go:1369` ran during cloud-init on first boot only. After `km stop` + `km resume`, cloud-init does not re-run, so the scope directory is absent. The two cgroup.procs writes at `userdata.go:1455` (inside `km-sandbox-shell`) and `userdata.go:1487` (inside `/etc/profile.d/km-cgroup.sh`) both attempt bash redirects to the missing path. The `2>/dev/null` guard does NOT suppress the error: bash opens the redirect target (performing the `open()` syscall) BEFORE applying the stderr redirect — if the open fails with ENOENT, bash prints the error to stderr (which has not yet been redirected) and the user sees `bash: /sys/fs/cgroup/.../cgroup.procs: No such file or directory`. The shell process is not placed in the eBPF cgroup, so eBPF enforcement is silently bypassed.

**Root cause 2 (/run/km wiped on resume):** `/run` is a tmpfs. It is recreated empty at every boot (including resume from stop). The `touch /run/km/learn-commands.log` at `userdata.go:426` and `mkdir -p /run/km` at `userdata.go:419` ran once during cloud-init. After resume, `/run/km/` does not exist. The learn-mode PROMPT_COMMAND append (`>> /run/km/learn-commands.log`) at `userdata.go:498` fails silently (the `2>/dev/null` on that line does work, because `>>` errors are stderr and the redirect is already in place). The `/run/km/audit-pipe` FIFO is also absent after resume — however, the km-audit-log sidecar already has a FIFO-retry loop (Phase 56.1 P56.1-04) that re-creates `/run/km/` and the FIFO. The main problem is the learn-mode log file and the ordering: the bootstrap service must run before SSM accepts sessions, so that `/run/km/` is ready before any shell hook runs.

**Solution:** Add a `km-bootstrap.service` systemd unit (Type=oneshot, RemainAfterExit=yes, Before=amazon-ssm-agent.service) that is written to `/etc/systemd/system/` during cloud-init and enabled there, so it runs on every subsequent boot (including resume). The unit's `ExecStart=/usr/local/bin/km-bootstrap` script sources `/etc/profile.d/km-identity.sh` to get the current `SANDBOX_ID` (correct on baked-AMI relaunch — same lesson as Phase 56.1 Bug 4), re-creates the cgroup scope and the `/run/km` tree idempotently on every boot. Separately, harden the two cgroup.procs redirect writes with `{ echo $$ > path; } 2>/dev/null || true` so redirect-open errors are actually suppressed in bash.

**Primary recommendation:** Two targeted edits to `pkg/compiler/userdata.go`: (1) emit `km-bootstrap.service` + `/usr/local/bin/km-bootstrap` script + `systemctl enable km-bootstrap` alongside the existing sidecar unit emission in section 5; (2) change the two `echo $$ > …/cgroup.procs` patterns to use the compound-command redirect-suppress form.

## Architecture Patterns

### Where systemd units are emitted in userdata.go

The existing sidecar unit sequence is:
1. Section 5 (line 380): `cat > /etc/systemd/system/km-dns-proxy.service` heredoc
2. Section 5 (line 398): `cat > /etc/systemd/system/km-http-proxy.service` heredoc
3. Section 5 (line 419-427): `mkdir -p /run/km`, `mkfifo /run/km/audit-pipe`, learnMode touch
4. Section 5 (line 430): `cat > /etc/systemd/system/km-audit-log.service` heredoc
5. Section 5 (line 454): `cat > /etc/systemd/system/km-tracing.service` heredoc
6. Section 5 (line 474): `cat > /etc/profile.d/km-audit.sh` (PROMPT_COMMAND hook)
7. Section 5 (line 528): `systemctl daemon-reload`
8. Later: `systemctl enable` + `systemctl start` sidecars

`km-bootstrap.service` should be emitted as a new unit in section 5, between the sidecar unit writes and the `systemctl daemon-reload` at line 528. The `/usr/local/bin/km-bootstrap` script must be written BEFORE the service unit (so it exists when systemd tries to validate ExecStart on first boot). The `systemctl enable km-bootstrap` must come AFTER `daemon-reload`.

### Recommended placement in the template

```
... (existing km-audit.sh hook at line 474)

# Write km-bootstrap script — runs on every boot to re-create
# ephemeral runtime state that cloud-init created only once.
cat > /usr/local/bin/km-bootstrap << 'BOOTSTRAPEOF'
#!/bin/bash
# km-bootstrap: recreates cgroup scope and /run/km on every boot.
# Sourced by km-bootstrap.service (Before=amazon-ssm-agent.service).
set -euo pipefail

# Source sandbox identity — correct SANDBOX_ID on baked-AMI relaunch.
# Phase 56.1 lesson: SANDBOX_ID is baked into /etc/profile.d/km-identity.sh,
# but systemd units on baked AMIs carry the old sandbox's env; sourcing here
# ensures we always use the current sandbox ID written by cloud-init.
# shellcheck source=/etc/profile.d/km-identity.sh
. /etc/profile.d/km-identity.sh 2>/dev/null || true

# (1) Cgroup scope: /sys/fs/cgroup is a kernel pseudo-fs — empty after every
#     stop+start. Re-create the sandbox's cgroup scope directory idempotently.
if [ -n "${KM_SANDBOX_ID:-}" ]; then
  CGROUP_DIR="/sys/fs/cgroup/km.slice/km-${KM_SANDBOX_ID}.scope"
  mkdir -p "${CGROUP_DIR}"
  # Allow sandbox user to write its PID into cgroup.procs.
  chown root:sandbox "${CGROUP_DIR}/cgroup.procs" 2>/dev/null || true
  chmod 664 "${CGROUP_DIR}/cgroup.procs" 2>/dev/null || true
  echo "[km-bootstrap] Cgroup scope ready: ${CGROUP_DIR}"
fi

# (2) /run/km: tmpfs — wiped at every boot. Re-create the directory and
#     sentinel files that shell hooks and sidecars expect to find.
mkdir -p /run/km
chown km-sidecar:km-sidecar /run/km 2>/dev/null || true

# audit-pipe FIFO: km-audit-log sidecar has FIFO-retry (Phase 56.1 P56.1-04)
# but the bootstrap ensures the dir exists before the sidecar starts.
# Note: do NOT create the FIFO here — the sidecar owns its creation to avoid
# ownership/permission races. Bootstrap only ensures the parent dir exists.

# learn-commands.log: written by _km_learn PROMPT_COMMAND when learnMode=true.
# File must exist before any shell session starts; sidecar does not own this file.
{{ if .LearnMode -}}
touch /run/km/learn-commands.log
chmod 666 /run/km/learn-commands.log
echo "[km-bootstrap] /run/km/learn-commands.log ready (learnMode)"
{{- end }}

echo "[km-bootstrap] /run/km ready"
BOOTSTRAPEOF
chmod +x /usr/local/bin/km-bootstrap

# km-bootstrap.service: run km-bootstrap on every boot before SSM accepts sessions.
# Type=oneshot + RemainAfterExit ensures the service remains "active" after the
# script exits so dependents can use After=km-bootstrap.service.
# Before=amazon-ssm-agent.service ensures cgroup scope + /run/km exist before
# any SSM session entry (km shell, km agent) runs the first command.
cat > /etc/systemd/system/km-bootstrap.service << 'UNIT'
[Unit]
Description=Klankrmkr sandbox bootstrap (per-boot cgroup and /run/km setup)
DefaultDependencies=no
Before=amazon-ssm-agent.service
[Service]
Type=oneshot
RemainAfterExit=yes
ExecStart=/usr/local/bin/km-bootstrap
[Install]
WantedBy=multi-user.target
UNIT

systemctl daemon-reload     # existing line — picks up km-bootstrap.service too
systemctl enable km-bootstrap   # persist across reboots (including resume)
```

Note: `DefaultDependencies=no` is important for a `Before=amazon-ssm-agent` unit to avoid implicit `After=basic.target` which could introduce ordering loops. This is the standard systemd oneshot setup pattern for pre-networking boot tasks that don't need the full target graph.

### Cgroup scope creation mechanics (cgroup-v2 unified hierarchy)

Direct inspection of `pkg/ebpf/cgroup.go` confirms: the project creates cgroup scopes via `os.MkdirAll` (`CreateSandboxCgroup` at cgroup.go:48-55). This is the correct cgroup-v2 approach — creating a directory under the cgroup2 mount creates a new cgroup group. No systemd `.scope` unit drop-in is needed; the directory IS the scope. The km.slice parent directory is likewise created with `mkdir -p` which auto-creates the `km.slice` parent directory if absent.

The `Wants=km.slice` directive mentioned in the ROADMAP.md goal is NOT appropriate and should be omitted. `km.slice` is not a registered systemd unit file; it is simply a directory under `/sys/fs/cgroup/`. `Wants=` must name a valid unit. Using `Wants=km.slice` would cause systemd to log a warning about a missing unit. The correct approach is to let `km-bootstrap` script create both `km.slice` and `km-${SANDBOX_ID}.scope` directories via `mkdir -p` — no unit dependency needed.

### Bash redirect error suppression — the critical fix

**Current code at userdata.go:1487 (`/etc/profile.d/km-cgroup.sh`):**
```bash
echo $$ > /sys/fs/cgroup/km.slice/km-{{ .SandboxID }}.scope/cgroup.procs 2>/dev/null || true
```

**Why this fails:** In bash, a redirect is processed BEFORE the command runs. When the redirect target does not exist, bash opens the file (performing `open(path, O_WRONLY|O_CREAT|O_TRUNC)`) and gets ENOENT. This error is reported to the current stderr BEFORE the `2>/dev/null` redirect takes effect (because the redirect processing order is: first open all redirects, then execute the command — the `2>/dev/null` is not in effect when the redirect-open error is generated for the first redirect). Result: the error message appears even with `2>/dev/null || true`.

**Fix — compound command wrapper:**
```bash
{ echo $$ > /sys/fs/cgroup/km.slice/km-{{ .SandboxID }}.scope/cgroup.procs; } 2>/dev/null || true
```

In this form, bash processes the `{ ... }` as a compound command. The `2>/dev/null` applies to the compound command's stderr stream. When bash opens `cgroup.procs` as a redirect for the inner `echo $$ >` and that open fails, the error message goes to the compound command's stderr — which IS `/dev/null`. The error is suppressed.

This is distinct from the `( ... ) 2>/dev/null` subshell form: the compound command `{ ... }` runs in the current shell (no subshell overhead; `$$` refers to the current PID, not a child PID).

**Same fix needed at userdata.go:1455 (`/usr/local/bin/km-sandbox-shell`):**
```bash
# Current:
if [ -w "$CGROUP_PROCS" ]; then
  echo $$ > "$CGROUP_PROCS" 2>/dev/null
fi

# Fix: the -w test already guards against missing file (returns false if path doesn't exist),
# but the path doesn't exist on resume, so -w returns false and the write is skipped entirely.
# However: after km-bootstrap.service runs, cgroup.procs DOES exist and IS writable by sandbox.
# The existing guard is fine once km-bootstrap.service is in place.
# The km-sandbox-shell path still benefits from adding the compound-command form as defense-in-depth.
```

Actually — on closer inspection, `km-sandbox-shell` (userdata.go:1454-1456) tests `[ -w "$CGROUP_PROCS" ]` first. If the file doesn't exist, `-w` returns false and the `echo $$ >` never runs, so no error is displayed. The km-sandbox-shell redirect path is already safe from the error leak in the missing-file case. The actual error seen in the live shell output is from `km-cgroup.sh` (userdata.go:1487) which does NOT have the `-w` guard:

```bash
# km-cgroup.sh (line 1487): no -w guard — unconditionally opens the redirect
echo $$ > /sys/fs/cgroup/km.slice/km-{{ .SandboxID }}.scope/cgroup.procs 2>/dev/null || true
```

This is the one that emits the error. The fix is compound-command form on this line. The km-sandbox-shell path at 1455 should also be fixed (it will correctly write after bootstrap, but the compound-command pattern is cleaner and provides defense-in-depth if bootstrap races).

### Ordering: does km-bootstrap.service complete before SSM accepts the first session?

`Before=amazon-ssm-agent.service` in the service unit establishes a systemd ordering dependency: systemd will not start `amazon-ssm-agent.service` until `km-bootstrap.service` has completed. This means cgroup scope and `/run/km` are guaranteed to exist before the first SSM session entry runs. This is HIGH confidence because:

1. `systemctl enable km-bootstrap` places a symlink in `/etc/systemd/system/multi-user.target.wants/km-bootstrap.service`.
2. On next boot (including resume from stop), systemd processes `multi-user.target` and honors the `Before=amazon-ssm-agent.service` ordering constraint.
3. `Type=oneshot` means systemd waits for `ExecStart` to complete before marking the unit active and allowing dependents.

However: SSM agent can have a `Wants=` pull from `multi-user.target` rather than `After=km-bootstrap.service` directly. The `Before=` directive ensures ordering from the bootstrap side — this is the correct and sufficient mechanism. Confirmed by systemd documentation: `Before=X` in unit A means A completes before X starts, regardless of whether X knows about A.

### /run/km tenant inventory (all paths that get wiped on resume)

From `grep -n "/run/km" userdata.go`:

| Path | Created by | Used by | Needed on resume? |
|------|-----------|---------|-------------------|
| `/run/km/` (dir) | cloud-init line 419 | all sidecars | YES — parent for all below |
| `/run/km/audit-pipe` (FIFO) | cloud-init line 421 | km-audit-log (read), km-audit.sh hook (write) | YES — but sidecar FIFO-retry (P56.1-04) re-creates it; bootstrap only needs to ensure the parent dir |
| `/run/km/learn-commands.log` | cloud-init line 426 (learnMode only) | `_km_learn()` PROMPT_COMMAND | YES — bootstrap must recreate when learnMode=true |
| `/run/km/http-proxy.pid` | km-http-proxy `ExecStartPost` (line 410) | km-ebpf-enforcer `ExecStartPre` (line 1387) | NO — sidecar creates it on start; not a bootstrap concern |
| `/run/km/enforcer.env` | km-ebpf-enforcer `ExecStartPre` (line 1387) | km-ebpf-enforcer `EnvironmentFile` | NO — sidecar creates it on start |
| `/run/km/budget_remaining` | km-http-proxy at runtime | budget enforcement | NO — sidecar creates it; not a bootstrap concern |

**Conclusion:** bootstrap must create `/run/km/` (parent dir) and conditionally `touch /run/km/learn-commands.log`. It must NOT create the FIFO (sidecar owns that). It must NOT create the PID file or env file (sidecars own those).

### LearnMode conditionality

The `learn-commands.log` touch is gated by `{{- if .LearnMode }}` in the template. This is already in the userDataParams struct (`LearnMode bool`, line 1836). The km-bootstrap.service is always emitted (not gated on enforcement mode) because:

1. The `/run/km` parent dir is needed by km-audit-log (FIFO parent) regardless of enforcement mode.
2. For non-eBPF profiles (enforcement=proxy), the cgroup scope creation is a no-op (mkdir -p on a non-existent parent succeeds but is unused). The script guards with `if [ -n "${KM_SANDBOX_ID:-}" ]` — effectively always true, but harmless.

Alternatively, the cgroup creation block in km-bootstrap can be gated on `{{- if or (eq .Enforcement "ebpf") (eq .Enforcement "both") }}` to avoid cluttering non-eBPF sandboxes with an unused cgroup scope. This is a planner decision.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Per-boot idempotent setup | Custom restart logic in cloud-init | `Type=oneshot RemainAfterExit=yes` systemd unit | Standard pattern; systemd manages boot ordering |
| Suppressing bash redirect errors | Wrapping every cgroup write site | Single `{ echo $$ > path; } 2>/dev/null` compound form | One pattern change per site; no custom error handler |
| FIFO re-creation on resume | Bootstrap creating the FIFO | Rely on Phase 56.1 km-audit-log FIFO-retry | Already implemented in P56.1-04; duplication would cause ownership races |
| km.slice systemd unit | Writing a .slice unit file | Let mkdir -p create the cgroup directory | cgroup-v2 doesn't need a registered .slice unit; mkdir is sufficient |

## Common Pitfalls

### Pitfall 1: `Wants=km.slice` causes "unit not found" warning

**What goes wrong:** `km.slice` is a cgroup directory, not a systemd unit file. Specifying `Wants=km.slice` causes systemd to emit a warning on every boot: `km-bootstrap.service: Found ordering cycle...` or `Failed to load unit km.slice: No such file or directory`. The service still runs, but the log is noisy and the dependency provides no value.

**How to avoid:** Omit `Wants=km.slice`. The km-bootstrap script creates the `km.slice` directory via `mkdir -p` as a side effect of creating the scope directory. No systemd unit dependency is needed.

**Warning signs:** `journalctl -u km-bootstrap` shows "unit not found" for km.slice.

### Pitfall 2: km-bootstrap sourcing km-identity.sh with `set -euo pipefail`

**What goes wrong:** `/etc/profile.d/km-identity.sh` may not exist on the very first boot before cloud-init's section 2.8 has run. If `set -euo pipefail` is active and the source fails, `km-bootstrap` exits with error. On the FIRST boot this would fire before the profile.d file is written — but on first boot, cloud-init runs and writes both the unit AND km-identity.sh in sequence, and `systemctl enable` won't take effect until next boot. So in practice, by the time km-bootstrap.service first runs (on resume = second boot), km-identity.sh exists.

**How to avoid:** Use `. /etc/profile.d/km-identity.sh 2>/dev/null || true` (suppress errors; do not `set -e` the source). Alternatively use `[[ -f /etc/profile.d/km-identity.sh ]] && . /etc/profile.d/km-identity.sh`.

**Warning signs:** `systemctl status km-bootstrap` shows `ExecStart failed` on the first resume.

### Pitfall 3: chown root:sandbox on cgroup.procs before sandbox user exists

**What goes wrong:** If `/etc/profile.d/km-identity.sh` exists but the `sandbox` group was never created (e.g., on a brand-new AMI resume where the useradd in cloud-init didn't persist to the AMI), the `chown root:sandbox` fails. However: the `sandbox` user is created by `useradd -m -s /bin/bash -d /home/sandbox sandbox` during cloud-init (line 372-373). This DOES persist to disk (it writes to `/etc/passwd` and `/etc/shadow`) and will be present on resume.

**How to avoid:** The `chown root:sandbox ... 2>/dev/null || true` form already guards this. No change needed. Confirmed: `useradd` writes to `/etc` (disk-backed, not tmpfs).

### Pitfall 4: `DefaultDependencies=no` removes implicit After=sysinit.target

**What goes wrong:** `DefaultDependencies=no` on a unit removes implicit dependencies including `After=sysinit.target`. If the cgroup filesystem isn't mounted yet when km-bootstrap runs, `mkdir -p /sys/fs/cgroup/km.slice/...` will fail or operate on a non-cgroup2 filesystem.

**How to avoid:** Add `After=sysinit.target` explicitly, or rely on the fact that `Before=amazon-ssm-agent.service` already creates a late-enough ordering (amazon-ssm-agent has `After=network.target` which pulls in sysinit). Alternatively, drop `DefaultDependencies=no` — the implicit ordering loop concern is theoretical on EC2 where the unit graph is fixed; keeping default dependencies is safer unless profiling reveals a real cycle.

**Recommendation:** Omit `DefaultDependencies=no` to keep the service in the standard `multi-user.target` dependency chain without risk of ordering gaps. The simpler unit header:

```ini
[Unit]
Description=Klankrmkr sandbox bootstrap (per-boot cgroup and /run/km setup)
Before=amazon-ssm-agent.service
```

### Pitfall 5: km-bootstrap and km-audit-log both creating /run/km in a race

**What goes wrong:** `km-bootstrap.service` has `Before=amazon-ssm-agent.service`. The km-audit-log sidecar is started by `systemctl start` in cloud-init (line 1293), and on resume it auto-starts because it has `Restart=always`. km-audit-log's FIFO-retry loop (Phase 56.1 P56.1-04) also calls `os.MkdirAll(filepath.Dir(pipePath), 0755)`. If km-bootstrap and km-audit-log both run simultaneously on resume, there is a `mkdir -p /run/km` race.

**Why this is safe:** `mkdir -p` is idempotent and `os.MkdirAll` (Go equivalent) is also idempotent. Both calls succeed if the directory exists. The race is benign — whichever runs first creates the directory; the second finds it already exists and succeeds. Confirmed by POSIX: `mkdir -p` returns 0 if directory already exists.

**Warning signs:** None — this is intentionally benign.

### Pitfall 6: Omitting the LearnMode conditional in km-bootstrap breaks non-learn profiles

**What goes wrong:** If the `touch /run/km/learn-commands.log` line runs on a non-learnMode profile, it creates a harmless file. But more importantly — if the `{{- if .LearnMode }}` conditional is also omitted from the cloud-init section 5 (lines 424-428), then the existing `_km_learn` function in km-audit.sh will write to a file that wasn't intentionally set up. The existing section 5 already has the conditional; the bootstrap script must mirror it.

**How to avoid:** Gate the `touch /run/km/learn-commands.log` in km-bootstrap behind `{{ if .LearnMode }}` — same template conditional already used at userdata.go:424.

### km-bootstrap script (key sections)

```bash
#!/bin/bash
# km-bootstrap: re-creates ephemeral state on every boot.
# Run by km-bootstrap.service (Type=oneshot, Before=amazon-ssm-agent.service).
set -euo pipefail

# Source current sandbox identity — must work after baked-AMI relaunch.
. /etc/profile.d/km-identity.sh 2>/dev/null || true

# Cgroup scope: only relevant for eBPF enforcement profiles
SANDBOX_ID="${KM_SANDBOX_ID:-}"
if [ -n "$SANDBOX_ID" ]; then
  CGROUP_DIR="/sys/fs/cgroup/km.slice/km-${SANDBOX_ID}.scope"
  mkdir -p "${CGROUP_DIR}"
  chown root:sandbox "${CGROUP_DIR}/cgroup.procs" 2>/dev/null || true
  chmod 664 "${CGROUP_DIR}/cgroup.procs" 2>/dev/null || true
  echo "[km-bootstrap] cgroup scope: ${CGROUP_DIR}"
fi

# /run/km parent dir: needed by km-audit-log (FIFO parent) and shell hooks
mkdir -p /run/km
chown km-sidecar:km-sidecar /run/km 2>/dev/null || true
echo "[km-bootstrap] /run/km ready"
# NOTE: do not create /run/km/audit-pipe here — km-audit-log owns FIFO creation (P56.1-04)
# NOTE: learn-commands.log is gated on LearnMode — handled in template conditional
```

### Compound-command cgroup.procs write (harden km-cgroup.sh)

```bash
# Source: fix for userdata.go:1487 (profile.d/km-cgroup.sh)
# OLD (broken — bash opens redirect BEFORE 2>/dev/null applies):
#   echo $$ > /sys/fs/cgroup/km.slice/km-{{ .SandboxID }}.scope/cgroup.procs 2>/dev/null || true
# NEW (fixed — compound command applies 2>/dev/null around the entire open+write):

```

### Compound-command cgroup.procs write (harden km-sandbox-shell)

```bash

# Current code already uses -w guard which prevents write when file missing.
```

## Open Questions

1. **Should the cgroup-creation block in km-bootstrap be gated on enforcement mode?**
   - What we know: For `enforcement=proxy` profiles, the cgroup scope is never used by eBPF enforcement. Creating it is harmless (mkdir-p is idempotent).
   - What's unclear: Whether the planner wants a clean separation ("only create cgroup for eBPF profiles") or a simple flat script ("always idempotent").
   - Recommendation: Gate on `{{ if or (eq .Enforcement "ebpf") (eq .Enforcement "both") }}` for cleanliness. The learn-mode FIFO parent and `/run/km/learn-commands.log` are always created regardless of enforcement (they are needed by proxy-mode profiles too).

2. **Should the km-bootstrap.service unit be gated on enforcement mode or always emitted?**
   - What we know: The `/run/km` directory creation is needed for all enforcement modes (km-audit-log always uses it). The cgroup section is only used for eBPF modes.
   - Recommendation: Always emit km-bootstrap.service. It is idempotent and adds <10ms to every boot. The script body handles the conditionality internally.

3. **Should km-sandbox-shell keep its `[ -w "$CGROUP_PROCS" ]` guard or switch to compound-command form?**
   - What we know: The `-w` guard prevents the write when the file doesn't exist (false if missing). After km-bootstrap.service runs, the file exists and is writable. The guard is no longer the primary protection.
   - Recommendation: Replace with compound-command form (simpler, consistent with km-cgroup.sh fix). No functional regression — if bootstrap didn't run (edge case), compound form silently suppresses the error.

4. **Does `/run/km` need `chown km-sidecar:km-sidecar` on every boot?**
   - What we know: Cloud-init line 420 does `chown km-sidecar:km-sidecar /run/km`. The km-audit-log FIFO-retry does `os.MkdirAll(dir, 0755)` which creates as root unless the process runs as km-sidecar. Since km-audit-log runs as km-sidecar (user in service unit), `os.MkdirAll` creates with km-sidecar ownership — but only if the directory doesn't already exist.
   - Recommendation: Include `chown km-sidecar:km-sidecar /run/km 2>/dev/null || true` in km-bootstrap to maintain the correct ownership established by cloud-init.
