<objective>
Eliminate two errors observed on shell entry after `km resume` of a stopped (or baked-then-resumed) sandbox:

1. **Cgroup scope vanished after resume (Bug A — userdata.go:1369, 1455, 1487):** `/sys/fs/cgroup` is a kernel-managed pseudo-filesystem. After `km stop` + `km resume`, it is empty. The `mkdir -p /sys/fs/cgroup/km.slice/km-${SANDBOX_ID}.scope` at userdata.go:1369 ran during cloud-init on first boot only. The two `echo $$ > .../cgroup.procs` writes at userdata.go:1455 (km-sandbox-shell) and userdata.go:1487 (km-cgroup.sh) emit `bash: cgroup.procs: No such file or directory` past `2>/dev/null` because bash opens the redirect target BEFORE applying the stderr redirect — net effect: the user sees the error and the shell process is NOT placed in the eBPF cgroup, so eBPF enforcement may be silently bypassed.

2. **/run/km tmpfs wiped on resume (Bug B — userdata.go:419-427):** `/run` is a tmpfs, recreated empty at every boot. The `touch /run/km/learn-commands.log` at userdata.go:426 ran once during cloud-init. After resume, the file is gone and the `_km_learn` PROMPT_COMMAND append at userdata.go:498 fails (silently in this case because the heredoc redirect IS already in place — but the file going missing breaks learn-mode capture entirely; the directory `/run/km/` is also needed by km-audit-log and the bootstrap must ensure it exists before the sidecar starts).

**Approach** (per RESEARCH.md):

- Add a `km-bootstrap.service` systemd unit (Type=oneshot, RemainAfterExit=yes, Before=amazon-ssm-agent.service) to the userdata template. The unit is `systemctl enable`d during cloud-init, so it runs on every subsequent boot — including resume from stop.
- Its ExecStart is `/usr/local/bin/km-bootstrap`, a small bash script also written to disk during cloud-init. The script sources `/etc/profile.d/km-identity.sh` (Phase 56.1 lesson — correct SANDBOX_ID on baked-AMI relaunch), then idempotently re-creates `/sys/fs/cgroup/km.slice/km-${KM_SANDBOX_ID}.scope` (with proper ownership/perms on `cgroup.procs`) and `/run/km/`. When `LearnMode=true`, the script also re-creates `/run/km/learn-commands.log`. Crucially, the script does NOT create `/run/km/audit-pipe` — the km-audit-log sidecar's Phase 56.1 P56.1-04 FIFO-retry loop owns that.
- Harden the two `echo $$ > .../cgroup.procs` writes with the compound-command form `{ echo $$ > path; } 2>/dev/null || true`. In bash, redirects are processed BEFORE the command runs; with the bare form `echo $$ > path 2>/dev/null`, the `open()` of `path` happens before stderr is redirected, so an ENOENT prints to the user's stderr. The compound `{ ...; } 2>/dev/null` form applies the stderr redirect to the entire compound command including the inner redirect's `open()`.
- Do NOT add `Wants=km.slice` to the unit (RESEARCH.md Pitfall 1) — `km.slice` is a cgroup directory, not a registered systemd unit. `Wants=km.slice` would emit a "unit not found" warning on every boot. The script's own `mkdir -p` creates the parent.

Purpose: Make `km resume` of a baked-AMI sandbox shell-clean. Phase 56.1 P56.1-05's `systemctl restart km-*` line handles sidecar identity refresh on first resume. This phase handles the cgroup scope and `/run/km` recreation that runs on EVERY boot — first boot, every resume, every reboot — via systemd's `[Install] WantedBy=multi-user.target` + `systemctl enable`.

Output:
- `pkg/compiler/userdata.go` — emit km-bootstrap script + km-bootstrap.service unit + `systemctl enable km-bootstrap`; harden two cgroup.procs write sites with compound-command form
- `pkg/compiler/userdata_test.go` — 4 new test functions covering all 7 phase requirements

Deployment: After this plan lands, the operator runs `make build` (CLI rebuild — userdata template is compiled into the km binary). NO `km init --sidecars` needed (no sidecar binary changed). The new km-bootstrap.service is emitted at `km create` time and baked into AMIs at `km bake` time. Existing AMIs do NOT get the fix retroactively — operator must re-bake or re-create the sandbox.

    # km-bootstrap: re-creates ephemeral state on every boot.
    # Run by km-bootstrap.service (Type=oneshot, Before=amazon-ssm-agent.service).
    set -uo pipefail

    # Source current sandbox identity — picks up baked-AMI's NEW SANDBOX_ID,

      mkdir -p "${CGROUP_DIR}"
      chown root:sandbox "${CGROUP_DIR}/cgroup.procs" 2>/dev/null || true
      chmod 664 "${CGROUP_DIR}/cgroup.procs" 2>/dev/null || true
      echo "[km-bootstrap] cgroup scope ready: ${CGROUP_DIR}"

    echo "[km-bootstrap] /run/km ready"

    #     File must exist before any shell session runs.
    touch /run/km/learn-commands.log
