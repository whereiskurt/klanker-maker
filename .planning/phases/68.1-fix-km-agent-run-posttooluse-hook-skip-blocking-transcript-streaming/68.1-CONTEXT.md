# Phase 68.1: Fix km agent run PostToolUse hook skip blocking transcript streaming - Context

**Gathered:** 2026-05-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Make Phase 68's per-turn Slack transcript streaming + final `.jsonl.gz` upload work for `km agent run` (operator-initiated, non-interactive `claude -p` mode). Today the streaming hook silently no-ops because Claude Code does not fire `PostToolUse` / `Stop` hooks in `-p` mode — only interactive `km shell` works.

**Reuse, don't rebuild.** Phase 68's hook script (`_km_stream_drain` in `pkg/compiler/userdata.go:526`) already does the real work: read transcript JSONL → render assistant text + tool one-liners → post via `km-slack` → write DDB row → update offset. Phase 68.1 invokes that same logic from a different driver (a sandbox-side wrapper poller) instead of relying on Claude Code hooks.

**In scope:** operator-initiated runs via `km agent run` (fire-and-forget AND `--wait`).

**Out of scope:**
- `km shell` — already works via real interactive PostToolUse/Stop hooks.
- Phase 67 Slack-inbound-poller's `claude -p` runs — their existing `.result` Slack post stays as-is.
- The Slack Connect `files.completeUploadExternal` rejection on externally-shared channels — that's a separate Phase 68.1 candidate.
- Subagent (Task-tool) thread fan-out — also a separate Phase 68.1 candidate.
- Phase B reaction-triggered session fork (uses the DDB stream-mapping table).

</domain>

<decisions>
## Implementation Decisions

### Approach: wrapper-side polling

- **Approach:** (A) wrapper-side polling. Background bash poller launched from `/tmp/km-agent-run.sh` tails the live transcript JSONL while `claude -p` runs and invokes the existing `_km_stream_drain` logic.
- **Rejected:** (B) interactive tmux pipe — would lose `--output-format json` and break `km agent results` consumers; (C) final-only — sacrifices real-time UX; (D) wait for upstream — no actionable timeline.
- **Why this works:** the streaming hook script already reads `transcript_path` from stdin payload. The same logic, fed a path discovered by the wrapper, produces identical Slack output regardless of who invoked it.

### Transcript JSONL discovery

- Wrapper pre-generates a UUID before launch and passes `--session-id <uuid>` to `claude -p`. Poller knows the path immediately: `~/.claude/projects/{cwd-encoded}/{uuid}.jsonl`.
- Deterministic, no race window between launch and discovery, no scan of `~/.claude/projects/` needed.
- Same UUID flows into output.json's `.session_id` (verifiable post-run).

### Run-done detection

- Poller watches for `/workspace/.km-agent/runs/<runID>/output.json` to appear.
- When `jq` parses it (or run dir has stderr indicating exit), poller does final-flush via `_km_stream_drain` once more, then `gzip` + `aws s3 cp` + bridge upload, then exits.
- Tied to existing run-dir convention (`agent.go` already creates `output.json` after `claude -p` exits).

### Poll cadence

- Interval-based, 2s default.
- Each tick calls `_km_stream_drain` (which already handles offset tracking + dedup).
- Bounded Slack post rate — matches semantics of the hook script's "one fire per PostToolUse."
- Configurable via env var (e.g., `KM_TRANSCRIPT_POLL_SECS`) for future tuning; default 2s.

### Gate logic

- Only the existing master gate: `KM_NOTIFY_SLACK_TRANSCRIPT_ENABLED=1` + `KM_SLACK_CHANNEL_ID` + `KM_SLACK_BRIDGE_URL` present.
- Poller creates auto-thread-parent if `KM_SLACK_THREAD_TS` is unset (mirrors `_km_stream_drain` behavior). Operator UX matches `km shell` exactly.

### Poller location & process model

- **Sandbox-side** — required for fire-and-forget runs (operator's `km agent run` exits immediately; nothing on the operator side to drive streaming).
- **Inline backgrounded bash from `km-agent-run.sh`** — add the poller body inline (or as a sourced helper) at the top of `/tmp/km-agent-run.sh`. Launch with `_km_stream_poller_loop &`, capture PID, install `trap` on EXIT to do final-flush + upload + reap.
- Zero new systemd units. Zero cloud-init changes (the poller body lives in the same heredoc that already builds `km-agent-run.sh`).
- Reuses `_km_stream_drain` and the `_km_stream_post_to_slack`-style helpers already in `pkg/compiler/userdata.go`.

### Concurrency

- Per-run poller, keyed on the pre-generated `session_id`.
- `_km_stream_drain` already keys offset/thread cache on `session_id`: `/tmp/km-slack-thread.{sid}`, `/tmp/km-slack-stream.{sid}.offset`. Distinct UUIDs → distinct keys → no interference between concurrent runs.

### Logging

- Poller logs land at `/workspace/.km-agent/runs/<runID>/poller.log` (sandbox:sandbox owned, alongside `output.json` + `stderr.log`).
- Lifecycle is the run-dir lifecycle.
- No journalctl, no logrotate, no SSM parameter changes.
- Future enhancement (out of scope): `km agent results <id> --include poller-log`.

### Phase 67 inbound poller scope

- **Stay scoped.** The Phase 67 `km-slack-inbound-poller.service` `claude -p` invocation in `pkg/compiler/userdata.go:1446` is **left untouched**. No streaming, no `.jsonl.gz` upload for inbound-driven turns.
- Inbound continues to post just `.result` text via the existing poller wrapper (current Phase 67 behavior).
- **No env-var guard needed** because the streaming code lives only in `/tmp/km-agent-run.sh`, not in shared bash helpers.
- **No thread-coordination logic needed** — inbound runs always have `KM_SLACK_THREAD_TS` pre-set from the originating Slack message; even if streaming code somehow ran on an inbound turn, `_km_stream_drain` would skip auto-thread-parent creation.
- Reasoning: inbound turns are conversational replies; per-tool 🔧 lines + a `.jsonl.gz` per turn would clutter the channel humans are chatting in.

### Failure mode (partial transcripts)

- **Upload partial with error footer.** On non-zero exit from `claude -p`, the trap-EXIT handler in `km-agent-run.sh` always runs final-flush + gzip + S3 upload of whatever transcript JSONL exists.
- **Synthetic footer line** appended to the JSONL before gzip:
  ```json
  {"type":"km_run_error","exit_code":N,"last_stderr":"<last 1024 bytes of stderr.log>","timestamp":"<RFC3339>"}
  ```
- **Slack thread surface:** post `⚠️ Run failed (exit N) — see attached transcript` to the same thread as per-turn lines, immediately after the `.jsonl.gz` upload (or instead of upload if upload itself fails).
- **Size cap (>100MB):** skip the Slack `files.completeUploadExternal` step entirely — always durably upload to S3, then post `⚠️ Transcript too large for Slack — see s3://${KM_ARTIFACTS_BUCKET}/transcripts/<sandbox-id>/<sid>.jsonl.gz` in the thread. Operator pulls full file from S3.

### Documentation

- **`docs/slack-notifications.md`** § Slack inbound — add: "Inbound-driven turns post the final reply text only; per-turn streaming + `.jsonl.gz` upload is enabled only for operator-initiated `km agent run` and `km shell`."
- **`docs/slack-notifications.md`** § Slack transcript streaming (Phase 68) — update the "Known Phase 68 limitations" subsection: remove the `km agent run` PostToolUse-skip bullet, replace with a "How it works for `km agent run`" paragraph describing the sandbox-side poller.
- **`CLAUDE.md`** — update the "Slack transcript streaming (Phase 68)" section to drop the "claude -p (km agent run) skips PostToolUse hooks" caveat and note that streaming now works for both interactive and fire-and-forget paths.
- **Phase 68 `deferred-items.md`** — mark this candidate `RESOLVED in Phase 68.1`.

### Claude's Discretion

- Exact poller bash idioms (e.g., `(while true; do _km_stream_drain; sleep 2; done) &` vs a function with explicit signal handling) — planner picks the cleanest pattern that's robust to `tmux kill-session`.
- Whether to extract the poller body into a separate helper file or keep inline in the `km-agent-run.sh` heredoc — Claude picks based on heredoc readability.
- Trap signal set (`EXIT` only vs `EXIT TERM INT`) — planner verifies which signals tmux session-kill propagates.
- Stderr-log truncation byte exact (1024 bytes is the target; planner picks `tail -c 1024` vs `awk` vs explicit byte slice).
- Whether the "⚠️ Run failed" Slack message uses the bridge `ActionPost` or a richer Block Kit format — Phase 74 (Slack mrkdwn) work is parallel; planner picks plain text + `*bold*` for now.
- OTEL span names if added (`km.transcript_poller.tick`, `km.transcript_poller.final_flush`) — planner decides instrumentation depth.

<specifics>
## Specific Ideas

- **Operator UX target:** firing `km agent run sb-X --prompt "audit and fix the failing tests" --transcript-stream` should produce the SAME Slack thread experience as `km shell` then typing the same prompt: per-turn 🔧 lines, auto-thread-parent at run start, gzipped JSONL upload at end. No "agent run is the loud cousin" caveat.
- **Pre-generated UUID** must be valid for Claude Code's `--session-id`. Spec-compliant UUIDv4 from `uuidgen` or Go's `uuid.NewString()` (already vendored).
- **Trap behavior:** if the operator `km destroy`s the sandbox mid-run, the trap-EXIT handler still runs (tmux session is killed before the EC2 instance terminates) and the partial transcript lands in S3 with the error footer. Operator gets evidence of what happened.
- **Test coverage parallel to Phase 68's `notify_hook_post_tool_use_test.go`** — fixture-based test file that drives the poller with a synthetic JSONL and asserts: per-turn lines posted in order, offset advanced correctly, final-flush captures everything, upload envelope built correctly, error footer added on simulated crash.
- **No Slack file upload retry loop in this phase.** If the Slack `files.completeUploadExternal` returns `internal_error` (Slack Connect channel limitation — separate Phase 68.1 candidate), the partial S3 upload is still durable; the failure surface is the existing Phase 68 thread message.

</specifics>
