2. Create `infra/modules/ses/v1.0.0/main.tf`:
   a. `aws_ses_domain_identity` for the subdomain
   b. `aws_ses_domain_dkim` for DKIM key generation
   c. `aws_route53_record` (x3) for DKIM CNAME records: `{token}._domainkey.{domain}` -> `{token}.dkim.amazonses.com`
   d. `aws_route53_record` for domain verification TXT record: `_amazonses.{domain}`
   e. `aws_route53_record` for MX record: `{domain}` -> `inbound-smtp.{region}.amazonaws.com`
   f. `aws_ses_receipt_rule_set` named "km-sandbox-email"
   g. `aws_ses_active_receipt_rule_set` pointing to "km-sandbox-email"
   h. `aws_ses_receipt_rule` named "sandbox-inbound" with:
      - recipients = [".{domain}"] (wildcard — catches all addresses at subdomain)
      - s3_action: bucket = var.artifact_bucket_name, object_key_prefix = "mail/"
      - position = 1
   i. `aws_s3_bucket_policy` (or data source for existing policy) allowing SES to write to the bucket under `mail/` prefix. Use `aws_iam_policy_document` with condition `StringEquals: aws:Referer = {account_id}`
   j. Use `data "aws_caller_identity" "current" {}` for account ID reference

3. Create `infra/modules/ses/v1.0.0/outputs.tf`:
   - `domain_identity_arn` — ARN of the SES domain identity
   - `domain_identity_verification_token` — for debugging
   - `receipt_rule_set_name` — "km-sandbox-email"

IMPORTANT: SES receipt rules require an active receipt rule set (Pitfall 5 from research). The module MUST create and activate the rule set.
IMPORTANT: The S3 action in the receipt rule stores with key prefix "mail/" — the sandbox-id is part of the recipient address, and SES stores the raw email as an object. The actual path in S3 will be "mail/{message-id}". The agent reads inbound mail by listing objects; cross-sandbox routing relies on the email content (headers) not the S3 key path. Adjust: the S3 action prefix should be "mail/" and the agent will need to parse the To/From headers to identify which sandbox the message is for.

2. Create `pkg/aws/ses.go`:
   - Define `SESV2API` narrow interface:
     ```go
     type SESV2API interface {
         CreateEmailIdentity(ctx context.Context, input *sesv2.CreateEmailIdentityInput, ...func(*sesv2.Options)) (*sesv2.CreateEmailIdentityOutput, error)
         DeleteEmailIdentity(ctx context.Context, input *sesv2.DeleteEmailIdentityInput, ...func(*sesv2.Options)) (*sesv2.DeleteEmailIdentityOutput, error)
         SendEmail(ctx context.Context, input *sesv2.SendEmailInput, ...func(*sesv2.Options)) (*sesv2.SendEmailOutput, error)
     }
     ```
   - `ProvisionSandboxEmail(ctx, client SESV2API, sandboxID, domain string) (string, error)`:
     Calls CreateEmailIdentity with `{sandboxID}@{domain}`, returns the address string.
   - `SendLifecycleNotification(ctx, client SESV2API, operatorEmail, sandboxID, event, domain string) error`:
     Calls SendEmail with From=`notifications@{domain}`, To=operatorEmail, Subject=`km sandbox {event}: {sandboxID}`, Body=plain text with event details. Events: "destroyed", "idle-timeout", "spot-interruption", "error".
   - `CleanupSandboxEmail(ctx, client SESV2API, sandboxID, domain string) error`:
     Calls DeleteEmailIdentity with `{sandboxID}@{domain}`. If the error is NotFoundException, return nil (idempotent).

3. Create `pkg/aws/ses_test.go`:
   - Define `mockSESV2API` struct with fields capturing each call's input and configurable return values.
   - Test all 7 behaviors above.

IMPORTANT: Use `service/sesv2` (v2 API), NOT `service/ses` (v1 API) per locked decision.
IMPORTANT: CleanupSandboxEmail must be idempotent — called during `km destroy` which may be retried.
IMPORTANT: Follow the same narrow-interface + mock pattern used in scheduler.go, mlflow.go, cloudwatch.go.
