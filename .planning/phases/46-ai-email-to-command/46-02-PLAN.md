**Refactor Handle() dispatch** (after KM-AUTH validation, Step 6):

Replace the current `switch` block with this logic:
1. Extract thread ID from MIME headers via `extractThreadID(msg)`
2. Try to load existing conversation from S3 via `loadConversation`
3. If conversation exists and state is "awaiting_confirmation":
   - Route to `handleConversationReply(ctx, senderEmail, bodyText, conversation)`
4. If no conversation exists:
   a. If YAML attachment present AND subject contains "create" -> existing `handleCreate` fast-path (no change)
   b. If subject contains "status" -> existing `handleStatus` (no change)
   c. If `h.BedrockClient != nil` -> new AI interpretation path: `handleAIInterpretation(ctx, senderEmail, bodyText, threadID)`
   d. Else -> existing `sendHelp` fallback

**New method: handleAIInterpretation(ctx, senderEmail, bodyText, threadID string) error**:
1. Build profile list from `profile.ListBuiltins()`
2. List running sandboxes from DynamoDB (use `awspkg.ListSandboxesDynamo` or scan — check what exists; if no list function, pass empty slice and document)
3. Call `callHaiku(ctx, h.BedrockClient, h.BedrockModelID, buildSystemPrompt(profiles, sandboxes), bodyText)`
4. If confidence < 0.7: send clarifying question reply, save conversation state as "new"
5. If confidence >= 0.7 AND cmd.Type == "info": execute immediately and reply with result
   - "list": query DynamoDB for running sandboxes, format as table, reply
   - "status": look up sandbox by ID or alias from cmd.Profile/Overrides, reply with status details
   - No confirmation needed, no conversation state saved
6. If confidence >= 0.7 AND cmd.Type == "action": build confirmation template, send reply, save conversation state as "awaiting_confirmation"

**New method: handleInfoCommand(ctx, senderEmail string, cmd *InterpretedCommand) error**:
- Switch on cmd.Command:
  - "list": scan DynamoDB km-sandboxes, format sandbox table (ID, alias, profile, status, TTL), send reply
  - "status": resolve sandbox ID from cmd.Profile or cmd.Overrides["sandbox_id"], fetch metadata, format status, send reply
- Always sends a direct reply — no conversation state, no confirmation loop

**Confirmation template format:**
```
I'll run:
  km {command} profiles/{profile}

With overrides:
  {key}: {value}
  ...

Confidence: {confidence}
Reasoning: {reasoning}

Reply YES to proceed, CANCEL to abort, or describe changes.
```

**New method: handleConversationReply(ctx, senderEmail, bodyText string, conv *ConversationState) error**:
1. Normalize body (trim, lowercase first word)
2. If body starts with "yes" (case-insensitive):
   - Switch on conv.ResolvedCmd.Command:
     - "create": generate sandbox ID, upload resolved profile YAML to S3, dispatch EventBridge create event
     - "destroy": dispatch EventBridge destroy event for the sandbox ID in ResolvedCmd
     - "extend": dispatch EventBridge extend event
     - "pause": dispatch EventBridge pause event
     - "resume": dispatch EventBridge resume event
   - Update conv.State = "confirmed", save conversation
   - Send "executing" reply with command details
3. If body starts with "cancel":
   - Update conv.State = "cancelled", save conversation
   - Send cancellation acknowledgment
4. Otherwise (revision):
   - Call Haiku with original resolved command context + new user message
   - Update conv.ResolvedCmd, conv.State = "awaiting_confirmation"
   - Save conversation, send updated confirmation template

**For the revision Haiku call:** Build a user message that includes the original interpretation context and the new revision request. Example: "Original request was interpreted as: {JSON of ResolvedCmd}. The user now says: {bodyText}. Please revise the command."
