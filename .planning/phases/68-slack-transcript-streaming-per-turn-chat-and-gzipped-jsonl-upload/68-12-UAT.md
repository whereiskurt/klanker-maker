## Scenario 6 — Operator warning at `km create`

```bash
km create profiles/test-transcript.yaml 2>&1 | grep "transcript streaming enabled"
```

**Pass criteria:**
- [x] Warning code path emits a single line on stderr with channel ID + member count (3 unit tests in `TestCreate_TranscriptWarning_*` PASS)
- [ ] Warning fires for `km create --remote ...` (FAILED — gap; see below)

**Result:** PARTIAL

**Evidence:**

UAT discovered a Plan 10 dispatch-path gap. The warning helper
`printTranscriptWarning` is wired into the **local** create flow only
(`internal/app/cmd/create.go` line 490, called from `runCreate`).
`runCreateRemote` (the `--remote` dispatch, default for EC2 substrates)
does NOT resolve the Slack channel locally — channel provisioning
happens inside the create-handler Lambda — so the warning never fires
on remote creates.

The warning code path itself is correct: Plan 10 ships 3 unit tests
that all PASS:
```
=== RUN   TestCreate_TranscriptWarning_PrintsWhenEnabled
--- PASS: TestCreate_TranscriptWarning_PrintsWhenEnabled (0.00s)
=== RUN   TestCreate_TranscriptWarning_AbsentWhenDisabled
--- PASS: TestCreate_TranscriptWarning_AbsentWhenDisabled (0.00s)
=== RUN   TestCreate_TranscriptWarning_IncludesMemberCount
--- PASS: TestCreate_TranscriptWarning_IncludesMemberCount (0.00s)
PASS
```

For UAT today the operator created `tt-7091bf92` via the default
`--remote` path and the warning was absent — expected given the gap.

**Follow-up filed in deferred-items.md as Phase 68.1 candidate:**
`runCreateRemote` should either (a) resolve the Slack channel locally
before dispatch and call `printTranscriptWarning`, or (b) the
create-handler Lambda should mirror the warning into the SES email
notification it sends to the operator on completion.

## Scenario 9 — `km destroy` cleanup decision

**Decision (recorded in plan):** Phase 68 relies on the existing artifacts
bucket lifecycle policy for transcript cleanup (NOT explicit `km destroy`
deletion). Operators who need faster cleanup can manually
`aws s3 rm --recursive s3://${BUCKET}/transcripts/{sandbox-id}/` after destroy.

**Pass criteria:**
- [x] Decision is documented in the UAT outcomes (this section)
- [x] Operator acknowledges the decision and confirms bucket lifecycle is
      configured (or manual cleanup is acceptable)

**Result:** PASS

**Evidence / Notes:**

The `slack_transcript_stale_objects` km doctor check (validated under
Scenario 7) operationalizes this decision: it surfaces orphaned
`transcripts/{sandbox_id}/` prefixes after `km destroy` and points
operators to the manual cleanup command:
```
aws s3 rm s3://<bucket>/<prefix> --recursive
```

This matches the design intent — Phase 68 does NOT add transcript
deletion to `km destroy`. Operators have three options:
1. Configure an S3 bucket lifecycle policy on `transcripts/*` (recommended
   for production deployments).
2. Periodically run `km doctor` and follow the suggested cleanup command.
3. Leave transcripts in S3 indefinitely (cheap; ~7 KB per session).

Operator acknowledged this decision and verified the doctor check
produces actionable output post-destroy.
