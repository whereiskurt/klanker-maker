## Harness caveat (important)

Two harness artifacts, both **expected and harmless**:

1. **Fabricated `ts` → `reactions.add: message_not_found`.** Every event uses a synthetic `ts`, so the 👀 reaction (and threaded reply) can't attach to a real Slack message. Verified via **side effects** (RUN_DIRs under `/workspace/.km-agent/runs/`, bridge `enqueued` log lines, SQS in-flight counts, DDB row), NOT the reaction.
2. **Claude not interactively authenticated** on the test box (the OAuth login is an operator browser flow; no credentials.json was rsync'd). Each dispatched `claude -p` therefore returns `"Not logged in · Please run /login"` and exits 1 in ~300 ms. **This does not invalidate the concurrency assertions** — the poller's dispatch / semaphore / RUN_DIR / heartbeat / dedup machinery is exercised fully regardless of whether the agent itself produces a real answer. The fast exit is actually convenient: it lets parallelism be observed cleanly (3 RUN_DIRs minted within a 4-second window).

### A3 — cap enforcement

```

MAX="${KM_SLACK_MAX_CONCURRENCY:-1}"; [ "$MAX" -lt 1 ] && MAX=1
```

- **Stale `VisibilityTimeout=30` test assertions.** `internal/app/cmd/create_slack_inbound_test.go:198` and `create_github_inbound_test.go:98` still asserted the pre-Phase-119 `"30"` value; the production `inboundQueueAttrs` helper (`pkg/aws/sqs.go:135`) was already raised to `"1800"` in Phase 119-02 but these two create-handler tests were missed in lockstep. Updated both to `"1800"` with a Phase-119 comment. `go test ./internal/app/cmd/` green after the fix. (Deviation Rule 1 — stale assertion from the Phase-119 queue base raise.)
- **Stale orphaned `km-sandboxes` row shadowed channel routing.** The first create attempt (`parallel-769afe41`) failed at provisioning (Terraform registry timeout, see Notes) but its `teardownPolicy: stop` row persisted with the same `slack_channel_id`. The bridge's first `FetchByChannel` resolved the *stale* row, routing event A1 to the dead sandbox. Deleted the orphan row; routing then correctly resolved to the live `parallel-3274551b`. (Operational artifact of `stop` policy + `archiveOnDestroy:false` channel reuse — not a Phase-119 defect; the bridge behaved per spec given an ambiguous channel→sandbox mapping.)

## Notes / findings

- **Transient Terraform-registry timeout during cold-create.** Two cold-creates failed with `Failed to install provider hashicorp/aws v6.46.0 ... registry.terraform.io ... Client.Timeout exceeded` from inside the create-handler Lambda (6 retries). This is an environment/network artifact (provider not pre-cached in the Lambda's `infra.tar.gz`; registry slow), **not** a Phase-119 code issue — a retry succeeded for `parallel-3274551b`. Flagged for a separate infra hardening item (pre-bundle the provider plugin cache into the toolchain tarball).
- **`[: : integer expression expected` at poller line 122** (harmless): the empty-batch `COUNT` check (`[ "$COUNT" -eq 0 ]`) emits this when SQS long-poll returns no messages and jq yields an empty string instead of `0`. Pre-existing cosmetic noise (the `|| continue` loop proceeds correctly); not introduced by Phase 119 and does not affect dispatch. Candidate one-line follow-up: `COUNT=${COUNT:-0}` or `[ "${COUNT:-0}" -eq 0 ]`.
- The synthetic `event_callback` POST must target **`/events`** (path-based dispatch); the bare bridge URL routes to the sandbox→Slack envelope handler.
