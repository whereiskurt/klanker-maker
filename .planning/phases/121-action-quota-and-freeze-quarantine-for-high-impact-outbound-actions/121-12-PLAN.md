<objective>
Close GAP 2: both bridges treat `quota.BreachFreeze` identically to `BreachBlock` — they block the
action and post a "Sandbox is now frozen — operator release required" notice, but NEVER call
`FreezeSandboxDynamo`, so `action_frozen` is never set on the `km-sandboxes` row. The frozen-dispatch
gate (`pkg/slack/bridge/events_handler.go` ~line 395) only fires when `action_frozen=true` already
exists — which auto-breach never writes. The notice text is a lie; only manual `km freeze` actually
quarantines. This breaks CONTEXT.md §7 decision 8 ("Auto-on-breach freeze trigger").

This plan threads a `Freezer` interface into each bridge `Handler`/`WebhookHandler`; on a
`BreachFreeze` trip the bridge calls `Freezer.FreezeSandbox(ctx, sandboxID, reason, by)` with
`by="auto:<action>:<window>"` BEFORE returning block=true / skipping the enqueue, latching
`action_frozen=true`. The concrete adapter (wrapping `kmaws.FreezeSandboxDynamo`) is wired in each
bridge's `aws_adapters.go`. `BreachWarn`/`BreachBlock` MUST NOT freeze.

Purpose: Make `onBreach: freeze` actually quarantine the sandbox automatically (BRG-01/H1-01) so the
frozen-dispatch gate (BRG-02) it relies on is reachable from the auto-breach path, not just `km freeze`.
Output: Auto-latch on freeze trips in both bridges + adapters + tests asserting freeze-only.
</objective>
