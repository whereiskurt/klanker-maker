<objective>
Close GAP 1: `pkg/quota.atomicIncrement` never writes `breached_at` or `on_breach` to the
action-quota table, so the `km-quota-alerter` Lambda's first-breach guard
(`if _, hasBreachedAt := newImg["breached_at"]; !hasBreachedAt { return nil }`) skips EVERY
production Stream record and no operator SES email / Slack control-channel notice ever fires.
The ALR-01 unit test only passes because `TestIdempotentAlert` hand-builds a fake DDB event with
`breached_at` already present — the live integration path is dead.

This plan makes `Record`, immediately AFTER an increment whose post-increment count exceeds the
window's limit, issue ONE additional conditional UpdateItem on the same row:
`SET breached_at = if_not_exists(breached_at, :now), on_breach = if_not_exists(on_breach, :policy)`.
`if_not_exists` makes it first-breach-only and idempotent under a hard loop. It adds a unit test
that drives the REAL `Record` (not a faked Stream event) and asserts the breach UpdateItem carries
those attrs on a trip and omits them on a non-trip.

Purpose: Restore the alerter's production path (ALR-01) and close the false-confidence hole the
verifier identified (the existing ALR-01 test bypasses the gap entirely).
Output: `breached_at` + `on_breach` written on first-breach; a REAL-path breach-write test.
</objective>

Current atomicIncrement signature (line 196):
```go
func atomicIncrement(ctx context.Context, client QuotaAPI, tableName, sandboxID string, action Action, w windowSpec) (int64, error)
```
Current UpdateExpression (lines 199-215): `ADD #c :one` (+ optional `SET #ttl = if_not_exists(#ttl, :ttl)` for hour/day). ReturnValues ALL_NEW.

DESIGN DECISION (settled — do NOT deliberate this in-task): the breach metadata is written by a
SECOND, separate conditional UpdateItem issued only when `count > w.limit`. This is the approach the
verification report itself sanctioned ("Or: pkg/quota.Record must issue a second UpdateItem to set
breached_at when count > limit"). Why a second write rather than folding it into the ADD: DynamoDB
cannot reference the post-increment count inside the SAME expression that produces it, so a
count-conditioned SET in the increment is not expressible. The second write is keyed on the SAME row
and both attrs use `if_not_exists`, so:
  - it fires only on rows that actually breached (the `count > w.limit` Go guard), AND
  - it is first-breach-only + idempotent under a hard loop (hundreds of blocked attempts → one set), AND
  - concurrent writers collapse (if_not_exists wins-once), AND
  - the alerter's "breached_at in NewImage, absent in OldImage" first-breach detection still fires on
    the MODIFY produced by this second write (NEW_AND_OLD_IMAGES stream view).
The two-event cost (the ADD MODIFY then the breach-SET MODIFY) is acceptable: the alerter ignores the
ADD-only MODIFY (no breached_at) and acts on the breach-SET MODIFY.
