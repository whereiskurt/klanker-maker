<objective>
Close GAP 1: `pkg/quota.atomicIncrement` never writes `breached_at` or `on_breach` to the
action-quota table, so the `km-quota-alerter` Lambda's first-breach guard
(`if _, hasBreachedAt := newImg["breached_at"]; !hasBreachedAt { return nil }`) skips EVERY
production Stream record and no operator SES email / Slack control-channel notice ever fires.
The ALR-01 unit test only passes because `TestIdempotentAlert` hand-builds a fake DDB event with
`breached_at` already present — the live integration path is dead.

This plan makes `atomicIncrement` `SET breached_at = if_not_exists(breached_at, :now)` AND
`SET on_breach = if_not_exists(on_breach, :policy)` on the SAME conditional increment when (and
only when) the post-increment count for that window exceeds its limit, and adds a unit test that
drives the REAL `Record` (not a faked Stream event) and asserts the breach UpdateItem expression
carries those attrs on a trip and omits them on a non-trip.

Purpose: Restore the alerter's production path (ALR-01) and close the false-confidence hole the
verifier identified (the existing ALR-01 test bypasses the gap entirely).
Output: `breached_at` + `on_breach` written on first-breach; a REAL-path breach-write test.
</objective>

From pkg/quota/quota.go — Record() builds windowSpecs then loops calling atomicIncrement.
`now` (time.Now().UTC()) is already in Record's scope at line 119; pass it (and the resolved
`onBreach` policy, line 155) into atomicIncrement so the breach write can use them. Each window
already knows its limit (`w.limit`); the post-increment `count` is the atomicIncrement return.

Current atomicIncrement signature (line 196):
```go
func atomicIncrement(ctx context.Context, client QuotaAPI, tableName, sandboxID string, action Action, w windowSpec) (int64, error)
```
Current UpdateExpression (lines 199-215): `ADD #c :one` (+ optional `SET #ttl = if_not_exists(#ttl, :ttl)` for hour/day). ReturnValues ALL_NEW. The breach attrs must be added to THIS SAME UpdateItem (a single conditional write per increment) so the breach is visible in the SAME Stream MODIFY that crosses the limit — DO NOT issue a second UpdateItem (avoids a second Stream event + a read-modify race).

Problem: ADD returns the post-increment count only AFTER the call, but the SET must be in the
SAME expression. Resolution: compute the breach decision from the limit ALONE is impossible
pre-call. Instead, structure the expression so breached_at is set unconditionally-but-idempotently
guarded by a DDB ConditionExpression on the count, OR — simpler and matching the existing
if_not_exists idiom — always include the breach SET clauses guarded by `if_not_exists` AND a
DynamoDB `ConditionExpression`-free approach is NOT possible because we cannot reference the
post-increment count inside the same SET. Use the two-write fallback that the verification report
itself sanctioned ("Or: pkg/quota.Record must issue a second UpdateItem to set breached_at when
count > limit"): keep the existing ADD increment as-is, and AFTER it returns, when
`count > w.limit`, issue ONE additional conditional UpdateItem:
`SET breached_at = if_not_exists(breached_at, :now), on_breach = if_not_exists(on_breach, :policy)`
keyed on the same row. `if_not_exists` makes it first-breach-only and idempotent under a hard loop
(hundreds of blocked attempts still set breached_at once). This is the cleaner, race-safe path
because count is known post-ADD and `if_not_exists` collapses concurrent writers.

From cmd/km-quota-alerter/alerter.go (handleRecord, ~line 123) — the consumer guard (DO NOT EDIT
