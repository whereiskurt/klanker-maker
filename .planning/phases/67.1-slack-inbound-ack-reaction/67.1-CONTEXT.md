# Phase 67.1: Slack inbound ACK reaction - Context

**Gathered:** 2026-05-03
**Status:** Ready for planning
**Source:** Chat brainstorm following Phase 67 close-out (UAT gap: users get no
visual feedback that Slack message was received until the agent boots and posts
its first reply, which can be 10-60s for paused sandboxes)

<domain>
## Phase Boundary

When the bridge Lambda accepts an inbound Slack message and successfully
enqueues it to SQS, it adds a 👀 emoji reaction (configurable) to the
originating message via `reactions.add`. This gives the user immediate
(<1s) visual confirmation that the sandbox saw their message — even before
the agent boots, before the paused-sandbox hint posts, before any reply.

**In scope:**
- Bridge-side Reactor interface + production HTTP adapter
- Fire-and-forget reaction call from `EventsHandler.Handle` after SQS write succeeds
- Configurable emoji via `KM_SLACK_ACK_EMOJI` env var (default `eyes`)
- Scope verification updates in `km slack init` and `km doctor`
- Unit tests covering happy path, reactor-failure-noop, bot-loop-still-skips
- Doc update: `docs/slack-notifications.md` § "Slack inbound" — new scope row + paragraph

**Out of scope (explicitly deferred):**
- Two-stage reaction (poller-side 🧠 when sandbox pulls from SQS) — needs `km-slack` sidecar `react` subcommand + sandbox redeploy
- Terminal reaction (✅/❌ on run completion) — needs run-completion callback path
- Reaction *removal* on success/failure — additive only
- Per-sandbox configurable emoji (profile field) — keep global for v1

</domain>

<decisions>
## Implementation Decisions

### Architectural pattern (LOCKED)
- **Mirror the existing `PauseHinter` pattern** (`pkg/slack/bridge/events_handler.go:197-209`)
  — fire-and-forget goroutine after step 8 (SQS write success) so Slack
  still gets a 200 within the 3s ack window
  - Use a fresh context (`context.WithTimeout(context.Background(), 5s)`)
    because the request ctx may be canceled after 200 is written
  - Reactor failures log a warning and never block the SQS path or the 200
- **Place the call AFTER successful SQS write, NOT before**
  — the 👀 reaction is a promise that "we accepted it for processing", not
  just "we received the HTTP request"

### Bridge code structure (LOCKED)
- New `Reactor` interface in `pkg/slack/bridge/events_interfaces.go`:
  ```go
  type Reactor interface {
      Add(ctx context.Context, channel, ts, emoji string) error
  }
  ```
- New production adapter `SlackReactorAdapter` in `pkg/slack/bridge/aws_adapters.go`
  — mirrors `SlackPosterAdapter` (same `BotTokenFetcher`, same `*http.Client`,
  POSTs JSON to `https://slack.com/api/reactions.add`)
- Add `Reactor` field to `EventsHandler` struct (nullable — nil means feature off)
- Cold-start wiring in `cmd/km-slack-bridge/main.go` reads `KM_SLACK_ACK_EMOJI`
  env (default `eyes`), constructs `SlackReactorAdapter`, attaches to handler

### Emoji configuration (LOCKED)
- **Bridge-global env var** `KM_SLACK_ACK_EMOJI` (default `eyes`)
- Set via Lambda env var update — no SSM, no profile field
- Operators can change without code deploy (just `km init --lambdas` to update env)
- Per-sandbox emoji configurability is **deferred to v2** if anyone asks

### Slack scope (LOCKED)
- New required scope: `reactions:write`
- Added to required-scope lists in:
  - `internal/app/cmd/slack.go` `VerifyEventsAPIScopes` (line 766)
  - `internal/app/cmd/doctor_slack.go` `checkSlackAppEventsScopes` (line 344)
- Both already have warning UX for missing scopes — just extend the constant
- Bot token does **not** change on reinstall, so no token rotation needed

### Bot-loop filter (LOCKED — already correct)
- The existing `isBotLoop` allow-list filter (`pkg/slack/bridge/events_handler.go:229`)
  already runs at step 4, BEFORE the SQS write at step 8
- Bot self-messages, channel_join, file_share, etc. are filtered out before
  the Reactor is ever invoked — no special-casing needed in the Reactor path
- Add a regression test confirming this (Reactor not called when isBotLoop matches)

### Failure handling (LOCKED)
- Reactor failure (missing scope, throttled, channel deleted, bot kicked):
  log `events: reaction failed` warning with channel/ts/err — continue
- **Never** block SQS enqueue or the 200 response on a Reactor error
- Slack rate limit for `reactions.add` is Tier 3 (~50/min) — not a concern
  at sandbox volumes; no need for backoff or queuing

### Operator migration path (LOCKED)
1. Slack App config → OAuth & Permissions → add Bot Token Scope `reactions:write`
2. Click "Reinstall to Workspace" (bot token unchanged)
3. `make build && km init --lambdas` to deploy bridge with new code + env var
4. `km doctor` confirms scope present
5. Smoke: post in any inbound-enabled `#sb-{id}`, see 👀 within 1s

Existing inbound-enabled sandboxes pick up the change automatically — no
`km destroy/create` needed because this is bridge-only.

### Claude's Discretion
- Exact wording/severity of warning log lines
- Internal function naming inside `SlackReactorAdapter`
- Whether to factor common HTTP-call boilerplate from `SlackPosterAdapter`
  + `SlackReactorAdapter` into a shared helper (acceptable if clean; not required)
- Test naming conventions (follow existing `TestEventsHandler_*` patterns)
- Whether to reuse `pkg/slack/client.go` `callJSON` helper instead of
  writing a fresh HTTP call in the adapter (preferred if it cleanly accepts
  the right shape)

### Test cases (specific scenarios)
1. **Happy path** — valid message → `Reactor.Add(channel, ts, "eyes")` called
   exactly once with the right args after SQS Send returns nil
2. **Reactor failure does not block** — `Reactor.Add` returns error → SQS
   still has the message, response is 200, no panic
3. **Bot loop skips Reactor** — `isBotLoop` returns true at step 4 →
   `Reactor.Add` is never called (use a counter on the fake)
4. **Nil Reactor is no-op** — `EventsHandler.Reactor == nil` → handler
   still works, no panic (back-compat for tests that don't wire Reactor)
5. **Doctor scope check** — extend `TestDoctor_SlackInboundEventsSubscription_*`
   tests to include `reactions:write` in required-set assertions

<deferred>
## Deferred Ideas

These are explicitly OUT of scope for 67.1 — defer to future phase or evaluate
after 67.1 ships if there's user demand:

- **Two-stage reaction (poller-side 🧠 ACK)** — when the sandbox poller
  pulls the message from SQS, add a second emoji indicating the sandbox
  is actively handling it. Requires `km-slack` sidecar to grow a `react`
  subcommand AND a sandbox redeploy for every existing inbound sandbox.
  Bigger blast radius. Defer.
- **Terminal reaction (✅/❌ on run finish)** — when `km agent run`
  completes, post a terminal reaction reflecting success/failure. Needs
  the run-completion path to call back to Slack. Possibly fits in Phase
  68 (transcript streaming) since that already adds run-finish callbacks.
- **Reaction removal** — when terminal reaction posts, remove the in-progress
  ones for cleanliness. Pure additive in v1; consider once we have terminal.
- **Per-sandbox emoji** (`spec.cli.slackAckEmoji` profile field) — niche;
  add only if requested. Bridge-global is sufficient for v1.
- **Ack on app_mention events** — Phase 67 deferred `app_mention` event
  type entirely; if/when that ships, the Reactor pattern extends trivially.

</deferred>
