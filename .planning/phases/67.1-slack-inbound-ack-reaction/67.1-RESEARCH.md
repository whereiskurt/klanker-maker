## Summary

Phase 67.1 is a small, bridge-only UX gap closure. The bridge Lambda already has all infrastructure in place — it just needs one new interface (`Reactor`), one new adapter (`SlackReactorAdapter`), and one new fire-and-forget call in `EventsHandler.Handle` after step 8. The entire scope fits in ~170 LoC across 9 files. No sandbox redeploy, no Terraform data-plane changes, no new SSM parameters.

The critical structural template is the `PauseHinter` goroutine block at `events_handler.go:197-209`. The Reactor call is structurally identical: spawn goroutine after successful SQS send, fresh context with 5s timeout, log warning on error, never surface errors to caller. The `SlackReactorAdapter` is structurally identical to `SlackPosterAdapter` — same `BotTokenFetcher`, same `*http.Client`, same `call()` method, same `slackAPIResponse` struct — it just calls `reactions.add` instead of `chat.postMessage`.

The Slack `reactions.add` API takes `channel`, `timestamp`, and `name` (emoji without colons — `"eyes"` not `":eyes:"`). It returns `{"ok":true}` on success. The `already_reacted` error is idempotency-safe: treat as success (log debug, don't return error). Rate limit is Tier 3 (~50/min), not a concern at sandbox volumes.

**Primary recommendation:** The `SlackPosterAdapter.call()` method in `aws_adapters.go` is the exact pattern to clone for `SlackReactorAdapter`. Do not use `pkg/slack/client.go` (it lacks the `BotTokenFetcher` lazy-fetch cache). Add `already_reacted` as an idempotent non-error in the adapter.

## Architecture Patterns

### Pattern 1: PauseHinter fire-and-forget (the structural template)

**Verified location:** `pkg/slack/bridge/events_handler.go:193-209`

The existing PauseHinter block is the exact structural template the Reactor call must mirror:

```go
// Source: pkg/slack/bridge/events_handler.go:193-209 (verified)
if info.Paused && h.PauseHinter != nil {
    ch, ts := msg.Channel, threadTS
    go func() {
        bgCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
        defer cancel()
        if err := h.PauseHinter.PostIfCooldownExpired(bgCtx, ch, ts); err != nil {
            h.log().Warn("events: pause hint post failed", "err", err, "channel", ch, "thread_ts", ts)
        }
    }()
}
```

The Reactor call follows the same pattern, placed immediately AFTER this block (step 9 in the Handle flow becomes step 10 effectively — the reaction fires after SQS write AND after the paused-sandbox logic):

```go
// New: Reactor call (Phase 67.1) — fires after step 8 SQS write success
if h.Reactor != nil {
    ch, ts, emoji := msg.Channel, msg.TS, h.AckEmoji
    go func() {
        bgCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
        defer cancel()
        if err := h.Reactor.Add(bgCtx, ch, ts, emoji); err != nil {
            h.log().Warn("events: reaction failed", "err", err, "channel", ch, "ts", ts, "emoji", emoji)
        }
    }()
}
```

**Important:** React on `msg.TS` (the originating message timestamp), NOT `threadTS`. The reaction marks the specific message the user sent, not the thread root. These differ when the user replies in-thread.

### Pattern 2: SlackPosterAdapter.call() — the HTTP adapter to clone

**Verified location:** `pkg/slack/bridge/aws_adapters.go:287-328`

`SlackReactorAdapter` is a near-verbatim copy of `SlackPosterAdapter` but with:
- Only one exported method: `Add(ctx, channel, ts, emoji string) error`
- Calls `reactions.add` instead of `chat.postMessage` or `conversations.archive`
- Treats `already_reacted` Slack error as idempotent success (log debug, return nil)

```go
// Source: Pattern from aws_adapters.go:271-328 (verified)
type SlackReactorAdapter struct {
    HTTPClient *http.Client
    BaseURL    string          // defaults to "https://slack.com/api"; overridden in tests
    Tokens     BotTokenFetcher // fetches and caches the bot token (reuse SSMBotTokenFetcher)
}

func (s *SlackReactorAdapter) Add(ctx context.Context, channel, ts, emoji string) error {
    // reuse the same call() pattern as SlackPosterAdapter
    resp, httpStatus, retryAfter, err := s.call(ctx, "reactions.add", map[string]any{
        "channel":   channel,
        "timestamp": ts,
        "name":      emoji, // WITHOUT colons: "eyes" not ":eyes:"
    })
    if err != nil {
        return err
    }
    if rlErr := checkRateLimit(httpStatus, retryAfter, "reactions.add"); rlErr != nil {
        return rlErr
    }
    if !resp.OK {
        if resp.Error == "already_reacted" {
            // Idempotent: Slack delivered the event twice, bridge already reacted.
            // Log at debug level and return nil — not an error.
            return nil
        }
        return fmt.Errorf("bridge: reactions.add: %s", resp.Error)
    }
    return nil
}
```

**Note on shared `call()` boilerplate:** `SlackPosterAdapter.call()` is unexported (lowercase). The planner must decide: (a) duplicate the method body in `SlackReactorAdapter` (simplest), (b) extract to a shared unexported `slackHTTPCall()` package-level function both adapters use, or (c) embed `SlackPosterAdapter` in `SlackReactorAdapter`. Option (b) is cleanest and is left to Claude's discretion per CONTEXT.md. The method signature would be `func slackCall(ctx context.Context, httpClient *http.Client, baseURL, token, method string, payload any) (*slackAPIResponse, int, string, error)`.

### Anti-patterns to Avoid

- **Reacting before SQS write succeeds:** The 👀 means "we accepted it for processing." If you react before SQS write, a subsequent SQS failure leaves a lying reaction. Always react in a block that only executes on the `err == nil` path after `h.SQS.Send(...)`.
- **Using `msg.ThreadTS` as the reaction timestamp:** The reaction must be on the specific message the user sent, not the thread root. Use `msg.TS`. (`threadTS` is derived from `msg.ThreadTS` or `msg.TS` — it's the session anchor, not the reaction target.)
- **Treating `already_reacted` as an error:** Slack delivers events at-least-once. On replay after a cold-start, the bridge may react twice. `already_reacted` must be handled as idempotent success, not logged as an error.
- **Using `pkg/slack/client.go` for the adapter:** The `Client` type in `pkg/slack/client.go` constructs with a pre-fetched token (`botToken string`). The bridge uses a lazy-cached `BotTokenFetcher` interface (`SSMBotTokenFetcher`) to avoid SSM calls on every Lambda invocation. Use `SlackPosterAdapter`'s pattern, not `pkg/slack.Client`.
- **Blocking the 200 response on Reactor failure:** The goroutine must be fire-and-forget. Any synchronous call to `Reactor.Add` would add latency to Slack's 3s ack window, potentially causing Slack to retry.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Emoji name format validation | Custom validator | Just pass through to Slack | Slack returns `invalid_name` for bad emoji; let Slack validate |
| HTTP retry for `already_reacted` | Retry loop | Treat as success (return nil) | Idempotency contract; retrying a 200-equivalent is wasteful |
| Bot token lazy fetch | Direct SSM call per reaction | Reuse `SSMBotTokenFetcher` via `BotTokenFetcher` interface | Already has 15-min cache; SSM costs money per call |
| Custom rate limit handling | Token bucket | Log warning, return error, let goroutine die | Tier 3 (~50/min) far exceeds sandbox volumes; no need for queuing |

## Common Pitfalls

### Pitfall 1: Reacting to Wrong Timestamp

**What goes wrong:** `reactions.add` returns `message_not_found` if the timestamp doesn't correspond to an actual message. Using `threadTS` when the message is a reply means reacting to the thread root instead of the user's specific message.

**Why it happens:** Confusing `msg.TS` (the originating message) with `threadTS` (the session anchor, which equals `msg.ThreadTS` for replies or `msg.TS` for top-level posts).

**How to avoid:** Always pass `msg.TS` to `Reactor.Add`. The `threadTS` variable (computed at line 161-163 in `events_handler.go`) is for SQS routing and DDB session lookup — not for reactions.

**Warning signs:** `message_not_found` errors in Lambda logs when user sends in-thread replies.

### Pitfall 2: already_reacted Treated as Error

**What goes wrong:** Slack delivers events at-least-once. After a Lambda cold start or a retry, the same event may arrive twice. If `already_reacted` is returned as an error, it gets logged at WARN level and potentially surfaced in metrics — creating false alarm noise.

**Why it happens:** Copy-pasting the `if !resp.OK { return fmt.Errorf(...) }` pattern from `PostMessage` without adding the `already_reacted` special case.

**How to avoid:** In `SlackReactorAdapter.Add()`, check `resp.Error == "already_reacted"` before the generic error return and return `nil` (optionally log at debug).

**Warning signs:** WARN logs in CloudWatch matching `reactions.add: already_reacted` on every Lambda cold start after deployment.

### Pitfall 3: Emoji Name Format

**What goes wrong:** Passing `":eyes:"` (with colons) instead of `"eyes"` (without colons) causes Slack to return `invalid_name`.

**Why it happens:** Emoji names in Slack message text use `:eyes:` format; the `reactions.add` `name` parameter does NOT.

**How to avoid:** The default `"eyes"` (no colons) is correct. If operators set `KM_SLACK_ACK_EMOJI`, document clearly in operator guide that colons must be omitted.

**Warning signs:** `invalid_name` errors in Lambda logs.

### Pitfall 4: Reactor Called When isBotLoop Should Have Filtered

**What goes wrong:** If the Reactor call is placed BEFORE the `isBotLoop` check, bot self-messages trigger a reaction on the bot's own message. This creates visual clutter and wastes API quota.

**Why it happens:** Incorrect placement of the Reactor call in the Handle() flow.

**How to avoid:** The `isBotLoop` filter is at step 4 (line 133). The SQS Send is at step 8 (line 183). Place the Reactor call only in the block after `h.SQS.Send()` returns `nil`. The existing `return` at line 135 for bot-loop cases exits before reaching the Reactor call.

**Warning signs:** 👀 reactions appearing on the bot's own outbound messages.

### Pitfall 5: Missing nil Check on Reactor Field

**What goes wrong:** Tests that construct `EventsHandler` without wiring a Reactor (all existing tests in `events_handler_test.go`) panic with nil pointer dereference.

**Why it happens:** The `Reactor` field is an interface; calling a method on a nil interface panics.

**How to avoid:** Guard the call with `if h.Reactor != nil`. This is already the pattern for `PauseHinter` at line 197. The Reactor field is nullable by design — nil means feature off.

### Complete SlackReactorAdapter (minimal viable implementation)

```go
// Source: Pattern from pkg/slack/bridge/aws_adapters.go SlackPosterAdapter (verified)
// New file location: pkg/slack/bridge/aws_adapters.go (add after SlackPosterAdapter)

type SlackReactorAdapter struct {
    HTTPClient *http.Client
    BaseURL    string          // defaults to "https://slack.com/api"; overridden in tests
    Tokens     BotTokenFetcher
}

func (s *SlackReactorAdapter) getBaseURL() string {
    if s.BaseURL != "" {
        return s.BaseURL
    }
    return "https://slack.com/api"
}

func (s *SlackReactorAdapter) Add(ctx context.Context, channel, ts, emoji string) error {
    token, err := s.Tokens.Fetch(ctx)
    if err != nil {
        return fmt.Errorf("bridge: get bot token for reactions.add: %w", err)
    }

    b, _ := json.Marshal(map[string]any{
        "channel":   channel,
        "timestamp": ts,
        "name":      emoji,
    })

    req, err := http.NewRequestWithContext(ctx, "POST", s.getBaseURL()+"/reactions.add", bytes.NewReader(b))
    if err != nil {
        return err
    }
    req.Header.Set("Authorization", "Bearer "+token)
    req.Header.Set("Content-Type", "application/json; charset=utf-8")

    hc := s.HTTPClient
    if hc == nil {
        hc = http.DefaultClient
    }
    resp, err := hc.Do(req)
    if err != nil {
        return err
    }
    defer resp.Body.Close()

    // Handle Slack rate limit (Tier 3; rare but possible under load)
    if resp.StatusCode == http.StatusTooManyRequests {
        return &ErrSlackRateLimited{RetryAfterSeconds: 1, Method: "reactions.add"}
    }

    var apiResp slackAPIResponse // reuse existing struct from aws_adapters.go
    if err := json.NewDecoder(resp.Body).Decode(&apiResp); err != nil {
        return fmt.Errorf("bridge: decode reactions.add: %w", err)
    }
    if !apiResp.OK {
        if apiResp.Error == "already_reacted" {
            return nil // idempotent success
        }
        return fmt.Errorf("bridge: reactions.add: %s", apiResp.Error)
    }
    return nil
}
```

**Note on code sharing:** If the planner opts for a shared `slackCall` helper instead of duplicating the HTTP call body, the signature would be:
```go
// package-level unexported helper in aws_adapters.go
func slackAPICall(ctx context.Context, hc *http.Client, baseURL, token, method string, payload any) (*slackAPIResponse, int, string, error)
```
Both `SlackPosterAdapter.call()` and `SlackReactorAdapter` would delegate to this. This is cleaner but not required.
