<objective>
Plan 03 closes Phase 67.1 with three concerns the prior two plans deliberately deferred:

1. **Terraform env passthrough** — the bridge code in Plan 01 reads `KM_SLACK_ACK_EMOJI` from the Lambda env, but nothing in the Lambda module declares or sets that variable. Without this, the env is unset and the bridge falls back to the hardcoded `eyes` default. That works (the default is correct for v1) but the operator has no way to override the emoji without code-editing the bridge — so we add the var + module passthrough so operators can set `KM_SLACK_ACK_EMOJI=hourglass_flowing_sand` (or any other emoji) via Terraform inputs.

2. **Operator documentation** — the new scope (`reactions:write`) and the new manual deploy step (add scope → reinstall app → make build && km init --lambdas) are not yet documented anywhere operators read. Without this, operators who deploy the new bridge code from Plan 01 hit a silent CloudWatch warning instead of a documented runbook.

3. **Manual UAT in real Slack** — the unit tests in Plans 01 and 02 cover code-level behavior, but the actual round-trip (Slack message → bridge Lambda → reactions.add → 👀 appears in Slack) cannot be tested without a real workspace. This plan includes a `checkpoint:human-verify` task with explicit step-by-step verification instructions.

This plan is **Wave 2** because it depends on Plans 01 and 02 having landed:
- Terraform passthrough (Task 1) needs nothing from Plan 01/02 — could in principle parallelize, but combining all three concerns into one Wave-2 plan keeps the work locally cohesive and avoids spreading docs across waves
- Docs (Task 2) describe the behavior implemented in Plan 01 and the scope check from Plan 02 — must come after both
- Manual UAT (Task 3) requires a deployed bridge Lambda with the Plan 01 code — must come last

Output: Operators can deploy Plan 01's bridge code via standard `make build && km init --lambdas` flow with a clear runbook; the 👀 reaction is verified working end-to-end in a real Slack workspace; Phase 67.1 ships GREEN.

```markdown
### ACK reaction (Phase 67.1)

When the bridge accepts an inbound message and successfully writes it to the
sandbox's SQS queue, it adds a 👀 emoji reaction to the originating Slack
message within ~1 second. This gives the user immediate visual confirmation
that the sandbox saw their message — even before the agent boots, before any
paused-sandbox hint posts, before any reply.

The 👀 means "we accepted this for processing" — not just "we received the
HTTP request". If the SQS write fails, no reaction is added (and the
operator sees the failure in CloudWatch logs).

**Required scope:** `reactions:write` (Bot Token Scope, added via Slack App
config → OAuth & Permissions; reinstall the app after adding).

**Bridge env var:** `KM_SLACK_ACK_EMOJI` (default `eyes`). Set on the Lambda
to override the emoji workspace-wide. Always omit the surrounding colons
(`hourglass_flowing_sand`, NOT `:hourglass_flowing_sand:`).

**Deploying:** `make build && km init --lambdas`. Existing inbound-enabled
sandboxes pick up the change automatically — no `km destroy/create` needed
because this is a bridge-only change.

**Failure modes** (all logged at WARN, none block delivery):
- Missing `reactions:write` scope → `events: reaction failed err=missing_scope`. Add scope and reinstall app.
- Bot kicked from channel → `events: reaction failed err=channel_not_found`. Re-invite the bot.
- Slack delivered the same event twice (cold-start replay) → `already_reacted`. Treated as idempotent success — NOT logged at WARN.
```
