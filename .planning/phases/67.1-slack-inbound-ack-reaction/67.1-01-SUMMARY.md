## Key Design Decisions

### HTTP call body duplicated (not shared helper)
`SlackReactorAdapter` has exactly one method (`Add`). Extracting a shared `slackAPICall` package-level function would save ~20 lines but requires generic-payload typing coordination with `SlackPosterAdapter.call()`. Decision: duplicate (~40 lines). If a third adapter appears, factor then. Documented in CONTEXT.md "Claude's Discretion" and code comment.

### Token fetcher sharing (YES — 15-min cache preserved)
`SlackReactorAdapter.Tokens` = same `tokenFetcher` instance as `SlackPosterAdapter`. Cold-start wiring passes `Tokens: tokenFetcher` (not a new `SSMBotTokenFetcher`). The `SSMBotTokenFetcher` caches the token for 15 minutes — creating a second instance would bypass the cache and add an SSM call on every reaction at warm Lambda invocations.

### Test helpers built fresh
No existing `buildMessageEventBody` or `buildBotMessageEventBody` helpers existed in the test file. Both were implemented freshly, following the existing inline JSON body pattern seen in tests like `TestEventsHandler_ValidMessage_HappyPath` but extracted as helpers for the 4 new reactor tests.

### Named reactorCall struct
`type reactorCall struct { channel, ts, emoji string }` chosen over anonymous struct for readable assertion output (`t.Errorf("unexpected reactor call: %+v", calls[0])`).
