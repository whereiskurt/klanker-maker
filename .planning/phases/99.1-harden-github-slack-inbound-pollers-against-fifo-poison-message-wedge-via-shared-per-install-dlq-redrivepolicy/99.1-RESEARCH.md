## Summary

Phase 99.1 adds a FIFO dead-letter queue to both the github-inbound and slack-inbound per-sandbox SQS queues to cure a production-confirmed head-of-line-blocking wedge. A poison message (e.g., a failed pre-auth `/review`) can loop forever in a FIFO group because neither poller deletes on failure — the AWS visibility timeout expires and the message requeues. With no `maxReceiveCount`, SQS never evicts it. The fix is entirely AWS-native: set `RedrivePolicy` on each source queue at creation time, pointing to a shared per-install FIFO DLQ. After `maxReceiveCount` receives the message moves to the DLQ automatically, unblocking the FIFO group. No poller-shell changes are needed.

**Primary recommendation:** Add a `CreateSharedInboundDLQ` / `GetOrCreateSharedInboundDLQ` helper in `pkg/aws/sqs.go`. Wire shared DLQ provisioning into `km init` via a new Terraform module (`infra/modules/sqs-inbound-dlq/v1.0.0`) with a live unit in `infra/live/use1/sqs-inbound-dlq/` registered in `regionalModules()`. Inject the DLQ ARN into `provisionGitHubInboundQueue` and `provisionSlackInboundQueue` via the existing deps structs. The create-handler Lambda does NOT need changes (it drains into a pre-existing queue, does not create it).

## Finding 3: Cold-Create Path — create-handler Lambda

**File:** `cmd/create-handler/main.go`

**CRITICAL CONCLUSION: The create-handler Lambda does NOT create the per-sandbox inbound queue.**

The Lambda's `Handle` method (line 160) runs `km create` as a subprocess (Step 4, line 296). The subprocess is a full `km create` invocation that calls `provisionGitHubInboundQueue` / `provisionSlackInboundQueue` internally. The Lambda simply runs the binary.

After the subprocess succeeds, Step 6 (line 413) calls `h.drainGithubEnvelope()` — this is the ONLY SQS operation performed by the Lambda directly. `drainGithubEnvelope` (line 427):
1. `GetQueueUrl` on the already-created per-sandbox queue (line 431) — queue must already exist
2. `SendMessage` into the queue (line 445)

The Lambda's `GithubInboundSQSAPI` interface (lines 76–79) only covers `GetQueueUrl` and `SendMessage` — no `CreateQueue`. This confirms the queue is created by the `km create` subprocess, not the Lambda itself.

**Deploy consequence (D4 compliance):** Only the warm `km create` path needs the redrive change. The Lambda does not need to be modified for queue creation. However, for the DLQ ARN to be available when the subprocess runs inside the Lambda, the DLQ must already exist at init time — confirmed by D1.

**Lambda rebuild requirement:** The `create-handler` binary (included in `lambdaBuilds()` at `init.go:2069`) embeds the `km` binary at cold-start download. Since the subprocess is a km binary downloaded from S3 at Lambda cold-start, and since `km create` internally calls `provisionGitHubInboundQueue`, the km binary must be rebuilt and re-uploaded for changes to reach the Lambda subprocess. `make build-lambdas` rebuilds the `create-handler.zip` but the km binary that the Lambda downloads comes from `toolchain/km` in S3, which is uploaded by `make build` + `km init` (via the S3 upload flow at init time).

## Finding 4: Shared Per-Install DLQ Provisioning — Recommended Home

### Analysis of Options

**Option A: Terraform module in `regionalModules()`** (recommended)
- Create `infra/modules/sqs-inbound-dlq/v1.0.0/` with `main.tf` creating two FIFO SQS queues (`{prefix}-github-inbound-dlq.fifo`, `{prefix}-slack-inbound-dlq.fifo`)
- Create `infra/live/use1/sqs-inbound-dlq/terragrunt.hcl` as the live unit
- Add `{name: "sqs-inbound-dlq", dir: filepath.Join(regionDir, "sqs-inbound-dlq")}` to `regionalModules()` in `internal/app/cmd/init.go` (after line 316, before `ses`)
- Output the DLQ ARNs; consuming code reads them via SSM or an env var written by `km init`
- **Advantages:** Idempotent by construction (Terraform state), consistent with existing shared-infra pattern (dynamodb-* modules), automatically destroyed by `km uninit` (which reverses `regionalModules()` order)
- **Deployment:** `km init --dry-run=false` creates both DLQs; `km uninit` destroys them
- **Reachability per project memories:** Must be in BOTH `infra/live/use1/` AND `regionalModules()` — module existence alone is invisible to `km init`

**Option B: Imperative get-or-create in Go at `km init` time**
- Add a `createSharedInboundDLQs()` call in `runInit()` after env export
- Uses `CreateQueue` idempotency (QueueNameExists treated as success, same pattern as `CreateSlackInboundQueue`)
- Advantages: No new Terraform state; no terragrunt unit; simpler for a two-queue operation
- Disadvantages: Not tracked by Terraform; `km uninit` must explicitly call delete (or it orphans); no plan-preview; inconsistent with the SES/DynamoDB shared-infra pattern

**Option C: Lazy get-or-create in the SQS helper** (not recommended)
- `GetOrCreateSharedDLQ` called at first source-queue creation
- Creates a race: two concurrent `km create` calls both call `CreateQueue` for the DLQ; SQS `QueueNameExists` handles idempotence but the DLQ may not exist when the second `km create` runs `SetQueueAttributes` if the first is still in-flight
- Also means no DLQ until first sandbox created — breaks D6 (doctor check skips when DLQ missing), recovers awkwardly

**Recommendation: Option A (Terraform module)** — matches existing patterns (`dynamodb-slack-threads`, `dynamodb-github-threads`), automatically tracked in state, compatible with `km uninit` reverse-order destroy, plan-preview-able.

### DLQ ARN Threading

After the Terraform module creates the DLQs, the DLQ ARNs must reach the Go provisioning code. Two mechanisms:

**Sub-option A1: SSM Parameter Store** — `km init` writes the ARN to `/{prefix}/config/github-inbound-dlq-arn` and `/{prefix}/config/slack-inbound-dlq-arn` (or the TF module outputs them and a `null_resource` local-exec writes to SSM). Provisioning code reads SSM once at `km create` time. Consistent with how bridge-url and other platform config is stored.

**Sub-option A2: Env var at init time** — `km init` writes `KM_GITHUB_INBOUND_DLQ_ARN` and `KM_SLACK_INBOUND_DLQ_ARN` to the Lambda env block (like `KM_ARTIFACTS_BUCKET`). km CLI reads via the same config.Load mechanism. However, the km CLI process does not get Lambda env vars — this only works for the Lambda subprocess path, not the local `km create` path.

**Sub-option A3: Derive ARN from account+region** — SQS ARN format is deterministic: `arn:aws:sqs:{region}:{account_id}:{queue_name}`. Given `resource_prefix`, region, and account ID (available via STS or from existing config), the ARN can be constructed without a lookup. This is the simplest approach: no SSM reads, no env vars. The `km create` flow already has region and can call `sts.GetCallerIdentity` for account ID (or reads it from existing AWS session context).

**Recommendation: Sub-option A3 (derive ARN)** — zero extra SSM reads, no new SSM writes at init, deterministic. Matches how SQS ARNs are constructed in IAM policy statements (e.g. `arn:aws:sqs:*:{account_id}:{prefix}-slack-inbound-*.fifo` in `km-operator-policy`).

### Required New Grants

**For the shared DLQ:**

1. **km-operator-policy must add DLQ lifecycle permissions** for the km CLI (creates/describes the DLQ at init time):
   - `sqs:CreateQueue, sqs:GetQueueAttributes, sqs:GetQueueUrl, sqs:TagQueue` on `arn:aws:sqs:*:{account}:{prefix}-*-inbound-dlq.fifo`

## Finding 7: Userdata Pollers — Behavior Unchanged

### Confirmed Poison-Message Lines

`pkg/compiler/userdata.go`:

- **Slack inbound** (line 2071):
  ```
  echo "[km-slack-inbound-poller] WARN: agent run failed (exit $RUN_EXIT), message returns to queue"
  ```
  — The `else` branch after the success block. The poller does NOT delete the message on failure; it falls off the bottom of the loop iteration, the AWS visibility timeout (30s) expires, and the message requeues automatically.

- **GitHub inbound** (line 2336):
  ```
  echo "[km-github-inbound-poller] WARN: agent run failed (exit $RUN_EXIT), message returns to queue"
  ```
  — Identical semantics.

### Why No Poller Change Is Required (D3)

SQS tracks `ApproximateReceiveCount` per message. After `maxReceiveCount` receives without a `DeleteMessage`, SQS moves the message to the DLQ. The poller's "fall off the bottom → visibility timeout expires → requeue" behavior already satisfies SQS's receive-count increment condition. The poller needs no modification — the DLQ eviction is handled entirely by SQS.

### Golden-File Tests

The poller content is embedded in `compiler.go` (the `userdata.go` template). If the poller shell is NOT changed, no golden-file test regeneration is needed. There is no test asserting exact poller shell content (only template-compilation tests). This is confirmed correct per D3.

| DLQ creation idempotency | Custom exists-check loop | `CreateQueue` with `QueueNameExists` error handling (already in `CreateSlackInboundQueue`/`CreateGitHubInboundQueue`) | SQS `CreateQueue` is already idempotent for same-name same-attribute queues |
| RedrivePolicy JSON | Custom JSON encoding | Standard map-to-JSON in `CreateQueue` Attributes: `{"deadLetterTargetArn":ARN,"maxReceiveCount":3}` | SQS accepts RedrivePolicy as a string-valued attribute |
| DLQ ARN lookup | SSM read at km create time | Derive: `arn:aws:sqs:{region}:{accountID}:{dlqName}` | Deterministic; no extra latency or SSM call |
| Module registration | Hand-edit all module loops | `regionalModules()` single-list pattern | `km uninit` already reverses this list; new entry auto-participates in both |

## Common Pitfalls

### Pitfall 1: FIFO DLQ Must Also Be FIFO
**What:** `RedrivePolicy` on a FIFO source queue requires the DLQ to also be FIFO. Setting a standard-queue DLQ ARN is an AWS validation error.
**How to avoid:** Create DLQs with `FifoQueue=true`. AWS FIFO DLQs do not require `ContentBasedDeduplication` (false is fine).

### Pitfall 2: RedrivePolicy Is a String Attribute, Not JSON Object
**What:** In `CreateQueue` `Attributes`, `RedrivePolicy` value must be a JSON-encoded string: `"{\"deadLetterTargetArn\":\"arn:...\",\"maxReceiveCount\":3}"` — not a nested object.
**How to avoid:** Use `json.Marshal(map[string]interface{}{...})` to produce the string value, then set it as a string in the attributes map.

### Pitfall 3: SQS 60-Second Creation Cooldown
**What:** Deleting a queue and immediately re-creating it with the same name fails for 60 seconds (AWS-enforced).
**How to avoid:** This affects the DLQ only if `km uninit` then `km init` are run within 60 seconds. Document this in the operator runbook. Per the existing `checkSlackInboundStaleQueues` code comment (doctor_slack.go line 343), this is a known pattern in the codebase.

### Pitfall 4: Existing Source Queues Are Not Updated
**What:** Existing per-sandbox inbound queues created before this phase have no `RedrivePolicy`. The `km create` change only affects new queues.
**How to avoid:** Per D7: document `km destroy && km create` as the migration path. The doctor check should WARN if a sandbox's source queue lacks a redrive policy (optional enhancement, not in scope per deferred list).

### Pitfall 5: DLQ Deletion Order at km uninit
**What:** If source queues still exist when the shared DLQ is destroyed (e.g., sandboxes not yet torn down), the source queues retain a dangling `RedrivePolicy` ARN. Future receives may fail validation.
**How to avoid:** `km uninit` already checks for active sandboxes before proceeding (uninit.go line 323). The safety gate means source queues should not exist at uninit time. Document this dependency.

### Pitfall 6: IAM Wildcard Scope
**What:** The existing `sqs_slack_inbound` IAM resource `{prefix}-slack-inbound-*.fifo` DOES match `{prefix}-slack-inbound-dlq.fifo` (the `*` matches `dlq`). This accidentally grants the km operator role `SetQueueAttributes` on the DLQ via the existing policy. This is acceptable (the DLQ is still owned by the same install), but it's worth noting as a non-obvious coverage.

### Attaching RedrivePolicy in CreateQueue (pattern)

```go
// Source: AWS SQS SDK v2 + existing CreateSlackInboundQueue pattern (pkg/aws/sqs.go:57)
// The RedrivePolicy value is a JSON-encoded string.
import "encoding/json"

func redriveAttr(dlqARN string, maxReceiveCount int) (string, error) {
    b, err := json.Marshal(map[string]interface{}{
        "deadLetterTargetArn": dlqARN,
        "maxReceiveCount":     maxReceiveCount,
    })
    if err != nil {
        return "", err
    }
    return string(b), nil
}

// In CreateGitHubInboundQueue / CreateSlackInboundQueue:
attrs := map[string]string{
    string(sqstypes.QueueAttributeNameFifoQueue):                 "true",
    string(sqstypes.QueueAttributeNameContentBasedDeduplication): "false",
    string(sqstypes.QueueAttributeNameVisibilityTimeout):         "30",
    string(sqstypes.QueueAttributeNameMessageRetentionPeriod):    "1209600",
}
if dlqARN != "" {
    rp, err := redriveAttr(dlqARN, 3)
    if err != nil {
        return "", fmt.Errorf("marshal redrive policy: %w", err)
    }
    attrs["RedrivePolicy"] = rp
}
```

### GetQueueARN helper (net-new)

```go
// Source: AWS SDK sqstypes.QueueAttributeNameQueueArn (existing SDK constant, not yet used in repo)
func GetQueueARN(ctx context.Context, c SQSClient, queueURL string) (string, error) {
    out, err := c.GetQueueAttributes(ctx, &sqs.GetQueueAttributesInput{
        QueueUrl:       aws.String(queueURL),
        AttributeNames: []sqstypes.QueueAttributeName{sqstypes.QueueAttributeNameQueueArn},
    })
    if err != nil {
        return "", fmt.Errorf("get queue ARN %s: %w", queueURL, err)
    }
    if arn, ok := out.Attributes["QueueArn"]; ok {
        return arn, nil
    }
    return "", fmt.Errorf("QueueArn attribute missing for %s", queueURL)
}
```

## Open Questions

1. **`maxReceiveCount` value**
   - What we know: CONTEXT.md says "~3"; 3 is the minimum non-trivial value AWS allows for FIFO DLQs.
   - What's unclear: Whether 3 is appropriate given the 30-second visibility timeout. A transient EC2 spike could cause 3 consecutive receive failures for a valid message. With `VisibilityTimeout=30s` and `maxReceiveCount=3`, a message reaches the DLQ after ~90 seconds of failure. Given the reproduced scenario (poller spinning ~50s per failed receive), 3 seems conservative.
   - Recommendation: Use 3 as locked by CONTEXT.md "~3". This is Claude's discretion.

2. **Whether to use Option A3 (derive ARN) or SSM for DLQ ARN**
   - Recommendation: A3 (derive). Region is in `awsconfig`, account ID can be fetched once at `km create` startup (STS is already called for other purposes — see doctor.go `STSCallerAPI`). Zero SSM reads.

3. **Whether to add `sqs.GetQueueUrl` to the `SQSClient` interface**
   - The create-handler's `GithubInboundSQSAPI` has `GetQueueUrl` (line 76) but the shared `SQSClient` interface does not. The new `GetOrCreateSharedInboundDLQ` helper could use `ListQueues`-by-prefix for URL resolution (same pattern as existing `QueueNameExists` fallback at sqs.go line 73). No interface change required.
