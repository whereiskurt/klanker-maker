# Phase 99.1: FIFO Poison-Message DLQ Hardening — Verification Report

**Phase Goal:** Fix a production-risk robustness gap in the per-sandbox github-inbound and slack-inbound SQS FIFO queues: a poison message head-of-line-blocks its FIFO group forever (only `purge-queue` recovers). Fix via a shared per-install DLQ + `RedrivePolicy` (maxReceiveCount=3) on both creation paths, with teardown safety, doctor visibility, and a documented deploy/migration surface — dormant when inbound config is absent.

### CRITICAL Goal-Backward Check (BOTH-PATHS)

The success-criterion-1 "both paths" concern was the central risk. Result: **NO GAP.**

- `cmd/create-handler/main.go` has **zero** `CreateQueue` calls (grep confirmed).
- The Lambda runs `km create` as a subprocess of the embedded `km` binary (`h.ToolchainDir/km`, Step 4 main.go:296). That subprocess executes the same `runCreate` → `provision{GitHub,Slack}InboundQueue` → `Create{GitHub,Slack}InboundQueue(…, deps.DLQArn)` warm code that now threads the redrive policy.
- The Lambda's only direct SQS op (`drainGithubEnvelope`) is `GetQueueUrl` (resolve pre-existing) + `SendMessage`.
- Plan 02's "no create-handler change" decision (RESEARCH Finding 3) is **correct**: both paths converge on the shared helper. The sole deploy caveat — the embedded km binary in S3 must be rebuilt (`make build` + `km init`) — is documented.
