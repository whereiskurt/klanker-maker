# Phase 99.1: Harden inbound pollers (DLQ) ŌĆö Context

**Gathered:** 2026-06-08
**Status:** Ready for planning
**Source:** Live finding in Phase 99 UAT + operator design decisions (AskUserQuestion, 2026-06-08). See memory `project_inbound_poller_fifo_poison_wedge`.

<domain>
## Phase Boundary

Fix a production-risk robustness gap in the per-sandbox **github-inbound** and **slack-inbound**
SQS **FIFO** queues. Today they have **no redrive policy**; the userdata poller shell loops
(`pkg/compiler/userdata.go` ŌĆö github ~2336, slack ~2071) silently requeue a failed `claude -p`
agent run. Because the queues are FIFO, a poison message **head-of-line-blocks its whole message
group forever** and wedges the poller ŌĆö `systemctl restart` does NOT clear it; only
`aws sqs purge-queue` recovers. Reproduced live: a pre-auth `/review` failed ŌåÆ requeued ŌåÆ poller
spun ~50s CPU, stuck ~20 min, blocked the next valid `/review`.

**Delivers:** a shared per-install DLQ + `RedrivePolicy` so SQS auto-evicts poison messages,
un-wedging the FIFO group, plus `km doctor` DLQ-depth visibility and teardown handling.

**Out of scope:** the Phase 99 `{{args}}` command layer (green); changing visibility-timeout or
retention semantics; a DLQ *consumer*/auto-reprocessor (depth visibility only); poller-side
failure-reply comments (the DLQ alone fixes the wedge ŌĆö a reply could be a later enhancement).
</domain>

<decisions>
## Implementation Decisions (locked)

### D1 ŌĆö DLQ topology: SHARED per-install (operator decision)
Create **ONE shared DLQ FIFO for github-inbound** and **ONE for slack-inbound** per install
(keyed off `resource_prefix`), created **once at `km init`/`bootstrap`** ŌĆö NOT per sandbox.
All per-sandbox source queues redrive to the shared DLQ. Rationale: far fewer queues, simpler
teardown, one place to watch. Names follow existing conventions (e.g.
`{prefix}-github-inbound-dlq.fifo`, `{prefix}-slack-inbound-dlq.fifo`).

### D2 ŌĆö Scope: BOTH pollers (operator decision)
Fix github-inbound **and** slack-inbound in lockstep ŌĆö identical root cause, identical redrive
pattern. Do not leave a known wedge in the Slack path.

### D3 ŌĆö Mechanism: SQS-native RedrivePolicy
Set `RedrivePolicy` (`deadLetterTargetArn` + `maxReceiveCount` Ōēł 3) on the per-sandbox inbound
FIFO source queues. SQS auto-moves a message to the DLQ after `maxReceiveCount` receives without
deletion ŌåÆ the poison message leaves the group ŌåÆ un-wedge. **No poller-loop change is required
for the un-wedge itself** (the existing "message returns to queue" behavior now terminates in the
DLQ after N tries instead of looping forever). FIFO DLQ requirements apply (DLQ must be FIFO; ARN
resolved via GetQueueAttributes `QueueArn`).

### D4 ŌĆö Both creation paths must set redrive
The per-sandbox inbound queue is created in TWO places ŌĆö both must set the RedrivePolicy:
- Warm `km create`: `pkg/aws/sqs.go CreateGitHubInboundQueue` (+ the slack equivalent), called via
  `provision*InboundQueue` (`create_github_inbound.go` / `create_slack_inbound.go`).
- Cold-create: the **create-handler Lambda** (`cmd/create-handler/main.go`) ŌĆö verify whether it
  creates the queue or only drains into it; if it creates, it needs the redrive too.
The shared DLQ must EXIST before a source queue references it (get-or-create at init; source
creation resolves the DLQ ARN).

### D5 ŌĆö Teardown
`km destroy` removes per-sandbox queue + its references but **leaves the shared DLQ** intact
(other sandboxes/installs depend on it). `km uninit` owns the shared DLQ lifecycle and must not
orphan sibling installs (mirror the SES per-prefix shared-resource pattern).

### D6 ŌĆö km doctor visibility
`km doctor` gains a check reporting **DLQ depth** (poison messages waiting) for the inbound DLQs:
clean/OK when empty, WARN when non-empty (so operators notice stuck runs). Dormant/skip when the
DLQ doesn't exist (no inbound configured).

### D7 ŌĆö Deploy / migration (document; no silent backfill)
Deploy = `make build-lambdas` + `km init --dry-run=false` (shared DLQ provisioning + create-handler
Lambda rebuild). Existing sandboxes' source queues do **NOT** gain redrive retroactively ŌåÆ
documented as requiring `km destroy && km create` (do not silently assume a backfill). No
SandboxProfile schema change ŌćÆ no `km init --sidecars` for schema, but shared-infra + Lambda
changes require a full `km init` apply. Per `feedback_verify_deploy_surface_not_just_code`, the
plan MUST include a deploy-surface verification task.

### Claude's Discretion
- Exact DLQ naming, `maxReceiveCount` value (Ōēł3), and where the shared DLQ get-or-create lives
  (km init regional infra vs a bootstrap helper vs lazy in the create helper) ŌĆö follow existing
  queue-provisioning + shared-resource conventions.
- Whether to thread the DLQ ARN via SSM/param or resolve it at queue-creation time.
- Test organization (table-driven, mock SQS client per existing `create_*_inbound_test.go`).
- `km doctor` check grouping/wording.
</decisions>

<specifics>
## Specific Ideas

- Central helper to mirror/extend: `pkg/aws/sqs.go CreateGitHubInboundQueue` (sets FIFO +
  `ContentBasedDeduplication=false` + `VisibilityTimeout=30` + `MessageRetentionPeriod=1209600`,
  no redrive today) and its slack counterpart.
- Provisioning callers: `internal/app/cmd/create_github_inbound.go` (`provisionGitHubInboundQueue`)
  and the slack equivalent; teardown `destroy_*_inbound.go`.
- Cold-create path: `cmd/create-handler/main.go` (drains via `drainGithubEnvelope`; verify queue
  ownership/creation there).
- No existing `RedrivePolicy`/DLQ pattern in the repo ŌĆö this is net-new; follow AWS FIFO-DLQ rules.
- The poison-loop log line to keep in mind: `userdata.go` `WARN: agent run failed (exit N),
  message returns to queue` (github ~2336, slack ~2071) ŌĆö behavior unchanged, but now terminates
  in the DLQ after `maxReceiveCount`.
</specifics>

<deferred>
## Deferred Ideas

- Poller-side failure-reply ("agent run failed repeatedly") posted back to GitHub/Slack.
- A DLQ consumer / auto-reprocessor or CloudWatch alarm on DLQ depth (beyond `km doctor`).
- Backfilling redrive onto already-existing sandbox queues without recreate.
- Generalizing the Gap-E (98-06) stale-`--resume` retry into the broader failure path.
</deferred>
