# Phase 100 — UAT: GitHub bridge federated relay (one App, many installs)

**Requirement:** GH-FED-E2E (manual-only — requires two live installs sharing one
GitHub App webhook + real GitHub delivery; cannot be exercised by unit mocks).
**Status:** ⬜ pending — operator runs this during `/gsd:verify-work` or post-deploy.
**Mode:** live, two installs, one GitHub App.

This runbook verifies the end-to-end behavior of the Phase 100 federated relay: a PR
comment on a repo owned by install **`sec`**, delivered by GitHub to the **`kph`**
front-door bridge, is relayed to `sec`, re-verified, and dispatched — producing
**exactly one** 👀 (from `sec`, the owner; `kph` posts none) and **no
double-processing**. Synthetic install names (`kph`, `sec`) and `on.aws` Function
URLs throughout.

---

## Pre-conditions

- Two km installs in the same account, distinct `resource_prefix`: **`kph`** (front
  door) and **`sec`**.
- One built `km` carrying the Phase 100 relay code (`make build`) and clean Lambda
  zips (`make build-lambdas`) available to both installs.
- A single **GitHub App** (one webhook URL, one webhook secret).
- A test repo **owned by `sec`** (e.g. `sec-org/demo`) with a PR open, and an
  allowlisted GitHub login to author the comment.

---

## Setup

### 1. One GitHub App, creds in BOTH installs

Create one GitHub App (`km github manifest` renders the manifest). Subscribe it to
the **`issue_comment`** webhook event. Then paste the **same** App creds + webhook
secret into **both** installs:

```bash
# In the kph install context:
km github init        # caches App client-id / private key / webhook-secret under /kph/config/github/
# In the sec install context:
km github init        # SAME App creds + webhook-secret under /sec/config/github/
```

Both installs now hold an identical copy of the App webhook secret — each can
independently HMAC-verify `X-Hub-Signature-256` on a relayed body.

### 2. Unique repo ownership

Each install lists only **its own** repos in `github.repos:` (the cross-fleet
ownership invariant):

```yaml
# kph km-config.yaml
github:
  repos:
    - match: "kph-org/*"
      profile: github-review

# sec km-config.yaml
github:
  repos:
    - match: "sec-org/*"
      profile: github-review
```

`sec-org/demo` is owned by **`sec` only**. `kph` does NOT match it.

### 3. Front door lists the peer

Get each install's bridge Function URL:

```bash
km github status      # → bridge-url, e.g.:
#   kph: https://kph000.lambda-url.us-east-1.on.aws/
#   sec: https://sec000.lambda-url.us-east-1.on.aws/
```

On the **front-door** install (`kph`), add `sec`'s URL to `github.peer_bridges`. For
full symmetry, also add `kph`'s URL on `sec` (so either install can front-door):

```yaml
# kph km-config.yaml
github:
  peer_bridges:
    - https://sec000.lambda-url.us-east-1.on.aws/

# sec km-config.yaml (symmetric — optional but recommended)
github:
  peer_bridges:
    - https://kph000.lambda-url.us-east-1.on.aws/
```

> **Self-loop guard check:** do NOT put an install's own bridge URL in its own
> `github.peer_bridges`. `km doctor` WARNs on this (`GitHub peer bridges` →
> self-loop). A self-loop costs one wasted hop but cannot loop — the relayed copy
> carries `X-KM-Relayed: 1` and is terminal.

### 4. Point the App webhook at the front door

In the GitHub App settings, set the **Webhook URL** to the **front-door** install's
bridge URL (`https://kph000.lambda-url.us-east-1.on.aws/`). GitHub will deliver
every `issue_comment` event to `kph`.

### 5. Deploy both installs (NOT `--sidecars`)

`github.peer_bridges` → `KM_GITHUB_PEER_BRIDGES` is an **env-block** change, so a full
terragrunt apply is required on each install where the config changed:

```bash
# On EACH install (kph and sec):
make build-lambdas       # CLEAN rebuild — make build alone does NOT rebuild the bridge zip
km init --dry-run=false  # full apply — updates the Lambda environment.variables block
km doctor                # → "GitHub peer bridges" OK (no malformed / no self-loop)
```

> **NOT `km init --sidecars`:** it rebuilds the zip + cold-starts but does NOT update
> the Lambda env block, so `KM_GITHUB_PEER_BRIDGES` would never reach the Lambda and
> the relay would stay silently off. (Memory: `project_km_init_lambdas_doesnt_deploy`,
> `feedback_km_init_full_apply`.)

## Test: comment on a `sec`-owned repo

1. On a PR in **`sec-org/demo`** (owned by `sec`, delivered to the `kph` front door),
   post a comment from an allowlisted login that @-mentions the bot:

   ```
   @km-bot review this PR
   ```

### Expected (PASS criteria)

- [ ] **Delivery → front door.** GitHub delivers the `issue_comment` webhook to the
      `kph` bridge (the App's configured webhook URL).
- [ ] **Front door relays, does NOT process.** `kph` runs `Resolve("sec-org/demo")`
      against its own `github.repos:` → **no match** → it POSTs the raw webhook
      verbatim (body + `X-Hub-Signature-256` + `X-GitHub-Event` + `X-GitHub-Delivery`,
      adding `X-KM-Relayed: 1`) to `sec`'s bridge. **`kph` posts no 👀.**
- [ ] **Owner re-verifies + dispatches.** `sec` re-verifies the forwarded
      `X-Hub-Signature-256` with its own copy of the App webhook secret, runs
      `Resolve("sec-org/demo")` → **match**, dedupes `X-GitHub-Delivery` in `sec`'s
      own nonces store, and dispatches the turn into a `sec` sandbox.
- [ ] **Exactly ONE 👀.** The owning install (`sec`) posts the single 👀 reaction on
      the comment. The front door (`kph`) posts none → **one** reaction total.
- [ ] **No double-processing.** Only one sandbox turn runs (in `sec`). `kph` never
      dispatches; the relayed event is terminal at `sec`.

### How to observe

- **Reaction:** the comment shows exactly one 👀 in the GitHub UI.
- **`kph` (front door) logs:** a relay line (broadcast to `sec`), **no** dispatch, no
  reaction.
- **`sec` (owner) logs:** HMAC verify OK on the relayed body, dispatch line, 👀 posted.
- **Sandbox:** `km agent list <sec-sandbox>` shows one new run; no run in any `kph`
  sandbox.

## Negative check: orphan repo (no install owns it)

Comment on a repo **no install owns** (e.g. `random-org/unconfigured`, delivered to
`kph`):

- [ ] `kph` resolves no match → relays to `sec`.
- [ ] `sec` resolves no match on a request carrying `X-KM-Relayed: 1` → **drops** it,
      logs `github_relay_no_owner`, returns 200.
- [ ] **No 👀, no dispatch anywhere.** (The helpful orphan reply is deferred to
      Phase 101.)

---

## Loop-guard check (self-loop is a no-op, not a loop)

Temporarily add `kph`'s **own** URL to `kph`'s `github.peer_bridges` (then `km init
--dry-run=false`). Comment on an unowned repo:

- [ ] `km doctor` WARNs: `GitHub peer bridges` → self-loop detected.
- [ ] The front door relays to itself once; the relayed copy carries
      `X-KM-Relayed: 1` and is **terminal** (dropped as `github_relay_no_owner`, never
      re-relayed). **No infinite loop.**
- [ ] Remove the self-loop entry and re-deploy after the check.

---

## Dormancy check (federation off ⇒ byte-identical to Phase 97/98)

- [ ] Remove `github.peer_bridges` from both installs, `km init --dry-run=false`.
- [ ] A comment on a `kph`-owned repo still dispatches + 👀 exactly as Phase 97/98.
- [ ] A comment on an unowned repo returns 200 with no broadcast (relayer nil).
- [ ] `km doctor` SKIPs the `GitHub peer bridges` check silently.
- [ ] **No sandbox recreate** was required at any point (no SandboxProfile schema
      change in this phase).

---

## Rollback

Federation is purely additive and env-driven:

1. Remove `github.peer_bridges` from `km-config.yaml` on the affected install(s).
2. `make build-lambdas && km init --dry-run=false` — `KM_GITHUB_PEER_BRIDGES` clears,
   the relayer goes nil, behavior reverts to Phase 97/98.
3. Optionally re-point the App webhook back to a single-install (Pattern A) URL.

No sandbox recreate, no DDB/SQS/schema changes to undo.
